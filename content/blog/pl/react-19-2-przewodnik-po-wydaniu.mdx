React 19.2 (wydany w październiku 2025) przynosi wiele nowości: nowych API, optymalizacji wydajności i ulepszeń SSR. To trzecie wydanie z serii React 19 (po 19.1 z połowy 2025 i 19.0 z końca 2024), kontynuujące dynamiczny rytm rozwoju frameworka. W tym artykule przechodzę przez wszystkie nowe funkcje, zmiany i ulepszenia w React 19.2 z perspektywy developera — od zmian w renderowaniu, przez nowe API (komponent i hook), aż po aktualizacje SSR, zmiany w linterze i poprawki.

React 19.2 przynosi kilka znaczących funkcji i usprawnień podnoszczących wydajność i doświadczenie dewelopera. Oto kluczowe dodatki:

- Komponent `<Activity />`: nowy komponent utrzymujący elementy UI zamontowane, ale ukryte — zachowując ich stan, jednocześnie odmontowując efekty uboczne i odraczając ich aktualizacje.
- Hook `useEffectEvent`: nowy hook oddzielający logikę „zdarzeniową” od reaktywnych efektów, zapewniający stabilne callbacki widzące zawsze bieżący stan bez ponownego uruchamiania efektu.
- API `cacheSignal`: API dla React Server Components dostarczające AbortSignal sygnalizujący, kiedy zbuforowany zasób nie jest już potrzebny, umożliwiając anulowanie lub wyczyszczenie długo trwających operacji.
- Partial Pre-rendering: nowe możliwości renderowania po stronie serwera pozwalające wstępnie wyrenderować część aplikacji do statycznego HTML (dla szybkiego shella), a następnie wznowić renderowanie, aby dodać dynamiczną zawartość.
- Batching Suspense w SSR: usprawnione zachowanie Suspense w streamowanym SSR — React grupuje odsłanianie granic Suspense, tak że zawartość pojawia się partiami (bardziej jak na kliencie) zamiast sekwencyjnie.
- Performance Tracks w DevTools: integracja nowych, specyficznych dla Reacta ścieżek wydajności w Chrome DevTools, dająca głębszy wgląd w harmonogramowanie Reacta i oś czasu renderowania komponentów.
- Plugin ESLint dla hooków v6: zaktualizowany plugin lintera wspierający nowy hook i nowy format konfiguracji, zapewniający spójność reguł hooków z funkcjami React 19.2.
- Inne usprawnienia: drobne zmiany, takie jak aktualizacja domyślnego prefiksu `useId` dla zgodności z nowymi standardami, wsparcie dla Web Streams w API SSR w Node.js oraz liczne poprawki błędów poprawiające stabilność i wydajność.

W dalszej części artykułu zagłębimy się w każdą z tych rzeczy szczegółowo. Przykłady kodu ilustrują, jak używać nowych funkcji w praktyce, a przy okazji omówimy wszelkie kwestie związane z aktualizacją lub dostosowywaniem istniejącego kodu. Na koniec powinieneś mieć jasny obraz tego, co oferuje React 19.2 i jak może to przynieść korzyści twoim projektom.

# Nowe funkcje rdzenia w React 19.2

## Komponent `<Activity />`

Jedną ze sztandarowych funkcji React 19.2 jest nowy komponent `<Activity>`. To wrapper, który ukrywa lub pokazuje fragment UI (drzewo komponentów potomnych) bardziej kontrolowanie niż zwykłe warunkowe renderowanie. Kluczowa idea: `<Activity>` utrzymuje swoje dzieci zamontowane w DOM nawet gdy są „ukryte”, zachowując ich stan, jednocześnie odmontowując efekty uboczne (takie jak `useEffect`) i odraczając ich aktualizacje do momentu potrzeby. Pozwala to fragmentom aplikacji pozostać aktywnymi w tle bez wpływu na wydajność widocznej zawartości.

### Użycie komponentu Activity

Aby użyć komponentu `<Activity>`, opakowujesz docelowe UI w `<Activity>` i sterujesz jego propem mode zamiast przełączać renderowanie. Na przykład tak można ukryć lub pokazać sidebar za pomocą Activity:

```jsx
// Bez Activity: warunkowe renderowanie (odmontowuje Sidebar, gdy ukryty)
{
  isShowingSidebar && <Sidebar />
}

// Z Activity: użycie trybów Activity (zachowuje Sidebar, gdy ukryty)
;<Activity mode={isShowingSidebar ? 'visible' : 'hidden'}>
  <Sidebar />
</Activity>
```

W powyższym przykładzie, gdy `isShowingSidebar` ma wartość `false`, `<Sidebar>` jest ukryty, ale nie odmontowany. `<Activity mode="hidden">` mówi Reactowi, by wizualnie ukrył sidebar (używając CSS `display: none`), zachowując jego drzewo komponentów w pamięci. Ważnie: w trybie `hidden` React automatycznie czyści wszelkie efekty wewnątrz `<Sidebar>`, aby uniknąć pracy w tle i zwolnić zasoby. Stan `<Sidebar>` pozostaje niezmieniony — gdy sidebar pojawi się ponownie, przywróci się z wcześniejszym stanem bez resetowania czy ponownego pobierania danych.

Jak to działa: API `<Activity>` obsługuje obecnie dwa tryby:

- `visible`: dzieci są pokazywane normalnie, ich efekty są zamontowane, a aktualizacje stanu przetwarzane jak zwykle.

- `hidden`: dzieci są ukryte (nierenderowane dla użytkownika, np. przez CSS), ich efekty są odmontowane, a aktualizacje stanu odroczone do chwili, gdy React nie ma innej pracy (wykonywane z priorytetem idle).

To pozwala wstępnie wyrenderować UI w tle lub utrzymać UI poza ekranem bez spowalniania widocznej zawartości. Możesz na przykład wyrenderować kolejną stronę lub modal w trybie `hidden` z wyprzedzeniem, tak by pojawił się natychmiast przy nawigacji użytkownika (bo był już wyrenderowany w tle). Albo utrzymać komponent formularza ukryty, zachowując dane, które użytkownik wprowadził, jeśli wróci (stan jest zapisany). W istocie `<Activity>` daje sposób na wdrożenie „natychmiastowej” nawigacji i buforowanego UI bez ręcznego żonglowania stanem.

### Przypadki użycia

Interfejsy z kartami (tabs), wieloetapowe formularze czy modale na żądanie zyskują dzięki `<Activity>`. Na przykład przełączanie kart może być płynniejsze, jeśli nieaktywne karty są w `<Activity hidden>` — ich stan (pola formularza, pozycje przewijania itd.) pozostaje dostępny, gdy karta pojawi się ponownie, ale ich efekty (timery, subskrypcje) nie działają, gdy są ukryte.

Kolejnym przypadkiem użycia jest prefetching: opakuj komponent w `<Activity hidden>` tuż przed tym, jak przewidujesz jego potrzebę (np. predykcyjne wstępne załadowanie kolejnego ekranu w kreatorze). Zawartość może ładować dane w tle bez blokowania UI, następnie przełączasz ją w tryb `visible`, gdy jest potrzebna, unikając opóźnienia ładowania dla użytkownika.

Warto zauważyć, że `<Activity>` to nowy prymityw i zespół Reacta planuje rozszerzać go o dodatkowe tryby. W React 19.2 dostępne są tylko „visible” i „hidden”, ale bardziej zniuansowane tryby (być może do zawieszania lub innych zachowań) mogą zostać dodane w przyszłości.

## Hook `useEffectEvent`

Następnym dużym dodatkiem jest hook `useEffectEvent`, który rozwiązuje długo obecny problem ze wzorcami efektów. Pozwala oddzielić logikę zdarzeniową od reaktywnego przepływu `useEffect`, tworząc stabilny callback („Effect Event”), który można wywoływać z wnętrza efektów bez ponownego utworzenia przy każdym renderze. Głównym celem jest uniknięcie zbędnego ponownego uruchamiania efektów z powodu zmieniających się wartości używanych wewnątrz callbacku, które jednak nie kontrolują momentu uruchamiania efektu.

Problem: w Reactcie częstym wzorcem jest użycie `useEffect` do subskrybowania zewnętrznych zdarzeń lub wykonania akcji w odpowiedzi na zdarzenia. Rozważmy efekt, który tworzy WebSocket i reaguje na zdarzenie, wywołując funkcję z bieżącym motywem (theme) lub stanem. Naiwna implementacja mogłaby wyglądać tak:

```jsx
useEffect(() => {
  const connection = createConnection(serverUrl, roomId)
  connection.on('connected', () => {
    showNotification('Connected!', theme)
  })
  connection.connect()
  return () => connection.disconnect()
}, [roomId, theme])
```

Efekt zależy od `roomId` i `theme`. Jednak tylko `roomId` jest naprawdę potrzebne, by określić, kiedy ponownie uruchomić efekt (np. przy przełączaniu pokojów czatu) — `theme` jest używane tylko wewnątrz obsługi zdarzenia do ostylowania powiadomienia. Jeśli `theme` się zmieni, efekt niepotrzebnie rozłożyłby i ponownie ustanowił połączenie, co jest niepożądane. Typowymi obejściami były albo pominięcie `theme` w tablicy zależności (i wyłączenie reguły ESLint), albo użycie ref-u do przechowywania bieżącej wartości `theme` — oba podejścia mają wady (nieaktualne wartości lub utrata wsparcia lintera).

Rozwiązanie — `useEffectEvent`: React 19.2 wprowadza `useEffectEvent` do elegancko rozwiązania tego problemu. Pozwala stworzyć wewnątrz komponentu specjalną funkcję obsługi zdarzenia, która zawsze ma dostęp do bieżących propsów/stanu, ale nie wyzwala ponownych uruchomień efektu, gdy te propsy/stan się zmienią. To tylko efekt korzystający z funkcji decyduje, kiedy się uruchamia.

Na przykład:

```jsx
function ChatRoom({ roomId, theme }) {
  // Utwórz obsługę effect event dla zdarzenia "connected":
  const onConnected = useEffectEvent(() => {
    // Ten kod może używać najświeższego `theme` (lub innego stanu)
    showNotification('Connected!', theme)
  })

  useEffect(() => {
    const connection = createConnection(serverUrl, roomId)
    connection.on('connected', () => {
      // Użyj stabilnej obsługi zamiast wpisywać logikę tutaj
      onConnected()
    })
    connection.connect()
    return () => connection.disconnect()
  }, [roomId]) // Uwaga: theme nie jest tutaj uwzględnione
}
```

Tablica zależności efektu to `[roomId]` — uruchomi się ponownie tylko gdy zmieni się ID pokoju (co ma sens, potrzebujesz nowego połączenia). Obsługa `onConnected` może bezpiecznie używać `theme` wewnątrz i zawsze, gdy zdarzenie „connected” się wyzwoli, wywoła `onConnected()` z bieżącą wartością `theme`. Zmiana propa `theme` nie powoduje już ponownego uruchomienia efektu ani zresetowania połączenia, ale każde nowe wywołanie `onConnected` widzi zaktualizowany `theme`. Innymi słowy, `useEffectEvent` daje aktualny callback dla scenariuszy zdarzeniowych, bez sprzęgania go z zależnościami efektu.

## Kluczowe cechy Effect Events

### Najświeższy stan zawsze widoczny

Funkcja effect event (jak `onConnected` powyżej) zawsze widzi w swoim ciele bieżące wartości propsów/stanu, podobnie jak obsługa zdarzenia w `onClick` zawsze widzi bieżący stan między renderami. Unika się tym nieaktualnych domknięć (stale closures).

### Sama w sobie niereaktywna

Nie uwzględniasz funkcji effect event w tablicy zależności efektu (plugin ESLint to wymusza). Obecność `onConnected` w efekcie nie powoduje jego ponownego uruchomienia przy każdym renderze, możesz swobodnie używać wewnątrz `onConnected` wartości takich jak `theme`.

### Deklarowana lokalnie

Musisz zadeklarować hook `useEffectEvent` w tym samym komponencie lub hooku, który używa go w efekcie; nie można go dowolnie przekazywać. Te ograniczenia są sprawdzane przez linter, aby zapewnić poprawne użycie effect event.

### Tylko dla logiki „zdarzeniowej”

Używaj `useEffectEvent` tylko dla kodu, który koncepcyjnie jest „obsługą zdarzenia” wyzwalaną wewnątrz efektu (callback z subskrypcji czy timera). Nie jest przeznaczony dla każdej funkcji ani do pomijania reguł lintera; nadużywanie go do ogólnej logiki może ukryć rzeczywiste problemy z zależnościami.

### Porównanie z alternatywami

Przed `useEffectEvent` developerzy sięgali po `useRef` do przechowywania mutowalnej referencji do wartości czy funkcji, tak by callback efektu mógł używać bieżącej wartości bez ponownego uruchamiania efektu. Można było przechowywać `theme` w ref-ie lub sprytnie używać `useCallback`. `useEffectEvent` upraszcza ten wzorzec, dostarczając dedykowany hook jasno sygnalizujący intencję „to jest callback zdarzeniowy z wnętrza efektu”. Poprawia to utrzymywalność złożonych efektów mieszających logikę reaktywną i niereaktywną, czyniąc kod czytelniejszym i bardziej niezawodnym. Wielu developerów uważało, że taki hook był „boleśnie potrzebny od czasu wprowadzenia hooków”.

Aby w pełni wykorzystać ten hook, zaktualizuj także `eslint-plugin-react-hooks` do v6+, który przychodzi z React 19.2 (patrz poniżej). Nowe reguły lintera rozumieją `useEffectEvent` — nie będą błędnie sugerować dodania effect event do tablic zależności i będą egzekwować poprawne wzorce użycia.

## API `cacheSignal` dla Server Components

React 19.2 dodaje nowe API `cacheSignal` ukierunkowane na React Server Components i strategie pobierania danych. Aby zrozumieć `cacheSignal`, przypomnij sobie, że React ma funkcję `cache()` (wprowadzoną w React 18+ dla Server Components), która memoizuje lub deduplikuje wywołania asynchroniczne — na przykład opakowując fetch, tak by wiele komponentów renderujących się na serwerze mogło współdzielić tę samą obietnicę żądania (unikając zdublowanych requestów). Nowy `cacheSignal` dostarcza AbortSignal powiązany z cyklem życia zbuforowanych danych, pozwalając wiedzieć, kiedy zbuforowany wynik nie jest już potrzebny (render się zakończył lub został przerwany) i anulować trwającą pracę.

### Cel

Gdy używasz `cache()` do opakowania funkcji asynchronicznej (jak `fetch`), React trzyma zbuforowany wynik przez czas trwania renderowania. Ale jeśli renderowanie zostanie przerwane (z powodu błędu czy nawigacji klienta) albo się zakończy, wynik może już nie być potrzebny. `cacheSignal()` daje sygnał AbortController, który wyzwoli się, gdy cykl życia zbuforowanego wyniku się kończy. Przekazując ten sygnał do operacji asynchronicznych, umożliwiasz im automatyczne przerwanie, gdy React uzna, że wynik nie jest potrzebny, oszczędzając transfer i moc obliczeniową.

Załóżmy, że mamy komponent serwerowy, który musi pobrać jakieś dane. Możemy użyć `cache()`, by zapewnić deduplikację fetcha, oraz `cacheSignal()`, by go anulować, jeśli to zasadne:

```jsx
import { cache, cacheSignal } from 'react'

// Opakuj fetch w cache() dla deduplikacji
const dedupedFetch = cache(async (url) => {
  const response = await fetch(url, { signal: cacheSignal() })
  return response.json()
})

// W komponencie serwerowym lub funkcji loadera:
async function MyServerComponent() {
  const data = await dedupedFetch('https://api.example.com/data')
  // ... renderuj z data ...
}
```

W tym fragmencie `cacheSignal()` dostarcza sygnał dla `fetch`. Jeśli React przerwie renderowanie `MyServerComponent` (użytkownik przeszedł gdzie indziej czy wystąpił błąd), sygnał zostanie przerwany, anulując żądanie `fetch`. Gdy render zakończy się pomyślnie, a wynik nie jest już potrzebny dla przyszłych renderów (wpis w cache wygasł), sygnał wyzwoli się, sygnalizując brak dalszej potrzeby. Zapobiega to zbędnej pracy i oszczędza zasoby, nie dokańczając żądań, których wyniki nie będą użyte. To szczególnie przydatne przy długo trwających fetchach lub kosztownych obliczeniach na serwerze.

Technicznie `cacheSignal` integruje się z cache Reacta i cyklem życia renderowania komponentów serwerowych. Ma sens tylko w kontekście renderowania po stronie serwera czy renderowania statycznego, gdzie używane jest `cache()` — nie to do czysto klienckiego komponentu. Ta funkcja pomaga unikać wycieków pamięci lub porzuconych procesów asynchronicznych na serwerach Node.js i poprawia efektywność Server Components i pobierania danych w Reactcie.

## Rozszerzone funkcje renderowania i SSR

React 19.2 to nie tylko nowe API; przynosi także solidne usprawnienia SSR i obsługi renderowania pod maską.

### Partial Pre-rendering dla szybszego ładowania

Jedną z najbardziej znaczących funkcji jest Partial Pre-rendering. Pozwala wstępnie wyrenderować części aplikacji do statycznego HTML, a następnie wznowić renderowanie, aby dodać dynamiczną lub spersonalizowaną zawartość. Możesz wygenerować „prerenderowany shell” aplikacji, serwowany natychmiast (np. z CDN), a przy rzeczywistym żądaniu React wznawia renderowanie, aby stworzyć finalne UI. Znacznie poprawia to Time-to-First-Paint i odczuwaną wydajność, dostarczając szybko użyteczny shell, a następnie uzupełniając interaktywne części.

### Jak to działa

React 19.2 dodaje nowe API w `react-dom` do renderowania częściowego:

- `prerender(element, options)`: wstępnie renderuje element React (zazwyczaj aplikację lub jej część) i zwraca prelude (HTML statycznego shella) oraz obiekt `postponed`.

- `resume(element, postponedState)`: przyjmuje wcześniejszy stan `postponed` z prerender i wznawia renderowanie, zwracając strumień (w streamingu) pozostałej zawartości.

- `resumeAndPrerender(element, postponedState)`: wznawia renderowanie i ponownie wykonuje prerender, aby wyprodukować pełny HTML (przydatne przy generowaniu stron statycznych).

Wyobraź sobie aplikację e-commerce. Mógłbyś prerenderować shell (nawigacja, stopka, layout — które są statyczne) i odroczyć renderowanie dynamicznych, specyficznych dla użytkownika części (spersonalizowane rekomendacje czy status magazynu). Oto ogólna ilustracja:

```jsx
// Przy buildzie lub w zadaniu serwerowym:
const controller = new AbortController()
const { prelude, postponed } = await prerender(<App />, { signal: controller.signal })
// Zapisz `postponed` do późniejszego użycia (baza danych, cache)
await savePostponedState(postponed)
// Serwuj HTML `prelude` do CDN dla szybkiej dostawy
```

Później, gdy użytkownik faktycznie zażąda strony, możesz zrobić:

```jsx
// Przy żądaniu:
const postponed = await getPostponedState(request)
// Wznów renderowanie z zapisanego stanu, produkując strumień pozostałego HTML
const stream = await resume(<App />, postponed)
// Wyślij strumień w odpowiedzi HTTP (streaming SSR)
```

W tym przepływie początkowy `prelude` to szkielet strony (z placeholderami lub stanami ładowania dla dynamicznych sekcji). Ta zawartość wysyła się do klienta natychmiast (zwłaszcza z CDN). Użytkownik szybko widzi strukturę strony. Krok `resume` dokańcza renderowanie, streamując odroczone części — zastępując placeholdery prawdziwymi danymi i komponentami. Dzieje się to, gdy użytkownik już ma shell, co skutkuje szybszym First Contentful Paint i wcześniejszą interaktywnością.

React dostarcza także `resumeAndPrerender`, które wznawia render, a następnie ponownie prerenderuje, aby uzyskać statyczny HTML całej strony (przydatne przy generowaniu nowej statycznej strony z kombinacji prelude i dynamicznych danych).

### Dlaczego to ma znaczenie

Partial pre-rendering jest potężnym narzędziem dla wydajności i skalowalności. Dzieląc pracę, możesz wykonać ciężkie zadania (renderowanie statycznej zawartości) z wyprzedzeniem, a na żądanie robić tylko minimalne renderowanie dynamiczne. Świetnie sprawdza się przy generowaniu stron (SSG) z dynamicznymi wstawkami czy aplikacjach dostarczających szybko część zawartości, odraczając resztę. Z perspektywy SEO krytyczna zawartość jest szybko serwowana crawlerom, a z perspektywy użytkownika strona nie wygląda na pustą, gdy czeka na personalizację.

Technicznie funkcja bazuje na streamowanym SSR i serializacji Reacta. Stan `postponed` z prerender to zserializowana reprezentacja niedokończonej pracy Reacta (części dynamiczne). Pakiety `react-dom/server` i `react-dom/static` zawierają te funkcje ze wsparciem Web Streams i tradycyjnych strumieni Node. Na przykład `react-dom/server` eksportuje `resume()` (dla Web Streams w Cloudflare Workers czy Node) i `resumeToPipeableStream()` (dla strumieni Node), oraz analogiczne warianty `prerender` i `resumeAndPrerender`. Oznacza to, że możesz używać partial pre-rendering w różnych środowiskach JavaScript.

Partial pre-rendering to zaawansowana funkcja wymagająca ostrożnego traktowania stanu i hydracją. Hydracja klienta musi wznowić się z tej częściowej zawartości. React zapewnia spójność procesu, ale developer musi napisać kod obsługujący przypadki, gdzie część zawartości była prerenderowana, a część nie. Dokumentacja Reacta dostarcza więcej szczegółów o wzorcach użycia.

## Grupowanie granic Suspense w SSR

Suspense to funkcja wyświetlająca fallback (spinner, placeholder), gdy zawartość się ładuje. Na kliencie, gdy wiele granic Suspense rozwiązuje się w tym samym czasie, React renderuje je razem. W streamowanym SSR jednak przed React 19.2 każda granica Suspense streamowała i zastępowała fallback natychmiast, gdy była gotowa, jedna po drugiej. Prowadziło to do efektu „wyskakiwania”, gdzie części HTML aktualizowały się w różnych momentach, nie pasując do zachowania na kliencie.

W React 19.2 renderer SSR grupuje odsłanianie granic Suspense, które kończą się mniej więcej jednocześnie, tak by wiele fragmentów wysłało się do klienta razem. W praktyce oznacza to, że jeśli masz kilka leniwych komponentów lub komponentów zależnych od danych, ich zawartość wstawia się do HTML w mniejszej liczbie większych porcji zamiast szybkiej sekwencji drobnych kawałków.

### Co się zmieniło

Podczas streamingu React wstrzyma się na chwilę z zastąpieniem fallbacku, by sprawdzić, czy inne granice też się rozwiążą. Następnie wysyła zawartość dla tych, które rozwiązały się w tym przedziale, w jednej partii. Na kliencie użytkownik widzi, jak wiele placeholderów zamieniają się na zawartość naraz, a nie jeden po drugim. To lepiej pasuje do renderowania klienta, gdzie React grupuje aktualizacje i painty razem.

#### Korzyści:

- Płynniejsze aktualizacje UI — grupowanie unika wizualnie rozpraszających sekwencyjnych zmian. Strona wygląda bardziej spójnie, gdy duże sekcje gotowe pojawiają się razem.
- Przygotowanie pod View Transitions: zmiana przygotowuje na API `<ViewTransition>`. View Transitions (API webowe) pozwalają na płynne animacje zmian DOM. Jeśli zawartość odsłania się w większych partiach, możesz animować całe sekcje zamiast drobnych zmian. Grupowanie w React 19.2 zapewnia, że View Transitions na SSR mogą obejmować więcej zawartości, prowadząc do ładniejszych animacji.
- Spójność: naprawia to dziwactwo, gdzie SSR streaming nie pasował do zachowania klienta. Developerzy mogą teraz rozumować o Suspense w SSR podobnie jak na kliencie.

### Kwestie wydajności

Zespół Reacta zaimplementował heurystyki, aby grupowanie nie szkodziło metrykom Core Web Vitals. Na przykład Largest Contentful Paint (LCP) to kluczowa metryka. Jeśli React wykryje, że wstrzymanie zawartości popchnęłoby LCP poza dobry próg (~2,5 sekundy), przestanie grupować i wyśle zawartość, aby uniknąć opóźnienia. Innymi słowy, grupowanie ma górny limit — nie będzie arbitralnie czekać i spowalniać ładowania. Celem jest balans: jak najwięcej grupowania dla UX bez regresji w czasie ładowania. Z perspektywy dewelopera ta zmiana jest przezroczysta — nie zmieniasz użycia Suspense, ale warto wiedzieć, że streaming SSR będzie teraz przychodzić nieco inaczej (w większych porcjach) niż wcześniej.

## Wsparcie Web Streams dla SSR w Node.js

React 19.2 rozszerza wsparcie Web Streams API w Node.js. Wcześniej streamowany SSR Reacta w Node używał głównie własnych strumieni Node (poprzez `renderToPipeableStream`). Teraz React dodaje możliwość używania Web Streams (standardowych strumieni z przeglądarek i środowisk edge) także na Node.js.

Funkcje takie jak `renderToReadableStream` (WHATWG ReadableStream) oraz `prerender` były pierwotnie dla przeglądarki i środowisk edge. W 19.2 są dostępne na Node.js. Dodatkowo nowe funkcje `resume` i `resumeAndPrerender` są dostarczone dla Node (z wariantami dla strumieni Node i Web Streams). Masz elastyczność: możesz używać Web Streams od początku, co daje jednolite API streamingu w różnych środowiskach JavaScript.

Zespół Reacta zaleca używanie API strumieni Node do SSR w Node, bo są szybsze i wspierają kompresję wbudowaną, której Web Streams nie mają. Web Streams w Node (od Node 18/19) mogą być wolniejsze i nie integrują się domyślnie z kompresją HTTP. React 19.2 daje opcję Web Streams w Node głównie dla zgodności czy eksperymentów. Do produkcyjnego renderowania serwerowego w Node prawdopodobnie zostaniesz przy `renderToPipeableStream`, które jest w pełni wspierane.

Przydatne, jeśli celujesz w Cloudflare Workers czy Deno Deploy (używające Web Streams) i chcesz uruchamiać kod na Node. Upraszcza to obsługę strumieni w różnych środowiskach. W miarę evolucji Node, Web Streams mogą stać się bardziej wydajne, więc React jest przygotowany. React 19.2 wypełnia lukę — czyniąc SSR bardziej elastycznym — ale daje developerom wybór zależnie od potrzeb wydajnościowych.

## Doświadczenie developera i różne usprawnienia

Poza sztandarowymi funkcjami React 19.2 zawiera kilka usprawnień nakierowanych na doświadczenie dewelopera i narzędzia.

### React Performance Tracks w DevTools

React 19.2 wprowadza Performance Tracks w Chrome DevTools — dobrodziejstwo dla developerów profilujących aplikacje. Nagrywając profil wydajności w Chrome (zakładka Performance), zobaczysz wyspecjalizowane ścieżki wizualizujące, co robi React pod maską. Dają wgląd w harmonogramowanie i renderowanie Reacta, pomagając znaleźć wąskie gardła i zrozumieć timing aktualizacji.

Dodano dwie główne ścieżki specyficzne dla Reacta:

- Scheduler Track: pokazuje aktywność wewnętrznego schedulera Reacta, skategoryzowaną wg priorytetu. Możesz zobaczyć, kiedy React obsługuje aktualizację „blokującą” (input użytkownika) kontra aktualizację „transition” (niskopriorytetową opakowaną w `startTransition`). Zdarzenia są oznaczone tym, co robi React — zdarzenie wyzwalające stan, start renderu, ustąpienia z powodu zmian priorytetu itd. Możesz zaobserwować, kiedy React czeka na paint przeglądarki czy aktualizacja jest w kolejce. Pomaga zwizualizować priorytetyzację pracy i może ujawnić, czy zbyt wiele pracy wykonuje się wysoko czy niskopriorytetowa aktualizacja opóźnia wysokopriorytetową.

- Components Track: pokazuje w czasie cykl życia renderowania i efektów komponentów. Wyświetla drzewo komponentów i kiedy każdy montuje, aktualizuje czy odmontowuje, i kiedy uruchamiają się efekty. Możesz zobaczyć blok „Render > MyComponent” wskazujący kiedy się renderował, czy etykiety takie jak „Effect mount” dla `useEffect`. Odnotowuje też, czy renderowanie było zablokowane/ustąpione (concurrent renderer). Niezwykle przydatne do identyfikacji wolnych komponentów czy zbędnych re-renderów.

Developer lub tech lead może łatwiej diagnozować wydajność. Na przykład, jeśli interakcja szarpie, ścieżka Scheduler może ujawnić, że „transition” aktualizacja jest przetwarzana jako blokująca. Ścieżka Components może pokazać, że głębokie poddrzewo niespodziewanie się re-renderuje. Te wglądy były trudne z ogólnej osi czasu, ale dzięki instrumentacji specyficznej dla Reacta profilowanie staje się o wiele bardziej intuicyjne.

Aby z tego korzystać, użyj Chrome (lub devtoolsów Chromium) i nagraj profil wydajności aplikacji React 19.2. Dokumentacja React DevTools zawiera szczegóły czytania tych ścieżek. Nie potrzeba zmian w kodzie — jest to wbudowane w React DevTools.

## Ulepszony plugin ESLint dla React Hooks (v6)

Wraz z React 19.2 zespół wydał `eslint-plugin-react-hooks` w wersji 6. Ta aktualizacja dostosowuje reguły lintera do nowych funkcji Reacta i nowych konfiguracji.

Kluczowe zmiany w pluginie ESLint dla hooków v6 obejmują:

- Wsparcie `useEffectEvent`: reguły lintera (szczególnie `exhaustive-deps`) rozpoznają `useEffectEvent` i traktują obsługi effect event odpowiednio. Nie będą ostrzegać o braku effect event w tablicach zależności. Zapobiega fałszywym alarmom i zachęca do poprawnego użycia bez wyłączania reguł. Aktualizując się do React 19.2 z `useEffectEvent`, zaktualizuj ten plugin, aby uniknąć konfuzji lintera.
- Flat config i nowe reguły: plugin domyślnie dostarcza nowszy format flat config ESLint w konfiguracji recommended. Wprowadza też reguły z React Compiler (opt-in), mogące wychwycić złożone problemy dzięki informacji Reacta z czasu kompilacji. Na przykład reguły do wykrywania nadużyć hooków czy egzekwowania dobrych praktyk.
- Konfiguracja legacy: jeśli musisz trzymać się starszej konfiguracji, jest preset `recommended-legacy`, ale domyślnie `recommended` odnosi się teraz do nowego stylu.

Z punktu widzenia migracji, po aktualizacji do React 19.2 powinieneś uruchomić:

```bash
npm install eslint-plugin-react-hooks@^6.0.0 --save-dev
```

i zaktualizować rozszerzenia (extends) swojej konfiguracji ESLint. Zapewni to, że kontrole lintera w twoim projekcie uwzględnią nowe zachowanie hooka. Dzięki tym usprawnieniom linting jest bardziej zgrany z runtime Reacta, pomagając ci wcześnie wychwytywać problemy i unikać wzorców mogących powodować błędy lub problemy z wydajnością. Na przykład linter będzie teraz poprawnie egzekwował reguły hooków dla `useEffectEvent` lub ignorował go w zależnościach, jak wspomniano, co utrzymuje twój kod czystym i zgodnym z intencjami Reacta.

Ogólnie ta aktualizacja pluginu to usprawnienie zza kulis, które wzmacnia dobre praktyki z hookami i daje developerom pewność przy adaptowaniu nowych API.

## Zaktualizowany domyślny prefiks `useId`

Hook `useId` w Reakcie generuje unikalne ID dla zawartości renderowanej po stronie serwera, by dopasować je do ID generowanych po stronie klienta (pomocne dla spójnej hydracji formularzy, id dostępności itd.). W React 19.0 i 19.1 ID miały domyślnie prefiksy takie jak `":r:"` lub `«r»`. Te prefiksy wybrano, ponieważ zawierały znaki nieprawidłowe w selektorach CSS, zmniejszając ryzyko kolizji z ID definiowanymi przez developera. Jednak w 19.2 zespół Reacta zmienił domyślny prefiks na `_r_`.

Nowy prefiks `_r_` jest prostszy (tylko alfanumeryczne znaki i podkreślnik), a powodem tej zmiany jest przyszła zgodność: nadchodzące funkcje takie jak View Transitions (pozwalające animować zmiany DOM) oraz pewne standardy webowe wymagają, by ID elementów były prawidłowymi identyfikatorami CSS i nazwami XML. Stary prefiks `:r:` nie był prawidłowym identyfikatorem CSS (z powodu dwukropka na początku), co mogło kolidować z użyciem tych ID w przejściach opartych na CSS lub innych API. Przez przełączenie na `_r_` wszelkie ID generowane przez `useId` są teraz prawidłowe do tych zastosowań.

Ta zmiana jest w większości przezroczysta. Jeśli używasz useId, po prostu zauważysz, że ID wyglądają teraz jak `_r_abcdef` zamiast czegoś w rodzaju `:r:abcdef`. Istnieje niewielka możliwość, że to będzie breaking change, jeśli kod lub testy twojej aplikacji zakładały coś o prefiksie albo jeśli miałeś CSS celujący w stary prefiks (co jest mało prawdopodobne, bo poleganie na nim było odradzane). Ogólnie nie powinno to wymagać zmian kodu dla większości aplikacji, ale to zauważalna poprawka dla tych, którym zależy na przewidywalnych ID. Korzyścią jest poprawiona zgodność z nowymi funkcjami przeglądarek (jak View Transitions) i zgodność ze standardami.

## Inne godne uwagi poprawki i usprawnienia

React 19.2 zawiera liczne poprawki błędów i drobne usprawnienia, które, choć nie tak przyciągające nagłówki, są ważne dla stabilności i komfortu pracy developera. Kilka przykładów problemów rozwiązanych w 19.2:

- Poprawki `useDeferredValue`: naprawiono błąd nieskończonej pętli związany z `useDeferredValue` przy użyciu ze zdarzeniami popstate historii przeglądarki. Zaadresowano też błąd inicjalizacji `useDeferredValue`. Te poprawki czynią użycie odroczonych wartości (do odraczania aktualizacji) bardziej odpornym.
- Usprawnienia Suspense: rozwiązano błąd z głęboko zagnieżdżonym `<Suspense>` wewnątrz drzew fallbacku oraz naprawiono problem, w którym zdehydratowane granice Suspense (z SSR) nie ukrywały poprawnie zawartości, jeśli ponownie się zawiesiły. Zapewnia to bardziej niezawodne działanie Suspense w złożonych scenariuszach zagnieżdżania.
- Poprawka crasha akcji formularza: naprawiono crash, który mógł wystąpić przy wysyłaniu formularzy używających nowych React Server Actions (dawniej znanych jako „Client Actions”).
- Ostrzeżenia dev i znaki ucieczki: React 19.2 przestaje teraz ostrzegać o prawidłowych atrybutach ARIA 1.3 (renderer DOM został zaktualizowany, by rozpoznawał nowsze atrybuty ARIA). Poprawiono też stack trace komponentów w komunikatach o błędach w kilku przypadkach brzegowych, dla łatwiejszego debugowania.
- Różne poprawki DOM: na przykład React DOM pozwala teraz na atrybut nonce na wynoszalnych (hoistable) tagach `<style>` (by lepiej wspierać polityki bezpieczeństwa CSP) i ostrzega, jeśli próbujesz wyrenderować węzeł kontrolowany przez Reacta do kontenera, który już zawiera tekst (co może wskazywać na problem z hydracją).

Te poprawki przyczyniają się do płynniejszego doświadczenia developera i mniejszej liczby pułapek przy aktualizacji. Choć żadna z nich nie jest breaking change, mogą nieco zmienić niektóre zachowania w runtime (zazwyczaj na lepsze, np. brak fałszywych ostrzeżeń czy crashy). Zawsze dobrą praktyką jest przejrzeć oficjalny changelog dla pełnej listy zmian i przetestować swoją aplikację, ale generalnie React 19.2 to wstecznie zgodne wydanie minor z głównie addytywnymi funkcjami i poprawkami błędów.

## Kwestie migracji i zgodności

Aktualizacja do React 19.2 powinna być względnie prosta dla większości projektów. Nie ma dużych breaking changes w API — istniejące komponenty i hooki działają nadal jak wcześniej. Nowe funkcje (Activity, useEffectEvent itd.) są w całości addytywne. Jednak oto kilka rzeczy, o których warto pamiętać przy migracji:

- Jeśli planujesz użyć `useEffectEvent`, zaktualizuj plugin ESLint dla hooków do v6, tak by twój linter nie flagował nowego hooka niepoprawnie. Ponadto, jeśli używasz TypeScript, upewnij się, że twoje `@types/react` są aktualne (definicje typów React 19.2 powinny zawierać nowe API).
- Zmiana prefiksu `useId` oznacza, że generowane ID będą się różnić. W mało prawdopodobnym przypadku, gdy testy lub kod twojej aplikacji polegają na dokładnym formacie tych ID, być może będziesz musiał je zaktualizować. Na przykład testy snapshot, które przechwytują ID DOM, mogą wymagać aktualizacji. Poleganie na tych ID jest generalnie odradzane, więc większość aplikacji nie będzie dotknięta.
- Jeśli zamierzasz użyć partial pre-rendering, zauważ, że wymaga to zgodnej konfiguracji serwera. Będziesz miał do czynienia ze streamingiem i być może utrwalaniem stanu postponed. Upewnij się, że twoja wersja Node jest wystarczająco nowoczesna (Node 18+ dla Web Streams lub użyj wersji API dla strumieni Node). Ta funkcja może też wiązać się ze zmianami w twoim pipeline'ie wdrożeniowym, jeśli używasz jej do SSG (np. generowania i przechowywania danych prelude/postponed).
- Jeśli twoja aplikacja intensywnie korzysta ze streamowanego SSR z Suspense, możesz chcieć przetestować, jak działa nowe grupowanie. W większości przypadków powinno ono tylko poprawić sytuację. Bądź jedynie świadomy tej zmiany, jeśli mierzyłeś lub animowałeś pojawianie się zawartości na kliencie — teraz wiele segmentów Suspense może wyskoczyć razem. Zazwyczaj jest to pożądane, ale zweryfikuj, czy nie wpływa to na twoje spinnery ładowania lub założenia UI.
- Jeśli twój projekt nie potrzebuje nowych API od razu, możesz aktualizować się we własnym tempie. Wiele korzyści z 19.2 (jak performance tracks, poprawki błędów, usprawnienia Suspense) przychodzi „za darmo”, po prostu przez aktualizację. Decyzja może zależeć od tego, czy planujesz użyć Activity lub useEffectEvent. Zawsze dobrym pomysłem jest uruchomienie zestawu testów i może wykonanie odrobiny ręcznego testowania po aktualizacji, zwłaszcza dla złożonych projektów, by wychwycić wszelkie nieprzewidziane problemy (które powinny być rzadkie).

Aktualizacja jest tak prosta jak uruchomienie:

```bash
npm install react@19.2 react-dom@19.2   # lub użyj @latest
```

(I podobnie zaktualizuj inne pakiety React, jak react-dom, do 19.2, by wersje były zsynchronizowane.)

Ogólnie React 19.2 to podbicie wersji minor z dużymi zyskami w produktywności i wydajności. Otrzymujesz nowe możliwości bez konieczności przepisywania istniejącego kodu. Zwróć tylko uwagę na kilka zmian w domyślnych ustawieniach (jak prefiks `useId`) i upewnij się, że twoje narzędzia (ESLint, rozszerzenie DevTools itd.) są aktualne, by w pełni wykorzystać usprawnienia.

## Podsumowanie

React 19.2 pakuje mnóstwo ekscytujących usprawnień jak na wydanie „point-two”. Wprowadza praktyczne nowe prymitywy, które czynią aplikacje React bardziej wydajnymi i łatwiejszymi w tworzeniu:

- Komponent `<Activity>` i hook `useEffectEvent` adresują częste wzorce (przenoszenie UI do tła i obsługa zdarzeń efektów) w czysty, deklaratywny sposób, prawdopodobnie upraszczając kod dla modali, kart, subskrypcji sieciowych i nie tylko.
- Usprawnienia w renderowaniu po stronie serwera — od partial pre-rendering po grupowanie Suspense — przybliżają React do bezszwowych, natychmiast ładujących się aplikacji, bez poświęcania SEO czy doświadczenia użytkownika. Aplikacje React mogą dostarczać zawartość szybciej i płynniej niż wcześniej.
- Doświadczenie developera zostaje podniesione dzięki lepszym narzędziom profilowania (ścieżki DevTools) i inteligentniejszemu lintingowi. Pomagają one zespołom optymalizować i utrzymywać duże bazy kodu z pewnością.
- Dziesiątki poprawek i drobnych zmian pod maską oznaczają większą niezawodność i zgodność ze standardami webowymi (jak dopasowanie do View Transitions i nowszych specyfikacji ARIA).

Podsumowując, nawet jeśli React 19.2 jest oficjalnie wydaniem minor, przynosi znaczące usprawnienia komfortu pracy i funkcje patrzące w przyszłość do ekosystemu React. Frontendowcy i liderzy techniczni docenią, że te zmiany nie tylko rozwiązują bieżące bolączki (jak niezgrabne wzorce efektów czy wolne aktualizacje SSR), ale też kładą fundament pod przyszłość (jak płynniejsze przejścia i bardziej efektywny streaming). Jeśli utrzymujesz aplikację React, warto ocenić, jak te nowe możliwości mogą przynieść korzyści twojemu projektowi. Aktualizacja jest generalnie niskiego tarcia, a zyskasz dostęp do tych potężnych funkcji i optymalizacji, które mogą uczynić twoje UI szybszym, a kod czystszym.
