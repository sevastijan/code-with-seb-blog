"Po prostu włącz React Compiler i wyrzuć wszystkie `useMemo`". Słyszałem to co najmniej kilka razy od premiery 1.0. I za każdym razem się wstrzymuję — bo to tylko połowa prawdy, a druga część potrafi zrujnować Ci wydajność.

Pojawił się w październiku 2025 w wersji 1.0. Jest stabilny, produkowany już w Instagrama i Quest Store, i domyślnie włączony w Expo SDK 54. To wszystko prawda: automatycznie memoizuje komponenty i hooki bez pisania ręcznie `useMemo` czy `useCallback`.

Ale jest haczyk. W rzeczywistej produkcyjnej aplikacji 15 000 linii kodu kompilator poprawnie skompilował 361 z 363 komponentów — a jednocześnie w pełni rozwiązał tylko 2 z 9 problemów wydajnościowych. Pozostałych 7 wymagało ręcznej naprawy.

Kompilator zmienia grę. Nie jest magią. Pokażę Ci dokładnie co się tam dzieje.

## Czym React Compiler naprawdę jest (a czym nie)

To **kompilator optymalizacyjny w fazie builda**, który przepisuje kod Reacta podczas bundlowania. Nie jest to biblioteka runtime'owa — nie wysyła do przeglądarki dodatkowego JS-a. Przetwarza komponenty w buildzie i automatycznie dodaje memoizację, którą inaczej musiałbyś kodować ręcznie.

Co zastępuje:
- `useMemo` do kosztownych obliczeń
- `useCallback` do stabilnych referencji funkcji
- `React.memo` do pomijania re-renderów

Czego **nie** zastępuje:
- Decyzji architektonicznych (podział komponentów, normalizacja danych)
- Zachowania zewnętrznych bibliotek (React Query, Redux itd.)
- Optymalizacji sieciowej (lazy loading, code splitting)
- Wydajności na poziomie DOM (wirtualizacja, CSS containment)

Patrz na niego jako na optymalizator, nie na czarownika. Przyspiesza kod dobrze zaprojektowany. Nie naprawi architektury, która jest źle pomyślana. Jeśli chcesz wiedzieć, gdzie dokładnie w cyklu renderowania kompilator wstawia memoizację, przeczytaj o [tym jak React Fiber robi rekoncyliację](/pl/blog/react-fiber-silnik-nowoczesnego-reacta).

## Jak to działa: pipeline kompilacji

Pod spodem robi coś zaskakująco zaawansowanego jak na tool JavaScriptowy: prawdziwy pipeline kompilatora z reprezentacjami pośrednimi, analizą przepływu danych i kilkoma przebiegami optymalizacyjnymi.

### Krok 1: Parsowanie do AST

Zaczyna się jako plugin Babela (ze wsparciem SWC w Next.js 15.3.1+). Bierze Twój kod źródłowy i zamienia go na drzewo składni (Abstract Syntax Tree).

### Krok 2: Budowa grafu przepływu sterowania (CFG)

Tu robi się interesująco. Kompilator buduje **graf przepływu sterowania** (Control Flow Graph) — taką samą strukturę której używają LLVM czy GCC. CFG mapuje każdą możliwą ścieżkę, którą komponent może podjąć.

```
// Twój komponent:
function Product({ item, onAdd }) {
  if (!item) return null;
  const price = formatPrice(item.price);
  return <Card price={price} onClick={() => onAdd(item.id)} />;
}

// CFG widzi dwie ścieżki:
// Ścieżka 1: item jest falsy → return null
// Ścieżka 2: item jest truthy → oblicz price → wyrenderuj Card
```

### Krok 3: Analiza z użyciem High-Level Intermediate Representation (HIR)

CFG trafia do **High-Level IR**, gdzie kompilator robi analizę przepływu danych. Śledzi:

- **Które wartości zależą od jakich inputów** (props, state, context)
- **Które operacje są czyste** (bez efektów ubocznych)
- **Co można bezpiecznie zachować między renderami**

I tutaj robi coś czego ręczna memoizacja nie potrzebuje: memoizuje **przez gałęzie warunkowe**. Przykład:

```javascript
function ThemeProvider(props) {
  if (!props.children) {
    return null;  // wczesny return
  }

  // useMemo ręczne NIE może być tu (Rules of Hooks mówią nie)
  // ale React Compiler potrafi zmemoizować to obliczenie
  const theme = mergeTheme(props.theme, use(ThemeContext));

  return (
    <ThemeContext value={theme}>
      {props.children}
    </ThemeContext>
  );
}
```

Z `useMemo` musiałbyś przepisać ten komponent aby uniknąć hooków po warunkowym returnie. Kompilator robi to samo automatycznie, bo patrzy na CFG, nie na składnię.

### Krok 4: Wstawianie memoizacji

Na podstawie analizy kompilator dodaje memoizację na poziomie pojedynczych wartości i elementy JSX. Nie owija całego komponentu w `React.memo` — ustala dokładnie co można zachować, a co trzeba przeliczać.

### Krok 5: Generowanie kodu

Kod trafia do przeglądarki. To normalna React, bez specjalnego runtime'u. Memoizacja używa zwykłych prymitywów Reacta, tylko że wstawianej automatycznie.

## Przed i po: jak zmienia się kod

Praktyczna różnica. Bez kompilatora pisałbyś:

```javascript
import { useMemo, useCallback, memo } from 'react';

const ExpensiveList = memo(function ExpensiveList({ items, onSelect }) {
  const sorted = useMemo(
    () => items.slice().sort((a, b) => b.score - a.score),
    [items]
  );

  const handleSelect = useCallback(
    (id) => onSelect(id),
    [onSelect]
  );

  return (
    <ul>
      {sorted.map(item => (
        <ListItem
          key={item.id}
          item={item}
          onSelect={handleSelect}
        />
      ))}
    </ul>
  );
});
```

Z kompilerem ta sama logika to:

```javascript
function ExpensiveList({ items, onSelect }) {
  const sorted = items.slice().sort((a, b) => b.score - a.score);

  const handleSelect = (id) => onSelect(id);

  return (
    <ul>
      {sorted.map(item => (
        <ListItem
          key={item.id}
          item={item}
          onSelect={handleSelect}
        />
      ))}
    </ul>
  );
}
```

To samo zachowanie, ta sama wydajność, zero boilerplate'u. Kompilator sam ustala co trzeba zachować i robi to w buildzie.

**Co się zmieniło:** zniknęły `memo()`, `useMemo()` i `useCallback()`. Kod jest czystszy, łatwiejszy do czytania. Optymalizacja jest transparentna — i to jest sens.

## Rules of React: dlaczego liczą się teraz bardziej

Kompilator zależy od **Rules of React** by robić analizę. Jak je łamiesz, kompilator albo cofa optymalizację (cicho), albo generuje błędy.

Trzy zasady, które liczą się najbardziej:

### Zasada 1: Komponenty muszą być czyste

To samo na wejściu = to samo na wyjściu. Bez efektów ubocznych w renderze. Kompilator na tym polega żeby wiedzieć co można zachować.

```javascript
// ❌ Łamie to
function BadComponent({ items }) {
  analytics.track('render');  // efekt uboczny w renderze!
  return <List items={items} />;
}

// ✅ Kompilator optymalizuje to
function GoodComponent({ items }) {
  useEffect(() => {
    analytics.track('render');
  });
  return <List items={items} />;
}
```

### Zasada 2: Props i state to immutable

Nie mutuj propsów ani state'u. Kompilator polega na tożsamości referencji aby wiedzieć co się zmieniło.

```javascript
// ❌ mutacja — kompilator nie widzi zmian
function BadForm() {
  const [user, setUser] = useState({ name: '', email: '' });

  const handleName = (e) => {
    user.name = e.target.value;  // bezpośrednia mutacja!
    setUser(user);  // ta sama referencja — kompilator myśli że nic się nie zmieniło
  };
}

// ✅ immutable update — kompilator optymalizuje
function GoodForm() {
  const [user, setUser] = useState({ name: '', email: '' });

  const handleName = (e) => {
    setUser({ ...user, name: e.target.value });
  };
}
```

### Zasada 3: Hooki zawsze bezwarunkowo

Statyczna analiza kompilatora potrzebuje hooków w tej samej kolejności każdy render. Chcesz się ulepszć? Zajrzyj do [przewodnika po React hooks](/pl/blog/zaawansowane-hooki-react-najlepsze-praktyki).

Dobra wiadomość: **Kompilator ma ESLint reguły** które łapią naruszenia. Zainstaluj `eslint-plugin-react-hooks` z presetem `recommended-latest` i dostajesz ostrzeżenia zanim kompilator pominie coś cicho.

## 5 wzorców które kompilator nie rozwiąże

Sekcja którą pomija każdy "React Compiler jest cudowny" artykuł. To izdajnościowe problemy na które musisz uważać:

### 1. Obiekty z bibliotek jako propsy

Najczęstsza produkcyjna bolączka. React Query i inne zwracają nowe obiekty każdy render — kompilator nie może zatrzymać re-rendera poniżej:

```javascript
// React Query zwraca nowy obiekt 'data' co render
const { data } = useQuery({ queryKey: ['users'], queryFn: fetchUsers });

// Kompilator memo UserList ale 'data' to nowy obiekt zawsze
// więc UserList się i tak re-renderuje
return <UserList users={data} />;
```

**Opcja:** Zdestrukturyzuj do scalars albo użyj `select` do stabilnych wartości:

```javascript
const { data } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  select: (data) => data.map(u => u.id)  // tablica scalars, stabilna
});
```

### 2. Efekty uboczne w renderze

Kompilator nie kompiluje komponentów z efektami w renderze. Jeśli czytasz z globalnego store'a, mutujesz ref'y albo woołasz API w renderze — brak optymalizacji.

### 3. Context — wszyscy słuchacze re-renderują

Jak context się zmienia, **wszyscy konsumenci re-renderują** — nawet jeśli używają tylko jednej części. Kompilator nie potrafi rozdzielić czytania.

```javascript
// każdy komponent czytający AppContext re-renderuje
// gdy JAKAŚ część kontekstu się zmieni
const { theme, user, locale } = useContext(AppContext);
```

**Opcja:** Podziel contextsy po częstości zmian albo użyj `use-context-selector`.

### 4. Duże drzewa montowane na click

Kompilator optymalizuje *re-rendery*, nie *initial mount*. Jeśli klik montuje 500 komponentów — brak pomocy, to architektura. Warto wiedzieć że kompilator to Reactowy odwet na [granular reactivity i signals](/pl/blog/sygnaly-drobnoziarnista-reaktywnosc-2026) które wszystkie inne frameworki już mają — ten sam cel (nie marnować pracy), inna strategia.

**Opcja:** Lazy loading, virtualization, progressive reveal. Szczegółowo w [React Suspense przewodniku](/pl/blog/react-suspense-poradnik-lazy-loading-async).

### 5. Obliczenia poza komponentami

Kompilator memoizuje tylko React komponenty i hooki. Helpery, klasy, module-level obliczenia — bez zmian.

```javascript
// to NIE jest memoizowane
const processedData = heavyTransform(rawData);

function Dashboard() {
  // kompilator tu memoizuje
  return <Chart data={processedData} />;
}
```

**Opcja:** Przenieś ciężkie obliczenia do wnętrza komponentu (gdzie kompilator dosięga) albo zmemoizuj ręcznie na module level.

### Rzeczywistość w produkcji

**Sanity Studio** (15K LOC):
- 1 231 z 1 411 komponentów skompilowanych
- 20-30% ogólnie szybsze rendery
- Największe zyski w deeply nested prop chains

**Wakelet:**
- LCP +10%
- INP +~15% (30% pure React)

**Real audit** (developerway.com):
- 361 z 363 komponentów skompilowanych
- Jedna strona: 280ms TBT → 0ms
- Druga: 130ms → 90ms
- **2 z 9 wydajnościowych problemów rozwiązane** tylko kompilator

Obraz jasny: kompilator daje wyniki, ale to nie silver bullet. Duże wygrane z głębokich prop chainów i częstych re-renderów. Małe (albo żadne) jeśli problem jest w architekturze albo bibliotekach.

## Setup: dodaj do projektu

### Krok 1: Install

```bash
npm install --save-dev --save-exact babel-plugin-react-compiler@latest
```

Pin wersję — zmian między wersjami mogą subtelnie zmienić zachowanie.

### Krok 2: Config

**Next.js** (next.config.js):

```javascript
module.exports = {
  experimental: {
    reactCompiler: true,
  },
};
```

**Vite** (vite.config.ts):

```typescript
import { reactCompiler } from 'eslint-plugin-react-compiler';

export default defineConfig({
  plugins: [
    react({
      babel: {
        plugins: ['babel-plugin-react-compiler'],
      },
    }),
  ],
});
```

### Krok 3: ESLint rules

```javascript
// eslint.config.js
import reactHooks from 'eslint-plugin-react-hooks';
export default [reactHooks.configs['flat/recommended-latest']];
```

Łapie naruszenia Rules of React zanim kompilator sie cofa cicho.

### Krok 4: Stopniowo

Nie kompiluj all at once. Zacznij od części:

```javascript
// babel-plugin-react-compiler config
{
  compilationMode: 'annotation',
}
```

Potem dodaj `'use memo'` do komponentów które chcesz:

```javascript
'use memo';
function MyComponent() {
  // ten komponent będzie skompilowany
}
```

Jak nabierzesz pewności, full compilation mode.

### Krok 5: Measure

React DevTools Profiler: porównaj render times before/after. Focus INP, nie tylko initial load.

## Kiedy zachować useMemo/useCallback

Te hooki nie zginęły. Keep them dla:

1. **Effect deps** — Jak zmemoizowana wartość jest dep w effect, jawny `useMemo` zapobiega re-run

```javascript
const config = useMemo(() => ({ threshold: 0.5, enabled }), [enabled]);

useEffect(() => {
  initializeSDK(config);  // runs tylko jak 'enabled' zmieni się
}, [config]);
```

2. **Library integration** — Niektóre libs chcą reference stability, którą kompilator nie gwarantuje między wersjami

3. **Critical hot paths** — Jak zmierzyłeś i znalazłeś bottleneck który kompilator nie rozwiązuje

4. **Incremental migration** — Jak migrujesz stopniowo, keep existing memo aż nie zweryfikujesz kompilatora

**Kciuk:** Nowy kod — nie dodawaj manual memo. Daj kompilatorowi. Stary kod — nie śpiesz się z usuwaniem, test first.

## Verdict

To największa rzecz w Reacta od Hooks — razem z [Activity comp i useEffectEvent w React 19.2](/pl/blog/react-19-2-przewodnik-po-wydaniu) zmienia jak myślimy o React performance. Zabija całą kategorię bugów — te z zapomnianych memo, złych deps, inline arrow functions.

To optymalizator, nie czary. Przyspiesza dobrą architekturę. Nie naprawi złej.

**Co się zmienia:**
- Koniec boilerplate `useMemo`/`useCallback` w nowym kodzie
- Mniej "czemu to się re-renderuje" sesji
- Więcej architectury, mniej micro-optimizations

**Co się nie zmienia:**
- Musisz myśleć o component boundaries
- Musisz obsługiwać library references
- Musisz virtualize duże listy
- Musisz code split duże bundles — szczególnie z [edge runtime i Server Components w Next.js](/pl/blog/nextjs-2026-server-components-edge-runtime)

Włącz. Ufaj w standard cases. Ale keep performance profiling sharp — kompilator robi 80%, a Twoja ekspertyza robi pozostałe 20%.
