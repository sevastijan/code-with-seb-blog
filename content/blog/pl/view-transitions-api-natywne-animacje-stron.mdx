W zeszłym miesiącu usunąłem z projektu klienta 14 kB kodu animacji (po gzipie), a przejścia między stronami stały się *lepsze*. Żadnego `AnimatePresence`, żadnych hacków na layout-shift, żadnego `motion.div` opakowującego każdą trasę. Tylko dwie reguły CSS i kilka deklaracji `view-transition-name`.

To jest właśnie obietnica **View Transitions API** i w 2026 roku wreszcie ją spełnia. Wariant single-page nadawał się do produkcji w Chromium już od jakiegoś czasu, ale prawdziwym przełomem są **cross-document view transitions**: płynne, morfujące animacje przy pełnych nawigacjach między stronami w zwykłej aplikacji multi-page, bez żadnego routera po stronie klienta. Chrome dowiózł to w wersji 126, Safari dołączyło w 18.2, a zespoły po cichu wyrywają ciężkie biblioteki do animacji ze swoich bundle'i.

Spędziłem z tym API wystarczająco dużo czasu — łącznie z fragmentami, które kazały mi przeklinać na monitor — żeby napisać przewodnik, którego sam bym potrzebował. Przejdźmy przez to, czym to naprawdę jest, przez obie powierzchnie API, jak dostosować animacje, jak wygląda historia z frameworkami oraz przez pułapki, które cię ugryzą, jeśli nikt cię nie ostrzeże.

## Co View Transitions API właściwie robi

Oto model mentalny, dzięki któremu wszystko mi wskoczyło na miejsce. Gdy uruchamiasz przejście, przeglądarka robi cztery rzeczy:

1. **Przechwytuje** bieżący stan wizualny strony (lub konkretnych elementów) jako migawki (snapshots) — w zasadzie zrzuty ekranu.
2. **Pozwala ci zmienić DOM** do nowego stanu.
3. **Przechwytuje** nowy stan jako drugi zestaw migawek.
4. **Animuje** przejście między starymi a nowymi migawkami, po czym oddaje kontrolę żywemu DOM-owi.

Genialny jest krok 4. Podczas animacji nie animujesz żywych węzłów DOM — przenikasz i transformujesz statyczne obrazy wygenerowane przez przeglądarkę. Dlatego jest to tanie, dlatego nie szarpie layoutem i dlatego domyślny efekt już wygląda przyzwoicie. Cała choreografia dzieje się na kompozytorze.

Są dwa sposoby, żeby to uruchomić: **same-document API** (wywołujesz funkcję JavaScript po zmianie DOM) oraz **cross-document API** (przeglądarka uruchamia je automatycznie przy nawigacji w tym samym originie). Ta sama idea, dwa punkty wejścia.

## Przejścia same-document: API JavaScript

Dla SPA, aktualizacji renderowanych po stronie klienta czy dowolnego momentu typu „DOM się właśnie zmienił" wywołujesz `document.startViewTransition()`:

```js
// Klasyczna forma: przekazujesz callback, który zmienia DOM.
// Przeglądarka robi migawkę "przed", uruchamia twój callback, robi migawkę "po",
// a następnie animuje przejście między nimi.
function updateView(newContent) {
  // Wykrywanie funkcji — zawsze. Starsze przeglądarki po prostu dostają natychmiastową aktualizację.
  if (!document.startViewTransition) {
    applyDOMUpdate(newContent);
    return;
  }

  document.startViewTransition(() => {
    applyDOMUpdate(newContent); // synchroniczna mutacja DOM idzie tutaj
  });
}
```

To cała ścieżka „szczęśliwego przypadku". Bez ani jednej dodatkowej reguły CSS dostajesz już **domyślne przenikanie (cross-fade)** całej strony, bo przeglądarka niejawnie przypisuje `view-transition-name: root` elementowi dokumentu.

`startViewTransition()` zwraca **obiekt `ViewTransition`** z trzema promise'ami i furtką awaryjną. Mają one większe znaczenie, niż sugeruje dokumentacja:

```js
const transition = document.startViewTransition(() => applyDOMUpdate(newContent));

// Rozwiązuje się, gdy callback DOM się zakończył ORAZ migawki są gotowe —
// to moment, żeby podpiąć własne animacje przez Web Animations API.
transition.ready.then(() => {
  // np. animuj pseudo-element ręcznie
});

// Rozwiązuje się, gdy twój callback aktualizacji się wykonał (DOM jest w nowym stanie),
// ale animacja może wciąż trwać.
transition.updateCallbackDone.then(() => {});

// Rozwiązuje się, gdy animacja jest całkowicie zakończona i widoczny jest żywy DOM.
await transition.finished;

// Przerwij animację, ale zachowaj aktualizację DOM. Świetne przy
// prefers-reduced-motion lub gdy użytkownik nawiguje ponownie w trakcie przejścia.
transition.skipTransition();
```

Jeśli twoja aktualizacja jest asynchroniczna — powiedzmy, że pobierasz dane przed renderowaniem — przekaż asynchroniczny callback. Przeglądarka trzyma migawkę „przed" zamrożoną, gdy czekasz na `await`, co jest potężne, a zarazem stanowi pułapkę, do której jeszcze wrócę.

### Forma z opcjami i typy przejść

Nowoczesne Chromium akceptuje też **obiekt opcji** zamiast samego callbacku, co odblokowuje `types`:

```js
document.startViewTransition({
  update: () => applyDOMUpdate(newContent),
  // Otaguj to konkretne przejście, aby CSS mógł je stylować inaczej.
  types: ['slide-forward'],
});
```

Typy pozwalają jednej stronie mieć kilka odrębnych animacji — nawigacja do przodu vs. do tyłu, rozwijanie vs. zwijanie — bez żonglowania klasami na elemencie root. Celujesz w nie pseudoklasą `:active-view-transition-type()`, której użyję poniżej.

## view-transition-name i morfowanie shared elements

Przenikanie jest w porządku. Magia to **shared element transitions** — miniatura, która płynnie rozrasta się w obraz hero, karta, która przechodzi w panel szczegółów. Uzyskujesz to, nadając „temu samemu" elementowi w obu stanach identyczny `view-transition-name`:

```css
/* Na stronie z listą */
.product-card__image {
  view-transition-name: product-hero;
}

/* Na stronie ze szczegółami — ta sama nazwa, inny element */
.product-detail__image {
  view-transition-name: product-hero;
}
```

Gdy nazwa istnieje zarówno w starej, jak i nowej migawce, przeglądarka przestaje ją przenikać, a zamiast tego **animuje jej pozycję, rozmiar i kształt** ze starego do nowego. Interpoluje `width`, `height`, `transform`, a nawet `border-radius`. To ten morph, do którego ludzie kiedyś ściągali całą bibliotekę fizyki.

**Jedna reguła, która cię ugryzie:** każdy `view-transition-name` musi być *unikalny w obrębie dokumentu w momencie przechwytywania*. Dwa elementy z `view-transition-name: product-hero` widoczne w tym samym czasie rzucają błąd i pomijają całe przejście. Więc na liście 20 produktów nie możesz statycznie nazwać ich wszystkich `product-hero` — generujesz unikalne nazwy (`product-hero-42`) i przypisujesz współdzieloną nazwę tylko *tej jednej* karcie, którą użytkownik faktycznie kliknął, zwykle przez handler zdarzenia w JS tuż przed rozpoczęciem przejścia.

### Drzewo pseudo-elementów

Gdy przejście się uruchamia, przeglądarka buduje drzewo pseudo-elementów `::view-transition` nałożone na wszystko. Zrozumienie go oddziela „wkleiłem snippet" od „potrafię zbudować dowolną animację, jaką chcę":

```
::view-transition                          (nakładka root, pokrywa viewport)
└─ ::view-transition-group(name)           (kontener z pozycją + rozmiarem per nazwa)
   └─ ::view-transition-image-pair(name)   (kontekst izolacji dla blendu)
      ├─ ::view-transition-old(name)       (migawka starego stanu)
      └─ ::view-transition-new(name)       (migawka nowego stanu)
```

`-group` obsługuje morfowanie rozmiaru/pozycji. `-old` i `-new` to faktyczne migawki przenikane wewnątrz `-image-pair`. Aby dostosować animację, celujesz w te pseudo-elementy zwykłymi `@keyframes` w CSS:

```css
/* Wsuń całą zawartość strony z prawej zamiast przenikania */
@keyframes slide-from-right {
  from { transform: translateX(30px); opacity: 0; }
}
@keyframes slide-to-left {
  to { transform: translateX(-30px); opacity: 0; }
}

::view-transition-old(root) {
  animation: 250ms ease both slide-to-left;
}
::view-transition-new(root) {
  animation: 250ms ease both slide-from-right;
}
```

Ponieważ to zwykła animacja CSS na pseudo-elementach, obowiązuje wszystko, co już wiesz — `animation-delay`, `cubic-bezier`, `steps()`, cały arsenał. Chcesz wsunięcia tylko przy nawigacji do przodu? Ogranicz to typem:

```css
/* Tylko gdy przejście zostało otagowane types: ['slide-forward'] */
html:active-view-transition-type(slide-forward)::view-transition-old(root) {
  animation: 250ms ease both slide-to-left;
}
```

## Przejścia cross-document: zero JavaScriptu dla MPA

To jest sztandarowa funkcja 2026 roku i powód, dla którego piszę ten tekst teraz. Dla **aplikacji multi-page** — stron renderowanych na serwerze, stron statycznych, aplikacji w Astro, Rails czy zwykłym HTML — możesz uzyskać dokładnie te same przejścia przy prawdziwych nawigacjach, *bez ani grama JavaScriptu*. Zarówno strona, którą opuszczasz, jak i ta, na której lądujesz, włączają się jedną regułą at-rule CSS:

```css
/* Umieść to w globalnym arkuszu stylów ładowanym na każdej stronie */
@view-transition {
  navigation: auto;
}
```

I tyle. Dwie strony z tego samego originu, obie niosące tę regułę, i każde kliknięcie linku czy naciśnięcie przycisku Wstecz przenika między nimi. **Nie ma żadnej funkcji do wywołania** — przeglądarka jest właścicielem triggera. Przechwytuje stary dokument, ładuje nowy, czeka, aż stanie się renderowalny, i animuje. Shared elements morfują przez nawigację używając tego samego mechanizmu `view-transition-name`, więc pasek nawigacji czy obraz hero istniejący na obu stronach zostaje na miejscu i płynnie wślizguje się w nową pozycję.

To naprawdę duża sprawa. Architektury multi-page — które wracają do łask z dobrych [powodów renderingowych](/blog/ssr-vs-ssg-vs-isr-vs-rsc-nextjs-strategies) — nie muszą już wybierać między „płynnością SPA" a „brakiem routera po stronie klienta". Zachowujesz prostotę MPA i dostajesz szlif SPA.

### Dostosowywanie per-nawigacja z pageswap i pagereveal

Goła reguła `@view-transition` przenika. Aby robić kierunkowe lub kontekstowe animacje między dokumentami, podpinasz dwa zdarzenia. **`pageswap`** odpala się na stronie *wychodzącej* tuż przed jej zrzuceniem do migawki; **`pagereveal`** odpala się na stronie *przychodzącej* przed jej pierwszym renderem. Oba udostępniają obiekt `viewTransition` (gdy przejście faktycznie się dzieje) oraz `activation` nawigacji z adresami URL from/to:

```js
// Strona wychodząca — zdecyduj o kierunku i otaguj przejście
window.addEventListener('pageswap', (event) => {
  if (!event.viewTransition) return;

  const fromPath = event.activation.from?.url;
  const toPath = event.activation.entry.url;

  // Przykład: przypisz współdzieloną nazwę tylko klikniętemu elementowi lub ustaw typ
  if (isForward(fromPath, toPath)) {
    event.viewTransition.types.add('forward');
  } else {
    event.viewTransition.types.add('back');
  }
});

// Strona przychodząca — odzwierciedl decyzję, aby stare/nowe migawki się zgadzały
window.addEventListener('pagereveal', (event) => {
  if (!event.viewTransition) return;
  // Ustaw tutaj view-transition-name na elemencie docelowym, aby morph trafił
});
```

Typowy wzorzec: na indeksie bloga w `pageswap` ustawiasz `view-transition-name: article-hero` na konkretnej karcie linkującej do celu, a w `pagereveal` na stronie artykułu ustawiasz tę samą nazwę na jego obrazie nagłówkowym. Przeglądarka je dopasowuje i morfuje. Posprzątaj nazwy potem (w `finished`), aby nie kolidowały przy następnej nawigacji.

## Historia z frameworkami: React, Next.js i Astro

Nie potrzebujesz frameworka do niczego z tego — o to właśnie chodzi. Ale ergonomia ręcznego nazywania elementów i żonglowania `pageswap` w SPA jest kiepska, więc frameworki to opakowują.

### React i Next.js

React dowozi eksperymentalny komponent **`<ViewTransition>`** (React Canary / 19.x experimental w chwili pisania — jeszcze nie stabilny, więc traktuj go tak samo). Deklaratywnie oznacza poddrzewo, które ma się animować, gdy wchodzi, wychodzi lub aktualizuje się wewnątrz `startTransition`:

```jsx
import { unstable_ViewTransition as ViewTransition } from 'react';

function Page({ children }) {
  return (
    <ViewTransition>
      {children}
    </ViewTransition>
  );
}
```

Ponieważ React steruje aktualizacjami DOM przez własny scheduler, podpina komponent do `document.startViewTransition()` za ciebie i animuje tylko wtedy, gdy aktualizacja pochodzi z tranzycji — co ładnie łączy się z modelem współbieżnego renderowania, który omówiłem w [przewodniku po React concurrent mode](/blog/react-concurrent-mode-practical-guide). Przyczynę animacji wybierasz przez **`addTransitionType`**:

```jsx
import { startTransition, addTransitionType } from 'react';

function navigate(url) {
  startTransition(() => {
    addTransitionType('nav-forward'); // mapuje się na typ możliwy do wskazania w CSS
    router.push(url);
  });
}
```

**Next.js** udostępnia to za flagą. Włącz w konfiguracji, opakuj swoją zawartość i nawigacje między trasami stają się przejściami automatycznie:

```js
// next.config.js
module.exports = {
  experimental: {
    viewTransition: true,
  },
};
```

Od Next.js 16.2 komponent `<Link>` przyjmuje prop `transitionTypes` — tablicę przekazywaną prosto do `addTransitionType` — więc możesz sterować kierunkowymi animacjami z samego linku. Jeśli jesteś już na ścieżce aktualizacji, wpasowuje się to obok innych zmian w [przewodniku migracji do Next.js 16](/blog/nextjs-16-migration-guide-breaking-changes).

### Astro

Astro było tu wcześnie i teraz oferuje oba podejścia. Jego komponent `<ClientRouter />` przechwytuje nawigacje, zamienia twoje MPA w soft-SPA i dodaje funkcje, których natywnemu API wciąż brakuje (utrzymywanie stanu między nawigacjami, drobniejsze dyrektywy). Nazywasz shared elements dyrektywą `transition:name`, a animacje wybierasz przez `transition:animate`:

```astro
---
import { ClientRouter } from 'astro:transitions';
---
<head>
  <ClientRouter />
</head>
<body>
  <img src={hero} transition:name="cover" transition:animate="fade" />
</body>
```

Ciekawy ruch 2026 roku polega na tym, że wiele stron w Astro **porzuca `ClientRouter` na rzecz natywnej reguły `@view-transition`**. Tracisz kilka frameworkowych udogodnień, ale kasujesz cały JavaScript routera i opierasz się na webowym standardzie. Dla stron z treścią, gdzie przejście jest czystą dekoracją, to często właściwy kompromis.

## Dostępność: uszanuj prefers-reduced-motion

To nie jest opcjonalne. Zaburzenia przedsionkowe wywoływane ruchem są realne, a slajd na całą stronę może kogoś fizycznie przyprawić o mdłości. Dobra wiadomość jest taka, że zabijasz *każdą* animację view transition jednym media query — to [fundament dostępności, który każdy frontend jest winien użytkownikom w 2026](/blog/web-accessibility-2026-eaa-ada-wcag-guide):

```css
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

Ustawienie `animation: none` *nie* psuje przejścia — DOM wciąż się aktualizuje, nowy stan wciąż się pokazuje. Pomijasz jedynie ruch, czyli dokładnie to, o co użytkownik poprosił. Dla przejść same-document możesz też rozgałęzić się w JS i wywołać `skipTransition()`, ale podejście CSS pokrywa cross-document za darmo, więc od niego zaczynam.

Jeszcze jedno: pseudo-elementy są `aria-hidden` i inert podczas animacji, a to żywy DOM widzą czytniki ekranu. To poprawne zachowanie, ale oznacza, że zbyt długie przejścia opóźniają moment, w którym technologia asystująca dociera do prawdziwej treści. Trzymaj czasy trwania krótkie (200–300 ms) także z tego powodu.

## Wydajność i pułapki, o których nikt nie wspomina

API jest z założenia szybkie — animujesz przyjazne kompozytorowi migawki — ale nie jest darmowe, a ma ostre krawędzie.

**Koszt migawki skaluje się z malowanym obszarem.** Każdy nazwany element staje się teksturą. Nazwanie dziesiątek elementów albo nazwanie czegoś ogromnego oznacza, że przeglądarka alokuje wielkie migawki, a krok przechwytywania może się zacinać na obciążonej klatce. Nazywaj tylko to, co musi morfować. To rzecz warta pilnowania względem twoich [budżetów wydajnościowych](/blog/performance-budgets-frontend-engineers-guide) — przejście, które wygląda maślanie na twoim laptopie z serii M, może gubić klatki na średniopółkowym Androidzie.

**Nazwane elementy są wyciągane z flow podczas przechwytywania.** `view-transition-name` promuje element do jego własnej warstwy migawki. Jeśli żyje wewnątrz kontenera z `overflow: hidden`, `clip-path` lub kontekstem stackingu, na którym polegałeś, obraz może uciec ze swojego przycięcia w trakcie animacji. Testuj faktyczne przejście, nie tylko stany końcowe.

**Cross-document i 4-sekundowy timeout.** Dla przejść MPA, jeśli strona przychodząca nie osiągnie stanu renderowalnego w ciągu **4 sekund** od rozpoczęcia nawigacji, przeglądarka porzuca przejście i po prostu pokazuje stronę. Wolna odpowiedź serwera lub zasób blokujący render po cichu zabija twoją animację. Przejścia cross-document są tak płynne, jak twój TTFB i krytyczna ścieżka renderowania — kolejny powód, żeby trzymać strony lekkie.

**Asynchroniczne callbacki zamrażają starą stronę.** W przejściach same-document, jeśli robisz `await` na wolnym fetchu wewnątrz callbacku aktualizacji, użytkownik gapi się na zamrożoną migawkę „przed" przez cały ten czas. Pobierz dane *najpierw*, a potem uruchom przejście z danymi już w garści. Nie opakowuj żądania sieciowego w `startViewTransition`.

**Pozycja scrolla i `position: fixed`.** Elementy fixed i sticky są przechwytywane względem migawki, nie viewportu, więc stały nagłówek może wyglądać, jakby dryfował podczas przejścia, chyba że nadasz mu własny `view-transition-name` i pozwolisz traktować go jak stabilny shared element. To zaskakuje każdego za pierwszym razem.

**Przerywanie przejść.** Użytkownicy klikają szybko. Jeśli nowe przejście startuje, gdy jedno wciąż trwa, przeglądarka pomija to w locie. Zabezpiecz stan w JS, aby na wpół zaaplikowana aktualizacja DOM z pominiętego przejścia nie zostawiła UI w niespójnym stanie — rozwiązuj na `finished`, nie na `updateCallbackDone`.

## Wsparcie przeglądarek i fallbacki w 2026

Oto uczciwa macierz na moment pisania:

- **Same-document** view transitions są **Baseline Newly available** (Chrome, Edge, Safari oraz Firefox od rolloutu w 2025). Bezpieczne do szerokiego użycia.
- **Cross-document** (MPA) transitions dowożą w **Chromium 126+ i Safari 18.2+**. **Firefox wciąż blokuje regułę at-rule `@view-transition`** — parsuje ją, ale nie animuje, więc ci użytkownicy dostają natychmiastową, nieanimowaną nawigację.

Ten ostatni punkt to piękna część: **fallbackiem jest nierobienie niczego**. Przeglądarka, która nie wspiera `@view-transition`, ignoruje regułę i nawiguje normalnie. Przeglądarka, której brakuje `document.startViewTransition`, trafia w twoją gałąź wykrywania funkcji i aplikuje aktualizację DOM natychmiast. Nikt nie widzi zepsutej strony; widzą po prostu zwykłe cięcie zamiast przenikania. To progressive enhancement w najczystszej postaci — dodajesz szlif, nie zależność, więc nigdy nie potrzebujesz polyfilla.

## Kiedy tego NIE używać

Jestem entuzjastą, ale to nie młotek na każdy gwóźdź:

- **Złożone animacje oparte na fizyce i sterowane gestami** — drag-to-dismiss dla arkuszy, interpolacja sprężynowa, skoordynowana choreografia wielu elementów z przerwaniem w locie. To wciąż teren Framer Motion / GSAP. View Transitions animują *między dwoma stanami*; to nie jest uniwersalny silnik animacji.
- **Ciągła lub zapętlona animacja.** Loadery, otoczkowy ruch, cokolwiek niepowiązanego ze zmianą stanu. Złe narzędzie.
- **Mikrointerakcje wewnątrz komponentu** — naciśnięcie przycisku, przełącznik. Zwykłe przejście CSS jest prostsze i nie ponosi narzutu migawek.
- **Gdy musisz animować żywy, interaktywny DOM** podczas przejścia. Migawki to statyczne obrazy; nie wpiszesz w nie tekstu.

Słodki punkt to dokładnie to, co zabiło biblioteki: **zmiany tras i widoków** — strona-do-strony, lista-do-szczegółów, przełączanie zakładek, rozwijanie/zwijanie dużych regionów. Do tego natywne API jest mniejsze, szybsze i bardziej niezawodne niż cokolwiek, co dowieziesz samodzielnie.

Wdrożyłem to już na trzech produkcyjnych projektach. Bundle stał się lżejszy, animacje płynniejsze, a kod krótszy. W 2026 roku sięganie po ciężką zależność do animacji tylko po to, żeby przenikać między stronami, to nawyk warty porzucenia.
