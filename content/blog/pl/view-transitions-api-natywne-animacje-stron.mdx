W zeszłym miesiącu usunąłem z projektu klienta 14 kB kodu animacji (po gzipie), a przejścia między stronami stały się *lepsze*. Żadnego `AnimatePresence`, żadnych hacków na layout-shift, żadnego `motion.div` opakowującego każdy widok. Tylko dwie reguły CSS i kilka deklaracji `view-transition-name`.

To jest właśnie obietnica **View Transitions API** i w 2026 roku wreszcie ją spełnia. Wariant single-page nadawał się do produkcji w Chromium już od jakiegoś czasu, ale prawdziwym przełomem są **cross-document view transitions**: płynne, morfujące animacje podczas prawdziwych nawigacji między stronami w aplikacji multi-page, bez żadnego routera po stronie klienta. Chrome to dowiózł w wersji 126, Safari dołączyło w 18.2, a zespoły cicho usuwają ciężkie biblioteki animacji z bundli.

Spędziłem z tym API wystarczająco dużo czasu — łącznie z fragmentami, które testowały moją cierpliwość — żeby napisać przewodnik, którego sam bym potrzebował. Przejdźmy przez to, czym to naprawdę jest, przez obie powierzchnie API, jak dostosować animacje, jak to wygląda w praktyce z frameworkami oraz przez pułapki, które cię zaskoczą, jeśli nikt cię nie ostrzeże.

## Co View Transitions API właściwie robi

Oto model mentalny, który mi wyjaśnił ideę. Kiedy uruchomisz przejście, przeglądarka wykonuje cztery kroki:

1. **Przechwytuje** bieżący stan wizualny strony (lub konkretnych elementów) jako snapshot — zasadniczo zrzut ekranu.
2. **Pozwala ci zmienić DOM** do nowego stanu.
3. **Przechwytuje** nowy stan jako drugi snapshot.
4. **Animuje** przejście między starymi a nowymi snapshotami, a następnie zwraca kontrolę żywemu DOM-owi.

Genialna jest czwarta część. Podczas animacji nie animujesz živych węzłów DOM — przenikasz i transformujesz statyczne obrazy wygenerowane przez przeglądarkę. Dlatego jest to tanie, dlatego nie powoduje zmian w layoutzie i dlatego domyślny efekt już wygląda porządnie. Całe działanie zachodzi na poziomie kompozytor przeglądarki.

Są dwa sposoby, żeby to uruchomić: **same-document API** (wywołujesz funkcję JavaScript po zmianie DOM) oraz **cross-document API** (przeglądarka uruchamia to automatycznie podczas nawigacji w obrębie tego samego origina). Ta sama idea, dwa punkty wejścia.

## Przejścia same-document: API JavaScript

Dla SPA, aktualizacji renderowanych po stronie klienta czy dowolnego momentu typu „DOM się właśnie zmienił" wywołujesz `document.startViewTransition()`:

```js
// Klasyczna forma: przekazujesz callback, który zmienia DOM.
// Przeglądarka robi migawkę "przed", uruchamia twój callback, robi migawkę "po",
// a następnie animuje przejście między nimi.
function updateView(newContent) {
  // Wykrywanie funkcji — zawsze. Starsze przeglądarki po prostu dostają natychmiastową aktualizację.
  if (!document.startViewTransition) {
    applyDOMUpdate(newContent);
    return;
  }

  document.startViewTransition(() => {
    applyDOMUpdate(newContent); // synchroniczna mutacja DOM idzie tutaj
  });
}
```

To całe podejście „happy path". Bez żadnej dodatkowej reguły CSS otrzymujesz już **domyślne przenikanie** całej strony, bo przeglądarka niejawnie przypisuje `view-transition-name: root` elementowi dokumentu.

`startViewTransition()` zwraca **obiekt `ViewTransition`** z trzema promise'ami i możliwością przerwania. Są ważniejsze, niż sugeruje dokumentacja:

```js
const transition = document.startViewTransition(() => applyDOMUpdate(newContent));

// Rozwiązuje się, gdy callback DOM się zakończył i snapshoty są gotowe —
// moment, aby dodać własne animacje przez Web Animations API.
transition.ready.then(() => {
  // np. animuj pseudo-element ręcznie
});

// Rozwiązuje się, gdy twój callback aktualizacji się wykonał (DOM jest w nowym stanie),
// ale animacja może wciąż trwać.
transition.updateCallbackDone.then(() => {});

// Rozwiązuje się, gdy animacja jest całkowicie zakończona i widoczny jest żywy DOM.
await transition.finished;

// Przerwij animację, ale zachowaj aktualizację DOM. Przydatne przy
// prefers-reduced-motion lub gdy użytkownik nawiguje ponownie w trakcie przejścia.
transition.skipTransition();
```

Jeśli twoja aktualizacja jest asynchroniczna — na przykład pobierasz dane przed renderowaniem — przekaż asynchroniczny callback. Przeglądarka trzyma snapshot „przed" zamrożony, gdy czekasz na `await`, co jest zarazem potężne i pułapką, do której jeszcze wrócę.

### Forma z opcjami i typy przejść

Nowoczesne Chromium akceptuje też **obiekt opcji** zamiast samego callbacku, co odblokowuje `types`:

```js
document.startViewTransition({
  update: () => applyDOMUpdate(newContent),
  // Otaguj to konkretne przejście, aby CSS mógł je stylować inaczej.
  types: ['slide-forward'],
});
```

Typy pozwalają jednej stronie mieć kilka odrębnych animacji — nawigacja do przodu vs. do tyłu, rozwijanie vs. zwijanie — bez żonglowania klasami na elemencie root. Celujesz w nie pseudoklasą `:active-view-transition-type()`, której użyję poniżej.

## view-transition-name i morfowanie shared elements

Przenikanie jest OK. Magią są **shared element transitions** — miniatura, która płynnie rozrasta się w duży obraz, karta przechodzaca w panel szczegółów. Uzyskujesz to, nadając temu samemu elementowi w obu stanach identyczną wartość `view-transition-name`:

```css
/* Na stronie z listą */
.product-card__image {
  view-transition-name: product-hero;
}

/* Na stronie ze szczegółami — ta sama nazwa, inny element */
.product-detail__image {
  view-transition-name: product-hero;
}
```

Gdy nazwa istnieje zarówno w starym, jak i nowym snapshotu, przeglądarka przestaje go przenikać i zamiast tego **animuje pozycję, rozmiar i kształt** ze starego do nowego. Interpoluje `width`, `height`, `transform`, a nawet `border-radius`. To efekt, do którego wcześniej trzeba było sięgać po biblioteki animacji.

**Reguła, która cię zaskoczy:** każdy `view-transition-name` musi być *unikalny w obrębie dokumentu w momencie przechwytywania*. Dwa elementy z `view-transition-name: product-hero` widoczne jednocześnie powodują błąd i całe przejście się pomija. Na liście 20 produktów nie możesz wszystkim statycznie przypisać `product-hero` — generujesz unikalne nazwy (`product-hero-42`) i przypisujesz współdzieloną nazwę tylko *tej jednej* karcie, którą użytkownik kliknął, zwykle przez event handler w JS tuż przed uruchomieniem przejścia.

### Drzewo pseudo-elementów

Kiedy przejście się uruchamia, przeglądarka buduje hierarchię pseudo-elementów `::view-transition` nałożoną na całą zawartość. Zrozumienie jej oddziela „skopiowałem kod" od „rozumiem, jak zbudować dowolną animację":

```
::view-transition                          (nakładka root, pokrywa viewport)
└─ ::view-transition-group(name)           (kontener z pozycją + rozmiarem per nazwa)
   └─ ::view-transition-image-pair(name)   (kontekst izolacji dla blendu)
      ├─ ::view-transition-old(name)       (migawka starego stanu)
      └─ ::view-transition-new(name)       (migawka nowego stanu)
```

`-group` obsługuje morfowanie rozmiaru/pozycji. `-old` i `-new` to rzeczywiste snapshoty przenikane wewnątrz `-image-pair`. Aby dostosować animację, celujesz w te pseudo-elementy zwykłymi `@keyframes` w CSS:

```css
/* Wsuń całą zawartość strony z prawej zamiast przenikania */
@keyframes slide-from-right {
  from { transform: translateX(30px); opacity: 0; }
}
@keyframes slide-to-left {
  to { transform: translateX(-30px); opacity: 0; }
}

::view-transition-old(root) {
  animation: 250ms ease both slide-to-left;
}
::view-transition-new(root) {
  animation: 250ms ease both slide-from-right;
}
```

Ponieważ to zwykła animacja CSS na pseudo-elementach, obowiązuje wszystko, co już wiesz — `animation-delay`, `cubic-bezier`, `steps()`, cały arsenał. Chcesz wsunięcia tylko przy nawigacji do przodu? Ogranicz to typem:

```css
/* Tylko gdy przejście zostało otagowane types: ['slide-forward'] */
html:active-view-transition-type(slide-forward)::view-transition-old(root) {
  animation: 250ms ease both slide-to-left;
}
```

## Przejścia cross-document: zero JavaScriptu dla MPA

To jest funkcja, którą czekaliśmy w 2026 roku. Dla **aplikacji multi-page** — stron renderowanych na serwerze, stron statycznych, aplikacji w Astro, Rails czy zwykłym HTML — możesz uzyskać dokładnie te same przejścia podczas rzeczywistych nawigacji, *bez żadnego JavaScriptu*. Zarówno strona, którą opuszczasz, jak i ta, na której lądujesz, wymagają tylko jednej reguły CSS at-rule:

```css
/* Umieść to w globalnym arkuszu stylów ładowanym na każdej stronie */
@view-transition {
  navigation: auto;
}
```

I tyle. Dwie strony z tego samego origina, obie zawierające tę regułę, a każde kliknięcie linku czy naciśnięcie przycisku Wstecz przenika między nimi. **Bez żadnych wywołań funkcji** — przeglądarka steruje tym automatycznie. Przechwytuje stary dokument, ładuje nowy, czeka, aż będzie renderowalny, i animuje. Shared elements morfują przez nawigację, używając tego samego mechanizmu `view-transition-name`, więc pasek nawigacji czy obraz hero obecny na obu stronach pozostaje na miejscu i płynnie przechodzi w nową pozycję.

To naprawdę znaczące osiągnięcie. Architektury multi-page — które powracają do łask z dobrych [powodów wydajnościowych](/pl/blog/ssr-ssg-isr-rsc-strategie-nextjs) — nie muszą już wybierać między „płynnością SPA" a „brakiem routera po stronie klienta". Zachowujesz prostotę MPA i zyskujesz doświadczenie SPA.

### Dostosowywanie per-nawigacja z pageswap i pagereveal

Goła reguła `@view-transition` przenika całą stronę. Aby tworzyć kierunkowe lub kontekstowe animacje między dokumentami, dodajesz obsługę dwóch zdarzeń. **`pageswap`** jest wyzwalane na stronie *wychodzącej* tuż przed przechwyceniem jej jako snapshot; **`pagereveal`** jest wyzwalane na stronie *przychodzącej* przed jej pierwszym renderem. Oba udostępniają obiekt `viewTransition` (gdy przejście faktycznie zachodzi) i `activation` z adresami URL source/destination:

```js
// Strona wychodząca — zdecyduj o kierunku i otaguj przejście
window.addEventListener('pageswap', (event) => {
  if (!event.viewTransition) return;

  const fromPath = event.activation.from?.url;
  const toPath = event.activation.entry.url;

  // Przykład: przypisz współdzieloną nazwę tylko klikniętemu elementowi lub ustaw typ
  if (isForward(fromPath, toPath)) {
    event.viewTransition.types.add('forward');
  } else {
    event.viewTransition.types.add('back');
  }
});

// Strona przychodząca — odzwierciedl decyzję, aby stare/nowe migawki się zgadzały
window.addEventListener('pagereveal', (event) => {
  if (!event.viewTransition) return;
  // Ustaw tutaj view-transition-name na elemencie docelowym, aby morph trafił
});
```

Typowy wzorzec: na stronie indeksu bloga w `pageswap` przypisujesz `view-transition-name: article-hero` konkretnej karcie, którą użytkownik kliknął, a w `pagereveal` na stronie artykułu przypisujesz tę samą nazwę jego obrazowi nagłówkowemu. Przeglądarka je dopasowuje i morfuje. Wyczyść nazwy potem (w `finished`), aby nie kolidowały podczas kolejnej nawigacji.

## Historia z frameworkami: React, Next.js i Astro

Nie potrzebujesz frameworka do żadnego z powyższych — o to właśnie chodzi. Ale ergonomia ręcznego nazewnictwa elementów i obsługi `pageswap` w SPA jest uciążliwa, więc frameworki opakowują te procesy.

### React i Next.js

React oferuje eksperymentalny komponent **`<ViewTransition>`** (React Canary / 19.x experimental w chwili pisania — jeszcze niestabilny, traktuj go odpowiednio). Deklaratywnie zaznacza poddrzewo, które powinno się animować, gdy się pojawia, znika lub aktualizuje wewnątrz `startTransition`:

```jsx
import { unstable_ViewTransition as ViewTransition } from 'react';

function Page({ children }) {
  return (
    <ViewTransition>
      {children}
    </ViewTransition>
  );
}
```

Ponieważ React steruje aktualizacjami DOM za pośrednictwem własnego schedulera, podłącza komponent do `document.startViewTransition()` za ciebie i animuje tylko, gdy aktualizacja pochodzi z tranzycji — co elegancko integruje się z modelem współbieżnego renderowania opisanym w [przewodniku po React concurrent mode](/pl/blog/react-concurrent-mode-praktyczny-przewodnik). Typ animacji wybierasz za pomocą **`addTransitionType`**:

```jsx
import { startTransition, addTransitionType } from 'react';

function navigate(url) {
  startTransition(() => {
    addTransitionType('nav-forward'); // mapuje się na typ możliwy do wskazania w CSS
    router.push(url);
  });
}
```

**Next.js** udostępnia to za flagą konfiguracyjną. Włącz ją w konfiguracji, opakuj swoją zawartość, a nawigacje między trasami automatycznie się animują:

```js
// next.config.js
module.exports = {
  experimental: {
    viewTransition: true,
  },
};
```

Od Next.js 16.2 komponent `<Link>` przyjmuje prop `transitionTypes` — tablicę bezpośrednio przesyłaną do `addTransitionType` — dzięki czemu sterować możesz kierunkowymi animacjami wprost z linku. Jeśli już aktualizujesz projekt, integruje się to naturalnie z innymi zmianami opisanymi w [przewodniku migracji do Next.js 16](/pl/blog/nextjs-16-przewodnik-migracji).

### Astro

Astro było tu wcześnie i teraz oferuje obie opcje. Komponent `<ClientRouter />` przechwytuje nawigacje, zamienia twoją MPA w quasi-SPA i dodaje funkcje, których natywnemu API wciąż brakuje (utrzymywanie stanu między nawigacjami, bardziej granularne dyrektywy). Shared elements oznaczasz dyrektywą `transition:name`, a animacje wybierasz za pośrednictwem `transition:animate`:

```astro
---
import { ClientRouter } from 'astro:transitions';
---
<head>
  <ClientRouter />
</head>
<body>
  <img src={hero} transition:name="cover" transition:animate="fade" />
</body>
```

Interesujące zjawisko 2026 roku to to, że wiele projektów w Astro **przechodzi z `ClientRouter` na natywną regułę `@view-transition`**. Tracisz kilka frameworkowych wygód, ale eliminujesz cały JavaScript routera i opierasz się na webowym standardzie. Dla witryn z treścią, gdzie przejście jest czystym upiększeniem, to często dobry kompromis.

## Dostępność: uszanuj prefers-reduced-motion

To jest obowiązkowe, nie opcjonalne. Zaburzenia przedsionkowe wywoływane ruchem są realne, a animacja przejścia na całą stronę może kogoś fizycznie przyprawić o mdłości. Dobrą wiadomością jest, że wyłączasz *każdą* animację view transition za pomocą jednego media query — to [fundamentalne podejście do dostępności, które każdy frontend powinien wdrożyć w 2026](/pl/blog/dostepnosc-web-2026-eaa-ada-wcag):

```css
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

Ustawienie `animation: none` *nie* psuje przejścia — DOM wciąż się aktualizuje, nowy stan wciąż się wyświetla. Eliminujesz jedynie ruch, czyli dokładnie to, o co użytkownik prosił. Dla przejść same-document możesz też rozgałęzić się w JS i wywołać `skipTransition()`, ale podejście CSS pokrywa cross-document bezpłatnie, więc zaczynam od niego.

Jeszcze jedno: pseudo-elementy mają `aria-hidden` i są inertem podczas animacji, ale żywy DOM widzą czytniki ekranu. To poprawne zachowanie, ale oznacza, że zbyt długie przejścia opóźniają moment, w którym technologia asystująca dociera do rzeczywistej treści. Utrzymuj czasy trwania krótkie (200–300 ms) także z tego powodu.

## Wydajność i pułapki, o których nikt nie wspomina

API jest z założenia szybkie — animujesz snapshoty optymalizowane dla kompozytora — ale nie jest bezkosztowe, i ma ostre krawędzie.

**Koszt snapshotu skaluje się z obszarem do renderowania.** Każdy nazwany element zostaje teksturą. Nazwanie dziesiątek elementów lub czegoś ogromnego oznacza, że przeglądarka alokuje duże snapshoty, a krok przechwytywania może się zacinać na obciążonej klatce. Nazwij tylko to, co musi się morfować. To kwestia godna uwagi względem twoich [budżetów wydajności](/pl/blog/budzety-wydajnosci-frontend) — przejście wyglądające płynnie na MacBooku Pro może tracić klatki na średniopółkowym Androidzie.

**Nazwane elementy są wyciągane z layout flow podczas przechwytywania.** `view-transition-name` promuje element do jego własnej warstwy snapshotu. Jeśli żyje wewnątrz kontenera z `overflow: hidden`, `clip-path` lub kontekstem stackingu, na którym polegałeś, obraz może wydostać się z przycięcia podczas animacji. Testuj faktyczne przejście, nie tylko stany końcowe.

**Cross-document i 4-sekundowy timeout.** Dla przejść MPA, jeśli strona przychodząca nie osiągnie stanu renderowalnego w ciągu **4 sekund** od rozpoczęcia nawigacji, przeglądarka porzuca przejście i zwyczajnie wyświetla stronę. Powolna odpowiedź serwera lub blokujący render zasób dyskretnie zabija twoją animację. Przejścia cross-document są tak płynne, jak twój TTFB i krytyczna ścieżka renderowania — kolejny powód, aby utrzymywać strony lekkie.

**Asynchroniczne callbacki zamrażają stary dokument.** W przejściach same-document, jeśli robisz `await` na wolnym pobieraniu wewnątrz callbacku aktualizacji, użytkownik patrzy na zamrożony snapshot „przed" przez całkowity czas oczekiwania. Pobierz dane *najpierw*, a potem uruchom przejście z danymi już dostępnymi. Nie zawijaj żądań sieciowych w `startViewTransition`.

**Pozycja scrolla i `position: fixed`.** Elementy fixed i sticky są przechwytywane względem snapshotu, nie viewportu, więc stały nagłówek może wyglądać, jakby dryfował podczas przejścia, chyba że przypisujesz mu własny `view-transition-name` i traktujesz go jako stabilny shared element. To zaskakuje każdego za pierwszym razem.

**Przerywanie przejść.** Użytkownicy klikają szybko. Jeśli nowe przejście startuje, gdy jedno wciąż trwa, przeglądarka je pomija. Zabezpiecz stan w JS, aby częściowo zastosowana aktualizacja DOM z pominiętego przejścia nie pozostawiła UI w niespójnym stanie — rozwiązuj problemy na `finished`, nie na `updateCallbackDone`.

## Wsparcie przeglądarek i fallbacki w 2026

Oto uczciwy obraz na moment pisania:

- **Same-document** view transitions są **Baseline Newly available** (Chrome, Edge, Safari i Firefox od rolloutu w 2025). Bezpieczne do powszechnego użycia.
- **Cross-document** (MPA) transitions dostępne w **Chromium 126+ i Safari 18.2+**. **Firefox wciąż nie wspiera reguły at-rule `@view-transition`** — parsuje ją, ale nie animuje, więc ci użytkownicy otrzymują natychmiastową, nieanimowaną nawigację.

To ostatni punkt to piękna strona: **fallbackiem jest brak działania**. Przeglądarka, która nie wspiera `@view-transition`, ignoruje regułę i nawiguje normalnie. Przeglądarka bez `document.startViewTransition` trafia w twój kod detekcji funkcji i aplikuje aktualizację DOM natychmiast. Nikt nie widzi broken strony; widzi po prostu cięcie zamiast przenikania. To progressive enhancement w czystej postaci — dodajesz ulepszenie, nie zależność, więc nigdy nie potrzebujesz polyfilla.

## Kiedy tego NIE używać

Jestem entuzjastą, ale to nie rozwiązanie do wszystkiego:

- **Złożone animacje oparte na fizyce i sterowane gestami** — drag-to-dismiss dla arkuszy, interpolacja sprężynowa, skoordynowana choreografia wielu elementów z przerwaniem w locie. To wciąż teren Framer Motion i GSAP. View Transitions animują *między dwoma stanami*; to nie jest uniwersalny silnik animacji.
- **Ciągła lub zapętlona animacja.** Loadery, ruch rotacyjny, cokolwiek niezwiązanego ze zmianą stanu. Złe narzędzie.
- **Mikrointerakcje wewnątrz komponentu** — naciśnięcie przycisku, przełącznik. Zwykłe przejście CSS jest prostsze i nie ponosi narzutu snapshotów.
- **Gdy musisz animować żywy, interaktywny DOM** podczas przejścia. Snapshoty to statyczne obrazy; nie wpiszesz w nie tekstu czy interaktywnych elementów.

Idealne zastosowanie to dokładnie to, co zabiło biblioteki: **zmiany tras i widoków** — strona-do-strony, lista-do-szczegółów, przełączanie zakładek, rozwijanie/zwijanie dużych sekcji. Native API jest mniejsze, szybsze i bardziej niezawodne niż wszystko, co samodzielnie budujesz.

Wdrożyłem to na trzech projektach produkcyjnych. Bundle stał się lżejszy, animacje płynniejsze, a kod krótszy. W 2026 roku sięganie po ciężką bibliotekę animacji tylko po to, aby przenikać między stronami, to nawyk godny porzucenia.
