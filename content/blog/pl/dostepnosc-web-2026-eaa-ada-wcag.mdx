Raport WebAIM Million wykazał, że 96,3% miliona najpopularniejszych stron nie przechodzi podstawowych testów dostępności. Przez lata była to wstydliwa statystyka bez większych konsekwencji. Już nie.

European Accessibility Act zaczął obowiązywać. Termin dla ADA Title II wypada w kwietniu 2026. Liczba pozwów wzrosła o 37% rok do roku. Jeśli jesteś frontend developerem w 2026 roku, **dostępność nie jest już mile widzianą umiejętnością — to wymóg prawny**.

Oto co naprawdę musisz wiedzieć i wdrożyć.

## Kalendarium dostępności na 2026 rok

Przejdźmy przez prawniczy żargon prosto do dat, które mają znaczenie:

| Regulacja             | Termin                       | Kogo dotyczy                         | Standard          |
| --------------------- | ---------------------------- | ------------------------------------ | ----------------- |
| **EAA (UE)**          | 28 czerwca 2025 (obowiązuje) | Każda firma obsługująca klientów z UE | WCAG 2.1 AA       |
| **ADA Title II (USA)** | 24 kwietnia 2026            | Administracja stanowa/lokalna (50k+ mieszkańców) | WCAG 2.1 AA |
| **ADA Title II (USA)** | 26 kwietnia 2027            | Administracja stanowa/lokalna (&lt;50k mieszkańców) | WCAG 2.1 AA |
| **WCAG 3.0**          | ~2027-2028                   | Wszyscy (docelowo)                   | Nowy model oceny  |

**Podsumowując:** jeśli Twoja strona obsługuje użytkowników w UE lub amerykański sektor publiczny, musisz być zgodny z WCAG 2.1 AA już teraz — albo grożą Ci kary sięgające 500 000 € i kroki prawne.

## WCAG 2.2 AA: standard, który musisz spełnić

WCAG porządkuje dostępność wokół czterech zasad, znanych jako **POUR**:

- **Perceivable (postrzegalność)** — użytkownicy mogą postrzegać treść (zobaczyć ją, usłyszeć itd.)
- **Operable (funkcjonalność)** — użytkownicy mogą nawigować i korzystać z interfejsu
- **Understandable (zrozumiałość)** — użytkownicy rozumieją treść i sposób korzystania z niej
- **Robust (kompatybilność)** — treść działa z obecnymi i przyszłymi technologiami wspomagającymi

Aby uzyskać zgodność na poziomie AA, musisz spełnić 50 kryteriów sukcesu w ramach tych zasad. Brzmi przytłaczająco, ale większość sprowadza się do garści podstawowych wzorców.

## Podstawowe wzorce dostępności

### 1. Semantic HTML: Twój fundament

Najbardziej znacząca rzecz, jaką możesz zrobić dla dostępności, nic nie kosztuje i nie wymaga żadnej specjalnej biblioteki: **używaj semantycznego HTML**.

```html
<!-- Źle: zupa z divów -->
<div class="header">
  <div class="nav">
    <div class="nav-item" onclick="goHome()">Home</div>
    <div class="nav-item" onclick="goAbout()">About</div>
  </div>
</div>
<div class="content">
  <div class="title">Welcome</div>
  <div class="text">Some content here...</div>
</div>

<!-- Dobrze: semantyczny HTML -->
<header>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
    </ul>
  </nav>
</header>
<main>
  <h1>Welcome</h1>
  <p>Some content here...</p>
</main>
```

Dlaczego to ma znaczenie? Użytkownicy czytników ekranu mogą przeskakiwać bezpośrednio do punktów orientacyjnych (`<header>`, `<nav>`, `<main>`, `<footer>`). Prawie 70% użytkowników czytników ekranu nawiguje po nagłówkach. Natywne elementy `<a>` i `<button>` są domyślnie dostępne z klawiatury.

**Zasada:** jeśli istnieje semantyczny element HTML do danego celu, użyj go. Po `<div>` i `<span>` sięgaj tylko wtedy, gdy naprawdę nie ma semantycznej alternatywy.

### 2. Nawigacja klawiaturą

Wszystko na Twojej stronie musi dać się obsłużyć bez myszki. Oznacza to:

```typescript
// Źle: interakcja tylko przez kliknięcie
<div onClick={handleAction} className="button">
  Do Something
</div>

// Dobrze: dostępne z klawiatury
<button onClick={handleAction} type="button">
  Do Something
</button>

// Dla komponentów niestandardowych obsłuż zdarzenia klawiatury
function CustomDropdown({ options, onSelect }) {
  const [isOpen, setIsOpen] = useState(false)
  const [focusedIndex, setFocusedIndex] = useState(0)

  const handleKeyDown = (e: React.KeyboardEvent) => {
    switch (e.key) {
      case 'Enter':
      case ' ':
        if (isOpen) {
          onSelect(options[focusedIndex])
        }
        setIsOpen(!isOpen)
        break
      case 'ArrowDown':
        e.preventDefault()
        setFocusedIndex(i => Math.min(i + 1, options.length - 1))
        break
      case 'ArrowUp':
        e.preventDefault()
        setFocusedIndex(i => Math.max(i - 1, 0))
        break
      case 'Escape':
        setIsOpen(false)
        break
    }
  }

  return (
    <div
      role="combobox"
      aria-expanded={isOpen}
      aria-haspopup="listbox"
      tabIndex={0}
      onKeyDown={handleKeyDown}
    >
      {/* implementacja dropdowna */}
    </div>
  )
}
```

**Podstawy zarządzania fokusem:**

- Każdy interaktywny element potrzebuje widocznego wskaźnika fokusu
- Kolejność fokusu powinna odpowiadać kolejności wizualnej (unikaj wartości `tabindex` > 0)
- Okna modalne powinny zatrzymywać fokus wewnątrz siebie
- Po zamknięciu modala zwróć fokus do elementu, który go otworzył

```css
/* Nigdy tak nie rób */
*:focus {
  outline: none;
}

/* Rób tak zamiast tego */
:focus-visible {
  outline: 2px solid #005fcc;
  outline-offset: 2px;
}
```

### 3. Kompatybilność z czytnikami ekranu: ARIA zrobione dobrze

ARIA (Accessible Rich Internet Applications) pomaga, gdy semantyka HTML nie wystarcza. Ale oto kluczowa zasada: **brak ARIA jest lepszy niż złe ARIA**.

```typescript
// Pierwsza zasada ARIA: nie używaj ARIA, jeśli go nie potrzebujesz
// Źle - zbędne ARIA
<button role="button" aria-label="Submit button">Submit</button>

// Dobrze - pozwól HTML wykonać robotę
<button type="submit">Submit</button>

// Kiedy JUŻ potrzebujesz ARIA - dynamiczna treść
function LiveRegion() {
  return (
    <div
      role="status"
      aria-live="polite"
      aria-atomic="true"
    >
      {/* Czytniki ekranu ogłaszają tutaj zmiany */}
      {statusMessage}
    </div>
  )
}

// Powiązania etykiet
<label id="email-label">Email Address</label>
<input
  type="email"
  aria-labelledby="email-label"
  aria-describedby="email-hint email-error"
/>
<span id="email-hint">We'll never share your email</span>
<span id="email-error" role="alert">
  {emailError}
</span>
```

**Wzorce ARIA, których faktycznie użyjesz:**

| Wzorzec              | Zastosowanie                                  |
| -------------------- | --------------------------------------------- |
| `aria-label`         | Gdy widoczny tekst nie wystarcza (przyciski z ikonami) |
| `aria-labelledby`    | Wskazanie innego elementu jako etykiety        |
| `aria-describedby`   | Dodatkowy tekst opisowy                        |
| `aria-expanded`      | Stan zwijanej treści                           |
| `aria-live`          | Dynamiczne aktualizacje treści                 |
| `aria-hidden="true"` | Ukrycie elementów dekoracyjnych przed czytnikami ekranu |

### 4. Kolor i kontrast

WCAG AA wymaga:

- **4,5:1** kontrastu dla zwykłego tekstu
- **3:1** kontrastu dla dużego tekstu (18px+ lub 14px+ pogrubiony)
- **3:1** kontrastu dla komponentów UI i grafik

```css
/* Sprawdź te kombinacje */
:root {
  /* Dobrze: kontrast 7,5:1 */
  --text-primary: #1a1a1a;
  --bg-primary: #ffffff;

  /* Dobrze: kontrast 4,6:1 */
  --text-secondary: #595959;
  --bg-secondary: #ffffff;

  /* Źle: kontrast 2,8:1 - nie przechodzi AA */
  --text-muted: #999999;
  --bg-muted: #ffffff;
}
```

**Poza kontrastem:** sam kolor nigdy nie powinien przekazywać informacji.

```typescript
// Źle: tylko kolor wskazuje błąd
<input style={{ borderColor: hasError ? 'red' : 'gray' }} />

// Dobrze: kolor + ikona + tekst
<div>
  <input
    aria-invalid={hasError}
    aria-describedby={hasError ? 'error-message' : undefined}
    style={{ borderColor: hasError ? 'red' : 'gray' }}
  />
  {hasError && (
    <span id="error-message" role="alert">
      <ErrorIcon aria-hidden="true" />
      Please enter a valid email address
    </span>
  )}
</div>
```

## Testowanie dostępności

Narzędzia automatyczne wychwytują około 30-40% problemów z dostępnością. Reszta wymaga testów manualnych.

### Testowanie automatyczne

Dodaj to do swojego pipeline'u CI:

```typescript
// Użycie @axe-core/react podczas developmentu
import React from 'react'

if (process.env.NODE_ENV === 'development') {
  import('@axe-core/react').then(({ default: axe }) => {
    axe(React, ReactDOM, 1000)
  })
}

// W plikach testowych z jest-axe
import { axe, toHaveNoViolations } from 'jest-axe'

expect.extend(toHaveNoViolations)

test('page has no accessibility violations', async () => {
  const { container } = render(<MyComponent />)
  const results = await axe(container)
  expect(results).toHaveNoViolations()
})
```

**Niezbędne narzędzia:**

- **axe DevTools** — rozszerzenie przeglądarki do audytów stron
- **Lighthouse** — wbudowane w Chrome, obejmuje podstawy dostępności
- **jest-axe** — testy automatyczne w CI
- **eslint-plugin-jsx-a11y** — wychwytywanie problemów podczas developmentu

### Checklista testów manualnych

Narzędzia automatyczne nie wychwycą wszystkiego. Przejdź przez to ręcznie:

1. **Nawigacja klawiaturą**

   - Przejdź Tabem przez całą stronę
   - Czy da się dotrzeć do każdego interaktywnego elementu?
   - Czy przez cały czas widać, gdzie jest fokus?
   - Czy da się obsłużyć wszystkie kontrolki klawiszami Enter/Spacja?

2. **Testowanie z czytnikiem ekranu**

   - Testuj z VoiceOver (Mac), NVDA (Windows) lub TalkBack (Android)
   - Czy treść ma sens, gdy jest czytana liniowo?
   - Czy wszystkie obrazy są odpowiednio opisane?
   - Czy błędy formularzy są ogłaszane?

3. **Testowanie powiększenia**

   - Powiększ do 200% — czy układ nadal działa?
   - Czy jakaś treść jest ucięta lub nachodzi na siebie?

4. **Preferencje ruchu**
   - Czy `prefers-reduced-motion` wyłącza animacje?

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

## WCAG 3.0: co nadchodzi

WCAG 3.0 nie zostanie sfinalizowany przed 2027-2028 rokiem, ale zrozumienie kierunku zmian pomaga się przygotować:

**Kluczowe zmiany:**

- **Testowanie oparte na wynikach** zamiast kryteriów sukcesu typu przechodzi/nie przechodzi
- **Poziomy zgodności Bronze/Silver/Gold** zamiast A/AA/AAA
- **Punktacja od 0 do 4** dla bardziej szczegółowej oceny
- **Szerszy zakres** obejmujący aplikacje, PDF-y, XR, interfejsy głosowe

**Co to oznacza w praktyce:** zamiast pytać „Czy to przechodzi test współczynnika kontrastu?", WCAG 3.0 pyta „Czy użytkownik ze słabym wzrokiem faktycznie może wykonać swoje zadanie?".

**Co robić już teraz:**

1. Trzymaj się WCAG 2.2 AA — to standard wymagany prawnie
2. Myśl w kategoriach ścieżek użytkownika, nie tylko pojedynczych komponentów
3. Zacznij włączać do testów prawdziwych użytkowników z niepełnosprawnościami
4. Wbudowuj dostępność w swój design system, a nie traktuj jej po macoszemu

## Checklista dostępności

Przed uruchomieniem zweryfikuj:

### Struktura

- [ ] Strona ma jeden `<h1>`, który opisuje jej cel
- [ ] Nagłówki zachowują logiczną hierarchię (bez pomijania poziomów)
- [ ] Regiony punktów orientacyjnych są poprawnie użyte (`<header>`, `<main>`, `<nav>`, `<footer>`)
- [ ] Język jest ustawiony atrybutem `lang` na elemencie `<html>`

### Obrazy i multimedia

- [ ] Wszystkie obrazy mają odpowiedni tekst `alt`
- [ ] Obrazy dekoracyjne używają `alt=""` lub `aria-hidden="true"`
- [ ] Filmy mają napisy
- [ ] Treści audio mają transkrypcje

### Formularze

- [ ] Wszystkie pola mają powiązane etykiety
- [ ] Komunikaty o błędach są jasne i powiązane z polami
- [ ] Pola wymagane są oznaczone (nie tylko kolorem)
- [ ] Formularz da się wypełnić wyłącznie klawiaturą

### Interakcja

- [ ] Wszystkie interaktywne elementy są dostępne z klawiatury
- [ ] Fokus jest widoczny na wszystkich elementach
- [ ] Kolejność fokusu jest logiczna
- [ ] Nie istnieją pułapki klawiaturowe

### Warstwa wizualna

- [ ] Tekst ma kontrast co najmniej 4,5:1
- [ ] Informacja nie jest przekazywana samym kolorem
- [ ] Strona jest użyteczna przy powiększeniu 200%
- [ ] Animacje respektują `prefers-reduced-motion`

## Dlaczego nakładki (overlays) nie działają

Widziałeś przyciski „widżetów dostępności" obiecujące zgodność jednym kliknięciem. Nie działają, a oto dlaczego:

1. **Nie naprawią problemów strukturalnych** — żadna nakładka nie doda brakującego tekstu alt, nie naprawi hierarchii nagłówków ani nie sprawi, że niestandardowe komponenty będą dostępne z klawiatury
2. **Często psują działanie** — nakładki potrafią zakłócać czytniki ekranu, które już działają poprawnie
3. **Nie dają ochrony prawnej** — sądy nie uznają stosowania nakładek za dowód zgodności
4. **Są uznawane za problematyczne** — National Federation of the Blind wprost potępiła nakładki dostępności

**Jedyna droga do zgodności** to tworzenie dostępnego kodu od samego początku. Nie ma dróg na skróty.

## Ruszamy dalej

Dostępność stron w 2026 roku nie jest opcjonalna. Terminy są realne, kary są znaczne, a co najważniejsze — tworzenie dostępnych produktów to po prostu właściwa rzecz.

Zacznij od semantycznego HTML. Dodaj obsługę klawiatury. Testuj z prawdziwą technologią wspomagającą. Wbuduj to w swój proces od pierwszego dnia, a nie jako checkbox przed uruchomieniem.

Te 96,3% niezgodnych stron to szansa. Bądź w tych 3,7%, które robią to dobrze.

Potrzebujesz pomocy w audycie swojej obecnej strony? Zacznij od Lighthouse i axe DevTools, a potem przejdź przez powyższą checklistę manualną. Krok po kroku, problem po problemie, dojdziesz do celu.
