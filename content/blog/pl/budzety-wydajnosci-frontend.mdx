Wydajność degraduje się jeden import na raz. Nikt nie dodaje 200KB JavaScriptu w pojedynczym commicie. Dzieje się to przez wiele miesięcy — biblioteka do wykresów tu, paczka ikon tam, polyfill, którego nikt nie usunął, biblioteka CSS-in-JS, która podwoiła bundle. Zanim ktokolwiek to zauważy, strona ładuje się w 4 sekundy i nikt nie potrafi wskazać, kiedy zrobiła się wolna.

Budżety wydajności zapobiegają temu, zamieniając mgliste cele („strona powinna być szybka") w twarde liczby egzekwowane w CI. Przekroczysz budżet → build się wywala → PR zostaje naprawiony przed mergem.

Tylko 47% stron zalicza wszystkie trzy Core Web Vitals w 2026. Pozostałe 53% traci od 8% do 35% konwersji, ruchu i przychodów. Budżet wydajności to najtańsza polisa ubezpieczeniowa przed byciem w tych 53%.

## Trzy budżety, których potrzebujesz

Zacznij od trzech budżetów. Więcej przytłacza; mniej zostawia luki.

### 1. Budżet JavaScriptu: < 200KB gzip (pierwsze załadowanie)

JavaScript to najdroższy zasób. Każdy KB kosztuje czas parsowania na urządzeniu mobilnym (~1ms/KB), czas wykonania i pamięć. Opisałem to dogłębnie w moim [przewodniku po optymalizacji bundle](/pl/blog/dynamiczna-optymalizacja-bundle-ponizej-200kb).

```json
// konfiguracja size-limit w package.json
{
  "size-limit": [
    {
      "path": ".next/static/**/*.js",
      "limit": "200 kB",
      "gzip": true
    }
  ]
}
```

### 2. Largest Contentful Paint (LCP): < 2,5 sekundy

LCP mierzy moment, w którym największy widoczny element kończy się renderować. Próg Google: poniżej 2,5s jest „good".

```json
// konfiguracja budżetu lighthouse — budget.json
[
  {
    "timings": [
      { "metric": "largest-contentful-paint", "budget": 2500 },
      { "metric": "interactive", "budget": 3500 },
      { "metric": "first-contentful-paint", "budget": 1500 }
    ],
    "resourceSizes": [
      { "resourceType": "script", "budget": 200 },
      { "resourceType": "total", "budget": 500 }
    ]
  }
]
```

### 3. Interaction to Next Paint (INP): < 200ms

INP mierzy responsywność — ile czasu mija między interakcją użytkownika a kolejną aktualizacją wizualną. 43% stron nie zalicza tego progu w 2026, co czyni go najczęściej oblewanym Core Web Vital.

INP trudniej zbudżetować w CI, bo wymaga prawdziwej interakcji użytkownika. Monitoruj go za pomocą Real User Monitoring (RUM) i ustaw alerty. Jeśli Twoja aplikacja ciągle go nie zalicza, przyczyny i konkretne poprawki rozłożyłem na czynniki pierwsze w moim [przewodniku po Interaction to Next Paint](/pl/blog/inp-interaction-to-next-paint-core-web-vital).

## Egzekwowanie budżetów w CI

### size-limit (rozmiar bundle)

```bash
npm install -D size-limit @size-limit/preset-app
```

```yaml
# .github/workflows/performance.yml
name: Performance Budget

on: pull_request

jobs:
  bundle-size:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: 'npm' }
      - run: npm ci
      - run: npm run build
      - uses: andresz1/size-limit-action@v1
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
```

To dodaje komentarz do każdego PR-a pokazujący zmiany rozmiaru bundle i wywala build, jeśli budżet zostanie przekroczony.

### Lighthouse CI (Core Web Vitals)

```bash
npm install -D @lhci/cli
```

```javascript
// lighthouserc.js
module.exports = {
  ci: {
    collect: {
      url: ['http://localhost:3000/', 'http://localhost:3000/blog'],
      startServerCommand: 'npm run start',
      numberOfRuns: 3,
    },
    assert: {
      assertions: {
        'categories:performance': ['error', { minScore: 0.9 }],
        'largest-contentful-paint': ['error', { maxNumericValue: 2500 }],
        'cumulative-layout-shift': ['error', { maxNumericValue: 0.1 }],
        'total-blocking-time': ['error', { maxNumericValue: 200 }],
        'first-contentful-paint': ['error', { maxNumericValue: 1500 }],
      },
    },
    upload: {
      target: 'temporary-public-storage',
    },
  },
};
```

```yaml
  lighthouse:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: 'npm' }
      - run: npm ci && npm run build
      - run: npx @lhci/cli autorun
```

### Real User Monitoring (RUM)

CI wyłapuje regresje przed mergem. RUM wyłapuje problemy na produkcji z prawdziwymi użytkownikami:

```typescript
// lib/web-vitals.ts
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendToAnalytics(metric: { name: string; value: number }) {
  // Wyślij do swojego dostawcy analityki
  fetch('/api/analytics/vitals', {
    method: 'POST',
    body: JSON.stringify(metric),
    keepalive: true, // Zapewnia dostarczenie nawet przy zamykaniu strony
  });
}

export function reportWebVitals() {
  onLCP(sendToAnalytics);
  onINP(sendToAnalytics);
  onCLS(sendToAnalytics);
}
```

Ustaw alerty, gdy wartości p75 przekroczą Twoje budżety:
- LCP p75 > 2,5s → alert na Slacku
- INP p75 > 200ms → alert na Slacku
- CLS p75 > 0,1 → alert na Slacku

## Cykl życia budżetu

1. **Zmierz stan obecny** — uruchom Lighthouse i size-limit na swoim buildzie produkcyjnym
2. **Ustaw początkowe budżety** — 10-20% powyżej obecnych wartości (osiągalne, nie aspiracyjne)
3. **Egzekwuj w CI** — wywalaj buildy przekraczające budżety
4. **Zaostrzaj kwartalnie** — obniżaj budżety w miarę optymalizacji
5. **Poluzuj, gdy uzasadnione** — nowe funkcje z ciężkimi zależnościami mogą wymagać korekty budżetu. Udokumentuj dlaczego.

## Praktyczne wnioski

1. **Minimum trzy budżety:** rozmiar JS (< 200KB), LCP (< 2,5s), Performance Score (> 90).
2. **Egzekwuj w CI, nie na spotkaniach.** Automatyczne kontrole wyłapują 100% regresji; code review wyłapuje może 20%.
3. **Używaj size-limit do sprawdzania bundle** — jest prosty, szybki i publikuje wyniki na PR-ach.
4. **Używaj Lighthouse CI do metryk renderowania** — uruchamiaj 3+ razy na URL, żeby zredukować szum.
5. **Monitoruj INP za pomocą RUM** — nie da się go zmierzyć w CI, tylko z prawdziwymi użytkownikami.
6. **Zacznij 10-20% powyżej obecnych wartości.** Aspiracyjne budżety, które zawsze oblewają, są ignorowane.
7. **Zaostrzaj budżety kwartalnie.** Dyscyplina wydajności kumuluje się w czasie.
8. **Każda nowa zależność to decyzja budżetowa.** Sprawdź [bundlephobia.com](https://bundlephobia.com) przed `npm install`.
9. **Priorytet LCP: preload obrazu hero, użyj ISR, inline'uj krytyczny CSS.**
10. **Priorytet INP: [useTransition](/pl/blog/react-concurrent-mode-praktyczny-przewodnik) dla kosztownych renderów, wirtualizuj listy, odraczaj niekrytyczny JS.**

Budżety wydajności to różnica między „powinniśmy kiedyś zoptymalizować" a „build nie przejdzie, dopóki tego nie zrobimy". Zespoły, które egzekwują budżety, dostarczają szybciej, bo wyłapują regresje wcześnie — zanim skumulują się w rodzaj spowolnienia, którego naprawa zajmuje cały sprint.
