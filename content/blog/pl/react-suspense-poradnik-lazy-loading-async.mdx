## Czym jest Suspense

`<Suspense>` to komponent Reacta, który „wyświetla fallback, dopóki jego dzieci nie zakończą ładowania”. Innymi słowy, każdy komponent, który się zawiesza (suspend) — z powodu kodu ładowanego leniwie albo oczekujących danych — powoduje wstrzymanie renderowania tego poddrzewa i wyświetlenie UI fallbacku. Umożliwia to wzorce async renderingu bez ręcznego zarządzania stanem ładowania. Pod maską React 18+ działa w trybie concurrent, dzięki czemu może opóźnić renderowanie części drzewa komponentów.

- Fallback UI: lekki placeholder (spinner, skeleton itp.) pokazywany, gdy komponenty się zawieszają. Suspense obsługuje wyłącznie operacje, które je obsługują.
- Przykłady: code-splitting (`React.lazy`), biblioteki do zarządzania danymi (Relay, SWR, Next.js) oraz nowy hook `use()` dla Promise. Pobieranie danych wewnątrz `useEffect` lub w event handlerze Suspense'a nie wyzwoli.

```tsx
import React, { Suspense } from 'react'

function App() {
  return (
    <div>
      <h1>My App</h1>
      {/* Każdy komponent wewnątrz Suspense może się „zawiesić” */}
      <Suspense fallback={<div>Loading...</div>}>
        <SomeComponent /> {/* może być ładowany leniwie lub pobierać dane */}
      </Suspense>
    </div>
  )
}
```

Tutaj `SomeComponent` mógłby być ładowany leniwie albo oczekiwać na Promise z danymi. React automatycznie wyrenderuje fallback `<div>Loading...</div>`, dopóki `SomeComponent` nie będzie gotowy. Zwróć uwagę, że Suspense zadziała tylko wtedy, gdy zawieszony komponent korzysta z obsługiwanego źródła async (np. dynamicznego `import`, Promise przez `use()` albo biblioteki zintegrowanej z Suspense).

## Lazy loading (code splitting)

API `lazy()` Reacta razem z Suspense umożliwia code splitting na poziomie tras i komponentów. Użyj `React.lazy()`, aby dynamicznie zaimportować komponent dopiero wtedy, gdy jest potrzebny. Na przykład:

```tsx
import React, { Suspense, lazy } from 'react'

// Import leniwy (komponent ładuje się w osobnym chunku dopiero przy renderze)
const HeavyComponent = lazy(() => import('./HeavyComponent'))

function App() {
  return (
    <div>
      <h1>Main App</h1>
      {/* Suspense pokazuje fallback, gdy HeavyComponent jest pobierany */}
      <Suspense fallback={<div>Loading heavy component...</div>}>
        <HeavyComponent /> {/* Ładuje się dopiero przy pierwszym renderze */}
      </Suspense>
    </div>
  )
}
```

W tym przykładzie `HeavyComponent` nie jest częścią początkowego bundla. Załaduje się dopiero w momencie, gdy React go wyrenderuje wewnątrz `<Suspense>`. Zmniejsza to początkowy rozmiar pobierania i poprawia wydajność — a w połączeniu z [automatyczną memoizacją React Compilera](/pl/blog/react-compiler-automatyczna-memoizacja) ponowne renderowanie tych leniwie ładowanych komponentów jest optymalizowane automatycznie. Zawsze dodawaj `fallback`; bez niego komponent ładowany leniwie wyrzuci błąd. Lazy loading możesz zastosować do każdego komponentu, ale zwyczajnie dotyczy to stron lub rzadko używanych części aplikacji.

## Przykład: lazy loading z React Router

React Router bezproblemowo współpracuje z `React.lazy` i Suspense przy leniwym ładowaniu stron. Na przykład w React Router v6+:

```tsx
import { BrowserRouter, Routes, Route } from 'react-router-dom'
const Home = React.lazy(() => import('./Home'))
const About = React.lazy(() => import('./About'))

function App() {
  return (
    <BrowserRouter>
      {/* Owiń Routes w Suspense dla code splittingu na poziomie tras */}
      <Suspense fallback={<p>Loading page...</p>}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/about" element={<About />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  )
}
```

Każdy komponent trasy (`Home` i `About`) ładuje się wyłącznie, gdy użytkownik wejdzie na daną ścieżkę. Fallback `<p>Loading page...</p>` jest wyświetlany w trakcie pobierania. To podejście utrzymuje początkowy bundel JavaScript mały i minimalizuje początkowy rozmiar aplikacji.

## Data fetching z Suspense

Poza code splittingiem Suspense potrafi zarządzać także danymi asynchronicznymi. W React 18 było to eksperymentalne, ale React 19 wprowadza nowy hook `use()`, który natywnie pozwala zawiesić się na Promise. Zamiast useEffect i ręcznych flag ładowania możesz po prostu przekazać Promise do `use()` wewnątrz granicy Suspense. Na przykład:

```tsx
import React, { use, Suspense } from 'react'

async function fetchTodo(id: number) {
  const res = await fetch(`https://jsonplaceholder.typicode.com/todos/${id}`)
  return res.json()
}

// Komponent nadrzędny (tu bez async/await – tworzymy Promise)
function App() {
  const todoPromise = fetchTodo(1) // utwórz Promise jednorazowo
  return (
    <Suspense fallback={<div>Loading data...</div>}>
      <Todo titlePromise={todoPromise} />
    </Suspense>
  )
}

// Komponent kliencki z use()
function Todo({ titlePromise }: { titlePromise: Promise<{ title: string }> }) {
  const data = use(titlePromise)
  return <h1>{data.title}</h1> // Renderuje się, gdy Promise się rozwiąże
}
```

Tutaj `<Todo>` zawiesza się na `titlePromise`. Podczas rozpatrywania Promise'a React wyświetla fallback ładowania. Gdy `use()` wydobędzie dane, wyrenderowana zostaje `<h1>`. Jeśli Promise zostanie odrzucone, najbliższy Error Boundary je przechwyci (zobacz następną sekcję).

### Kluczowe punkty

Nie twórz Promise'a w ciele komponentu przy każdym renderowaniu. Nie pisz na przykład `const data = use(fetchTodo(id));` bezpośrednio w `Todo` — to tworzy nowe Promise'a przy każdym renderowaniu i powoduje nieskończone pętle. Zamiast tego stwórz Promise raz (w komponencie nadrzędnym albo wcześniej) i przekaż go dalej. Dzięki temu Promise pozostaje stabilny i unikasz zbędnych ponownych renderów.

Hook `use()` ma kilka ograniczeń:

- Możesz go wywoływać wyłącznie w ciele komponentu Reacta lub własnego hooka (nie w obsługiwaczach zdarzeń ani metodach)
- W przeciwieństwie do innych hooków `use()` może się pojawić w pętlach lub warunkach wewnątrz komponentu
- W Server Components (np. w Next.js) zwykle wolisz `async/await` zamiast `use()`, albo tworzysz Promise na serwerze i transmitujesz go do komponentu klienckiego z `'use client'`

## Przykład: use() kontra useEffect

Porównaj stare podejście z `useEffect` z `use()` (React 19):

```tsx
// Stary wzorzec (React 18):
function TodoOld() {
  const [todo, setTodo] = useState<{ title: string } | null>(null)
  useEffect(() => {
    fetch(`.../todos/1`)
      .then((res) => res.json())
      .then(setTodo)
  }, [])
  if (!todo) return <div>Loading...</div>
  return <h1>{todo.title}</h1>
}

// Nowy wzorzec (React 19+):
function TodoNew() {
  const titlePromise = fetchTodo(1) // utworzone poza lub powyżej
  const todo = use(titlePromise)
  return <h1>{todo.title}</h1> // Zawiesza się automatycznie
}
```

Podejście z `use()` wymaga mniej boilerplate'u: bez stanu i bez efektu. Komponent po prostu zawiesza się, dopóki Promise się nie rozwiąże. Zmniejsza to narzut związany z zarządzaniem „stanem ładowania”. (Jeśli jednak sam musisz obsłużyć ładowanie, możesz też użyć `useTransition` albo pokazać ręczny spinner.)

## Error Boundaries

Suspense obsługuje stany ładowania, ale nadal musisz wyłapywać błędy za pomocą Error Boundaries (odrzucone Promise'y lub wyjątki). Error Boundary to komponent, który wyłapuje błędy z poddrzewa i wyświetla UI fallbacku. Dla Suspense i `use()` odrzucony Promise traktowany jest jak błąd — przepływa w górę do najbliższego Error Boundary.

**Dobra praktyka: owiń swoje granice Suspense w Error Boundary. Na przykład:**

```tsx
import { use, Suspense } from 'react'
import { ErrorBoundary } from 'react-error-boundary'

// Owiń Suspense w ErrorBoundary
function MessageContainer({ messagePromise }: { messagePromise: Promise<string> }) {
  return (
    <ErrorBoundary fallback={<p>⚠️ Something went wrong</p>}>
      <Suspense fallback={<p>⌛ Loading message...</p>}>
        <Message messagePromise={messagePromise} />
      </Suspense>
    </ErrorBoundary>
  )
}

function Message({ messagePromise }: { messagePromise: Promise<string> }) {
  const content = use(messagePromise) // może rzucić, jeśli Promise zostanie odrzucone
  return <p>Message: {content}</p>
}
```

Jeśli messagePromise zostanie odrzucone, `<ErrorBoundary>` wyświetli swój fallback. Error Boundary owija `<Suspense>`, aby obsłużyć odrzucenia. Bez niego nieobsługiwane odrzucenie spowoduje błąd w aplikacji. Możesz też dodać obsługę błędów bezpośrednio na Promise (np. `fetch().then().catch()`), ale Error Boundary jest zazwyczaj czystszym rozwiązaniem.

## Integracja z React Router i data routers

Nowoczesne biblioteki routingowe mają wbudowane wsparcie dla Suspense. W React Router v6+ możesz używać Suspense do leniwego ładowania komponentów tras (jak pokazano wyżej). Data API React Routera mogą też współpracować z Suspense, ale to zależy od implementacji. Podstawowa zasada: owijaj leniwie ładowane komponenty tras w Suspense, aby każda ścieżka pobierała swój kod na żądanie.

Przykład:

```tsx
<Suspense fallback={<p>Loading page...</p>}>
  <Routes>
    <Route path=”/” element={<Home />} />
    <Route path=”/about” element={<About />} />
    {/* ... */}
  </Routes>
</Suspense>
```

Dzięki temu każda trasa ładuje kod dopiero, gdy użytkownik na nią wejdzie — to sprawia, że aplikacja jest szybka i efektywna.

## Integracja z Server Components w Next.js

Next.js (v13+) w pełni wykorzystuje Suspense i `use()` do streamingu danych z serwera. Typowy wzorzec: pobierasz dane w Server Component bez await, następnie przekazujesz nierozwiązany Promise do Client Component owiniętego w `<Suspense>`. Client Component (z `'use client'`) wywołuje `use(promise)`, aby zawiesić się aż do nadejścia danych:

```tsx
// app/post/page.tsx (Server Component)
import { Suspense } from 'react'
import PostList from './PostList'

export default function Page() {
  const postsPromise = fetch('https://api.example.com/posts').then((r) => r.json())
  return (
    <Suspense fallback={<div>Loading posts...</div>}>
      <PostList postsPromise={postsPromise} />
    </Suspense>
  )
}

// app/post/PostList.tsx (Client Component)
;('use client')
import { use } from 'react'

export default function PostList({ postsPromise }: { postsPromise: Promise<Post[]> }) {
  const posts = use(postsPromise) // Zawiesza się tutaj do nadejścia danych
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

Server Component inicjuje Promise bez await, a Suspense wyświetla fallback do czasu gotowości. Client Component następnie wydobywa dane hokiem `use()`. Ważne: Client Component wymaga `'use client'` na górze.

Poza ręcznym Suspense Next.js oferuje też pliki `loading.js` dla segmentów tras. W praktyce użycie `use()` zapewnia efektywne streamowanie danych z serwera do klienta.

## Częste błędy i pułapki

- Brakujący `fallback`: zawsze dodawaj UI fallbacku w `<Suspense>`. Bez niego komponent ładowany leniwie lub zawieszony na danych wyrzuci błąd. Fallback powinien być lekki — spinner albo skeleton.
- Promise tworzony w renderze: nie twórz nowego Promise'a przy każdym renderowaniu. Unikaj `use(fetchData())` wewnątrz komponentu — tworzy to nowe Promise'a przy każdym renderze i powoduje nieskończoną pętlę. Stwórz Promise poza komponentem (lub w komponencie nadrzędnym), aby był stabilny.
- Niepoprawne użycie `use()`: hook `use()` możesz wywoływać wyłącznie w ciele komponentu Reacta lub własnego hooka. Nie używaj go w obsługiwaczach zdarzeń, efektach ani poza Reactem. `use()` działa z Promise'ami i Contextami — nic więcej.
- Brak Error Boundaries: bez Error Boundary wokół Suspense każde odrzucenie Promise spowoduje błąd aplikacji. Zawsze owijaj granice Suspense Error Boundary'mi.
- Data fetching w `useEffect`: Suspense nie wykrywa danych pobieranych w `useEffect` ani w callbackach. Jeśli pobierasz tam dane, Suspense nie zadziała — potrzebujesz ręcznych stanów ładowania. Aby Suspense obsługiwał dane, Promise musi być zgłoszony (przez `use()` lub bibliotekę) podczas renderowania.
- Zbyt granularne granice Suspense: nie otaczaj Suspense każdego małego komponentu — wyświetlisz mnóstwo izolowanych spinnerów. Zamiast tego grupuj komponenty, które powinny ładować się razem. Możesz też zagnieżdżać granice Suspense dla progresywnego ładowania. Dobierz to do Twojego interfejsu: zdecyduj, które części pojawiają się razem, a które mogą ładować się niezależnie.

Stosując te dobre praktyki — zawsze dodawaj `fallback`, twórz Promise'a poza renderem i używaj Error Boundaries — unikniesz większości pułapek. Zwróć też uwagę na wersję Reacta: `use()` i Suspense do danych są oficjalne w React 19; w React 18 korzystasz z bibliotek takich jak React Query albo starszego podejścia ze zgłaszaniem Promise'ów.

## Podsumowanie

React Suspense to potężna funkcja dla async renderingu. Upraszcza lazy loading i data fetching — automatycznie zarządza stanami ładowania. Dzięki `React.lazy` dzielisz aplikację na chunki, komponenty ładują się na żądanie. Nowy hook `use()` z React 19 pozwala zawiesić się na Promise'ach podczas renderowania. W połączeniu z Error Boundaries Suspense zapewnia czysty wzorzec: fallbacki ładowania dla zasobów i danych, fallbacki błędów dla niepowodzeń.

W praktyce Suspense i `use()` prowadzą do czystszego kodu. Unikasz zarządzania `useEffect` i `useState` dla każdego async — zamiast tego piszesz UI, jakby dane były synchroniczne. Duże frameworki polegają na tym: streaming w Next.js, przyszłe data API React Routera — wszystko zależy od Suspense. W miarę dojrzewania funkcji concurrent w Reakcie, Suspense będzie coraz bardziej centralny. Opanowując Suspense i pilnując typowych błędów, będziesz pisać szybkie, odporne aplikacje React.
