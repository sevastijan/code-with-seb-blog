React Fiber to całkowita przebudowa algorytmu reconciliation w rdzeniu Reacta. Jego celem było przezwyciężyć ograniczenia starego „stack reconcilera”, który działał całkowicie synchronicznie. Zanim pojawił się Fiber (React 16), aktualizacje przetwarzane były liniowo — React przechodził całe dotknięte poddrzewo, wyliczał diff, aktualizował DOM i dopiero wtedy oddawał kontrolę. Problem: nie można było przerwać tego procesu ani zmienić priorytetu. Ciężka aktualizacja (np. filtrowanie listy) blokowała główny wątek, co prowadziło do skokowych animacji, zamarzającego UI i braku odpowiedzi na kliknięcia. Pojedynczy, drogi render mógł zatamować cały interfejs. Te bolączki były dokładnie tym, czego Fiber miał rozwiązać: umożliwić przerywalne, priorytetyzowane renderowanie rozłożone na wiele klatek, by UI zawsze reagował na interakcje użytkownika.

Fiber zmienia tę sytuację, reprezentując drzewo komponentów jako listę powiązaną (linked list) węzłów, gdzie każdy fiber to jednostka pracy — coś w rodzaju ramki stosu w pamięci. W odróżnieniu od starego stack reconcilera, Fiber nie musi przejść całego drzewa bez przeszkód. Może przerwać się w dowolnym momencie, zwrócić sterowanie przeglądarce (żeby obsłuży zdarzenia, rysowała animacje), a potem wznowić od miejsca, w którym się zatrzymał. To zmienia wszystko: React nagle potrafi wstrzymywać pracę i wrażać ją później, przypisywać priorytet różnym typom operacji, ponownie używać już wykonanej pracy, a nawet porzucać aktualizacje, które się wyczuliły. Fiber to właściwie „wirtualny stos” — ramki trzymane w obiektach JavaScript zamiast w natywnym stosie, dlatego JavaScript ma nad nimi pełną kontrolę. Ta zmiana architektury jest fundamentem dla wszystkich nowoczesnych funkcji Reacta: Concurrent Mode, Suspense, SSR ze streamingiem, automatycznego batchowania aktualizacji.

# Struktura węzła fiber

Pod maską każdy fiber to zwykły obiekt JavaScript przechowujący wszystko, co React potrzebuje wiedzieć o jednym komponencie: jego typ, props, state, gdzie siedzi w drzewie. Kluczowe pola to:

- `type` i `key`: Opisują typ komponentu (funkcja lub klasa dla komponentów złożonych, string dla elementów hostowych) oraz klucz reconciliation.
- `child` i `sibling`: Wskaźniki tworzące strukturę listy powiązanej dla drzewa. Pole `child` wskazuje na pierwszego fibera-dziecko (czyli element zwrócony przez render tego komponentu), a `sibling` każdego dziecka wskazuje na kolejne dziecko. (Ta reprezentacja lewe-dziecko/prawy-brat sprawia, że drzewo fiberów łatwo przechodzić po jednym węźle naraz.) Na przykład, mając `Parent` z dziećmi `<Child1/>` i `<Child2/>`, `child` fibera Parent wskazuje na `Child1`, a `Child1.sibling` wskazuje na `Child2`.
- `return`: Wskazuje na fibera-rodzica („adres powrotu” tej ramki stosu). Gdy komponent skończy renderować swoje poddrzewo, React używa `return`, by wrócić do rodzica.
- `stateNode`: Referencja do lokalnej instancji (dla komponentów klasowych) lub węzła DOM (dla komponentów hostowych). To łączy fibera z faktycznymi elementami UI.
- `pendingProps` i `memoizedProps`: Nowe props do zastosowania (`pendingProps`) kontra ostatnio wyrenderowane props (`memoizedProps`). Gdy się zgadzają, React może pominąć ponowne renderowanie tego poddrzewa — to dokładnie ten mechanizm, który [automatyczna memoizacja React Compilera](/pl/blog/react-compiler-automatyczna-memoizacja) wykorzystuje na etapie budowania.
- `pendingWorkPriority`: Numeryczny priorytet wskazujący, jak pilna jest ta aktualizacja. Wyższe wartości numeryczne oznaczają niższy priorytet (0 oznacza brak pracy). Scheduler Reacta używa tego pola, by zdecydować, nad którymi fiberami pracować w pierwszej kolejności.
- `alternate`: Wskaźnik na „drugą kopię” tego fibera. React utrzymuje dwa drzewa równolegle: Current Tree (to, co już wysłane do DOM) i Work-in-Progress Tree (to, nad czym pracuje teraz). Każdy fiber ma wskaźnik na jego odpowiednik w innym drzewie. Ten podwójny bufor (double-buffering) pozwala Reactowi bezpiecznie budować nowe UI bez ryzyka zepsucia tego już wyświetlanego. Gdy nowa praca jest gotowa, React zamienia wskaźniki i work-in-progress staje się current.

Wszystko razem: fiber przechowuje swoją tożsamość (type, key), strukturę drzewa (child, sibling, return), wejściowe props, stan, wyjście (np. węzeł DOM) oraz metadane schedulingu. Jeden z autorów podsumowuje: „Fiber to zwykły obiekt JS, jednostka pracy reprezentująca jeden komponent w drzewie. Każdy fiber zawiera:

- Typ komponentu
- Props i state
- Powiązania do fiberów child/parent/sibling

Wskaźnik do swojego węzła DOM; efekty zakolejkowane do commitu”. W porównaniu z nieprzejrzystym stosem wywołań dawnego stack reconcilera, ten jawny graf obiektów pozwala Reactowi wstrzymać się w połowie pracy, zajrzeć do niej lub cofnąć ją, a następnie wznowić później.

# Fazy render i commit

Aktualizacja w Reakcie przebiega w dwóch niezależnych, dobrze rozdzielonych etapach.

## Faza render (reconciliation)

Tutaj React pracuje nad drzewem bez dotykania DOM. Przechodzi fiberów, wywołuje render każdego komponentu, porównuje nowy stan ze starym — całe to robić bez jakiejkolwiek widocznej dla użytkownika zmiany. Praca nad fiber to jednostka, którą React może przerwać w dowolnym momencie. Jeśli budżet na klatkę (16 ms) się kończy, Fiber oddaje kontrolę przeglądarce: pozwala jej narysować bieżący obraz, obsłużyć wejście użytkownika, a sam wraca do pracy w następnej klatce. Dzięki temu UI nigdy nie zamraza się na długi czas.

## Faza commit

Gdy render się skończy i całe drzewo jest gotowe, React przechodzi do commitu. To krótkie, synchroniczne okno, podczas którego React rzeczywiście mutuje DOM i uruchamia hooki cyklu życia (`useLayoutEffect`, `componentDidMount`, `useEffect`). Commit nie może być przerywany — musi zakończyć się atomowo. To gwarancja spójności: albo zmiany są w DOM w całości, albo wcale. Nawet jeśli render został wstrzymany wiele razy, commit trafia do DOM tylko raz i od razu.

Oto kluczowa różnica: Fiber pozwala na wiele przejść render przed jednym commitem. W React 18+ komponent może być wyrenderowany, wznowiony, zmieniony, a nawet porzucony — wszystko to zanim jakikolwiek píxel zmieni się na ekranie. React czeka, czeka i czeka, aż ma całkowity pewność wyniku, dopiero wtedy robi commit. To architektura, która gwarantuje responsywność i spójność jednocześnie.

# Scheduling, priority lanes i współbieżność

Sercem Fibera jest **system schedulingu**. React nie przetwarza aktualizacji liniowo, w kolejności ich nadejścia — przypisuje każdej priorytet (zbiór bitów zwany „lane"). Scheduler bierze najistotniejszą pracę najpierw: kliknięcia użytkownika zawsze przed pobieraniem danych w tle. Dawny React miał kilka sztywnych poziomów priorytetów; Fiber używa masek bitowych — każdy bit to jeden lane, dlatego można mieć dokładnie tyle priorytetów, ile bitów w liczbie (31 lanes). Aktualizacja dostaje jeden konkretny lane. Gdy React ma wiele pracy oczekującej, scheduler wybiera lane o najwyższym priorytecie i zaczyna na nim pracę.

## Poziomy priorytetów (lanes)

Istnieje wiele lanes odpowiadających klasom priorytetów. Na przykład wszystkie aktualizacje synchroniczne używają tego samego bitu SyncLane. Jeśli oczekuje wiele aktualizacji sync, batchują się razem i renderują za jednym razem. Dla aktualizacji typu transition (oznaczonych przez `startTransition`) React przypisuje różne lanes, tak by nie blokowały się nawzajem. W praktyce oznacza to, że dwie niezależne tranzycje mogą postępować równolegle, podczas gdy cała praca sync pozostaje w jednym atomowym batchu. Matematycznie React używa operacji bitowych do zarządzania lanes (kod źródłowy używa masek `renderLanes`). Istotne jest to: „Każdy bit w masce bitowej nazywany jest Lane. Każdy render Reacta jest powiązany z jednym lub więcej lanes…” oraz „każda aktualizacja w Reakcie dostaje dokładnie jeden lane. Jeśli dwie aktualizacje mają ten sam lane, zawsze wyrenderują się w tym samym batchu”. Grupując aktualizacje w lanes, React może opóźnić lub scalić pracę o niskim priorytecie bez blokowania zadań o wysokim priorytecie.

## Jak to działa w praktyce

Scheduler pracuje w ten sposób: zawsze bierze fiberów z najwyższym priorytetem (lane). Jeśli w trakcie pracy pojawi się aktualizacja z wyższym priorytetem (np. użytkownik nagle kliknie coś ważnego), React może zawiesić bieżącą pracę i przełączyć się na nowe zadanie. To elastyczność, której stack reconciler nigdy nie miał.

## Time-slicing — przerywanie renderowania

Fiber mierzy czas podczas pracy. Budżet to około 16 ms na klatkę (żeby utrzymać 60 FPS). Gdy render zaczyna zbliżać się do tego limitu, Fiber mówi „stop” i oddaje kontrolę przeglądarce. Dzięki temu:

- Przeglądarka może narysować bieżący obraz
- Obsłużyć kliknięcia i wpisywanie użytkownika
- Wykonać inne ważne zadania
- React wraca do pracy w następnej klatce

Stary stack reconciler nie rozumiał pojęcia “oddać kontrolę” — musiał skończyć całą aktualizację bez przeszkód. Fiber rozbija ogromne renderowanie na kawałki i wykonuje je fragmentem po fragmentem.

## Automatyczne batchowanie

Fiber grupuje wiele `setState` w jeden render automatycznie, nawet poza react event handlerami. W React 17 kod taki:

```jsx
setTimeout(() => {
  setState(a)
  setState(b)
}, 0)
```

Wyzwalał dwa oddzielne rendery. Teraz, dzięki schedulerowi Fibera, jest to jeden render. Aktualizacje w tym samym “zadaniu” dzielą ten sam lane i automatycznie się mergują.

## Podsumowanie schedulingu

Fiber zmienia React z systemu „rób wszystko od razu w kolejności nadejścia” na system z priorytetami i elastycznym czasem. Ważne rzeczy (kliknięcia, animacje) idą do przodu; mniej ważne (pobieranie danych, aktualizacje UI w tle) czekają. Rezultat: interfejs zawsze czuje się responsywny, nawet jeśli w tle pracuje duża aktualizacja.

# Co dał nam Fiber

Fiber to więcej niż techniczny szczegół — to fundament dla całego nowoczesnego ekosystemu Reacta. Dzięki przerwalnemu, priorytetyzowanemu renderowaniu mamy dostęp do funkcji, które były niemożliwe ze starym stack reconcilerem.

## Concurrent rendering — responsywne UI nawet z dużymi aktualizacjami

React 18 domyślnie włącza Concurrent Mode, który jest możliwy tylko dzięki Fiberowi. Oznacza to, że React potrafi wstrzymać render, obsłużyć pilne zdarzenia, a potem wznowić pracę. Żaden render nie może zamrozić UI.

Praktycznie to wygląda tak: oznaczasz pewną aktualizację stanu jako “mniej ważną” za pomocą `startTransition`. React potraktuje ją z niższym priorytetem — jeśli użytkownik w tym czasie kliknie guzik, React przerywa tę aktualizację i najpierw obsługuje kliknięcie. To nadaje aplikacji nadzwyczajną responsywność.

```jsx
import { startTransition, useState } from 'react'

function App() {
  const [tab, setTab] = useState('home')
  // Zmiana zakładki bez blokowania UI
  function selectTab(nextTab) {
    startTransition(() => {
      setTab(nextTab)
    })
  }
  // ...
}
```

W tym przykładzie kliknięcie różnych zakładek opakowuje aktualizację stanu w `startTransition`. React potraktuje zmianę zakładki jako aktualizację „transition” o niższym priorytecie. Jeśli użytkownik szybko przełączy zakładki ponownie, React przerwie poprzednią pracę transition i spriorytetyzuje najnowsze kliknięcie (Fiber restartuje render po obsłużeniu wejścia). Innymi słowy, pilne wejście użytkownika nie jest blokowane przez wolniejszy render. Ta zdolność bierze się ze schedulera Fibera: przypisuje on te aktualizacje do odroczonych lanes i może je wstrzymać w połowie renderu.

## Suspense — oczekiwanie na dane bez zamrażania UI

Suspense to API pozwalające komponentom “czekać” na dane asynchronicznie, bez blokowania reszty interfejsu. Komponent rzuca Promise (np. podczas pobierania danych), React wyświetla fallback (spinner), a gdy dane przyjdą, kontynuuje render. Bez Fibera byłoby to niemożliwe — musielibyśmy wstrzymać całe renderowanie. Z Fiberem React po prostu wstrzymuje granię Suspense i renderuje resztę drzewa.

```jsx
import { Suspense } from 'react'

function Profile({ userId }) {
  return (
    <Suspense fallback={<div>Ładowanie…</div>}>
      <ProfileDetails userId={userId} />
    </Suspense>
  )
}
```

React może nawet streamować HTML na serwer, wysyłając części strony w miarę ich gotowości — wszystko dzięki Fiberowi, który potrafi wznowić render tam, gdzie się zatrzymał.

## Automatyczne batchowanie

W React 17 wiele `setState` poza event handlerami wyzwalało osobne rendery. Teraz scheduler Fibera automatycznie łączy je. Dwa `setState` w `setTimeout`? Jeden render. W `Promise`? Jeden render. Zawsze jeden render, o ile znajdują się w tym samym “zadaniu”. To redukuje niepotrzebną pracę, a jest możliwe bo Fiber grupuje aktualizacje po lanach.

## Streaming SSR — wysyłanie HTML stopniowo

Fiber zmienia też całą grę SSR. Zamiast czekać, aż serwer wyrenderuje całą stronę (co może chwilę potrwać, jeśli coś pobiera dane), React 18 wysyła HTML stopniowo — kawałek po kawałku. Granice Suspense na serwerze są obsługiwane automatycznie: strona wyświetla się szybko z fallbackami, a zawartość dołącza się, gdy jest gotowa.

```jsx
import { renderToPipeableStream } from 'react-dom/server'

const { pipe } = renderToPipeableStream(<App />, {
  onAllReady() {
    pipe(response) // wysyłaj HTML do klienta jak jest gotowy
  },
})
```

To zmienia UX dla użytkownika — widzi część treści od razu zamiast czekać na ładowanie całej strony.

---

Podsumowując: bez Fibera żadna z tych funkcji nie istniałaby. Concurrent Mode, Suspense, streamingowe SSR, automatyczne batchowanie — wszystko zbudowane na fundamentach architektury Fibera. To nie jest tylko optymalizacja — to zmiana sposobu myślenia o renderowaniu w Reactie.

# Fiber w praktyce — przykłady kodu

## Jak startTransition zmienia responsywność

Bez Fibera i concurrent rendering, nawet proste kliknięcie mogło być opóźnione długą aktualizacją. Oto problem:

```jsx
function App() {
  const [text, setText] = useState('')
  const [result, setResult] = useState('')

  function handleChange(e) {
    setText(e.target.value)
    const out = heavyCalculation(e.target.value) // To blokuje pisanie!
    setResult(out)
  }

  return <>
    <input value={text} onChange={handleChange} />
    <div>{result}</div>
  </>
}
```

Problem: każdy `handleChange` robi ciężkie obliczenie. Jeśli trwa 500 ms, użytkownik czeka 500 ms między każdym naciśnięciem. Morderstwo dla UX.

Z `startTransition`:

```jsx
import { startTransition } from 'react'

function App() {
  const [text, setText] = useState('')
  const [result, setResult] = useState('')

  function handleChange(e) {
    setText(e.target.value) // natychmiast
    startTransition(() => {
      const out = heavyCalculation(e.target.value)
      setResult(out) // może czekać
    })
  }

  return <>
    <input value={text} onChange={handleChange} />
    <div>{result}</div>
  </>
}
```

Teraz React priorytetyzuje `setText` (obliczenia są w transition — niższy priorytet). Pola input aktualizuje się od razu, a obliczenie Fibera może wstrzymać jeśli użytkownik znowu coś napisze.

## Automatyczne batchowanie wszędzie

W React 17 wiele `setState` wewnątrz event handlera batchowało się (jeden render). Ale poza handlerami — w `setTimeout`, promise, `fetch` — każdy `setState` wyzwalał osobny render. Frustrujące dla wydajności.

Fiber zmienia to całkowicie. Teraz *zawsze* batchuje:

```jsx
// React 17 — dwa osobne rendery
setTimeout(() => {
  setCount(c => c + 1)
  setFlag(f => !f)
}, 0)

// React 18 z Fiberem — jeden render
setTimeout(() => {
  setCount(c => c + 1)
  setFlag(f => !f)
  // Fiber: "To w tym samym zadaniu, łączę i robię jeden render"
}, 0)
```

Scheduler Fibera widzi obie aktualizacje w tym samym "oknie" (tym samym lane) i robi jeden render zamiast dwóch. Nie musisz nic robić — działa domyślnie.

## Streaming z serwera na klienta

Zamiast czekać aż serwer wyrenderuje stronę 100%, Fiber wysyła ją fragmentami:

```jsx
import { renderToPipeableStream } from 'react-dom/server'
import App from './App'

const { pipe } = renderToPipeableStream(<App />, {
  onAllReady() {
    pipe(res) // wysyłaj HTML jak jest gotowy
  },
})
```

Strona wyświetla się w przeglądarce *zaraz*, a slowData (`<Suspense>` fallback) pojawia się, gdy będzie gotowa. Starego `renderToString` trzeba czekać na wszystko. Fiber wysyła to co ma, a resztę później.
