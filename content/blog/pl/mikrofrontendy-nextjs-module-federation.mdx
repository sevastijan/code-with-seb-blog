Pracowałem z mikrofrontendami, które uratowały organizacje, i tymi, które rujnowały developer experience. Różnica nigdy nie jest techniczna — chodzi o to, czy organizacja naprawdę ich potrzebowała.

Konsultowałem firmę fintechową z 8 zespołami pracującymi na tym samym Next.js monolicie. Deployment wymagał koordynacji wszystkich ośmiu — jeden nieudany test zatrzymywał całą linię, a release cycle to były dwa tygodnie merge-test-deploy. Mikrofrontendy dały każdemu zespołowi niezależny deployment. Tempo się zmieniło z raz na dwa tygodnie na kilka razy dziennie.

Auditowałem startup z czterema programistami, którzy wdrożyli mikrofrontendy, bo „tak robią duże firmy". Przez 3 miesiące budowali infrastrukturę federacji. Więcej było YAML-a niż rzeczywistego kodu aplikacji. Dwóch z nich spędzało większość czasu na debugowaniu integracji między mikrofrontendami zamiast pisać features.

**Zasada:** Mikrofrontendy rozwiązują problemy **organizacyjnego skalowania**, nie techniczne. Jeśli wąskim gardłem jest koordynacja deploymentów między zespołami — mikrofrontendy pomagają. Jeśli problem jest gdzie indziej, pogorszą sytuację.

## Kiedy mikrofrontendy mają sens

### Kiedy rzeczywiście mają sens:
- **5+ niezależnych zespołów** musi wdrażać bez koordynacji
- **Różne kadencje wydań** — zespół A wydaje codziennie, zespół B kwartalnie
- **Odrębne domeny produktowe** — checkout, magazyn, analityka, admin to osobne bounded contexts
- **Autonomia organizacyjna** jest ważniejsza niż prostota techniczna
- **Próbowałeś już** dobrze zorganizowanego monorepo i trafiłeś na limity koordynacji

### Kiedy na pewno nie mają sensu:
- **Masz mniej niż 30 inżynierów** — dobrze zorganizowane monorepo z [Turborepo lub Nx](/pl/blog/monorepo-turborepo-nx-react-nextjs) poradzi sobie lepiej
- **Zespoły mocno współdzielą UI** — jeśli 80% UI jest współdzielone, mikrofrontendy dodają koszt bez korzyści
- **Jesteś startupem** — podatek infrastrukturalny cię spowolni
- **Twoim problemem jest jakość kodu** — mikrofrontendy nie naprawiają złej architektury, one ją rozpraszają

## Trzy podejścia w Next.js

### Podejście 1: Multi-Zones (najprostsze)

Next.js Multi-Zones pozwalają komponować wiele aplikacji Next.js pod jedną domeną. Każda „strefa" (zone) to samodzielna aplikacja Next.js z własnym buildem i deploymentem:

```javascript
// main-app/next.config.mjs
export default {
  async rewrites() {
    return [
      {
        source: '/checkout/:path*',
        destination: 'https://checkout.internal.example.com/:path*',
      },
      {
        source: '/admin/:path*',
        destination: 'https://admin.internal.example.com/:path*',
      },
    ];
  },
};
```

Główna aplikacja routuje requesty do odpowiednich stref na podstawie ścieżki URL. Każda strefa deployuje się niezależnie.

**Plusy:** Brak współdzielonego runtime'u, każda strefa to zwykła aplikacja Next.js, minimalna konfiguracja.

**Minusy:** Full page reload przy przechodzeniu między strefami, brak shared state, brak component sharing.

**Idealne gdy:** Sekcje produktu są całkowicie odseparowane (main app vs admin panel vs docs).

### Podejście 2: Module Federation (najbardziej elastyczne)

Module Federation umożliwia mikrofrontendom współdzielenie komponentów i dependencji w runtime — bez page reloads, z jedną instancją Reacta i component-level composition:

```javascript
// host-app/next.config.mjs
import { NextFederationPlugin } from '@module-federation/nextjs-mf';

export default {
  webpack(config) {
    config.plugins.push(
      new NextFederationPlugin({
        name: 'host',
        remotes: {
          checkout: 'checkout@https://checkout.example.com/_next/static/chunks/remoteEntry.js',
          analytics: 'analytics@https://analytics.example.com/_next/static/chunks/remoteEntry.js',
        },
        shared: {
          react: { singleton: true, requiredVersion: '^19.0.0' },
          'react-dom': { singleton: true, requiredVersion: '^19.0.0' },
        },
      })
    );
    return config;
  },
};
```

```javascript
// checkout-app/next.config.mjs
import { NextFederationPlugin } from '@module-federation/nextjs-mf';

export default {
  webpack(config) {
    config.plugins.push(
      new NextFederationPlugin({
        name: 'checkout',
        filename: 'static/chunks/remoteEntry.js',
        exposes: {
          './CheckoutForm': './components/CheckoutForm',
          './CartSummary': './components/CartSummary',
        },
        shared: {
          react: { singleton: true },
          'react-dom': { singleton: true },
        },
      })
    );
    return config;
  },
};
```

```typescript
// host-app — konsumowanie zdalnego komponentu
import dynamic from 'next/dynamic';

const CheckoutForm = dynamic(
  () => import('checkout/CheckoutForm'),
  { ssr: false, loading: () => <Skeleton /> }
);

export default function CheckoutPage() {
  return (
    <div>
      <h1>Checkout</h1>
      <CheckoutForm onComplete={(orderId) => router.push(`/orders/${orderId}`)} />
    </div>
  );
}
```

**Plusy:** Component-level sharing, dependency deduplication, zero page reloads.

**Minusy:** Złożona webpack config, ograniczone wsparcie App Router, trudne debugowanie, version mismatches = subtelne bugs.

**Idealne gdy:** Duże organizacje, gdzie zespoły współdzielą komponenty między aplikacjami.

### Podejście 3: Vercel @vercel/microfrontends (zarządzane)

Zarządzana usługa Vercela abstrakcjonizuje całą infrastrukturę:

```javascript
// vercel.json
{
  "microfrontends": {
    "applications": {
      "web": { "defaultRoute": true },
      "checkout": { "routing": [{ "prefix": "/checkout" }] },
      "admin": { "routing": [{ "prefix": "/admin" }] }
    }
  }
}
```

**Plusy:** Zero webpack config, auto routing, managed infrastructure, App Router support.

**Minusy:** Vendor lock-in, enterprise pricing.

## Komunikacja między mikrofrontendami

Najtrudniejsza część: jak niezależnie deployowane aplikacje mają sobie przekazywać dane?

### Custom events (najlepsze)

```typescript
// Shared event types (publikuj jako npm package)
interface MicroFrontendEvents {
  'cart:updated': { items: CartItem[]; total: number };
  'user:authenticated': { userId: string; token: string };
  'checkout:completed': { orderId: string };
  'theme:changed': { theme: 'light' | 'dark' };
}

// Emitter
function emitMFEvent<K extends keyof MicroFrontendEvents>(
  event: K,
  data: MicroFrontendEvents[K]
) {
  window.dispatchEvent(new CustomEvent(`mfe:${event}`, { detail: data }));
}

// Listener (w każdym MFE)
function useMFEvent<K extends keyof MicroFrontendEvents>(
  event: K,
  handler: (data: MicroFrontendEvents[K]) => void
) {
  useEffect(() => {
    const listener = (e: CustomEvent) => handler(e.detail);
    window.addEventListener(`mfe:${event}`, listener as any);
    return () => window.removeEventListener(`mfe:${event}`, listener as any);
  }, [event, handler]);
}
```

### URL jako współdzielony stan

URL to najbardziej niezawodny kanał komunikacji między MFE — survives page reload, bookmarkable, zero runtime coupling:

```typescript
// Checkout MFE reads cart from URL params
const searchParams = useSearchParams();
const cartId = searchParams.get('cartId');
// Fetch cart data independently using cartId
```

## Anty-wzorce, które rujnują wdrożenia

### 1. Współdzielenie wszystkiego

Jeśli twoje MFE współdzielą design system, state management, API client, routing i build pipeline — zbudowałeś rozproszoną monolityczną bajkę. Gorsza niż oryginalny monolit, z którego uciekałeś.

**Zasada:** Share tylko React, react-dom i design system. Reszta = full ownership.

### 2. Sprzężenie przez global state

```typescript
// ❌ ANTY-WZORZEC: Shared Redux store
const globalStore = createStore(rootReducer);
// Każdy MFE jest teraz sprzężony z każdym innym
```

Każdy MFE powinien mieć własny state. Komunikacja = events, nie shared stores.

### 3. Brak design systemu

Bez shared component library każdy zespół pisze własne buttony, formy, layouty. User widzi 3 różne button styles na jednej stronie. Zainwestuj w design system ZANIM zaczyną się MFE.

### 4. Ignorowanie bundle size

Jeśli każdy MFE ładuje swoją kopię Reacta, ładujesz React 5 razy. Używaj `singleton: true` w shared Module Federation config i monitoruj łączny bundle rozmiar.

## Praktyczne wnioski

1. **Nie wdrażaj MFE zanim nie masz 5+ zespołów z bólem koordynacji.** Monorepo są lepsze dla mniejszych organizacji.
2. **Zacznij od Multi-Zones** — najprostsze podejście do independent deployments.
3. **Przejdź na Module Federation**, gdy potrzebujesz component-level sharing między appami.
4. **Komunikuj się custom events** — typed, loosely coupled, brak shared state.
5. **Share React jako singleton** — ładowanie Reacta 5 razy = performance disaster.
6. **Zainwestuj w design system FIRST** — visual consistency = pierwsza korzyść MFE.
7. **Każdy MFE = own state** — zero shared Redux, żaden globalny Zustand między MFE.
8. **Monitor łączny bundle** — suma wszystkich MFE, a nie każdy osobno.
9. **URL jako shared state** — najniezawodniejszy kanał komunikacji między MFE.
10. **Cel = team autonomy, nie techniczna elegancja.** Jeśli zespoły nie mogą deployować niezależnie, architektura nie działa — bez względu na jej geniusz.

Mikrofrontendy to organizacyjna architektura w technicznej masce. Pytanie nie brzmi „czy Module Federation?", tylko „czy nasze zespoły potrzebują independent deploymentu?". Jeśli tak — wybierz najprostsze podejście, które to daje. Jeśli nie — [dobrze zorganizowane monorepo](/pl/blog/monorepo-turborepo-nx-react-nextjs) zawsze wygra.
