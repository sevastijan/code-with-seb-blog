Tydzień temu usunąłem 400 linii JavaScriptu, a komponent zadziałał _lepiej_. Bez `ResizeObserver`. Bez listenera obserwującego kliknięcie poza elementem. Bez Floating UI. Bez `MutationObserver` śledzczącego zmianę checkboxa, żeby móc toggle'ować klasę na jego rodzicu. Po prostu CSS, który wreszcie robi dokładnie to, co chciałem.

Od kilkunastu lat bycie „frontend developerem" oznaczało w praktyce „pisanie JavaScriptu, aby nadrobić to, czego CSS nie potrafi". Stylowanie na podstawie stanu rodzica? JavaScript. Responsywne komponenty niezależne od viewport? JavaScript. Tooltip, który przeskakuje gdy zbliża się do krawędzi okna? Cała biblioteka. Zaakceptowaliśmy, że dostarczamy dziesiątki kilobajtów kodu tylko po to, aby wymienić nazwy klas.

To się skończyło. W **2026 roku funkcje CSS weszły na Baseline** — pracują w Chrome, Firefox, Safari i Edge bez flag — i każda z nich eliminuje konkretny, istotny kawałek JavaScriptu. Chcę Ci opowiedzieć o tym, co się zmienia: jakie funkcje są teraz dostępne, jaki dokładnie JavaScript można wyrzucić, realne uwagi na temat wsparcia przeglądarek i szczera lista rzeczy, które _nadal_ potrzebują skryptu.

## :has() — selektor rodzica, który eliminuje obserwatory DOM

`:has()` to funkcja, która zmienia grę. To **relacyjna pseudoklasa**: stylizuje element na podstawie tego, co _zawiera_, albo jaki element go następuje. CSS nigdy tego nie oferował. Brak możliwości stylowania rodzica na podstawie jego potomków — to był główny powód, dla którego JavaScript przenikał do logiki prezentacji.

Funkcja jest **na Baseline od grudnia 2023** (Chrome/Edge 105+, Safari 15.4+, Firefox 121+), więc w 2026 możesz jej używać bez żadnych obaw.

Najprostszy przykład — stylowanie karty na podstawie stanu checkboxa w jej wnętrzu. Tradycyjny sposób wyglądał tak:

```js
// Tradycyjnie: nasłuchuj zmian, przełączaj klasę na rodzicu
const inputs = document.querySelectorAll('.card input[type="checkbox"]');
inputs.forEach((input) => {
  input.addEventListener('change', () => {
    input.closest('.card').classList.toggle('is-selected', input.checked);
  });
});
```

To listener dla każdego inputu, synchronizacja klas, i zawsze ryzyko braku synchronizacji. Nowy sposób? Jeden selektor:

```css
/* Styluj rodzica na podstawie stanu potomka — czysty CSS */
.card:has(input[type='checkbox']:checked) {
  border-color: oklch(60% 0.2 250);
  background: oklch(97% 0.02 250);
}
```

Kluczowe zrozumienie: `:has()` łączy się ze wszystkim — `:checked`, `:invalid`, `:focus-within`, kombinatorami. Oto wzorce, które teraz regularnie stosuję:

```css
/* Marginesy warunkowe: dodaj spacja pod figurą tylko jeśli ma napis */
figure:has(figcaption) {
  padding-block-end: 1rem;
}

/* Zapytania o liczbę elementów — bez nth-child rachunków:
   przełącz layout na dwie kolumny, gdy pojawi się 4. element */
.gallery:has(> :nth-child(4)) {
  grid-template-columns: 1fr 1fr;
}

/* Stylowanie elementu PRZED innym — wcześniej niemożliwe.
   Wygaś etykietę jeśli pole obok niej jest wyłączone */
label:has(+ input:disabled) {
  opacity: 0.5;
}
```

Uczciwa uwaga o wydajności: **`:has()` może być drogi przy bardzo częstych zmianach DOM**. Przeglądarka musi ponownie obliczyć dopasowanie dla każdej mutacji. Szeroki selektor jak `body:has(...)` przeliczany przy każdym naciśnięciu klawisza to realny koszt. Trzymaj go ograniczony do konkretnego zakresu. W statycznych sytuacjach i przy zmianach stanu — jak w przykładach wyżej — jest praktycznie darmowy i eliminuje całe biblioteki kodu.

## Container queries — komponenty dostosowujące się do swojego kontenera

Media queries pytają jedno: jak duży jest _viewport_? Ale karta nie żyje w viewport. Żyje w sidebarze, modalu, trzech kolumnach — i powinna dostosować się do _swojej_ dostępnej przestrzeni. Wcześniej jedynym sposobem był `ResizeObserver` na każdy komponent.

**Container queries osiągnęły Baseline** (Chrome/Edge 105+, Firefox 110+, Safari 16+) i działają dla 93%+ globalnego ruchu. Oto JavaScript, który można wyrzucić:

```js
// Tradycyjnie: obserwuj rozmiar, zmień klasę na podstawie szerokości
const observer = new ResizeObserver((entries) => {
  for (const entry of entries) {
    const w = entry.contentRect.width;
    entry.target.classList.toggle('card--wide', w > 400);
  }
});
document.querySelectorAll('.card').forEach((el) => observer.observe(el));
```

Wersja CSS obejdzie się bez obserwatora, bez ręcznego zarządzania klasami, i wykona się na poziomie layoutu przeglądarki:

```css
/* Oznacz rodzica jako kontener dla zapytań */
.card-wrapper {
  container-type: inline-size;
  container-name: card;
}

/* Odpytuj szerokość kontenera, nie viewport */
@container card (min-width: 400px) {
  .card {
    display: grid;
    grid-template-columns: 120px 1fr;
    gap: 1rem;
  }
}
```

Umieść tę samą `.card` w wąskim sidebarze — ułoży się w pionie. Umieść w szerokim pasku — zajmie przestrzeń w poziomie. Bez zmian w kodzie, bez JavaScriptu. Dostępne są też **container query units** (`cqi`, `cqb`, `cqw`, `cqh`) — skalują rozmiary względem kontenera, idealne do responsywnej typografii w komponencie.

W 2026 funkcja jest w pełni gotowa: **style queries** (`@container style(...)`) są na Baseline — Firefox 151 dodał je w maju 2026 — więc możesz warunkować na podstawie wartości zmiennych CSS kontenera, nie tylko jego rozmiaru. Dla bibliotek komponentów to naturalnie wpisuje się w [podejście systemów komponentowych](/pl/blog/od-komponentow-do-systemow-atomic-design): komponenty stają się naprawdę niezależne od kontekstu.

## @scope — style, które działają wyłącznie w wyznaczonym zakrebie

Ograniczanie zasięgu stylów wymagało CSS Modules, kroku w buildu, dyscypliny BEM lub CSS-in-JS. `@scope` robi to natywnie, bez toolingu. Jest **na Baseline od początku 2026** (Chrome 118+, Safari 17.4+, Firefox dołącza wsparcie).

Dwie rzeczy czynią go mocnym: korzenie zasięgu i opcjonalna **granica zasięgu** (`to (...)`), która tworzy „pęcherzyk" — reguły działają wewnątrz korzenia, ale zatrzymują się na granicy.

```css
/* Reguły działają dla .card i jej potomków... */
@scope (.card) to (.card__content) {
  /* ...ale PRZERWĘ na .card__content — dzieci nie podlegają regułom.
     img tutaj = obrazki w .card, ale NOT w .card__content */
  img {
    border-radius: 8px;
  }

  :scope {
    /* :scope to sam korzeń zakresu — tutaj .card */
    border: 1px solid oklch(90% 0 0);
  }
}
```

Druga, nie mniej ważna rzecz: `@scope` zmienia to, jak **bliskość** rozstrzyga konflikty. Dwie reguły z równą specyficzności — wygrywa ta, której korzeń jest _bliżej_ w DOM. To dokładnie problem, który kiedyś wymagał JavaScript albo skomplikowanych hacków ze specyficzności przy stylowaniu zagnieżdżonych komponentów. Znacznie zdrowszy domyślny wybór niż wojna specyficzności, którą opisuję w [przewodniku po architekturze CSS](/pl/blog/zasady-czystego-kodu-frontend-dry-kiss-solid).

## Natywne zagnieżdżanie CSS — Sass bez buildu

Sass uruchamialiśmy głównie dla zagnieżdżania i `&`. **Natywne zagnieżdżanie CSS osiągnęło Baseline** we wszystkich przeglądarkach, więc wiele konfiguracji Sassa istnieje wyłącznie z przyzwyczajenia.

```css
.nav {
  display: flex;
  gap: 1rem;

  /* Zagnieżdżaj selektory bezpośrednio */
  a {
    color: inherit;

    /* & to referencja do rodzica — jak w Sassie */
    &:hover {
      color: oklch(55% 0.22 250);
    }
  }

  /* Można nawet zagnieżdżać zapytania */
  @container (min-width: 500px) {
    gap: 2rem;
  }
}
```

Dwie rzeczy warte zapamiętania. Po pierwsze, natywne zagnieżdżanie **nie skaduje stringów** jak Sass — `&--active` nie zadziała; piszesz `&.nav--active`. Po drugie, selektor jak `a { }` to OK, ale goły selektor zaczynający się od nazwy w starszych przeglądarkach wymagał `&`. Nowoczesne silniki to rozluźniają, ale zaczynanie od `&` to nawyk, który nigdy Cię nie podłamie.

## Cascade layers — zarządzanie specyficzności bez `!important`

`@layer` pozwala zadeklarować jednoznaczną kolejność priorytetów. Późniejsze warstwy zawsze wygrywają z wcześniejszymi, **niezależnie od specyficzności**. To koniec wojen o `!important` i w końcu kontrola nad CSS firm trzecich.

```css
/* Ustal kolejność raz — ostatnia wygrywa */
@layer reset, base, components, utilities;

@layer components {
  /* Nawet wysoko-specyficzny selektor tutaj... */
  .button.button--primary {
    background: oklch(55% 0.2 255);
  }
}

@layer utilities {
  /* ...przegrywa nisko-specyficzną regułą w utilities, bo jest później */
  .bg-transparent {
    background: transparent;
  }
}
```

Dlatego utility frameworki przeszły na warstwy. Jeśli pracowałeś z nowym Tailwindem, widziałeś to w akcji — wyjaśniłem to szczegółowo w [przewodniku migracji Tailwind CSS 4](/pl/blog/tailwind-css-4-co-nowego-migracja). Reguły bez warstwy domyślnie wygrywają z regułami wewnątrz warstwy — przydatne wyjście awaryjne do zapamiętania.

## Subgrid — wyrównanie zagnieżdżonych siatek bez JS

Zagnieżdżone siatki były kiedyś izolowane: karta wewnętrzna nie mogła się wyrównać do linii rodzica, więc nagłówki i stopki w rzędzie rozjeżdżały się. Rozwiązaniem był JavaScript mierzący wysokości. **Subgrid jest na Baseline** (Safari 16, Firefox 71, Chrome/Edge 117) i pozwala potomkom przejąć strukturę rodzica:

```css
.cards {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 1rem;
}

.card {
  display: grid;
  /* Rozciągnij na 3 wiersze rodzica i odziedzicz jego linie wierszy */
  grid-row: span 3;
  grid-template-rows: subgrid;
}
```

Teraz tytuł, treść i stopka każdej karty wyrównują się na wspólnych liniach w całym rzędzie. Bez skryptów synchronizacyjnych, bez zgadywania wymiarów.

## Walidacja formularzy z :user-valid — UI bez maszyny stanów

`:valid` i `:invalid` były teoretycznie OK, ale niepraktyczne: puste pole wymagane jest „nieprawidłowe" od załadowania, więc jest czerwone zanim użytkownik wpisze cokolwiek. Więc budowaliśmy w JS maszynę stanów — śledząc „touched", „dirty", „blurred" — żeby wiedzieć, kiedy pokazać błąd.

**`:user-valid` i `:user-invalid`** rozwiązują to na poziomie przeglądarki: aktywują się tylko _po_ interakcji użytkownika z polem. Już są na Baseline.

```css
/* Stylizuj dopiero, gdy użytkownik rzeczywiście wchodził w pole */
input:user-invalid {
  border-color: oklch(60% 0.2 25);
}

input:user-valid {
  border-color: oklch(60% 0.17 150);
}

/* Połącz z :has(), żeby pokazać błąd bez JavaScript */
.field:has(input:user-invalid) .error-text {
  display: block;
}
```

Dwie kolejne rzeczy dla formularzy. `field-sizing: content` sprawia, że input i textarea rosną z zawartością — zastępuje skrypt auto-expanding textarea:

```css
/* Auto-rosnąca textarea, natywnie */
textarea {
  field-sizing: content;
  min-height: 3lh; /* zacznij od 3 wysokości linii */
  max-height: 12lh;
}
```

`field-sizing` jest najpierw w Chromium i rozszerza się w 2026, więc traktuj jako **progresywne ulepszenie** — przegrywa się do zwykłej textarea. Dostępny, odporny UX formularzy jak ten pojawia się w całym [przewodniku dostępności web 2026](/pl/blog/dostepnosc-web-2026-eaa-ada-wcag), a robienie tego w CSS oznacza, że działa nawet jeśli Twój JS się nie załaduje.

## text-wrap: balance / pretty — typografia bez bibliotek

Wyważone nagłówki wymagały biblioteki typu Balance Text, która mierzyła i wstawiała `<br>`. Teraz to jedna właściwość, i jest dostępna:

```css
/* Rozmieść wieloliniowe nagłówki równomiernie — bez sierot */
h1,
h2,
h3 {
  text-wrap: balance;
}

/* Dla tekstu: unikaj jednowyrazowych ostatnich linii,
   zoptymalizowane dla dłuższych paragyrafów */
p {
  text-wrap: pretty;
}
```

Użyj `balance` dla krótkich tekstów (nagłówki, cytaty) — jest droższe, przeglądarki ograniczają ilość linii — a `pretty` dla akapitów.

## Anchor positioning + popover — Floating UI to teraz CSS

To funkcja, która nadal wydaje się niemożliwa. Tooltips, dropdowny, popovery wymagały JavaScript do dwóch trudnych rzeczy: **pozycjonowania** względem triggera i **przeskakiwania** gdy wychodzą poza viewport. Dlatego istnieje Floating UI — cztery do pięciu kilobajtów kodu, patrz [optymalizacja bundle](/pl/blog/dynamiczna-optymalizacja-bundle-ponizej-200kb).

Łącz **natywny atrybut `popover`** (na Baseline — gratis obsługuje pokaż/ukryj, Escape zamyka, light-dismiss na klik poza, layer stacking) z **CSS anchor positioning**:

```html
<button popovertarget="menu" id="trigger">Options</button>
<div popover id="menu">…menu items…</div>
```

```css
#trigger {
  anchor-name: --trigger; /* nazwij zakotwiczenie */
}

#menu {
  position: absolute;
  position-anchor: --trigger; /* przywiąż do zakotwiczenia */

  /* Umieść poniżej, wyrównaj do lewej krawędzi */
  position-area: bottom span-right;
  margin-top: 0.5rem;

  /* Automatycznie przeskakuj, aby zostać widocznym */
  position-try-fallbacks: flip-block, flip-inline;
}
```

Stary sposób: listener na `scroll`/`resize`, przeliczanie współrzędnych, ręczne sprawdzanie przepełnienia w każdej ramce. Tutaj `position-area` umieszcza, `position-try-fallbacks` obsługuje przeskok — deklaratywnie, bez głównego wątku.

**Uczciwość o wsparciu:** najmniej gotowa funkcja tutaj. Jest **Baseline 2026** w sensie Chrome 125+, Firefox 132+, Safari 18.2+, ale wylądowała najpierw w Chromium i krawędzie są nierówne — Safari 18.4+ potrzebował dla pełnego wsparcia. Używaj jako ulepszenia, zawsze miej statyczne fallbacki, nie bierz na to całej funkcji menu.

## color-mix() i oklch() — pełne design systemy w CSS

Generowanie wariantów kolorów — jasne, ciemne, hover — było drugą główną rzeczą, do której używaliśmy Sassa. Teraz przeglądarka oblicza percepcyjnie równomierną matematykę kolorów w runtime. **`oklch()` i `color-mix()` są na Baseline**.

```css
:root {
  --brand: oklch(55% 0.2 255);
}

.button {
  background: var(--brand);
}

.button:hover {
  /* Wymieszaj 15% czerni — prawdziwy, percepcyjny ciemny variant */
  background: color-mix(in oklch, var(--brand), black 15%);
}

.badge {
  /* Wariant względny: ten sam odcień, znacznie jaśniejszy */
  background: oklch(from var(--brand) 92% c h);
}
```

Ponieważ OKLCH jest percepcyjnie równomierny, `+10%` jasności wygląda identycznie wszędzie na kolorze — w odróżnieniu od HSL. Możesz wygenerować całą paletę z jednej zmiennej, w runtime, wrażliwą na motyw, bez buildu.

## Scroll-driven animations — efekty przewijania bez głównego wątku

Progress bary i efekty reveal przy scrollu wymagały `scroll` listenera odpalanego na głównym wątku (jank!) albo `IntersectionObserver`. **`animation-timeline` z `scroll()` i `view()` jest już dostępne**, i przeglądarka uruchamia go na compositorze:

```css
@keyframes grow {
  from {
    transform: scaleX(0);
  }
  to {
    transform: scaleX(1);
  }
}

/* Progress bar powiązany ze scrollem strony — bez JS */
.progress {
  transform-origin: left;
  animation: grow linear;
  animation-timeline: scroll(root block);
}

/* Fade in element przy wejściu w viewport */
.reveal {
  animation: fade-in linear both;
  animation-timeline: view();
  animation-range: entry 0% cover 40%;
}
```

Ponieważ działa na compositorze, nie konkuruje z Twoim kodem o czas ramki — ważne jeśli śledzisz metryki w [przewodniku wydajności](/pl/blog/budzety-wydajnosci-frontend).

## Gdzie JavaScript wciąż jest konieczny

CSS nie wyeliminował _całego_ Twojego JavaScript — i pretensja, że to zrobił, kończy się rozczarowaniem. Wciąż potrzebujesz skryptu do:

- **Logiki aplikacji i stanu** — pobieranie danych, zarządzanie stanem, reguły biznesowe. CSS stylizuje; nie oblicza.
- **Zaawansowanego zarządzania fokusem** — roving tabindex, pułapkowanie fokusu poza `popover`, większość ARIA interakcji klawiatury. Dostępność żyje w HTML + JS.
- **Manipulacji DOM** — dodawanie/usuwanie/przesuwanie węzłów, listy wirtualne, drag and drop.
- **Odczytu pomiarów** — jeśli potrzebujesz wartości pikselowych w JS (analityka, warunkowe pobieranie), `ResizeObserver`/`IntersectionObserver` są nadal konieczne. CSS reaguje, ale nie daje Ci liczby.
- **Dynamicznych kotwic** i złożonych animacji opartych na gestach, które deklaratywne osie czasu nie mogą wyrazić.

Zmiana: JavaScript wraca do bycia o _zachowaniu i danych_, CSS przejmuje _prezentację_ — podział, którego zawsze chcieliśmy.

## Plan migracji

Nie przepisujesz wszystkiego w piątek. Oto co zadziałało dla mnie:

1. **Włącz nowe rzeczy od razu.** `text-wrap: balance/pretty`, natywne zagnieżdżanie, `oklch()`/`color-mix()`, cascade layers, subgrid, `:user-valid` — to Baseline i niskie ryzyko. Wdróż teraz.
2. **Następnie zamień obserwatory.** Wyszukaj `ResizeObserver` i podmień na container queries; wyszukaj event listenery do toggle'owania klas i zamień na `:has()`. Tu liczba linii spada szybko.
3. **Ulepszaj progresywnie na froncie.** Anchor positioning i `field-sizing` to są rzeczy, ale nierówne — zawsze miej fallbacki.
4. **Usuń zależności.** Gdy `:has()` i container queries działają, mnóstwo małych narzędzi i cała Floating UI pojawi się w Twoim `node_modules` jako zbędne. To najpotężniejsza optymalizacja tego kwartału.

Piszę CSS od ponad dekady. 2026 to pierwszy rok, kiedy język jest _przed_ tym, czego potrzebuję, zamiast dwa kroki za. Otwórz codebase, grep'uj `ResizeObserver` i `classList.toggle`, zacznij usuwać. To najsatysfakcjonujący refaktor roku.
