W zeszłym roku wypuściłem na produkcję buga, którego TypeScript powinien był wychwycić. Zewnętrzne API zmieniło format odpowiedzi — pole, które kiedyś zwracało liczbę, zaczęło zwracać string. TypeScript tego nie złapał, bo dane przychodziły w runtime, długo po tym jak kompilator przyklepał kod. Typ mówił `number`, wartość była `"42"`, a moja arytmetyka po cichu produkowała `NaN`, który kaskadowo przeszedł przez trzy kolejne serwisy, zanim ktokolwiek to zauważył.

To był tydzień, w którym przestałem traktować walidację runtime jako opcjonalną.

TypeScript jest niesamowity do wychwytywania błędów podczas developmentu. Ale wyparowuje w czasie kompilacji — każda adnotacja `interface` i `type` zostaje usunięta z JavaScriptu, który faktycznie się wykonuje. Jeśli twoje dane pochodzą z API, formularza, bazy danych, pliku albo LLM-a, TypeScript nie ma żadnego zdania na temat tego, czy pasują do twoich typów. Jest to szczególnie krytyczne, gdy masz do czynienia z [niezaufanym wejściem w aplikacjach opartych na AI](/pl/blog/prompt-injection-bezpieczenstwo-llm). **Zod wypełnia tę lukę.**

## Dlaczego Zod wygrał

Istnieją dziesiątki bibliotek walidacyjnych. Joi, Yup, Ajv, class-validator, io-ts, Valibot. Zod wygrał ekosystem TypeScript z trzech powodów:

1. **Inferencja typów oparta na schemacie** — definiujesz schemat raz, wnioskujesz z niego typ TypeScript. Zero duplikacji.
2. **Natywny dla TypeScript** — zbudowany od podstaw z myślą o TypeScript, a nie doklejony później.
3. **Integracja z ekosystemem** — pierwszorzędne wsparcie w React Hook Form, tRPC, Next.js, Astro, Hono i każdym większym frameworku.

```typescript
import { z } from 'zod';

// Definiujemy schemat raz
const UserSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1).max(100),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'viewer']),
  createdAt: z.coerce.date(),
});

// Wnioskujemy typ — zero duplikacji
type User = z.infer<typeof UserSchema>;
// { id: string; name: string; email: string; role: "admin" | "user" | "viewer"; createdAt: Date }
```

Koniec z utrzymywaniem osobno interfejsu `User` i osobnej funkcji walidacyjnej, które mogą się z czasem rozjechać. Schemat JEST typem.

## parse() kontra safeParse() — poznaj różnicę

To najważniejsza decyzja dotycząca API, jaką podejmiesz w Zodzie.

### parse() — rzuca wyjątek przy niepowodzeniu

```typescript
try {
  const user = UserSchema.parse(unknownData);
  // user jest tutaj w pełni otypowany
} catch (error) {
  if (error instanceof z.ZodError) {
    console.error(error.issues);
  }
}
```

Używaj `parse()`, gdy niepoprawne dane są sytuacją wyjątkową — wczytywanie konfiguracji, zmienne środowiskowe, dane, co do których naprawdę nie spodziewasz się, że będą błędne.

### safeParse() — zwraca obiekt wyniku

```typescript
const result = UserSchema.safeParse(unknownData);

if (result.success) {
  const user = result.data; // w pełni otypowany
} else {
  const errors = result.error.issues; // ustrukturyzowane informacje o błędach
  console.error(errors);
}
```

Używaj `safeParse()` wszędzie indziej — walidacja formularzy, dane wejściowe API, przepływy widoczne dla użytkownika. Jest szybszy niż try/catch, gdy niepowodzenia są częste, i daje ci ustrukturyzowane dane o błędach bez narzutu wyjątków.

**Moja praktyczna zasada:** `safeParse()` jest domyślnym wyborem. `parse()` jest wyjątkiem.

## Transformacje i pipe'y — tu Zod pokazuje pazur

Walidacja to dopiero początek. Zod potrafi też **transformować** dane w trakcie walidacji:

```typescript
// String na liczbę
const StringToNumber = z.string().transform((val) => parseInt(val, 10));
StringToNumber.parse("42"); // => 42

// Przycięcie i zamiana na małe litery
const NormalizedEmail = z.string().email().transform((val) => val.trim().toLowerCase());

// Parsowanie stringów z datą
const DateString = z.string().transform((val) => new Date(val));

// Złożone transformacje
const ApiResponseSchema = z.object({
  user_name: z.string(),
  created_at: z.string(),
  is_active: z.union([z.boolean(), z.string()]),
}).transform((data) => ({
  userName: data.user_name,
  createdAt: new Date(data.created_at),
  isActive: data.is_active === true || data.is_active === 'true',
}));

type ApiResponse = z.infer<typeof ApiResponseSchema>;
// { userName: string; createdAt: Date; isActive: boolean }
```

### Pipe'y — łączenie walidacji w łańcuch

`.pipe()` pozwala łączyć schematy sekwencyjnie — zwaliduj, przetransformuj, a potem zwaliduj przetransformowany wynik:

```typescript
// Parsuj string jako JSON, a potem zwaliduj strukturę
const JsonConfig = z.string()
  .transform((val) => JSON.parse(val))
  .pipe(z.object({
    host: z.string(),
    port: z.number().int().min(1).max(65535),
    debug: z.boolean().default(false),
  }));

const config = JsonConfig.parse('{"host":"localhost","port":3000}');
// { host: "localhost", port: 3000, debug: false }
```

## Unie dyskryminowane — bezpieczny typowo polimorfizm

Tutaj Zod naprawdę błyszczy. Unie dyskryminowane pozwalają modelować dane, które mogą przyjmować różne kształty w zależności od pola dyskryminującego:

```typescript
const NotificationSchema = z.discriminatedUnion('type', [
  z.object({
    type: z.literal('email'),
    to: z.string().email(),
    subject: z.string(),
    body: z.string(),
  }),
  z.object({
    type: z.literal('sms'),
    phoneNumber: z.string().regex(/^\+\d{10,15}$/),
    message: z.string().max(160),
  }),
  z.object({
    type: z.literal('push'),
    deviceToken: z.string(),
    title: z.string(),
    body: z.string(),
    badge: z.number().int().nonnegative().optional(),
  }),
]);

type Notification = z.infer<typeof NotificationSchema>;

// TypeScript zna dokładny kształt na podstawie `type`
function sendNotification(notification: Notification) {
  switch (notification.type) {
    case 'email':
      // TypeScript wie: to, subject, body
      sendEmail(notification.to, notification.subject, notification.body);
      break;
    case 'sms':
      // TypeScript wie: phoneNumber, message
      sendSMS(notification.phoneNumber, notification.message);
      break;
    case 'push':
      // TypeScript wie: deviceToken, title, body, badge
      sendPush(notification.deviceToken, notification.title, notification.body);
      break;
  }
}
```

W przeciwieństwie do zwykłego `z.union()`, unie dyskryminowane używają klucza dyskryminującego, żeby od razu wybrać właściwy schemat — nie trzeba po kolei próbować każdej opcji. Jest to zarówno szybsze, jak i daje lepsze komunikaty o błędach.

## Własna walidacja z refine() i superRefine()

Gdy wbudowane walidatory nie wystarczają:

```typescript
// Prosta rafinacja
const Password = z.string()
  .min(8, 'Must be at least 8 characters')
  .refine((val) => /[A-Z]/.test(val), 'Must contain an uppercase letter')
  .refine((val) => /[0-9]/.test(val), 'Must contain a number')
  .refine((val) => /[^A-Za-z0-9]/.test(val), 'Must contain a special character');

// Walidacja między polami z superRefine
const RegistrationSchema = z.object({
  password: z.string().min(8),
  confirmPassword: z.string(),
  email: z.string().email(),
}).superRefine((data, ctx) => {
  if (data.password !== data.confirmPassword) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Passwords must match',
      path: ['confirmPassword'],
    });
  }
});

// Asynchroniczna rafinacja — np. sprawdzenie, czy email istnieje w bazie danych
const UniqueEmailSchema = z.string().email().refine(
  async (email) => {
    const exists = await checkEmailInDB(email);
    return !exists;
  },
  { message: 'Email already registered' }
);
```

## Wzorzec z prawdziwego świata: walidacja na granicy API

Tutaj Zod naprawdę zarabia na siebie. Waliduj na granicy, ufaj wszędzie indziej:

```typescript
// schemas/api.ts — zdefiniuj raz, używaj wszędzie
export const CreateOrderSchema = z.object({
  items: z.array(z.object({
    productId: z.string().uuid(),
    quantity: z.number().int().positive().max(100),
    price: z.number().positive(),
  })).min(1, 'Order must have at least one item'),
  shippingAddress: AddressSchema,
  paymentMethod: z.enum(['card', 'paypal', 'bank_transfer']),
  couponCode: z.string().optional(),
  notes: z.string().max(500).optional(),
});

export type CreateOrderInput = z.infer<typeof CreateOrderSchema>;

// api/orders/route.ts — waliduj na granicy API
export async function POST(request: Request) {
  const body = await request.json();
  const result = CreateOrderSchema.safeParse(body);

  if (!result.success) {
    return Response.json(
      { errors: result.error.flatten().fieldErrors },
      { status: 400 }
    );
  }

  // result.data jest w pełni otypowany i zwalidowany
  // Koniec z niespodziankami w runtime
  const order = await createOrder(result.data);
  return Response.json(order, { status: 201 });
}
```

### Spłaszczanie błędów pod API

Metoda `flatten()` w Zodzie zamienia zagnieżdżone obiekty błędów na czysty, przyjazny dla API format:

```typescript
const result = CreateOrderSchema.safeParse(badData);

if (!result.success) {
  const flat = result.error.flatten();
  // {
  //   formErrors: [],
  //   fieldErrors: {
  //     items: ["Order must have at least one item"],
  //     paymentMethod: ["Invalid enum value..."]
  //   }
  // }
}
```

## React Hook Form + Zod + Next.js Server Actions

Zabójcze combo do walidacji formularzy full-stack. Zdefiniuj schemat raz, waliduj zarówno po stronie klienta, jak i serwera:

```typescript
// schemas/contact.ts
import { z } from 'zod';

export const ContactSchema = z.object({
  name: z.string().min(1, 'Name is required').max(100),
  email: z.string().email('Please enter a valid email'),
  subject: z.enum(['general', 'support', 'partnership', 'other']),
  message: z.string().min(10, 'Message must be at least 10 characters').max(2000),
});

export type ContactFormData = z.infer<typeof ContactSchema>;
```

```typescript
// components/ContactForm.tsx
'use client';

import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { ContactSchema, type ContactFormData } from '@/schemas/contact';
import { submitContact } from '@/actions/contact';

export function ContactForm() {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<ContactFormData>({
    resolver: zodResolver(ContactSchema),
  });

  const onSubmit = async (data: ContactFormData) => {
    const result = await submitContact(data);
    if (result.error) {
      // Obsłuż błędy walidacji po stronie serwera
    }
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')} placeholder="Name" />
      {errors.name && <span>{errors.name.message}</span>}

      <input {...register('email')} placeholder="Email" />
      {errors.email && <span>{errors.email.message}</span>}

      <select {...register('subject')}>
        <option value="general">General</option>
        <option value="support">Support</option>
        <option value="partnership">Partnership</option>
        <option value="other">Other</option>
      </select>
      {errors.subject && <span>{errors.subject.message}</span>}

      <textarea {...register('message')} placeholder="Message" />
      {errors.message && <span>{errors.message.message}</span>}

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Sending...' : 'Send'}
      </button>
    </form>
  );
}
```

```typescript
// actions/contact.ts
'use server';

import { ContactSchema } from '@/schemas/contact';

export async function submitContact(data: unknown) {
  // Walidacja po stronie serwera — nigdy nie ufaj klientowi
  const result = ContactSchema.safeParse(data);

  if (!result.success) {
    return { error: result.error.flatten().fieldErrors };
  }

  // dane są zwalidowane i otypowane
  await sendEmail(result.data);
  return { success: true };
}
```

**Kluczowy wniosek:** ten sam `ContactSchema` waliduje zarówno po stronie klienta (natychmiastowy feedback przez React Hook Form), jak i serwera (bezpieczeństwo przez Server Action). Jeśli ktoś ominie klienta, serwer i tak to wyłapie.

## Walidacja zmiennych środowiskowych

Jeden z moich ulubionych wzorców w Zodzie — waliduj całe środowisko przy starcie:

```typescript
// env.ts
import { z } from 'zod';

const EnvSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']),
  DATABASE_URL: z.string().url(),
  API_SECRET: z.string().min(32, 'API_SECRET must be at least 32 characters'),
  REDIS_URL: z.string().url().optional(),
  PORT: z.coerce.number().int().min(1).max(65535).default(3000),
  LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
  ALLOWED_ORIGINS: z.string().transform((val) => val.split(',')).pipe(
    z.array(z.string().url()).min(1)
  ),
});

// Waliduj przy starcie — zawódź szybko
export const env = EnvSchema.parse(process.env);

// Teraz env jest w pełni otypowany i zwalidowany
// env.PORT to liczba, a nie string
// env.ALLOWED_ORIGINS to string[], a nie string
```

Jeśli którejś zmiennej środowiskowej brakuje lub jest źle sformatowana, twoja aplikacja natychmiast crashuje przy starcie z czytelnym komunikatem błędu — a nie 3 godziny później, gdy ta ścieżka kodu wykona się po raz pierwszy.

## Walidacja odpowiedzi API

Nie ufaj zewnętrznym API. Waliduj to, co zwracają:

const GitHubUserSchema = z.object({
  login: z.string(),
  id: z.number(),
  avatar_url: z.string().url(),
  name: z.string().nullable(),

  public_repos: z.number().int(),
  followers: z.number().int(),

async function fetchGitHubUser(username: string): Promise<GitHubUser> {

## Zod v4: co się zmieniło

Zod v4 przyszedł ze znaczącymi usprawnieniami:

- **10-20x szybsze parsowanie** dla większości schematów
- **Mniejszy rozmiar bundla** — lepszy tree-shaking
- **Ulepszone komunikaty o błędach** — domyślnie bardziej pomocne
- **Lepsze unie dyskryminowane** — komponuj je, używaj dyskryminatorów union/pipe
- **`z.interface()`** — dla typów rekurencyjnych bez obejść

```typescript
// Zod v4: typy rekurencyjne z z.interface()
const CategorySchema: z.ZodType<Category> = z.interface({
  name: z.string(),
  children: z.array(z.lazy(() => CategorySchema)).default([]),
});

type Category = z.infer<typeof CategorySchema>;
```

Jeśli wciąż jesteś na v3, migracja jest w większości bezbolesna — API jest kompatybilne wstecz w 95% przypadków użycia.

## Zod kontra Valibot kontra Yup — kiedy czego używać

| Cecha | Zod | Valibot | Yup |
|---------|-----|---------|-----|
| **Rozmiar bundla** | ~17.7 kB | ~1.4 kB | ~12 kB |
| **Inferencja typów** | Doskonała | Doskonała | Dobra |
| **Styl API** | Metody łańcuchowe | Funkcyjny/modularny | Metody łańcuchowe |
| **Ekosystem** | Ogromny | Rosnący | Duży (React/Formik) |
| **Tree-shaking** | Dobry (v4) | Doskonały | Ograniczony |
| **Wydajność** | Szybki (v4) | Bardzo szybki | Umiarkowana |
| **Krzywa uczenia** | Niska | Niska | Niska |

**Wybierz Zod**, gdy potrzebujesz najszerszego wsparcia ekosystemu, najlepszej dokumentacji i pracujesz z tRPC, React Hook Form albo Next.js.

**Wybierz Valibot**, gdy rozmiar bundla jest kluczowy — funkcje edge, aplikacje mocno obciążające klienta, albo gdy potrzebujesz poniżej 2 kB.

**Wybierz Yup**, jeśli twój zespół już go używa z Formikiem, a koszt migracji nie jest uzasadniony.

Dla nowych projektów w 2026 domyślnie wybieram Zod. Samo wsparcie ekosystemu jest warte tych paru dodatkowych kilobajtów. Jeśli używasz TanStack w swoim stacku, Zod świetnie z nim współgra — cały ekosystem opisałem w moim [przewodniku po TanStack](/pl/blog/tanstack-ekosystem-kompletny-przewodnik-2026).

## Typowe pułapki

### 1. Zapominanie o coerce przy danych z formularzy

Formularze HTML zawsze wysyłają stringi. Jeśli spodziewasz się liczby, użyj `z.coerce.number()`:

```typescript
// Źle — dane z formularza to zawsze stringi
const schema = z.object({ age: z.number() });
schema.parse({ age: "25" }); // ZodError!

// Dobrze — wymuś konwersję stringów na liczby
const schema = z.object({ age: z.coerce.number().int().positive() });
schema.parse({ age: "25" }); // { age: 25 } ✓
```

### 2. Optional kontra nullable kontra nullish

```typescript
z.string().optional()  // string | undefined
z.string().nullable()  // string | null
z.string().nullish()   // string | null | undefined

// One się różnią! Wybierz właściwą dla swoich danych.
```

### 3. Schematy obiektów domyślnie usuwają nieznane klucze

```typescript
const schema = z.object({ name: z.string() });
schema.parse({ name: 'Seb', age: 30 }); // { name: 'Seb' } — age zostaje usunięte!

// Aby zachować nieznane klucze:
schema.passthrough().parse({ name: 'Seb', age: 30 }); // { name: 'Seb', age: 30 }

// Aby odrzucić nieznane klucze:
schema.strict().parse({ name: 'Seb', age: 30 }); // ZodError!
```

### 4. Zamieszanie wokół formatowania błędów

```typescript
const result = schema.safeParse(badData);
if (!result.success) {
  // Dla API: płaski obiekt z nazwami pól jako kluczami
  result.error.flatten();

  // Dla formularzy: format zagnieżdżony pasujący do struktury obiektu
  result.error.format();

  // Do logowania: tablica pojedynczych problemów
  result.error.issues;
}
```

## Praktyczne wnioski

1. **Definiuj schematy na granicy** — trasy API, obsługa formularzy, wczytywanie konfiguracji. Waliduj raz, ufaj wewnętrznie.
2. **Wnioskuj typy ze schematów** — nigdy nie utrzymuj typów i schematów osobno.
3. **Domyślnie używaj `safeParse()`** — `parse()` tylko wtedy, gdy niepoprawne dane są naprawdę wyjątkowe.
4. **Waliduj zmienne środowiskowe przy starcie** — zawódź szybko, a nie o 3 nad ranem.
5. **Waliduj odpowiedzi zewnętrznych API** — dostawcy zmienią swoje API bez uprzedzenia.
6. **Używaj unii dyskryminowanych** dla danych polimorficznych — są szybsze i dają lepsze błędy niż zwykłe unie.
7. **Łącz z React Hook Form** dla walidacji full-stack z jednym schematem.
8. **Rozważ Valibot**, jeśli rozmiar bundla jest twoim głównym zmartwieniem.

Zod to nie tylko biblioteka walidacyjna — to most między obietnicami TypeScript z czasu kompilacji a rzeczywistością runtime. Połącz go z [zasadami czystego kodu](/pl/blog/zasady-czystego-kodu-frontend-dry-kiss-solid) i [zaawansowanymi wzorcami hooków w React](/pl/blog/zaawansowane-hooki-react-najlepsze-praktyki), a masz solidny fundament dla każdego projektu TypeScript. Jeśli nie używasz go na granicach swojego systemu, ufasz danym, którym ufać nie powinieneś.
