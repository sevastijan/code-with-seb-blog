Buduję z Claude API od ponad roku. Własne prompty, łańcuchy tool-use, logika ponawiania prób, obsługa streamingu — cała warstwa hydrauliki, którą piszesz, gdy chcesz, żeby LLM faktycznie coś robił. To działa, ale każdy projekt zaczyna się od tych samych 400 linii boilerplate'u, zanim dotrzesz do ciekawej części.

Potem Anthropic wypuścił Claude Agent SDK i usunąłem większość tego kodu.

Agent SDK to nie kolejny wrapper wokół Messages API. To ten sam runtime, który napędza Claude Code — te same implementacje narzędzi, ten sam model uprawnień, ta sama pętla agentowa — udostępniony jako biblioteka TypeScript, którą instalujesz w swoich projektach. Jeśli używałeś Claude Code i marzyłeś, żeby móc go skryptować, to dokładnie to. Wspólnym fundamentem pod tym wszystkim jest Bun, który [Anthropic przejął w grudniu 2025](/pl/blog/anthropic-przejecie-bun-strategia-claude-code) właśnie dlatego, że zarówno dystrybucja Claude Code, jak i historia runtime'u Agent SDK zależą od niego.

## Czym właściwie jest Agent SDK

Claude Code to interaktywny agent CLI. Wpisujesz prompt, on rozumuje, wywołuje narzędzia, edytuje pliki, uruchamia komendy. Agent SDK wyciąga silnik stojący za tym doświadczeniem i daje Ci nad nim programową kontrolę. (Jeśli chcesz, żeby Anthropic hostował pętlę i sandbox zamiast Ciebie, to inny produkt — [Managed Agents z Fable 5 do wielodniowych przebiegów](/pl/blog/fable-5-managed-agents-wielodniowe-agenty) omówiłem osobno.)

Dostajesz:

- **Te same wbudowane narzędzia** — Read, Write, Edit, Bash, Glob, Grep, WebSearch, WebFetch, Monitor, Agent
- **Tę samą pętlę agentową** — model decyduje, które narzędzia wywołać, przetwarza wyniki i iteruje, aż zadanie jest gotowe
- **Ten sam model uprawnień** — tryby sandbox, listy allow/deny, callbacki zatwierdzające
- **Podejście TypeScript-first** — pełne bezpieczeństwo typów, async/await, wsparcie dla streamingu

Model mentalny: Claude Code to GUI. Agent SDK to API. Ten sam silnik, inny interfejs.

## Instalacja i szybki start

```bash
npm install @anthropic-ai/claude-agent-sdk
```

Potrzebujesz klucza Anthropic API. Ustaw go jako zmienną środowiskową albo przekaż bezpośrednio:

```typescript
import { Agent } from '@anthropic-ai/claude-agent-sdk';

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  apiKey: process.env.ANTHROPIC_API_KEY,
});

const result = await agent.run('Read the README.md and summarize it');
console.log(result.text);
```

To kompletny, działający agent. Potrafi czytać pliki, rozumować o nich i zwrócić ustrukturyzowaną odpowiedź. Metoda `run` uruchamia pętlę agentową — model wywołuje narzędzia w miarę potrzeb, przetwarza wyniki i kończy, gdy ma ostateczną odpowiedź.

Dla streamingu (który będzie Ci potrzebny na produkcji do śledzenia postępów):

```typescript
const stream = agent.stream('Analyze the src/ directory and find security issues');

for await (const event of stream) {
  if (event.type === 'tool_use') {
    console.log(`Using tool: ${event.tool}`);
  }
  if (event.type === 'text') {
    process.stdout.write(event.content);
  }
}
```

## Wbudowane narzędzia: co dostajesz od razu

SDK jest dostarczany z tym samym zestawem narzędzi, który sprawia, że Claude Code jest skuteczny. Oto co robi każde z nich i kiedy model po nie sięga:

| Narzędzie | Cel | Kiedy się uruchamia |
|------|---------|---------------|
| **Read** | Odczyt zawartości plików | Analiza kodu, konfiguracji, dokumentacji |
| **Write** | Tworzenie nowych plików | Generowanie kodu, konfiguracji |
| **Edit** | Modyfikacja istniejących plików | Refaktoryzacja, naprawa błędów |
| **Bash** | Wykonywanie komend powłoki | Uruchamianie testów, buildów, git |
| **Glob** | Wyszukiwanie plików po wzorcu | Odkrywanie struktury projektu |
| **Grep** | Przeszukiwanie zawartości plików | Znajdowanie użyć, wzorców |
| **WebSearch** | Przeszukiwanie sieci | Pobieranie dokumentacji, research |
| **WebFetch** | Pobieranie adresu URL | Czytanie stron WWW, API |
| **Monitor** | Obserwowanie procesów | Długotrwałe buildy, serwery |
| **Agent** | Uruchamianie subagentów | Praca równoległa, wyspecjalizowane zadania |

Nie musisz ich rejestrować. Są dostępne domyślnie. Każde z nich możesz wyłączyć:

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  disabledTools: ['Bash', 'Write', 'Edit'], // agent tylko do odczytu
});
```

Tak właśnie budujesz bezpiecznego agenta analitycznego, który może eksplorować bazę kodu, ale nigdy jej nie modyfikuje. Używam tego wzorca do botów do code review — potrzebują Read, Glob i Grep, ale nigdy nie powinny dotykać pliku.

## Budowanie własnych narzędzi

Tutaj SDK robi się interesujący. Własne narzędzia rejestruje się jako serwery [MCP](/pl/blog/mcp-vs-a2a-protokoly-agentow-ai-2026) działające w procesie. Nie musisz uruchamiać osobnego procesu ani konfigurować endpointów HTTP — SDK hostuje serwer MCP wewnątrz Twojego procesu Node.js.

```typescript
import { Agent, Tool } from '@anthropic-ai/claude-agent-sdk';

const jiraLookup: Tool = {
  name: 'jira_lookup',
  description: 'Look up a Jira ticket by ID and return its details',
  inputSchema: {
    type: 'object',
    properties: {
      ticketId: { type: 'string', description: 'Jira ticket ID, e.g. PROJ-123' },
    },
    required: ['ticketId'],
  },
  async execute({ ticketId }) {
    const response = await fetch(
      `https://your-org.atlassian.net/rest/api/3/issue/${ticketId}`,
      {
        headers: {
          Authorization: `Basic ${Buffer.from(`${JIRA_EMAIL}:${JIRA_TOKEN}`).toString('base64')}`,
        },
      }
    );
    const issue = await response.json();
    return {
      summary: issue.fields.summary,
      status: issue.fields.status.name,
      assignee: issue.fields.assignee?.displayName ?? 'Unassigned',
      description: issue.fields.description,
    };
  },
};

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  tools: [jiraLookup],
});

await agent.run('Look up PROJ-456 and tell me if it is ready for code review');
```

Model widzi Twoje własne narzędzie obok wbudowanych i decyduje, kiedy go użyć, na podstawie opisu. Dobre opisy mają znaczenie — to jedyna dokumentacja, jaką model ma dla Twojego narzędzia.

Możesz zarejestrować tyle własnych narzędzi, ile potrzebujesz. Mam agenty z 15+ własnymi narzędziami, które łączą się ze Slackiem, GitHubem, Jirą, wewnętrznymi API i bazami danych. Model radzi sobie z doborem narzędzi zaskakująco dobrze, o ile opisy są jasne i nie zachodzą na siebie.

## Uruchamianie subagentów do pracy równoległej

Jednym z najpotężniejszych wzorców w SDK jest uruchamianie subagentów do pracy równoległej. Jeśli czytałeś mój przewodnik o [subagentach Claude Code i orkiestracji wieloagentowej](/pl/blog/claude-code-subagenty-systemy-wieloagentowe), wiesz, że wzorzec orkiestrator-worker potrafi dać ponad 90% wzrostu wydajności przy złożonych zadaniach. SDK czyni ten wzorzec programowalnym.

```typescript
const orchestrator = new Agent({
  model: 'claude-opus-4-20250514',
  systemPrompt: `You are a code review orchestrator.
    Break the review into parallel tasks and delegate to sub-agents.
    Synthesize their findings into a single report.`,
  maxSubAgents: 5,
  subAgentConfig: {
    model: 'claude-sonnet-4-20250514',
    disabledTools: ['Write', 'Edit', 'Bash'], // workery tylko do odczytu
  },
});
```

Orkiestrator uruchamia subagenty za pomocą wbudowanego narzędzia Agent. Każdy subagent dostaje własne okno kontekstowe, działa niezależnie i zwraca wyniki do orkiestratora. Model, narzędzia i limity dla subagentów kontrolujesz osobno od orkiestratora.

To ma znaczący wpływ na koszty: Twój orkiestrator używa Opusa do strategicznego rozumowania, ale workery działają na Sonnecie do mechanicznej pracy analitycznej. To ta sama zasada `opusplan`, którą omówiłem w moim [przewodniku o optymalizacji tokenów](/pl/blog/claude-code-optymalizacja-tokenow-rachunek) — planowanie klasy Opus, wykonanie klasy Sonnet.

## Hooki do zdarzeń cyklu życia

SDK udostępnia hooki cyklu życia, które pozwalają Ci przechwytywać i modyfikować zachowanie agenta na każdym etapie. Jeśli znasz [hooki Claude Code](/pl/blog/claude-code-hooki-custom-agents-pipeline-ai), hooki SDK działają według tego samego modelu zdarzeń, ale z interfejsem callbacków TypeScript zamiast skryptów powłoki.

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  hooks: {
    onToolUse: async (event) => {
      // Loguj każde wywołanie narzędzia do swojego stacku obserwowalności
      await telemetry.track('tool_use', {
        tool: event.toolName,
        input: event.input,
        sessionId: event.sessionId,
      });

      // Blokuj niebezpieczne operacje
      if (event.toolName === 'Bash' && event.input.command.includes('rm -rf')) {
        return { decision: 'deny', reason: 'Destructive command blocked by policy' };
      }

      return { decision: 'allow' };
    },

    onToolResult: async (event) => {
      // Dodaj kontekst po wykonaniu narzędzia
      if (event.toolName === 'Read' && event.result.includes('TODO')) {
        return {
          additionalContext: 'Note: this file contains TODO comments that may need attention.',
        };
      }
    },

    onAgentStop: async (event) => {
      // Przetwarzanie końcowe, gdy agent kończy pracę
      await saveReport(event.finalOutput);
    },
  },
});
```

Hooki to sposób, w jaki dodajesz zabezpieczenia, obserwowalność i własną logikę bez modyfikowania podstawowego zachowania agenta. Używam ich intensywnie do:

- **Logowania audytowego** — każde wywołanie narzędzia rejestrowane z wejściami i wyjściami
- **Śledzenia kosztów** — liczba tokenów na sesję i na narzędzie
- **Bramek bezpieczeństwa** — blokowanie zapisów do plików w określonych katalogach
- **Auto-formatowania** — uruchamianie Prettiera po każdym Write lub Edit

## Integracja z serwerami MCP

Poza narzędziami działającymi w procesie możesz łączyć się z zewnętrznymi serwerami MCP. Tak właśnie integrujesz się z szerszym [ekosystemem MCP](/pl/blog/mcp-vs-a2a-protokoly-agentow-ai-2026) — bazami danych, platformami SaaS, usługami wewnętrznymi, które już udostępniają endpointy MCP.

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  mcpServers: [
    {
      name: 'postgres',
      command: 'npx',
      args: ['-y', '@modelcontextprotocol/server-postgres'],
      env: {
        POSTGRES_CONNECTION_STRING: process.env.DATABASE_URL,
      },
    },
    {
      name: 'github',
      command: 'npx',
      args: ['-y', '@modelcontextprotocol/server-github'],
      env: {
        GITHUB_TOKEN: process.env.GITHUB_TOKEN,
      },
    },
  ],
});

await agent.run('Check the latest PRs on our repo and summarize any that touch the auth module');
```

SDK zarządza cyklem życia serwerów MCP — uruchamia je, utrzymuje połączenia, zamyka je, gdy agent kończy pracę. Ty deklarujesz serwery, SDK zajmuje się resztą.

Jest to szczególnie potężne w połączeniu z własnymi narzędziami. Twoje narzędzia działające w procesie obsługują logikę specyficzną dla biznesu (wyszukiwania w Jirze, powiadomienia Slack), podczas gdy serwery MCP obsługują znormalizowane integracje (bazy danych, systemy plików, API).

## Przykład produkcyjny: agent do code review

Pokażę Ci prawdziwego agenta, którego uruchamiam w CI. Recenzuje każdy PR, sprawdza kwestie bezpieczeństwa, weryfikuje pokrycie testami i publikuje komentarz podsumowujący na GitHubie.

```typescript
import { Agent, Tool } from '@anthropic-ai/claude-agent-sdk';

const postGithubComment: Tool = {
  name: 'post_review_comment',
  description: 'Post a code review comment on a GitHub pull request',
  inputSchema: {
    type: 'object',
    properties: {
      prNumber: { type: 'number' },
      body: { type: 'string' },
      repo: { type: 'string' },
    },
    required: ['prNumber', 'body', 'repo'],
  },
  async execute({ prNumber, body, repo }) {
    const response = await fetch(
      `https://api.github.com/repos/${repo}/issues/${prNumber}/comments`,
      {
        method: 'POST',
        headers: {
          Authorization: `Bearer ${process.env.GITHUB_TOKEN}`,
          'Content-Type': 'application/json',
        },
        body: JSON.stringify({ body }),
      }
    );
    return { success: response.ok, status: response.status };
  },
};

const reviewer = new Agent({
  model: 'claude-sonnet-4-20250514',
  systemPrompt: `You are a senior code reviewer. For each PR:
    1. Read the changed files using Read and Grep
    2. Check for security vulnerabilities (SQL injection, XSS, auth bypasses)
    3. Verify test coverage exists for new functionality
    4. Check for performance anti-patterns
    5. Post a structured review comment using post_review_comment

    Format your review as:
    ## Summary
    ## Security
    ## Testing
    ## Suggestions

    Be concise. Flag issues, not style preferences.`,
  tools: [postGithubComment],
  disabledTools: ['Write', 'Edit'], // nigdy nie modyfikuj kodu
  hooks: {
    onToolUse: async (event) => {
      if (event.toolName === 'Bash') {
        const allowed = ['git diff', 'git log', 'git show', 'npm test'];
        const isAllowed = allowed.some((cmd) => event.input.command.startsWith(cmd));
        if (!isAllowed) {
          return { decision: 'deny', reason: 'Only git and npm test commands are allowed' };
        }
      }
      return { decision: 'allow' };
    },
  },
});

// Wywoływane z GitHub Actions
const prNumber = parseInt(process.env.PR_NUMBER!);
const repo = process.env.GITHUB_REPOSITORY!;

await reviewer.run(
  `Review PR #${prNumber} in ${repo}. Run git diff origin/main...HEAD to see changes. Then post your review.`
);
```

Ten agent działa w mniej niż 60 sekund dla typowych PR-ów. Całkowity koszt na jedną recenzję: mniej więcej 0,03–0,08 USD w zależności od rozmiaru diffa. Przy 20 PR-ach dziennie to około 1,20 USD dziennie za automatyczne code review, które wyłapuje realne problemy.

Kluczowe decyzje projektowe:

1. **Tylko do odczytu** — Write i Edit są wyłączone. Agent nigdy nie może modyfikować kodu.
2. **Whitelist dla Bash** — dozwolone są tylko komendy git i test, przez hook.
3. **Własne narzędzie do outputu** — zamiast zapisywać do pliku, publikuje bezpośrednio na GitHubie.
4. **Sonnet, nie Opus** — code review nie potrzebuje najdroższego modelu. Sonnet wyłapuje te same kategorie problemów za ułamek kosztu.

## Kwestie kosztowe

Sam SDK jest darmowy i open source. Płacisz za tokeny Anthropic API, tak samo jak przy bezpośrednim korzystaniu z API. Ale wbudowane narzędzia SDK dodają narzut tokenowy — opisy narzędzi zużywają tokeny wejściowe, a wyniki narzędzi dokładają się do kontekstu.

Praktyczne przedziały kosztów, jakie widziałem na produkcji:

| Przypadek użycia | Model | Typowy koszt |
|----------|-------|-------------|
| Analiza plików (5-10 plików) | Sonnet | 0,01–0,05 USD |
| Code review (diff PR-a) | Sonnet | 0,03–0,08 USD |
| Refaktoryzacja wielu plików | Sonnet | 0,10–0,30 USD |
| Research + raport | Opus | 0,50–2,00 USD |
| Orkiestracja wieloagentowa | Opus + Sonnet | 0,20–1,00 USD |

Największą dźwignią kosztową jest wybór modelu. Używaj Opusa do orkiestracji i złożonego rozumowania. Używaj Sonneta do wszystkiego innego. Jeśli chcesz pełny playbook optymalizacji kosztów, mój [przewodnik o optymalizacji tokenów](/pl/blog/claude-code-optymalizacja-tokenow-rachunek) omawia dziesięć konkretnych zmian, które obcięły mój rachunek za Claude Code o 70% — większość tych technik stosuje się bezpośrednio także do korzystania z SDK.

Inne wskazówki kosztowe specyficzne dla SDK:

- **Wyłączaj nieużywane narzędzia** — każdy opis narzędzia dokłada ~200-500 tokenów wejściowych. Wyłącz narzędzia, których agent nie będzie potrzebował.
- **Limituj subagenty** — ustaw `maxSubAgents`, aby zapobiec niekontrolowanemu rozmnażaniu.
- **Używaj streamingu** — streamuj wyniki, aby wcześnie wykryć, kiedy agent schodzi z toru, i wtedy anuluj.
- **Cache'uj prompty systemowe** — SDK wspiera cache'owanie promptów systemowych, co redukuje koszty wejściowe nawet o 90% przy powtarzanych przebiegach.

## Kiedy używać Agent SDK, kiedy Claude Code CLI, a kiedy API bezpośrednio

Oto framework decyzyjny, którego używam:

**Używaj Agent SDK, gdy:**

- Budujesz zautomatyzowany pipeline (CI/CD, zadania harmonogramowane, webhooki)
- Potrzebujesz własnych narzędzi łączących się z Twoją infrastrukturą
- Chcesz programowej kontroli nad zachowaniem agenta (hooki, uprawnienia, konfiguracja subagentów)
- Osadzasz agenty AI w większej aplikacji
- Musisz uruchamiać agenty w środowiskach, gdzie CLI nie jest dostępne

**Używaj Claude Code CLI, gdy:**

- Pracujesz interaktywnie w terminalu
- Chcesz pełnego doświadczenia w stylu IDE (UI uprawnień, historia konwersacji, slash commands)
- Prototypujesz przepływy agentowe, zanim skodyfikujesz je w SDK
- Potrzebujesz [systemu skills](/pl/blog/claude-code-skills-wielokrotne-workflow-ai) do wielokrotnego użytku szablonów promptów

**Używaj API bezpośrednio, gdy:**

- Potrzebujesz drobnoziarnistej kontroli nad każdą wiadomością w konwersacji
- Budujesz interfejs czatu, w którym sam zarządzasz historią wiadomości
- W ogóle nie potrzebujesz tool use (czyste generowanie tekstu)
- Pracujesz w języku innym niż TypeScript/JavaScript
- Potrzebujesz funkcji, których SDK jeszcze nie udostępnia

SDK znajduje się pomiędzy CLI a surowym API. Daje Ci moc systemu narzędzi Claude Code bez interaktywnego interfejsu terminala i oszczędza Ci reimplementacji pętli agentowej, której wymaga surowe API.

## Jak zacząć

Jeśli już używasz Claude Code, przeskok mentalny do SDK jest niewielki. Zacznij od prostego agenta tylko do odczytu, dodaj jedno własne narzędzie, a potem buduj od tego dalej.

```typescript
import { Agent } from '@anthropic-ai/claude-agent-sdk';

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  disabledTools: ['Write', 'Edit', 'Bash'],
});

const result = await agent.run(
  'Read the package.json and tell me which dependencies are outdated'
);

console.log(result.text);
```

Uruchom go. Zobacz, co się stanie. Potem dodaj hook. Potem dodaj własne narzędzie. Potem uruchom subagenta. SDK nagradza stopniowe wdrażanie.
