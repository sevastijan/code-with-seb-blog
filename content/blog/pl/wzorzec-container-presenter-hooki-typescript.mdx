"Nie potrzebujemy już Container/Presenter. Hooki go zastąpiły." Słyszę to przynajmniej raz w miesiącu. To jedna z tych półprawd, które brzmią mądrze, a prowadzą do fatalnej architektury.

To, co zastąpiły Hooki, to **implementacja oparta na klasach** — klasa `ContainerComponent` z `componentDidMount` i render propsami. Czego Hooki nie zastąpiły, to **zasada**: oddzielenie pobierania danych i logiki od renderowania i prezentacji. To rozdzielenie liczy się dziś bardziej niż w 2018 roku — szczególnie przy React Server Components, gdzie granica serwer/klient wymusza podobny podział.

Wzorzec ewoluował. Kontenery stały się custom hookami. Prezentery stały się czystymi, typowanymi komponentami. Efekt jest czystszy niż oryginał.

## Klasyczny wzorzec (to, co mieliśmy)

```typescript
// 2018: kontener oparty na klasie
class UserListContainer extends React.Component {
  state = { users: [], loading: true };

  componentDidMount() {
    fetchUsers().then(users => this.setState({ users, loading: false }));
  }

  render() {
    return <UserListPresenter users={this.state.users} loading={this.state.loading} />;
  }
}

// Presenter: czyste renderowanie, brak logiki danych
function UserListPresenter({ users, loading }) {
  if (loading) return <Spinner />;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
```

## Nowoczesny wzorzec: custom hooki jako kontenery

"Kontenerem" jest teraz custom hook, który enkapsuluje pobieranie danych i logikę biznesową. "Prezenterem" jest czysty komponent, który przyjmuje typowane propsy:

```typescript
// hooks/useUserList.ts — KONTENER
import { useQuery } from '@tanstack/react-query';

interface UseUserListOptions {
  role?: string;
  limit?: number;
}

export function useUserList(options: UseUserListOptions = {}) {
  const { role, limit = 20 } = options;

  const query = useQuery({
    queryKey: ['users', { role, limit }],
    queryFn: () => fetchUsers({ role, limit }),
  });

  const sortedUsers = useMemo(
    () => query.data?.sort((a, b) => a.name.localeCompare(b.name)) ?? [],
    [query.data]
  );

  return {
    users: sortedUsers,
    isLoading: query.isLoading,
    error: query.error,
    refetch: query.refetch,
    isEmpty: !query.isLoading && sortedUsers.length === 0,
  };
}
```

```typescript
// components/UserList.tsx — PREZENTER
interface UserListProps {
  users: User[];
  isLoading: boolean;
  isEmpty: boolean;
  error: Error | null;
  onRefresh?: () => void;
}

export function UserList({ users, isLoading, isEmpty, error, onRefresh }: UserListProps) {
  if (isLoading) return <UserListSkeleton />;
  if (error) return <ErrorCard message={error.message} onRetry={onRefresh} />;
  if (isEmpty) return <EmptyState message="No users found" />;

  return (
    <ul className="space-y-2">
      {users.map((user) => (
        <li key={user.id} className="flex items-center gap-3 p-3 rounded-lg bg-gray-800">
          <Avatar src={user.avatar} alt={user.name} size="sm" />
          <div>
            <p className="font-medium">{user.name}</p>
            <p className="text-sm text-gray-400">{user.email}</p>
          </div>
          <Badge variant={user.role} className="ml-auto">{user.role}</Badge>
        </li>
      ))}
    </ul>
  );
}
```

```typescript
// pages/UsersPage.tsx — KOMPOZYCJA
export function UsersPage() {
  const { users, isLoading, isEmpty, error, refetch } = useUserList({ role: 'active' });

  return (
    <div>
      <h1>Team Members</h1>
      <UserList
        users={users}
        isLoading={isLoading}
        isEmpty={isEmpty}
        error={error}
        onRefresh={refetch}
      />
    </div>
  );
}
```

### Dlaczego to lepsze niż oryginał

1. **Hook jest reużywalny** — używaj `useUserList` w dowolnym komponencie, nie tylko w jednym kontenerze
2. **Prezenter jest testowalny** — przekazujesz propsy, sprawdzasz wynik, bez mockowania
3. **TypeScript wymusza kontrakt** — interfejs JEST dokumentacją
4. **Prezenter działa w Storybooku** — bo to po prostu propsy → JSX
5. **Wiele hooków się komponuje** — połącz `useUserList`, `useFilters`, `usePagination` na jednej stronie

## Wzorzec z Server Components

React Server Components w naturalny sposób wymuszają podział Container/Presenter:

```typescript
// app/users/page.tsx — SERVER COMPONENT (kontener)
// To JEST kontener — pobiera dane na serwerze
export default async function UsersPage() {
  const users = await fetchUsers({ role: 'active', limit: 50 });

  // Prezenter to komponent kliencki obsługujący interaktywność
  return (
    <div>
      <h1>Team Members</h1>
      <UserList users={users} />
    </div>
  );
}
```

```typescript
// components/UserList.tsx — CLIENT COMPONENT (prezenter)
'use client';

export function UserList({ users }: { users: User[] }) {
  const [search, setSearch] = useState('');
  const filtered = users.filter(u =>
    u.name.toLowerCase().includes(search.toLowerCase())
  );

  return (
    <div>
      <input
        value={search}
        onChange={(e) => setSearch(e.target.value)}
        placeholder="Search users..."
      />
      <ul>
        {filtered.map(user => <UserCard key={user.id} user={user} />)}
      </ul>
    </div>
  );
}
```

Server Component pobiera dane (kontener). Client Component renderuje i obsługuje interaktywność (prezenter). Granica `'use client'` JEST granicą Container/Presenter.

## Zaawansowane: wzorzec Headless Component

Pójdźmy dalej — prezenter nie renderuje nawet markup. Dostarcza zachowanie poprzez render propsy lub hooki:

```typescript
// Headless component — dostarcza logikę sortowania, bez UI
function useSortableList<T>(items: T[], defaultKey: keyof T) {
  const [sortKey, setSortKey] = useState<keyof T>(defaultKey);
  const [direction, setDirection] = useState<'asc' | 'desc'>('asc');

  const sorted = useMemo(() => {
    const sorted = [...items].sort((a, b) => {
      const aVal = a[sortKey];
      const bVal = b[sortKey];
      if (aVal < bVal) return direction === 'asc' ? -1 : 1;
      if (aVal > bVal) return direction === 'asc' ? 1 : -1;
      return 0;
    });
    return sorted;
  }, [items, sortKey, direction]);

  const toggleSort = (key: keyof T) => {
    if (key === sortKey) {
      setDirection(d => d === 'asc' ? 'desc' : 'asc');
    } else {
      setSortKey(key);
      setDirection('asc');
    }
  };

  return { sorted, sortKey, direction, toggleSort };
}

// Użycie — prezenter dostarcza własny markup
function UserTable({ users }: { users: User[] }) {
  const { sorted, sortKey, direction, toggleSort } = useSortableList(users, 'name');

  return (
    <table>
      <thead>
        <tr>
          <SortHeader label="Name" active={sortKey === 'name'} direction={direction}
            onClick={() => toggleSort('name')} />
          <SortHeader label="Email" active={sortKey === 'email'} direction={direction}
            onClick={() => toggleSort('email')} />
          <SortHeader label="Role" active={sortKey === 'role'} direction={direction}
            onClick={() => toggleSort('role')} />
        </tr>
      </thead>
      <tbody>
        {sorted.map(user => (
          <tr key={user.id}>
            <td>{user.name}</td>
            <td>{user.email}</td>
            <td>{user.role}</td>
          </tr>
        ))}
      </tbody>
    </table>
  );
}
```

## Kiedy stosować każdy z wariantów

| Scenariusz | Wzorzec | Dlaczego |
|----------|---------|-----|
| **Dane z API + UI** | Custom hook + komponent | Hook obsługuje TanStack Query, komponent jest czysty |
| **Dane renderowane na serwerze** | RSC + Client Component | RSC pobiera, Client obsługuje interaktywność |
| **Złożona logika biznesowa** | Hook wyodrębnia logikę | Komponenty skupione na renderowaniu |
| **Reużywalne zachowanie** | Headless hook | Sortowanie, filtrowanie, paginacja — reużywalne w każdym UI |
| **Storybook/testowanie** | Prezenter tylko na propsach | Łatwy do testowania i dokumentowania |

## Praktyczne wnioski

1. **Wzorzec nie umarł — ewoluował.** Kontenery to hooki, prezentery to typowane komponenty.
2. **Custom hooki SĄ kontenerami.** Enkapsulują pobieranie danych, transformacje i logikę biznesową.
3. **Prezentery przyjmują propsy, zwracają JSX.** Bez hooków (poza tymi wyłącznie UI, jak `useState` dla lokalnego stanu).
4. **Server Components są naturalnymi kontenerami.** Granica `'use client'` wymusza podział.
5. **Typuj interfejs prezentera jawnie.** To kontrakt między logiką a renderowaniem.
6. **Testuj prezentery propsami, nie mockami.** Jeśli musisz mockować hook, twój prezenter ma za dużo logiki.
7. **Headless hooki do reużywalnego zachowania.** Wzorce sortowania, paginacji i filtrowania działają w każdym UI.
8. **Komponuj hooki, nie zagnieżdżaj kontenerów.** Strona łączy `useUsers` + `useFilters` + `usePagination` — a nie zagnieżdżone komponenty-kontenery.

Wzorzec Container/Presenter to najbardziej niedoceniany wzorzec architektoniczny w Reakcie. Nie jest efektowny, nie wymaga biblioteki i nie ma logo. Ale sprawia, że kodbazy są testowalne, czytelne i łatwe do refaktoryzacji — a to właśnie liczy się naprawdę, gdy utrzymujesz kod przez lata.
