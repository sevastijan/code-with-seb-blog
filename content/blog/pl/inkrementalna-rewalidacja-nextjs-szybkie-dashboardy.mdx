W zeszłym kwartale zbudowałem dashboard, który ładował się 3,2 sekundy. Dane pochodziły z czterech różnych API, a najwolniejsze odpowiadało po 1,8 sekundy. Każde załadowanie strony czekało na wszystkie cztery, zanim cokolwiek się wyrenderowało. Użytkownicy narzekali. PM wpadł w panikę. Zarząd zaczął się zastanawiać, dlaczego wybraliśmy Next.js.

Dwa dni później ten sam dashboard ładował się w 180ms. Te same dane, te same API, ten sam sprzęt. Różnicą było zrozumienie, jak naprawdę działa cache w Next.js — a konkretnie, jak Incremental Static Regeneration zamienia wolne wywołania API w natychmiastowe odpowiedzi.

Sztuczka jest zwodniczo prosta: serwuj zbuforowaną stronę natychmiast, a cache aktualizuj w tle. Użytkownik widzi TTFB na poziomie 180ms zamiast 3,2 sekundy. Dane mogą być sprzed 60 sekund, ale dla dashboardu pokazującego metryki analityczne jest to całkowicie akceptowalne. A gdy dane naprawdę muszą być świeże — na przykład po akcji użytkownika — rewalidacja on-demand aktualizuje cache natychmiast.

To właśnie ten wzorzec sprawia, że dashboardy w Next.js wydają się aplikacjami desktopowymi. Pokażę Ci dokładnie, jak to działa.

## Cztery warstwy cache w Next.js

Zanim zagłębimy się w ISR, musisz zrozumieć architekturę cache'owania. Next.js ma cztery odrębne warstwy cache i działają one wspólnie:

```
Browser (Client)
  │
  ▼
┌──────────────────┐
│  Router Cache    │ ← Po stronie klienta, w pamięci. Buforuje payloady RSC
│  (Client)        │   dla nawigacji. Czyszczony po odświeżeniu lub timeoucie.
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  Full Route Cache│ ← Po stronie serwera. Przechowuje kompletny HTML + payload RSC
│  (Server)        │   dla tras statycznych. Czyszczony przy redeployu.
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  Data Cache      │ ← Po stronie serwera. Buforuje odpowiedzi fetch() między
│  (Server)        │   żądaniami. Przetrwa redeploy.
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  Request         │ ← Per-żądanie. Deduplikuje identyczne wywołania fetch()
│  Memoization     │   w obrębie pojedynczego przebiegu renderowania.
└──────────────────┘
```

**Dlaczego to ważne dla dashboardów:** Gdy użytkownik przechodzi do Twojego dashboardu, Next.js może zaserwować Full Route Cache (natychmiast), który został zbudowany z Data Cache (wcześniej pobrane odpowiedzi API). Użytkownik od razu widzi w pełni wyrenderowaną stronę. W tle ISR rewaliduje Data Cache, co wyzwala nowy Full Route Cache. Kolejny odwiedzający dostaje zaktualizowaną wersję.

## ISR w App Routerze: podstawy

W App Routerze ISR jest kontrolowany przez opcje fetcha oraz konfigurację segmentu trasy:

### Rewalidacja czasowa

```typescript
// app/dashboard/page.tsx
export const revalidate = 60; // Rewaliduj co 60 sekund

async function getMetrics() {
  const res = await fetch('https://api.example.com/metrics', {
    next: { revalidate: 60 }, // Rewalidacja na poziomie fetcha
  });
  return res.json();
}

async function getUsers() {
  const res = await fetch('https://api.example.com/users/stats', {
    next: { revalidate: 300 }, // Inny TTL: 5 minut
  });
  return res.json();
}

export default async function DashboardPage() {
  const [metrics, users] = await Promise.all([
    getMetrics(),
    getUsers(),
  ]);

  return (
    <div>
      <MetricsGrid data={metrics} />
      <UserStatsTable data={users} />
    </div>
  );
}
```

**Jak to działa:**

1. **Pierwsze żądanie:** Cache MISS → pobranie z API → renderowanie strony → zapis do Data Cache + Full Route Cache
2. **Kolejne żądanie (w ciągu 60s):** Cache HIT → serwowanie z Full Route Cache natychmiast (< 100ms TTFB)
3. **Żądanie po 60s:** Cache STALE → serwowanie nieświeżej treści natychmiast → rewalidacja w tle
4. **Kolejne żądanie po rewalidacji:** Cache HIT → serwowanie świeżej treści

Kluczowy wniosek: **użytkownicy zawsze dostają natychmiastową odpowiedź.** Otrzymują albo świeże dane (HIT), albo lekko nieświeże dane (STALE). Nigdy nie czekają na wywołanie API.

### Rewalidacja on-demand

Rewalidacja czasowa to zabezpieczenie na wszelki wypadek. Rewalidacja on-demand jest wtedy, gdy potrzebujesz natychmiastowej świeżości — po akcji użytkownika, webhooku lub mutacji danych.

#### revalidateTag — chirurgiczna inwalidacja cache

Otaguj swoje fetche, a potem inwaliduj po tagu:

```typescript
// app/dashboard/page.tsx
async function getMetrics() {
  const res = await fetch('https://api.example.com/metrics', {
    next: {
      tags: ['metrics'],     // Otaguj ten wpis cache
      revalidate: 60,        // Rewaliduj też co 60s jako zabezpieczenie
    },
  });
  return res.json();
}

async function getOrders() {
  const res = await fetch('https://api.example.com/orders', {
    next: {
      tags: ['orders', 'dashboard'],
      revalidate: 120,
    },
  });
  return res.json();
}
```

```typescript
// app/actions/order.ts
'use server';

import { revalidateTag } from 'next/cache';

export async function createOrder(formData: FormData) {
  await api.orders.create(formData);

  // Inwaliduj wszystkie fetche otagowane 'orders'
  revalidateTag('orders');
  // Dashboard pokaże nowe zamówienie przy kolejnej wizycie
}
```

```typescript
// app/api/webhooks/stripe/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const event = await parseStripeWebhook(request);

  if (event.type === 'payment_intent.succeeded') {
    revalidateTag('orders');
    revalidateTag('metrics');
    // Dane dashboardu odświeżają się natychmiast
  }

  return Response.json({ received: true });
}
```

**Dlaczego tagi są potężne:** Jedno `revalidateTag('orders')` inwaliduje każdą stronę i każdy fetch używający taga `orders` — dashboard, stronę listy zamówień, stronę analityki. Nie musisz wiedzieć, które strony używają których danych.

#### revalidatePath — inwalidacja po ścieżce

Gdy potrzebujesz zinwalidować konkretną stronę:

```typescript
'use server';

import { revalidatePath } from 'next/cache';

export async function updateDashboardSettings() {
  // ... aktualizacja ustawień ...

  // Inwaliduj całą trasę dashboardu
  revalidatePath('/dashboard');

  // Lub zinwaliduj konkretny layout (i wszystkie strony pod nim)
  revalidatePath('/dashboard', 'layout');
}
```

**Tag vs Path — kiedy czego używać:**
- **revalidateTag:** Gdy zmiany danych dotyczą wielu stron (utworzono zamówienie → dashboard + lista zamówień + analityka)
- **revalidatePath:** Gdy konkretna strona wymaga odświeżenia (użytkownik edytował profil → zinwaliduj stronę profilu)
- **Obie naraz:** Dla maksymalnej świeżości — zinwaliduj dane (tag) ORAZ stronę (path)

## Budowanie superszybkiego dashboardu

Oto kompletna architektura dashboardu, który działa natychmiastowo:

### Layout: streamowana powłoka

```typescript
// app/dashboard/layout.tsx
import { Suspense } from 'react';
import { DashboardNav } from '@/components/DashboardNav';

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex min-h-screen">
      <DashboardNav />
      <main className="flex-1 p-6">
        {children}
      </main>
    </div>
  );
}
```

Layout jest statyczny — nawigacja renderuje się natychmiast. Treść strony napływa streamem.

### Strona: równoległe pobieranie danych z Suspense

```typescript
// app/dashboard/page.tsx
import { Suspense } from 'react';
import { RevenueChart } from './RevenueChart';
import { MetricsGrid } from './MetricsGrid';
import { RecentOrders } from './RecentOrders';
import { TopProducts } from './TopProducts';
import { Skeleton } from '@/components/Skeleton';

export const revalidate = 60;

export default function DashboardPage() {
  return (
    <div className="space-y-6">
      <h1 className="text-2xl font-bold">Dashboard</h1>

      {/* Kluczowe metryki — ładują się pierwsze */}
      <Suspense fallback={<Skeleton className="h-32 grid grid-cols-4 gap-4" />}>
        <MetricsGrid />
      </Suspense>

      {/* Wykresy — mogą załadować się nieco później */}
      <div className="grid grid-cols-2 gap-6">
        <Suspense fallback={<Skeleton className="h-80" />}>
          <RevenueChart />
        </Suspense>
        <Suspense fallback={<Skeleton className="h-80" />}>
          <TopProducts />
        </Suspense>
      </div>

      {/* Tabela — może załadować się na końcu */}
      <Suspense fallback={<Skeleton className="h-96" />}>
        <RecentOrders />
      </Suspense>
    </div>
  );
}
```

**Magia granic Suspense:** Każda sekcja pobiera dane niezależnie i streamuje je do klienta, gdy tylko stają się dostępne. Metryki mogą załadować się w 100ms, wykres w 300ms, a tabela w 500ms — ale użytkownik od razu widzi powłokę i metryki. Bez czekania na najwolniejsze API.

### Komponenty danych: otagowane fetche

```typescript
// app/dashboard/MetricsGrid.tsx
async function getMetrics() {
  const res = await fetch(`${process.env.API_URL}/metrics/summary`, {
    next: { tags: ['metrics'], revalidate: 60 },
  });
  if (!res.ok) throw new Error('Failed to fetch metrics');
  return res.json() as Promise<DashboardMetrics>;
}

export async function MetricsGrid() {
  const metrics = await getMetrics();

  return (
    <div className="grid grid-cols-4 gap-4">
      <MetricCard
        label="Revenue"
        value={`$${metrics.revenue.toLocaleString()}`}
        change={metrics.revenueChange}
      />
      <MetricCard
        label="Orders"
        value={metrics.orderCount.toLocaleString()}
        change={metrics.orderCountChange}
      />
      <MetricCard
        label="Customers"
        value={metrics.customerCount.toLocaleString()}
        change={metrics.customerCountChange}
      />
      <MetricCard
        label="Avg. Order Value"
        value={`$${metrics.avgOrderValue.toFixed(2)}`}
        change={metrics.aovChange}
      />
    </div>
  );
}
```

```typescript
// app/dashboard/RecentOrders.tsx
async function getRecentOrders() {

}
```

## Zaawansowane wzorce

### Wzorzec 1: hybrydowa strategia rewalidacji

Połącz rewalidację czasową (zabezpieczenie) z on-demand (świeżość):

```typescript
// Dane zmieniające się przy akcji użytkownika → on-demand + krótki TTL
async function getOrders() {
  return fetch(url, {
    next: { tags: ['orders'], revalidate: 30 }, // Zabezpieczenie 30s
  });
}

// Dane zmieniające się codziennie → dłuższy TTL
async function getAnalytics() {
  return fetch(url, {
    next: { tags: ['analytics'], revalidate: 3600 }, // 1 godzina
  });
}

// Dane zmieniające się prawie nigdy → bardzo długi TTL
async function getCategories() {
  return fetch(url, {
    next: { tags: ['categories'], revalidate: 86400 }, // 24 godziny
  });
}
```

### Wzorzec 2: rewalidacja sterowana webhookami

Najbardziej niezawodny wzorzec dla zmian danych zewnętrznych:

```typescript
// app/api/webhooks/cms/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const secret = request.headers.get('x-webhook-secret');
  if (secret !== process.env.WEBHOOK_SECRET) {
    return Response.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const payload = await request.json();

  // Mapuj zdarzenia CMS na tagi cache
  const tagMap: Record<string, string[]> = {
    'product.updated': ['products', 'catalog'],
    'product.created': ['products', 'catalog', 'metrics'],
    'order.completed': ['orders', 'metrics', 'analytics'],
    'user.created': ['users', 'metrics'],
  };

  const tags = tagMap[payload.event] || [];
  for (const tag of tags) {
    revalidateTag(tag);
  }

  return Response.json({ revalidated: tags });
}
```

### Wzorzec 3: aktualizacje optymistyczne + rewalidacja

Dla akcji, w których użytkownik oczekuje natychmiastowej informacji zwrotnej:

```typescript
// components/OrderActions.tsx
'use client';

import { useOptimistic, useTransition } from 'react';
import { updateOrderStatus } from '@/actions/orders';

export function OrderStatusButton({ order }: { order: Order }) {
  const [isPending, startTransition] = useTransition();
  const [optimisticStatus, setOptimisticStatus] = useOptimistic(order.status);

  const handleShip = () => {
    startTransition(async () => {
      setOptimisticStatus('shipped'); // Natychmiastowa aktualizacja UI
      await updateOrderStatus(order.id, 'shipped'); // Server action + rewalidacja
    });
  };

  return (
    <button onClick={handleShip} disabled={isPending}>
      {optimisticStatus === 'shipped' ? 'Shipped ✓' : 'Mark as Shipped'}
    </button>
  );
}
```

```typescript
// actions/orders.ts
'use server';

import { revalidateTag } from 'next/cache';

export async function updateOrderStatus(orderId: string, status: string) {
  await api.orders.update(orderId, { status });

  revalidateTag('orders');
  revalidateTag('metrics');
}
```

Użytkownik widzi zmianę statusu natychmiast (aktualizacja optymistyczna), podczas gdy server action aktualizuje bazę danych i rewaliduje cache. Wszyscy pozostali zobaczą aktualizację przy kolejnej wizycie.

### Wzorzec 4: różne TTL-e dla różnych ról użytkowników

```typescript
// app/dashboard/page.tsx
import { getCurrentUser } from '@/lib/auth';

export default async function DashboardPage() {

  // Admini dostają świeższe dane
  const revalidateTime = user.role === 'admin' ? 10 : 60;

  const metrics = await fetch(url, {
    next: { tags: ['metrics'], revalidate: revalidateTime },
  });

  // ...
}
```

## Wyniki wydajnościowe

Oto, co wzorzec stale-while-revalidate osiąga w praktyce:

| Metryka | Przed (bez cache) | Po (ISR) | Poprawa |
|--------|-------------------|-------------|-------------|
| **TTFB** | 1 800ms | 85ms | **95% szybciej** |
| **LCP** | 3 200ms | 180ms | **94% szybciej** |
| **Wywołania API/min** | 600 | 10 | **98% redukcji** |
| **CPU serwera** | 78% | 12% | **85% redukcji** |
| **Wskaźnik błędów** | 2,1% (timeouty) | 0,01% | **99,5% redukcji** |

Redukcja wywołań API jest dramatyczna. Zamiast tego, że każdy użytkownik wyzwala wywołania API, tylko jedno żądanie na okres rewalidacji faktycznie trafia do backendu. 100 jednoczesnych użytkowników przy 60-sekundowej rewalidacji oznacza 1 wywołanie API na minutę, a nie 100.

## Pułapki i debugowanie

### 1. Cache nie działa w trybie deweloperskim

Next.js domyślnie wyłącza cache'owanie w trybie deweloperskim. Aby przetestować ISR lokalnie:

```bash
npm run build && npm run start
```

Albo użyj `next dev --experimental-https` z włączonym debugowaniem cache.

### 2. Sprawdzanie statusu cache

Dodaj sprawdzenie nagłówka `x-nextjs-cache`:

```bash
curl -I https://your-site.com/dashboard
# Szukaj:
# x-nextjs-cache: HIT    (zaserwowane z cache)
# x-nextjs-cache: STALE  (zaserwowane z cache, trwa rewalidacja)
# x-nextjs-cache: MISS   (nie w cache, wyrenderowane od nowa)
```

### 3. Trasy dynamiczne blokujące ISR

Jeśli Twoja strona używa `cookies()`, `headers()` lub `searchParams` bez owinięcia ich w Suspense, cała trasa staje się dynamiczna, a ISR zostaje wyłączony:

```typescript
// ŹLE: Sprawia, że cała strona jest dynamiczna
export default async function Page() {
  const session = await getSession(); // Używa cookies — strona jest dynamiczna!
  const metrics = await getMetrics(); // ISR wyłączony też dla tego

  return <Dashboard metrics={metrics} session={session} />;
}

// DOBRZE: Tylko część zależna od auth jest dynamiczna
export default function Page() {
  return (
    <div>
      <Suspense fallback={<Skeleton />}>
        <MetricsGrid /> {/* Statyczne, ISR działa */}
      </Suspense>
      <Suspense fallback={<Skeleton />}>
        <UserGreeting /> {/* Dynamiczne, używa cookies */}
      </Suspense>
    </div>
  );
}
```

### 4. revalidateTag nie aktualizuje się natychmiast dla bieżącego użytkownika

`revalidateTag` inwaliduje Data Cache, ale Router Cache (po stronie klienta) może nadal serwować nieświeże dane. Użyj `router.refresh()` po stronie klienta, aby wymusić świeży fetch:

```typescript
'use client';

import { useRouter } from 'next/navigation';

export function RefreshButton() {
  const router = useRouter();
  return <button onClick={() => router.refresh()}>Refresh Data</button>;
}
```

## Praktyczne wnioski

1. **Używaj rewalidacji czasowej jako zabezpieczenia** — każdy fetch powinien mieć wartość `revalidate`, nawet jeśli używasz też rewalidacji on-demand.
2. **Taguj wszystkie fetche** — `next: { tags: ['orders'] }` umożliwia chirurgiczną inwalidację cache.
3. **Owijaj komponenty danych w Suspense** — każda sekcja ładuje się niezależnie, najwolniejsze API nie blokuje strony.
4. **Używaj granic błędów per sekcja** — jedno padające API nie kładzie całego dashboardu.
5. **Łącz on-demand + czasową** — webhooki dla natychmiastowej świeżości, TTL jako zabezpieczenie.
6. **Nie czyń stron dynamicznymi przypadkiem** — `cookies()` i `headers()` w Server Component wyłączają ISR dla całej trasy.
7. **Stosuj aktualizacje optymistyczne** dla akcji użytkownika — pokaż natychmiastową informację zwrotną, rewaliduj w tle.
8. **Monitoruj wskaźniki trafień cache** — sprawdzaj nagłówki `x-nextjs-cache`. Niskie wskaźniki trafień oznaczają zbyt agresywną rewalidację.
9. **Różne TTL-e dla różnych danych** — zamówienia (30s), metryki (60s), analityka (1h), kategorie (24h).
10. **Testuj z `next build && next start`** — tryb deweloperski wyłącza cache'owanie.

Jeśli budujesz na [stacku Next.js 2026 z Server Components i Edge Runtime](/pl/blog/nextjs-2026-server-components-edge-runtime), ISR jest naturalnym uzupełnieniem — Server Components zajmują się renderowaniem, a ISR świeżością. A jeśli wdrażasz funkcje dashboardu stopniowo, połącz to z [feature flagami i progresywnymi rolloutami](/pl/blog/feature-flags-progresywne-wdrozenia-nextjs) dla maksymalnego bezpieczeństwa.

ISR to najbardziej wpływowa technika wydajnościowa w Next.js. Zamienia każdy dashboard z „czekaj na API” na „natychmiastowa odpowiedź, świeżość w tle”. Twoi użytkownicy widzą TTFB szybszy o 95%, backend obsługuje o 98% mniej żądań, a wskaźnik błędów spada niemal do zera. Implementacja jest prosta — trudną częścią jest zrozumienie, kiedy zastosować którą strategię rewalidacji.
