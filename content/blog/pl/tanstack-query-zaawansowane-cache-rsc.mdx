Kiedy programista Reacta poznaje TanStack Query, pojawia się ulga. Żegnaj, boilerplate `useEffect` + `useState`, żegnaj, ręczne zarządzanie stanami loading/error, żegnaj race conditions z porzuconych requestów. Biblioteka obsługuje to elegancko.

Ale potem trafiają na brzegowe scenariusze. Dashboard pokazujący przez 5 sekund nieaktualne dane po mutacji. Nieskończona lista, która odświeża wszystko, gdy zmieni się jeden element. Server Component, który nie mogę poddać swoimi prefetchowanymi danymi z Client Components. Optimistic update, który się wycofuje i miga.

Tu nie ma winy biblioteki — to wynik nieprawidłowej konfiguracji i architektury. Domyślne ustawienia są umyślnie konserwatywne (staleTime: 0, gcTime: 5 minut), a większość projektów musi je świadomie przepisać.

Ten przewodnik obejmuje zaawansowane wzorce: prefetching po stronie serwera z RSC i streamingiem, konfigurację czasów cache, strategie invalidacji, optimistic updates, zapytania równoległe i zależne, infinite queries oraz model mentalny, który sprawia, że wszystko wskakuje na swoje miejsce.

## Model mentalny: staleTime vs gcTime

Dwa parametry — `staleTime` i `gcTime` — kontrolują całe zachowanie cache'u w TanStack Query, a zdecydowana większość programistów myli ich role:

```
Query fetched
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│                    FRESH (staleTime)                      │
│  No background refetch. Data is served from cache.        │
│  Default: 0ms (data is immediately stale)                 │
└──────────────────────────────┬───────────────────────────┘
                               │ staleTime expires
                               ▼
┌──────────────────────────────────────────────────────────┐
│                    STALE                                   │
│  Data is served from cache, but refetch triggers on:      │
│  - Window focus (refetchOnWindowFocus)                     │
│  - Component mount (refetchOnMount)                       │
│  - Network reconnect (refetchOnReconnect)                 │
│  - Manual invalidation                                    │
└──────────────────────────────┬───────────────────────────┘
                               │ All observers unmount
                               ▼
┌──────────────────────────────────────────────────────────┐
│               INACTIVE (gcTime countdown)                 │
│  No components are using this query.                      │
│  Data stays in cache for gcTime, then garbage collected.  │
│  Default: 5 minutes                                       │
└──────────────────────────────┬───────────────────────────┘
                               │ gcTime expires
                               ▼
                          GARBAGE COLLECTED
                          (removed from cache)
```

### Ustawianie właściwych wartości domyślnych

```typescript
// lib/query-client.ts
import { QueryClient } from '@tanstack/react-query';

export function createQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // Dane świeże przez 60 sekund — w tym oknie brak refetchu w tle
        staleTime: 60 * 1000,

        // Nieaktywne zapytania trzymaj w cache 10 minut
        gcTime: 10 * 60 * 1000,

        // Większość aplikacji: nie refetchuj gdy użytkownik wróci do okna
        refetchOnWindowFocus: false,

        // Ponów nieudane zapytania 2 razy z exponential backoffem
        retry: 2,
        retryDelay: (attemptIndex) => Math.min(1000 * 2 ** attemptIndex, 30000),
      },
      mutations: {
        // Ponów mutacje raz (ostrożnie: mutacje nie są idempotentne!)
        retry: 1,
      },
    },
  });
}
```

**Dlaczego zmieniać staleTime z 0?** Domyślnie 0ms oznacza, że każde zamontowanie komponentu wyzwala refetch w tle. Dla większości danych (profile użytkowników, listy produktów, metryki dashboardu) to marnotrawstwo zasobów. 60 sekund nieaktualności jest akceptowalne i radykalnie zmniejsza liczbę wywołań API.

### Nadpisania per-query

Różne dane mają różne wymagania co do świeżości:

```typescript
// Powiadomienia: czas rzeczywisty — zawsze świeże
const { data: notifications } = useQuery({
  queryKey: ['notifications'],
  queryFn: fetchNotifications,
  staleTime: 0,           // Zawsze refetchuj
  refetchInterval: 10000, // Co 10 sekund
});

// Profil użytkownika: zmienia się rzadko
const { data: user } = useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId),
  staleTime: 5 * 60 * 1000, // Świeże przez 5 minut
  gcTime: 30 * 60 * 1000,   // W cache przez 30 minut
});

// Kategorie: dane statyczne, niemal nigdy się nie zmieniają
const { data: categories } = useQuery({
  queryKey: ['categories'],
  queryFn: fetchCategories,
  staleTime: Infinity,       // Nigdy nieaktualne (tylko invalidacja ręczna)
  gcTime: 24 * 60 * 60 * 1000, // W cache przez 24 godziny
});
```

## Prefetching z React Server Components

Największy zysk wydajnościowy z TanStack Query v5 w Next.js to prefetching po stronie serwera — dane są już w cache w momencie montowania Client Componentu, co całkowicie eliminuje stany ładowania.

### Konfiguracja

```typescript
// app/providers.tsx
'use client';

import { QueryClientProvider } from '@tanstack/react-query';
import { createQueryClient } from '@/lib/query-client';
import { useState } from 'react';

export function Providers({ children }: { children: React.ReactNode }) {
  // Instancja QueryClient — raz na session użytkownika
  const [queryClient] = useState(() => createQueryClient());

  return (
    <QueryClientProvider client={queryClient}>
      {children}
    </QueryClientProvider>
  );
}
```

```typescript
// app/layout.tsx
import { Providers } from './providers';

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html>
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}
```

### Prefetching w Server Components

```typescript
// app/dashboard/page.tsx (Server Component)
import { dehydrate, HydrationBoundary, QueryClient } from '@tanstack/react-query';
import { DashboardContent } from './DashboardContent';

export default async function DashboardPage() {
  const queryClient = new QueryClient();

  // Pobierz wszystkie dane na serwerze — klient dostanie je już w cache
  await Promise.all([
    queryClient.prefetchQuery({
      queryKey: ['metrics'],
      queryFn: fetchMetrics,
    }),
    queryClient.prefetchQuery({
      queryKey: ['recent-orders'],
      queryFn: () => fetchOrders({ limit: 20 }),
    }),
    queryClient.prefetchQuery({
      queryKey: ['user', 'current'],
      queryFn: fetchCurrentUser,
    }),
  ]);

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <DashboardContent />
    </HydrationBoundary>
  );
}
```

```typescript
// app/dashboard/DashboardContent.tsx (Client Component)
'use client';

import { useQuery } from '@tanstack/react-query';

export function DashboardContent() {
  // Dane już w cache z prefetchu serwerowego — bez stanu ładowania!
  const { data: metrics } = useQuery({
    queryKey: ['metrics'],
    queryFn: fetchMetrics,
  });

  const { data: orders } = useQuery({
    queryKey: ['recent-orders'],
    queryFn: () => fetchOrders({ limit: 20 }),
  });

  // Render natychmiastowy — dane przygotowane na serwerze
  return (
    <div>
      <MetricsGrid data={metrics!} />
      <OrderTable orders={orders!} />
    </div>
  );
}
```

**Przepływ:**
1. Server Component prefetchuje dane
2. `dehydrate(queryClient)` serializuje cache
3. `HydrationBoundary` przekazuje zserializowany cache do klienta
4. `useQuery` w Client Componencie znajduje dane już w cache → natychmiastowy render

### Streaming Prefetch (bez czekania)

Począwszy od TanStack Query v5.40, możesz prefetchować bez await — dane streamują do klienta w miarę gotowości:

```typescript
// app/dashboard/page.tsx
export default async function DashboardPage() {
  const queryClient = new QueryClient();

  // Bez await — prefetch startuje, ale serwer nie czeka na rezultat
  queryClient.prefetchQuery({
    queryKey: ['slow-analytics'],
    queryFn: fetchAnalytics, // 2 sekundy
  });

  // Szybkie dane: na te czekaj
  await queryClient.prefetchQuery({
    queryKey: ['metrics'],
    queryFn: fetchMetrics, // 100ms
  });

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      {/* Metryki od razu, analytics streamuje w tle */}
      <DashboardContent />
    </HydrationBoundary>
  );
}
```

## Invalidacja cache: najdelikatniejsza część

Invalidacja cache — szczególnie decyzja, kiedy to zrobić — to klasycznie jeden z najtrudniejszych problemów w architekturze danych. TanStack Query oferuje kilka sprawdzonych strategii:

### Strategia 1: Invalidacja po mutacji

Najczęstszy wzorzec — po udanej mutacji invaliduj dotknięte zapytania:

```typescript
const createOrder = useMutation({
  mutationFn: (data: CreateOrderInput) => api.orders.create(data),
  onSuccess: () => {
    // Unieważnij wszystkie zapytania z klucz em ['orders']
    queryClient.invalidateQueries({ queryKey: ['orders'] });

    // Unieważnij metryki — liczba zamówień się zmieniła
    queryClient.invalidateQueries({ queryKey: ['metrics'] });
  },
});
```

### Strategia 2: Hierarchia query key

Strukturyzuj swoje query keys hierarchicznie dla chirurgicznej invalidacji:

```typescript
// Fabryka kluczy — hierarchiczna struktura
export const orderKeys = {
  all: ['orders'] as const,
  lists: () => [...orderKeys.all, 'list'] as const,
  list: (filters: OrderFilters) => [...orderKeys.lists(), filters] as const,
  details: () => [...orderKeys.all, 'detail'] as const,
  detail: (id: string) => [...orderKeys.details(), id] as const,
};

// Stosowanie
useQuery({ queryKey: orderKeys.detail('abc123'), queryFn: ... });
useQuery({ queryKey: orderKeys.list({ status: 'pending' }), queryFn: ... });

// Invalidacja — precyzja chirurgiczna
queryClient.invalidateQueries({ queryKey: orderKeys.detail('abc123') }); // Jedno zamówienie
queryClient.invalidateQueries({ queryKey: orderKeys.lists() });          // Tylko listy
queryClient.invalidateQueries({ queryKey: orderKeys.all });              // Wszystko
```

### Strategia 3: Aktualizacje z odpowiedzi mutacji

Jeśli mutacja zwraca zaktualizowaną encję, pomiń refetch w całości:

```typescript
const updateOrder = useMutation({
  mutationFn: (data: UpdateOrderInput) => api.orders.update(data),
  onSuccess: (updatedOrder) => {
    // Zaktualizuj konkretne zamówienie w cache — bez refetchu
    queryClient.setQueryData(
      orderKeys.detail(updatedOrder.id),
      updatedOrder
    );

    // Listy: invaliduj (kolejność mogła się zmienić)
    queryClient.invalidateQueries({ queryKey: orderKeys.lists() });
  },
});
```

## Optimistic Updates

Pokaż wynik mutacji natychmiast, a następnie wycofaj go, jeśli się nie powiedzie:

```typescript
const toggleFavorite = useMutation({
  mutationFn: (productId: string) => api.products.toggleFavorite(productId),

  onMutate: async (productId) => {
    // Anuluj refetche w locie — nie chcemy, by nadpisały optimistic update
    await queryClient.cancelQueries({ queryKey: ['products', productId] });

    // Zrób snapshot przed zmianą
    const previous = queryClient.getQueryData<Product>(['products', productId]);

    // Aktualizuj UI od razu (optymistycznie)
    queryClient.setQueryData<Product>(['products', productId], (old) => {
      if (!old) return old;
      return { ...old, isFavorite: !old.isFavorite };
    });

    // Zwróć snapshot do wycofania
    return { previous };
  },

  onError: (_error, productId, context) => {
    // Błąd API: przywróć poprzednią wartość
    if (context?.previous) {
      queryClient.setQueryData(['products', productId], context.previous);
    }
  },

  onSettled: (_data, _error, productId) => {
    // Na koniec: refetch dla potwierdzenia spójności
    queryClient.invalidateQueries({ queryKey: ['products', productId] });
  },
});
```

**Przepływ:**
1. Użytkownik klika „ulubione" → UI aktualizuje się natychmiast (optymistycznie)
2. Wywołanie API startuje w tle
3. Jeśli sukces → refetch dla potwierdzenia (dane mogą mieć inne zmiany)
4. Jeśli błąd → wycofaj do poprzedniej wartości

## Zapytania równoległe i zależne

### Zapytania równoległe

Niezależne zapytania automatycznie działają równolegle:

```typescript
function Dashboard() {
  // Wszystkie trzy uruchamiają się równolegle
  const metrics = useQuery({ queryKey: ['metrics'], queryFn: fetchMetrics });
  const orders = useQuery({ queryKey: ['orders'], queryFn: fetchOrders });
  const users = useQuery({ queryKey: ['users'], queryFn: fetchUsers });

  // ...
}
```

### Równoległe zapytania Suspense

Z Suspense użyj `useSuspenseQueries`, by uniknąć waterfalla:

```typescript
function Dashboard() {
  // ŹLE: Sekwencyjnie — drugie czeka na pierwsze
  const { data: metrics } = useSuspenseQuery({ queryKey: ['metrics'], queryFn: fetchMetrics });
  const { data: orders } = useSuspenseQuery({ queryKey: ['orders'], queryFn: fetchOrders });
  // metrics zawiesza komponent → orders nawet nie startuje

  // DOBRZE: Równolegle — oba jednocześnie
  const [metricsResult, ordersResult] = useSuspenseQueries({
    queries: [
      { queryKey: ['metrics'], queryFn: fetchMetrics },
      { queryKey: ['orders'], queryFn: fetchOrders },
    ],
  });
}
```

### Zapytania zależne

Gdy zapytanie B potrzebuje danych z zapytania A:

```typescript
function UserOrders({ userId }: { userId: string }) {
  // Krok 1: pobierz użytkownika
  const { data: user } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  });

  // Krok 2: zamówienia — ale tylko gdy mamy ID klienta
  const { data: orders } = useQuery({
    queryKey: ['orders', { customerId: user?.customerId }],
    queryFn: () => fetchOrders({ customerId: user!.customerId }),
    enabled: !!user?.customerId, // Startuje gdy customerId dostępne
  });
}
```

## Infinite Queries do paginacji

```typescript
function OrderList() {
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
  } = useInfiniteQuery({
    queryKey: ['orders', 'infinite'],
    queryFn: ({ pageParam }) => fetchOrders({
      cursor: pageParam,
      limit: 20,
    }),
    initialPageParam: undefined as string | undefined,
    getNextPageParam: (lastPage) => lastPage.nextCursor,
  });

  const allOrders = data?.pages.flatMap((page) => page.items) ?? [];

  return (
    <div>
      {allOrders.map((order) => (
        <OrderCard key={order.id} order={order} />
      ))}

      {hasNextPage && (
        <button
          onClick={() => fetchNextPage()}
          disabled={isFetchingNextPage}
        >
          {isFetchingNextPage ? 'Ładowanie...' : 'Więcej'}
        </button>
      )}
    </div>
  );
}
```

### Aktualizacja pojedynczego elementu w infinite query

Trudna część — aktualizacja jednego elementu bez refetchu wszystkich stron:

```typescript
const updateOrderStatus = useMutation({
  mutationFn: (data: { orderId: string; status: string }) =>
    api.orders.updateStatus(data),
  onSuccess: (updatedOrder) => {
    // Wyszukaj element w infinite query i zaktualizuj — nie refetchuj wszystko
    queryClient.setQueryData<InfiniteData<OrderPage>>(
      ['orders', 'infinite'],
      (oldData) => {
        if (!oldData) return oldData;
        return {
          ...oldData,
          pages: oldData.pages.map((page) => ({
            ...page,
            items: page.items.map((order) =>
              order.id === updatedOrder.id ? updatedOrder : order
            ),
          })),
        };
      }
    );
  },
});
```

## Zapobieganie waterfallom requestów

Waterfalle — sekwencyjne, nieintencjonalne czekania między requestami — są cichym wrogiem wydajności. Komponent A pobiera dane, komponent B (jego dziecko) czeka, by dostać je do użytku:

```
Component A fetches    ████████████
  Component B fetches              ████████████
    Component C fetches                        ████████████
Total:                 ████████████████████████████████████████
                       ^                                     ^
                       0ms                                 3000ms
```

### Zapobieganie: prefetch na poziomie route'a

```typescript
// app/dashboard/page.tsx — prefetchuj WSZYSTKIE dane, których strona potrzebuje
export default async function DashboardPage() {
  const queryClient = new QueryClient();

  await Promise.all([
    queryClient.prefetchQuery({ queryKey: ['user'], queryFn: fetchUser }),
    queryClient.prefetchQuery({ queryKey: ['metrics'], queryFn: fetchMetrics }),
    queryClient.prefetchQuery({ queryKey: ['orders'], queryFn: fetchOrders }),
    queryClient.prefetchQuery({ queryKey: ['products'], queryFn: fetchProducts }),
  ]);

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <DashboardContent /> {/* Wszystkie dane w cache — zero waterfalli */}
    </HydrationBoundary>
  );
}
```

Cztery zapytania startują równolegle na serwerze. Zanim Client Components się renderują, dane są już dostępne.

## Praktyczne wnioski

1. **Zmień domyślny `staleTime` na 60s** — 0ms to marnotrawstwo dla większości danych. Nadpisuj per-query tylko dla scenariuszy czasu rzeczywistego.
2. **Prefetchuj na poziomie Server Components** — kombinacja `dehydrate` + `HydrationBoundary` eliminuje stany ładowania przy pierwszym renderze.
3. **Twórz fabryki query key** — ustrukturyzowane klucze (`orderKeys.list(filters)`) umożliwiają precyzyjną invalidację cache.
4. **Optimistic updates dla akcji użytkownika** — przełączanie ulubionych, zmiana statusu, usunięcia. Zawsze gotuj wycofanie przy błędzie.
5. **Używaj `useSuspenseQueries` dla równoległego Suspense** — pojedynczy `useSuspenseQuery` automatycznie tworzy waterfalle.
6. **Dla paginacji zastosuj infinite queries** — `useInfiniteQuery` + paginacja kursorowa elegancko obsługuje „load more".
7. **Aktualizuj cache z odpowiedzi mutacji** — `setQueryData` z danymi z serwera eliminuje zbędne refetche.
8. **Prefetchuj na poziomie route'a** — wszystkie dane zbieraj w Server Component, by uniknąć waterfalli po stronie klienta.
9. **Invaliduj chirurgicznie** — hierarchiczne query keys umożliwiają unieważnienie tylko tego, co faktycznie się zmieniło.
10. **`gcTime` na serwerze ustawiaj na Infinity** — QueryClienty serwerowe żyją króko, więc garbage collection nie stanowi problemu.
