Przez wiele lat każda aplikacja Vue, którą wdrażałem, ponosiła niewidoczny koszt. Przy każdej zmianie stanu Vue musiało alokować nowe obiekty JavaScript, przejść dwa drzewa, porównać je i dopiero wtedy zmienić DOM. Tę pracę — **virtual DOM** — wykonywała bateria telefonu użytkownika, jego procesor. To była cena za elegancki developer experience, który uwielbiałem.

Vue 4 oferuje sposób na wyeliminowanie tego kosztu. **Vapor mode** kompiluje komponenty bezpośrednio do operacji na DOM, omijając pośrednika. Bez virtual DOM, bez diffowania, bez reconciliacji. Utrzymujesz znany `ref` i `computed`, ale wynikiem są precyzyjne aktualizacje DOM, które stawiają Vue na równi z SolidJS i Svelte w testach wydajnościowych.

Kluczowa różnica: **nie wymaga zmian w sposobie pisania kodu**. Nie poznajemy nowego modelu reactivity. Nie przepisujemy komponentów w nowej składni. Dodajemy jeden atrybut do `<script setup>` i kompilator zajmuje się resztą. Przejdźmy przez to, jak to działa pod maską, dlaczego ma znaczenie i w jakich scenariuszach warto to wdrożyć.

## Dlaczego virtual DOM był kompromisem od samego początku

Virtual DOM nigdy nie stanowił najszybszego rozwiązania. Był wygodny — to była jego rzeczywista wartość. Gdy React go spopularyzował, a Vue 2 przejęło ten paradygmat, obietnica brzmiała: opisujesz pożądany stan UI, a framework wyznacza minimalny zestaw mutacji DOM niezbędnych do jego osiągnięcia. Znikała konieczność ręcznego manipulowania DOM i śledzenia, który element się zmienił.

Mechanizmem jest funkcja renderująca. Przy każdej zmianie stanu komponent uruchamia swoją funkcję renderującą, tworząc nowe drzewo zwykłych obiektów JavaScript — vnode'ów — reprezentujących żądany UI. Framework porównuje je z poprzednim drzewem, **diffuje**, a potem aplikuje różnice do rzeczywistego DOM. Ponieważ bezpośrednie mutacje DOM są kosztowne, batchowanie zmian przez diff stanowiło znaczną poprawę wobec naiwnego ponownego renderowania.

Problem leży w tym, gdzie ten proces się odbywa. Diffowanie wykonuje się **przy każdej zmianie stanu, dynamicznie w przeglądarce**. Zaktualizowanie pojedynczego węzła tekstowego wymaga alokacji vnode'ów dla całej otaczającej struktury, przechodzenia drzewa i porównania. Vue 3 zoptymalizowało to, dodając wskazówki na etapie kompilacji — patch flagi określające dynamiczne części szablonu i static hoisting, który pomija niezmieniające się węzły. Te optymalizacje zmniejszyły pracę diffu, ale jej nie wyeliminowały. Framework wciąż musiał *obliczyć* w runtime, co się zmieniło.

SolidJS i Svelte postawili to pytanie inaczej: skoro kompilator ma dostęp do szablonu na etapie kompilacji, dlaczego odsuwać decyzję "co się zmieniło" na czas runtime? Zamiast tego można bezpośrednio połączyć każdą reaktywną wartość z konkretnym węzłem DOM, który nią steruje — zmiana wartości aktualizuje wyłącznie ten węzeł, nic więcej. To jest fine-grained reactivity, i to właśnie podejście wprowadza Vapor mode do Vue.

## Czym jest Vapor mode

**Vapor mode to alternatywna strategia kompilacji.** Syntax szablonów i Composition API pozostają identyczne, ale kompilator produkuje zupełnie inny kod. Zamiast generować funkcję renderującą produkującą vnode'y, kompilator Vapor emituje kod, który tworzy prawdziwe węzły DOM jednorazowo, a następnie łączy reaktywny stan bezpośrednio z konkretnymi węzłami, które go wykorzystują.

Weźmy najnudniejszy komponent świata:

```vue
<script setup vapor>
import { ref } from 'vue'

const count = ref(0)
</script>

<template>
  <button @click="count++">Count: {{ count }}</button>
</template>
```

W klasycznym Vue kliknięcie przycisku ponownie uruchamia funkcję renderującą, generuje nowy vnode, porównuje go ze starym, wykrywa zmianę w tekście i aktualizuje go. W Vapor mode kompilator ustala na etapie budowania, że `count` wpływa dokładnie na jeden węzeł tekstowy wewnątrz przycisku. W przybliżeniu emituje kod podobny do poniższego (uproszczony — nigdy tego nie napiszesz ani nie zobaczysz):

```js
// Wyjście koncepcyjne — Vapor wiąże reaktywną wartość z jednym węzłem
const t0 = template('<button>Count: </button>')

function render() {
  const button = t0()            // utwórz prawdziwy węzeł DOM raz
  const text = createTextNode()  // dynamiczne dziecko tekstowe
  button.append(text)

  // Wiązanie: gdy `count` się zmienia, aktualizuj TYLKO ten węzeł tekstowy
  renderEffect(() => {
    text.data = `Count: ${count.value}`
  })

  on(button, 'click', () => count.value++)
  return button
}
```

Brak drzewa vnode'ów, brak diffowania. Gdy zmieni się `count`, efekt uruchamia się i aktualizuje pojedynczy węzeł tekstowy. Nie przychodzi do przechodzenia drzewa, nie ma porównywania. Zmiana stanu bezpośrednio prowadzi do precyzyjnej aktualizacji DOM — kompilator już wiedział w momencie budowania, który węzeł trzeba zmienić.

W tym leży sedno: **klasyczne Vue odkrywa dynamicznie, co się zmieniło** — przez diff w runtime. **Vapor Vue rozpoznaje to statycznie** — na etapie kompilacji — i koduje połączenia na sztywno.

## Reactivity jako fundament

To jest elegancka część rozwiązania i wyjaśnia, dlaczego Vue mogło wdrożyć Vapor bez podziału na dwa niezależne frameworki: **Vapor mode ponownie wykorzystuje istniejący system reactivity Vue.** Fundament oparty na signals, który napędza `ref`, `reactive`, `computed` i `watchEffect`, jest dokładnie tym mechanizmem, który steruje precyzyjnymi aktualizacjami DOM.

Znasz już to, jeśli pytałeś się kiedyś, jak `computed` wie, kiedy trzeba się przeliczyć. System reactivity Vue pracuje na zasadzie dependency tracking: gdy efekt uruchamia się i odczytuje `count.value`, Vue rejestruje, że efekt zależy od `count`. Gdy `count` jest aktualizowany, Vue powiadamia dokładnie te efekty, które go czytały. To ten sam prymityw, który stanowi podstawę nowoczesnych signals we wszystkich frameworkach — szczegółowo opisuję go w przewodniku [fine-grained reactivity](/pl/blog/sygnaly-drobnoziarnista-reaktywnosc-2026), wart przeczytania razem z tym artykułem, bo Vapor to właśnie ten model zastosowany bezpośrednio do DOM.

W tradycyjnym Vue śledzoną jednostką reactivity jest *cała funkcja renderująca komponentu*. Zmiana jakiejkolwiek zależności wyzwala pełny re-render (a diff vnode'ów zmniejsza następnie rzeczywiste zmiany w DOM). Vapor mode zmienia to fundamentalnie: kompilator tworzy **osobny, minimalny efekt dla każdego dynamicznego wiązania**. Interpolacja tekstowa ma własny efekt. Binding `:class` ma własny. Dyrektywa `v-if` ma własny. Każdy efekt zależy wyłącznie od reaktywnych wartości, które czyta, więc zmiana pojedynczej wartości uruchamia tylko odpowiedni efekt i zmienia wyłącznie powiązany węzeł.

Dla Vue 4 zespół całkowicie przepisał rdzeń reactivity przy użyciu **alien-signals** — implementacji dependency tracking'u zoptymalizowanej pod minimalny overhead: szybsze powiadamianie obserwatorów, bardziej agresywne batching zmian, mniej alokacji pamięci przy czyszczeniu starych zależności. Twoje wywołania `ref()` i `computed()` działają identycznie, ale podstawowa architektura jest znacznie lżejsza. Robi to dużą różnicę, gdy tysiące małych efektów wykonują pracę, którą wcześniej wykonywał jeden duży efekt renderujący.

## Adoptacja i współistnienie z klasycznym Vue

Vapor mode jest **opcjonalny na poziomie poszczególnych komponentów** — przełącznik jest celowo prosty. Wystarczy dodać atrybut `vapor` do `<script setup>`:

```vue
<script setup vapor>
// Ten komponent kompiluje się do wyjścia Vapor — bez virtual DOM
import { ref, computed } from 'vue'

const items = ref([])
const total = computed(() => items.value.length)
</script>

<template>
  <p>{{ total }} items</p>
</template>
```

To cała zmiana wymagana na poziomie kodu komponentu. **Composition API pozostaje niezmieniony** — `ref`, `reactive`, `computed`, `watch`, `watchEffect` działają dokładnie jak zawsze. Brak nowej składni do nauczenia się; jedyna różnica to wyjście kompilatora. (Potwierdź dokładną składnię w dokumentacji Vue dla Twojej wersji — atrybut `vapor` na `<script setup>` utrzymał się przez beta 3.6 aż do Vue 4, ale Vapor był długo eksperymentalnym API i ostateczne szczegóły ustaliły się stosunkowo niedawno.)

Ponieważ adopcja jest na poziomie komponentu, można migrować stopniowo. Konwertujesz komponenty wrażliwe na wydajność — duże tabele, wykresy aktualizujące się w czasie rzeczywistym, wirtualizowane listy — a pozostałą część pozostawiasz w klasycznym modelu. Ta przyrostowość jest kluczowa dla strategii adopcji i jest możliwa, ponieważ komponenty Vapor i VDOM **mogą współistnieć**:

- Komponenty Vapor mogą zawierać komponenty VDOM jako dzieci i vice versa. Vue automatycznie mostkuje oba modele na granicy komponentów.
- Interop obsługiwany jest przez **plugin `vaporInteropPlugin`**, umożliwiający istniejącym bibliotekom komponentów VDOM takim jak Element Plus czy Ant Design Vue pracować wewnątrz drzewa Vapor.
- W pełni Vaporowa aplikacja (każdy komponent to Vapor) może użyć **`createVaporApp`** zamiast `createApp`, całkowicie usuwając runtime virtual DOM z bundle'a. To daje największe oszczędności na rozmiarze kodu.

Istnieją faktycznie dwa poziomy adopcji: (1) Dodaj Vapor do kluczowych komponentów, utrzymując runtime VDOM dla interopu z istniejącymi bibliotekami. (2) Przejdź całkowicie na Vapor i wyeliminuj z bundle'a cały kod virtual DOM.

## Wydajność i rozmiar bundle'a

Wpływ na liczbę to główny powód zainteresowania tym podejściem. Testy wydajności zespołu Vue pokazują, że Vapor **renderuje 100 000 komponentów w około 100 ms** — wynik porównywalny z SolidJS, frameworkiem, który dominuje benchmarkami fine-grained reactivity. Największe zyski widoczne są w scenariuszach z dużą ilością aktualizacji, gdzie kumuluje się koszt diffowania — w ekstremalnych przypadkach obserwuje się **97% szybsze renderowanie** w porównaniu z odpowiadającą ścieżką VDOM.

Źródła tych zysków to dwie rzeczy: (1) Wyeliminowanie diffu zmniejsza alokacje pamięci i przechodzenie drzewa przy każdej zmianie — przeglądarka nie musi już porównywać drzew w każdym cyklu. (2) Zmniejsza się presja na pamięć, bo nie generujesz drzew vnode'ów przy każdym renderze — mniej pracy dla garbage collector, mniej mikrozacięć frame-time'u, które są szczególnie widoczne na średnio-zasilonych urządzeniach z Androidem.

Na rozmiar bundle'a wpływ jest znaczący. Komponenty wyłącznie Vapor mogą być **20–50% mniejsze** od ich odpowiedników VDOM, bo skompilowany kod jest lżejszy i wymaga mniej runtime'u. Pełna aplikacja Vapor (z `createVaporApp`) całkowicie eliminuje runtime VDOM. To nie jest mikro-optymalizacja — to mierzalne zmniejszenie bundle'a o dziesiątki kilobajtów i zauważalnie szybsza ścieżka interakcji, którą można zaobserwować w [performance budgecie](/pl/blog/budzety-wydajnosci-frontend). Dodaj do tego, że Vue 4 ma kompilator przepisany w Rust, który znacznie skraca czasy HMR w dużych projektach — szczegóły porównuję w artykule [Node vs Bun vs Deno](/pl/blog/nodejs-bun-deno-2026-porownanie-runtime).

Ważne wyjaśnienie: benchmarki to nie rzeczywiste aplikacje. Większość komponentów nie renderuje 100 000 elementów, a dla typowego formularza czy panelu ustawień różnica między Vapor a klasycznym Vue jest prawie niezauważalna. Zyski dotyczą komponentów, które mają dużo elementów i aktualizują się częstą. Profiluj swoje rzeczywiste wąskie gardła przed podjęciem decyzji o globalnej migracji.

## Porównanie z SolidJS i Svelte 5

Vapor mode reprezentuje konwergencję, nad którą frameworki pracowały przez wiele lat. **SolidJS** nigdy nie miał virtual DOM — od początku zbudowany na signals z fine-grained aktualizacjami DOM. **Svelte 5** przeprojektował reactivity wokół runes i śledzenia na etapie kompilacji, również bez VDOM. React wybrał inną ścieżkę — utrzymał VDOM i zautomatyzował memoization (niegdyś ręczna), co omawiamy w [analizie React Compilera](/pl/blog/react-compiler-automatyczna-memoizacja). React również znacząco zainwestował w runtime scheduler; jeśli chcesz zobaczyć, jak dużą ilość kodu trzeba, aby utrzymać VDOM, przeczytaj moją [analizę React Fiber](/pl/blog/react-fiber-silnik-nowoczesnego-reacta).

Wszystkie frameworki dochodzą do tego samego wniosku z różnych początkowych założeń: **kompilator powinien przejmować pracę, którą wcześniej wykonywał runtime.** SolidJS i Svelte 5 zostały zbudowane na tej idei od samego początku. Vue implementuje to w dojrzałym frameworku z wielkiego istniejącą bazą kodu — jest to trudniejsze, ale i bardziej wartościowe, bo Vue nie zmusza do porzucenia istniejących komponenty, aby to uzyskać.

Kluczowa różnica Vue polega na **ciągłości doświadczenia programistycznego**. W SolidJS signals definiują sposób pisania kodu — musisz pamiętać o wywoływaniu funkcji, aby czytać wartości i respektowaniu reguł dependency tracking. Svelte 5 wprowadza runes jako nowy dialekt, wymagający nowego sposobu myślenia. Vapor Vue zachowuje znane `ref`, `computed` i single-file components bez zmian. Dostajesz wydajność klasy Solid z kodem, który wygląda jak zwykłe Vue. To realna przewaga dla zespołów, które chcą fine-grained reactivity, ale nie chcą zmienić modelu mentalnego ani przeznaczać całego zespołu na ponowne szkolenie.

## Ograniczenia i kwestie praktyczne

Vapor mode to wciąż stosunkowo nowa funkcja i ważne jest wymienić rzeczywiste wyzwania. Oto kilka faktycznych ograniczeń:

- **Wyłącznie Composition API z `<script setup>`.** Vapor nie wspiera Options API — brak `data()`, `methods` czy tradycyjnego `setup()`. Jeśli codebase używa głównie Options API, to wymaga najpierw migracji do Composition API.
- **Brak hoków lifecycle per-element.** Atrybuty specyficzne dla VDOM, jak `@vue:mounted` na elemencie, nie istnieją w Vapor.
- **Interop ma ograniczenia.** Komponenty Vapor mogą zawierać komponenty VDOM i vice versa, ale niektóre zaawansowane scenariusze reactivity z props i slots mogą się zachowywać inaczej na granicy. Głębokie zagnieżdżanie bibliotek VDOM wewnątrz drzewa Vapor czasami prowadzi do błędów runtime — są otwarte issue dotyczące tego z bibliotekami UI.
- **Ekosystem się dopiero dostosowuje.** Core biblioteki — Pinia, VueUse, Nuxt — działają, ale wiele komponentów i custom dyrektyw trzecich firm robi założenia specyficzne dla VDOM. Te założenia pękają w Vapor, a pełne wsparcie ekosystemu zajmie jeszcze czas.
- **SSR, `Transition` i `KeepAlive` wymagają weryfikacji.** To były ostatnie elementy, które osiągnęły pełnię funkcjonalności w Vapor. Sprawdź dokumentację dla konkretnej wersji Vue zamiast zakładać parytet z klasycznym modelem.

Żaden z tych problemów nie dyskwalifikuje Vapor dla zmyślnych scenariuszy. Wszystkie stanowią powody, by nie migrować całej aplikacji na Vapor bez dokładnego planowania i testowania.

## Czy wdrożyć Vapor mode?

Oto moja praktyczna ocena:

**Adoptuj selektywnie, jeśli** masz konkretny wąski problem wydajnościowy — duża tabela z danymi, dashboard aktualizujący się w czasie rzeczywistym, wirtualizowana lista, która zacina się na średnich urządzeniach. To jest dokładnie scenariusz, do którego Vapor został zaprojektowany. Konwertuj tylko te komponenty, zmierz poprawę i zostaw resztę w klasycznym modelu. Granularność per-komponent oznacza niskie ryzyko — zawsze możesz cofnąć zmianę w jednym pliku.

**Poczekaj, jeśli** budujesz nową aplikację opartą głównie na bibliotekach UI (Element Plus, Ant Design Vue) albo twój zespół używa głównie Options API. Interop jest możliwy, ale nie całkowicie transparentny, a ekosystem wciąż się dopasowuje. Problemy interopu mogą pochłonąć więcej czasu niż oszczędzisz na wydajności.

**Nie migruj całej aplikacji do `createVaporApp`** chyba że kontrolujesz wszystkie komponenty i masz pełną kontrolę nad zależnościami. Oszczędność na bundle'u jest realna, ale wymóg "każdy komponent musi być Vapor" nakłada silne ograniczenia.

**Strategia przyrostowa:** (1) Najpierw migrate na Composition API i `<script setup>` — to prerekwizyt i wartościowy sam w sobie. (2) Profiluj aplikację, aby znaleźć rzeczywiste wąskie gardła. (3) Konwertuj te komponenty jedno po drugim, używając interopu. (4) Utrzymuj runtime VDOM dopóki możesz uczynić wszystkie komponenty Vaporowymi. (5) Nie konwertuj komponentów, które nie mają problemów wydajnościowych — zysk byłby zerowy, a ograniczenia rzeczywiste.

**Podsumowanie:** Nagłówek nie mówi, że virtual DOM jest martwy. Mówi, że w Vue 4 jest **opcjonalny** — decyzja na poziomie komponentu, z takim samym reactivity i modelem kodowania. Po latach płacenia kosztu diffowania możliwość jego eliminacji tam, gdzie ma to znaczenie, jest idealnym rozwiązaniem. Używaj tam, gdzie liczy się wydajność, ignoruj tam, gdzie się nie liczy, i pozwól kompilatorowi pracować.
