"JavaScript jest wolny, bo jest interpretowany." Słyszałem to na każdej rozmowie technicznej, którą prowadziłem przez ostatnie trzy lata. To nieprawda — a zrozumienie, dlaczego to nieprawda, fundamentalnie zmieni sposób, w jaki piszesz JavaScript.

Nowoczesny V8 nie interpretuje Twojego kodu. On go kompiluje. Wielokrotnie. Poprzez 4-poziomowy pipeline, który stopniowo optymalizuje hot paths do postaci kodu maszynowego dorównującego C++ w ciasnych pętlach. Silnik uruchamiający Twoje `Array.map()` to ten sam silnik, który napędza Chrome, Node.js, Deno i Cloudflare Workers. I to ten sam silnik stojący za [reconcilerem React Fiber](/blog/deep-dive-into-react-fiber-the-engine-behind-modern-react), dzięki któremu Twój interfejs sprawia wrażenie natychmiastowego. Obsługuje miliardy żądań dziennie i renderuje każdą stronę internetową, którą odwiedzasz.

Ale jest coś, co większość programistów przeoczy: V8 nie optymalizuje całego kodu jednakowo. Robi zakłady dotyczące zachowania Twojego kodu, optymalizuje na podstawie tych zakładów, a jeśli naruszysz jego założenia, wyrzuca zoptymalizowany kod i zaczyna od nowa. Zrozumienie tych mechanizmów to różnica między kodem, który lata, a kodem, który się wlecze.

## Pipeline wykonania: od źródła do kodu maszynowego

Kiedy V8 napotyka Twój JavaScript, nie przechodzi od razu do kodu maszynowego. Jeśli kiedykolwiek zastanawiałeś się nad [zaawansowanymi koncepcjami JavaScriptu, które pojawiają się na rozmowach senioralnych](/blog/deep-javascript-interview-guide-for-2025–2026), to jest fundament, na którym opiera się cała reszta. V8 przechodzi przez wyrafinowany, wielopoziomowy pipeline, gdzie każdy poziom wymienia szybkość kompilacji na wydajność wykonania:

```
Source Code
    │
    ▼
┌─────────┐
│ Parser  │ → Abstract Syntax Tree (AST)
└────┬────┘
     │
     ▼
┌──────────┐
│ Ignition │ → Bytecode (interpreted)
└────┬─────┘
     │  (code gets warm)
     ▼
┌───────────┐
│ Sparkplug │ → Baseline machine code (fast compile, basic perf)
└────┬──────┘
     │  (code gets hot)
     ▼
┌─────────┐
│ Maglev  │ → Optimized machine code (10x faster compile than TurboFan)
└────┬────┘
     │  (code is very hot)
     ▼
┌──────────┐
│ TurboFan │ → Highly optimized machine code (speculative optimization)
└──────────┘
```

### Poziom 1: Ignition — interpreter

Każdy fragment JavaScriptu zaczyna się tutaj. Ignition to rejestrowy interpreter bytecode'u, który kompiluje Twój kod źródłowy do zwartego bytecode'u i natychmiast go wykonuje.

```javascript
function add(a, b) {
  return a + b;
}
```

Ignition kompiluje to do bytecode'u, który wygląda mniej więcej tak:

```
Ldar a1          // Wczytaj argument 'b' do akumulatora
Add a0, [0]      // Dodaj argument 'a' do akumulatora
Return           // Zwróć akumulator
```

Ignition kompiluje szybko, ale wykonuje wolno — interpretuje jedną instrukcję bytecode'u naraz. Ale to w porządku w przypadku kodu, który uruchamia się tylko raz czy dwa. Większość kodu w typowej aplikacji wykonuje się zaledwie kilka razy — kod startowy, inicjalizacja, rzadko trafiane gałęzie.

**Kluczowy wniosek:** Ignition nie tylko wykonuje kod — on go profiluje. Zbiera informacje zwrotne o tym, jakie typy mają Twoje zmienne, które gałęzie są wybierane, jak często funkcje są wywoływane. Ten feedback jest krytyczny dla kolejnych poziomów.

### Poziom 2: Sparkplug — kompilacja bazowa

Kiedy funkcja została wywołana wystarczająco wiele razy, do gry wchodzi Sparkplug. Wprowadzony w 2021 roku Sparkplug to błyskawiczny kompilator bazowy, który tłumaczy bytecode bezpośrednio na kod maszynowy — niemal natychmiastowo.

Sparkplug nie optymalizuje. Generuje mechaniczne, jeden-do-jednego mapowanie z bytecode'u na kod maszynowy. Ale nawet ta "głupia" kompilacja jest 5-10x szybsza od interpretacji, ponieważ eliminuje narzut dispatchu interpretera.

**Dlaczego to ma znaczenie:** Sparkplug wypełnia lukę między "zbyt zimne, by optymalizować" a "wystarczająco gorące dla Maglev/TurboFan". Bez niego istniał wydajnościowy klif — kod był albo interpretowany (wolno), albo w pełni zoptymalizowany (szybko), bez niczego pomiędzy.

### Poziom 3: Maglev — optymalizator średniego poziomu

Wprowadzony w Chrome 117 Maglev to odpowiedź V8 na "lukę optymalizacyjną". Kompiluje ~10x wolniej niż Sparkplug, ale ~10x szybciej niż TurboFan, produkując kod znacznie lepszy od bazowego.

Maglev używa tradycyjnego podejścia opartego na SSA (Static Single Assignment) z CFG (Control Flow Graph). Wykonuje:

- Specjalizację typów na podstawie feedbacku z Ignition
- Alokację rejestrów
- Podstawową eliminację martwego kodu
- Proste inlining

**Kiedy Maglev ma znaczenie:** Aplikacje interaktywne, renderowanie po stronie serwera i każdy workload, w którym kod robi się ciepły, ale nie rozgrzany do czerwoności. Maglev daje Ci 80% korzyści TurboFana przy 10% kosztu kompilacji.

### Poziom 4: TurboFan — ciężki optymalizator

TurboFan to klejnot koronny V8. To pełny spekulacyjny kompilator optymalizujący, który produkuje kod maszynowy dorównujący językom kompilowanym z wyprzedzeniem (ahead-of-time). TurboFan używa unikalnej reprezentacji pośredniej (IR) "Sea of Nodes" — modelu opartego na grafie, który jest bardziej elastyczny niż tradycyjne formaty IR.

TurboFan wykonuje:

- **Optymalizację spekulacyjną** — zakłada typy na podstawie danych z profilowania
- **Inlining funkcji** — zastępuje wywołania funkcji ciałem funkcji
- **Analizę ucieczki (escape analysis)** — eliminuje zbędne alokacje obiektów
- **Rozwijanie pętli (loop unrolling)** i wektoryzację
- **Eliminację martwego kodu**
- **Zwijanie stałych (constant folding)**
- **Analizę zakresów (range analysis)** — eliminuje sprawdzanie granic, gdy jest to bezpieczne

```javascript
function sum(arr) {
  let total = 0;
  for (let i = 0; i < arr.length; i++) {
    total += arr[i];
  }
  return total;
}

// Po wywołaniu sum() z tablicami liczb setki razy
// TurboFan kompiluje ją, zakładając, że:
// - arr jest zawsze tablicą SMI (Small Integers)
// - do arr.length dostęp jest zawsze w ten sam sposób
// - sprawdzanie granic nie jest potrzebne (udowodnione przez range analysis)
// Rezultat: kod maszynowy niemal identyczny z C
```

### Deoptymalizacja — gdy zakłady zawodzą

Optymalizacje TurboFana są *spekulacyjne*. Jeśli wywołasz `sum()` z tablicą liczb 1000 razy, a potem przekażesz mu tablicę stringów, V8 musi wykonać **deoptimization** — wyrzucić zoptymalizowany kod maszynowy i wrócić do bytecode'u Ignition.

```javascript
sum([1, 2, 3]);       // V8 optymalizuje pod tablice liczb
sum([1, 2, 3]);       // Nadal liczby, nadal zoptymalizowane
sum(['a', 'b', 'c']); // BUM — deoptimization!
```

Deoptimization jest kosztowna. V8 musi zrekonstruować stan interpretera ze zoptymalizowanego kodu maszynowego, co może powodować zauważalne przycięcia. Funkcja ostatecznie zostanie ponownie zoptymalizowana z nowymi informacjami o typach, ale powtarzające się cykle deoptimization marnują znaczną ilość czasu.

## Hidden classes: sekret szybkiego dostępu do właściwości

Obiekty w JavaScripcie są dynamiczne — możesz dodawać i usuwać właściwości w dowolnym momencie. To sprawia, że dostęp do właściwości jest z gruntu trudniejszy niż w językach statycznie typowanych, gdzie układ pamięci jest znany w czasie kompilacji.

Rozwiązanie V8: **hidden classes** (wewnętrznie nazywane "Maps").

```javascript
function Point(x, y) {
  this.x = x; // Hidden class C0 → C1 (dodaje 'x' na offsecie 0)
  this.y = y; // Hidden class C1 → C2 (dodaje 'y' na offsecie 4)
}

const p1 = new Point(1, 2);
const p2 = new Point(3, 4);
// Zarówno p1, jak i p2 współdzielą hidden class C2
// V8 wie: x jest na offsecie 0, y jest na offsecie 4
// Dostęp do właściwości to pojedynczy odczyt z pamięci — tak szybki jak struct w C
```

Kiedy tworzysz obiekt, V8 przypisuje mu hidden class. Za każdym razem, gdy dodajesz właściwość, V8 tworzy nową hidden class połączoną z poprzednią przez **transition**. Obiekty tworzone w ten sam sposób współdzielą ten sam łańcuch hidden class.

### Co psuje hidden classes

```javascript
// ŹLE: Niespójna kolejność właściwości
function createUser(name, age) {
  const user = {};
  if (age) {
    user.age = age;   // Hidden class: {} → {age}
    user.name = name;  // Hidden class: {age} → {age, name}
  } else {
    user.name = name;  // Hidden class: {} → {name} — INNA!
    user.age = 0;      // Hidden class: {name} → {name, age} — INNA!
  }
  return user;
}
// Obiekty z różnych gałęzi mają różne hidden classes
// V8 nie może współdzielić zoptymalizowanego kodu między nimi

// DOBRZE: Spójna kolejność właściwości
function createUser(name, age) {
  return { name, age: age || 0 };
  // Zawsze: {} → {name} → {name, age}
  // Wszystkie obiekty współdzielą tę samą hidden class
}
```

### Katastrofa słowa kluczowego delete

```javascript
const obj = { x: 1, y: 2, z: 3 };
delete obj.y; // Zmusza V8 do przełączenia się w Dictionary Mode!
```

Kiedy `delete`'ujesz właściwość, V8 nie może po prostu zaktualizować hidden class. Musi przełączyć obiekt w **dictionary mode** — wyszukiwanie w hash mapie zamiast dostępu po stałym offsecie. Dostęp do właściwości zmienia się z pojedynczego odczytu z pamięci w obliczenie hasha + wyszukanie. **Nigdy nie używaj `delete` na obiektach w hot paths.** Zamiast tego ustaw właściwość na `undefined`.

## Inline caching: pamięć V8

Inline caching (IC) to sposób, w jaki V8 pamięta to, co widział. Każde miejsce dostępu do właściwości w Twoim kodzie ma powiązany inline cache, który zapisuje hidden classes napotkanych obiektów.

### Stany IC

- **Uninitialized** — pierwsze uruchomienie tego kodu, brak danych
- **Monomorphic** — widział dokładnie jedną hidden class. To fast path — V8 generuje pojedyncze sprawdzenie typu + bezpośredni odczyt spod offsetu w pamięci
- **Polymorphic** — widział 2-4 różne hidden classes. V8 przechowuje małą tablicę wyszukiwania. Wciąż szybko, ale nie tak szybko jak monomorphic
- **Megamorphic** — widział więcej niż 4 hidden classes. V8 poddaje się cachowaniu i wraca do ogólnego (wolnego) wyszukiwania właściwości

```javascript
// Monomorphic — najszybciej
function getX(point) { return point.x; }
getX({ x: 1, y: 2 }); // IC uczy się: hidden class A, x na offsecie 0
getX({ x: 3, y: 4 }); // Ta sama hidden class A — trafienie monomorphic!

// Polymorphic — wolniej
getX({ x: 1, y: 2 });       // Hidden class A
getX({ x: 1, y: 2, z: 3 }); // Hidden class B — teraz polymorphic

// Megamorphic — najwolniej
getX({ x: 1 });
getX({ x: 1, y: 2 });
getX({ x: 1, y: 2, z: 3 });
getX({ x: 1, y: 2, z: 3, w: 4 });
getX({ x: 1, a: 2 }); // 5. kształt — megamorphic!
```

**Praktyczna wskazówka:** Przekazuj obiekty o tym samym kształcie do tych samych funkcji. Funkcje fabrykujące i klasy naturalnie to wymuszają.

## Garbage Collection: Orinoco

Garbage collector V8 nazywa się **Orinoco**. To generacyjny, w większości współbieżny kolektor, który zarządza pamięcią w dwóch przestrzeniach:

### Young Generation (Scavenger)

Nowe obiekty lądują w **young generation** — małej, szybkiej do sprzątnięcia przestrzeni. Hipoteza generacyjna mówi, że większość obiektów umiera młodo, i ma rację. V8 używa **równoległego Scavengera** (semi-space copying collector):

1. Nowe obiekty alokowane w przestrzeni "from"
2. Gdy się zapełni, żywe obiekty są kopiowane do przestrzeni "to"
3. Martwe obiekty są po prostu zostawiane — bez markowania, bez sweepowania
4. Przestrzenie zamieniają się rolami

To ekstremalnie szybkie, ponieważ większość obiektów JEST śmieciami — tylko ocaleńcy muszą zostać skopiowani.

### Old Generation (Mark-Compact)

Obiekty, które przetrwają dwie kolekcje young generation, są promowane do **old generation**. Ta przestrzeń jest sprzątana przez kolektor **Mark-Compact**:

1. **Mark** — przejdź wszystkie osiągalne obiekty z referencji korzeniowych
2. **Sweep** — odzyskaj pamięć niezamarkowanych obiektów
3. **Compact** — defragmentuj, przesuwając żywe obiekty razem

### Co czyni Orinoco wyjątkowym

- **Concurrent marking** — markowanie odbywa się w wątkach w tle, podczas gdy Twój JavaScript działa
- **Incremental marking** — markowanie jest dzielone na małe fragmenty przeplatane z wykonaniem JavaScriptu
- **Parallel scavenging** — wiele wątków jednocześnie sprząta young generation
- **Lazy sweeping** — sweepowanie odbywa się przyrostowo, w miarę zapotrzebowania na pamięć

Rezultat: pauzy GC są zwykle poniżej 1ms dla young generation i poniżej 10ms dla pełnych kolekcji. Równoległy Scavenger zredukował czas GC young generation na głównym wątku o 20-50%.

### Unikanie presji na GC

```javascript
// ŹLE: Tworzy nowy obiekt w każdej iteracji
function processItems(items) {
  return items.map(item => ({
    ...item,
    processed: true,
    timestamp: new Date(),
  }));
}

// LEPIEJ: Ponownie wykorzystuj obiekty, gdy to możliwe
function processItems(items) {
  for (const item of items) {
    item.processed = true;
    item.timestamp = Date.now(); // Liczba, nie obiekt
  }
  return items;
}

// ŹLE: Konkatenacja stringów w pętlach tworzy wiele tymczasowych stringów
let result = '';
for (let i = 0; i < 10000; i++) {
  result += items[i].name + ', ';
}

// LEPIEJ: Array join
const result = items.map(i => i.name).join(', ');
```

## Praktyczny przewodnik optymalizacji

Bazując na tym, jak V8 działa wewnętrznie, oto wzorce, które pozwalają silnikowi Ci pomóc:

### 1. Utrzymuj spójne kształty obiektów

```javascript
// Wszystkie obiekty powinny mieć te same właściwości, w tej samej kolejności
class User {
  constructor(name, email, age) {
    this.name = name;
    this.email = email;
    this.age = age;
    // Wszystkie instancje User współdzielą jedną hidden class
  }
}
```

### 2. Unikaj mieszania typów w hot functions

```javascript
// ŹLE: V8 nie może zoptymalizować pod konkretny typ
function process(value) {
  return value + 1;
}
process(42);     // Zoptymalizowane pod liczby
process("hello"); // Deoptimization!

// DOBRZE: Osobne funkcje dla różnych typów
function processNumber(value: number) { return value + 1; }
function processString(value: string) { return value + "1"; }
```

### 3. Używaj TypedArrays do pracy numerycznej

```javascript
// Zwykła tablica — V8 musi obsłużyć mieszane typy
const arr = [1, 2, 3, 4, 5];

// TypedArray — V8 zna dokładny układ pamięci
const arr = new Float64Array([1, 2, 3, 4, 5]);
// Gwarantowana ciągła pamięć, brak boxowania, brak sprawdzania typów
```

### 4. W hot paths preferuj pętle for zamiast abstrakcji

```javascript
// forEach/map mają narzut wywołania funkcji + alokację domknięcia
// W naprawdę gorącym kodzie (miliony iteracji):
const len = arr.length;
for (let i = 0; i < len; i++) {
  // Bezpośredni dostęp, bez narzutu wywołania funkcji
}
```

### 5. Nie używaj obiektu arguments

```javascript
// ŹLE: arguments blokuje wiele optymalizacji
function sum() {
  let total = 0;
  for (let i = 0; i < arguments.length; i++) {
    total += arguments[i];
  }
  return total;
}

// DOBRZE: Rest parameters
function sum(...numbers) {
  let total = 0;
  for (const n of numbers) {
    total += n;
  }
  return total;
}
```

## Debugowanie optymalizacji V8

Możesz faktycznie zobaczyć, co V8 robi z Twoim kodem:

```bash
# Pokaż, które funkcje są optymalizowane/deoptymalizowane
node --trace-opt --trace-deopt your-script.js

# Pokaż stany inline cache
node --trace-ic your-script.js

# Pokaż zdarzenia GC
node --trace-gc your-script.js

# Wygeneruj szczegółowy log optymalizacji
node --trace-opt --trace-deopt --trace-ic --log-file=v8.log your-script.js
```

**W Chrome DevTools:**
- Zakładka Performance → zaznacz "V8 Runtime Call Stats"
- Zakładka Memory → heap snapshots i oś czasu alokacji
- `chrome://tracing` dla niskopoziomowych zdarzeń trace V8

## Kiedy naprawdę warto się tym przejmować

Powiem pragmatycznie: **większość kodu JavaScript nie potrzebuje optymalizacji na poziomie V8.** Jeśli Twój komponent React renderuje się w 2ms, ścinanie 0,1ms przez optymalizację hidden classes jest bez znaczenia.

Powinieneś przejmować się wnętrznościami V8, gdy:

- Przetwarzasz **duże zbiory danych** (>10K elementów) w ciasnych pętlach
- Budujesz **systemy czasu rzeczywistego** — gry, audio, przetwarzanie wideo
- Uruchamiasz **serwer Node.js o wysokiej przepustowości** obsługujący tysiące żądań na sekundę
- Widzisz **pauzy GC** w swoich profilach wydajności
- Budujesz **biblioteki krytyczne wydajnościowo** używane przez tysiące programistów

W przypadku wszystkiego innego pisz czysty, czytelny kod i pozwól V8 wykonać swoją robotę. Skup się na [zasadach czystego kodu frontendowego](/blog/principles-of-clean-frontend-code-dry-kiss-solid) i [właściwej abstrakcji](/blog/wet-vs-aha-avoiding-premature-abstraction-in-frontend-development) zamiast na mikrooptymalizacji. Silnik jest niesamowicie dobry w optymalizowaniu normalnych wzorców kodu. Najgorsze, co możesz zrobić, to napisać "sprytny", mikrooptymalizowany kod, który jest trudniejszy w utrzymaniu i tak naprawdę nie pomaga.
