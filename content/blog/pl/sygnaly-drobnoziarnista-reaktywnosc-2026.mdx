Czytając podziękowania w propozycji TC39 Signals — Angular, Solid, Vue, Svelte, Preact, Qwik, MobX — szybko zauważysz, że brakuje tam jednej wielkiej nazwy. React, framework, do którego większość programistów sięga odruchowo, jest jedynym dużym graczem nieobecnym w grupie, która próbuje ustandaryzować reaktywność w samym JavaScripcie.

To nie zbieg okoliczności. To świadoma decyzja. Przez ostatnie dwa lata niemal każdy inny framework zmierzył się z **fine-grained reactivity** opartą na sygnałach, a React postawił na kompilator — na „przerenderuj wszystko, potem porównaj". W 2026 to największy podział architektoniczny we frontendzie. I tu jest problem: większość developerów Reacta, z którymi rozmawiam, nie potrafi wyjaśnić, co to w ogóle jest sygnał.

Naprawmy to. Praktyczną rzecz ujmując: czym naprawdę jest fine-grained reactivity, jak się pisze w Svelte 5, SolidJS, Angularze i Vue, dlaczego React powiedział „nie" — i co naprawdę powinieneś z tym zrobić, jeśli Reactem zarabiasz chleb.

## Co naprawdę oznacza „drobnoziarnista reaktywność"

Najprostsza forma: **sygnał śledzi, które konkretne wartości się zmieniły, i aktualizuje tylko te fragmenty DOM-u, które od nich zależą — bez przerenderowania całego drzewa komponentów i bez porównywania.**

Model Reacta działa inaczej. Gdy stan się zmienia, React znovuprzebiegnie przez funkcję Twojego komponentu, buduje nowe virtual DOM i uzgadnia go ze starym, aby dowiedzieć się, co zmienić. To model „pull" — React pyta „jak wyglądać powinien teraz UI?" i cofając się, wylicza zmiany. To eleganckie, przewidywalne, i robi dużo niepotrzebnej pracy, którą nigdy nie widzisz.

Sygnały idą odwrotnym kierunkiem. Budują graf zależności w trakcie działania. Sygnał `state` przechowuje wartość; sygnały `computed` z niej czerpią; efekty się do niej łączą. Gdy wartość się zmienia, runtime już wie dokładnie, które computed i efekty jej zależą, więc odświeża tylko je. Nic się nie przerenderowuje. Nic się nie porównuje. Funkcja komponentu, która utworzyła sygnał, uruchamia się zazwyczaj **jeden raz**, i tyle.

To właśnie to — granularnością jest zmiana wartości, a nie komponent. Benchmarki nie kłamią: Solid i Svelte 5 rządzą na JS Framework Benchmark, a różnica między nimi a virtual DOM to przepaść — znacznie większa niż między nimi dwiema.

## Składnia, obok siebie

Każdy framework sygnałowy ma te same trzy prymitywy pod różnymi nazwami: **stan** (zapisywalny), **computed/derived** (tylko do odczytu, cache'owany) i **efekt** (efekty uboczne, które uruchamiają się ponownie, gdy zmieniają się zależności). Kiedy to zobaczysz, frameworki przestają wyglądać egzotycznie.

### SolidJS — sygnały jako jawne wywołania funkcji

Solid to najczystszy wyraz tego modelu. Sygnały to funkcje. *Wywołujesz* je, żeby odczytać.

```jsx
import { createSignal, createMemo, createEffect } from "solid-js";

function Counter() {
  const [count, setCount] = createSignal(0);
  const doubled = createMemo(() => count() * 2); // cache'owana wartość pochodna

  createEffect(() => {
    console.log("count wynosi teraz", count()); // uruchamia się tylko przy zmianie count
  });

  // Ciało tej funkcji wykonuje się RAZ. JSX poniżej jest skompilowany
  // do bezpośrednich aktualizacji DOM podpiętych do sygnałów — bez przerenderowania.
  return (
    <button onClick={() => setCount(count() + 1)}>
      {count()} (podwojone: {doubled()})
    </button>
  );
}
```

Adaptacja mentalna dla programisty Reacta: `count` to nie wartość, to getter. Jeśli zdestrukturyzujesz albo przekażesz `count()` za wcześnie, odczytasz migawkę i tracisz reaktywność. Ceną za tę dyscyplinę jest to, że Solid nigdy niczego nie przerenderowuje.

### Svelte 5 — ten sam model, ukryty przez kompilator

Runes w Svelte 5 to dokładnie ten sam schemat, ale kompilator pozwala pisać je jak zwyczajne zmienne. Bez getterów — `$state` daje Ci wartość, którą mutujesz na miejscu.

```svelte
<script>
  let count = $state(0);
  let doubled = $derived(count * 2);

  $effect(() => {
    console.log("count wynosi teraz", count);
  });
</script>

<button onclick={() => count++}>
  {count} (podwojone: {doubled})
</button>
```

Runes to bezpośrednia inspiracja modelem sygnałów Solida — różnica jest czysto syntaktyczna. Solid wymaga wywołania (`count()`), Svelte — dzięki kompilatorowi — pozwala pisać `count` i `count++` bezpośrednio. Ważne: runes działają w plikach `.svelte.js` poza komponentami, więc stan reaktywny nie jest przywiązany do warstwy widoku.

### Angular — sygnały jako obywatele pierwszej klasy

Angular przebrnął na sygnały, żeby porzucić Zone.js i jego „sprawdź wszystko przy każdym zdarzeniu". W Angularze 20 to natywne, bezzonowe (zoneless) domyślne rozwiązanie.

```typescript
import { Component, signal, computed, effect } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `<button (click)="increment()">
    {{ count() }} (podwojone: {{ doubled() }})
  </button>`,
})
export class CounterComponent {
  count = signal(0);
  doubled = computed(() => this.count() * 2);

  constructor() {
    effect(() => console.log('count wynosi teraz', this.count()));
  }

  increment() {
    this.count.update((n) => n + 1);
  }
}
```

Te same trzy prymitywy, ten sam model czytania-przez-wywołanie. Dla Angulara to przełom: fine-grained aktualizacje pozwalają całkowicie porzucić Zone.js i detektywkę drzewa przy każdym zdarzeniu.

### Vue — framework, który był już w połowie drogi

Vue miał `ref` i `computed` od Vue 3 w 2020, zanim „sygnały" stały się branżowym terminem. W kodzie czytasz ref przez `.value`, w szablonach Vue rozpakuje go sam.

```javascript
import { ref, computed, watchEffect } from 'vue';

const count = ref(0);
const doubled = computed(() => count.value * 2);

watchEffect(() => console.log('count wynosi teraz', count.value));

count.value++; // aktualizuje tylko computed i efekty
```

Vue 4 posunęło się dalej z trybem Vapor — komponenty kompiluje do bezpośrednich operacji na DOM, bez virtual DOM dla wybranych drzew — zamykając przepaść między Vue a Solid.

Spójrz na te cztery fragmenty razem, a zbieżność jest oczywista. Różne słowa kluczowe, jedna idea.

## Propozycja, która może wbudować to w język

2026 to punkt zwrotny nie dlatego, że pojedynczy framework wygrał, ale dlatego, że TC39 próbuje wbudować sygnały *w sam JavaScript*. Propozycja [Signals](https://github.com/tc39/proposal-signals) pięła się do Stage 1 w kwietniu 2024, przy udziale prawie każdego frameworka wymienionego wyżej.

API jest celowo minimalne — dwa typy, które faktycznie będziesz używać, plus niskopoziomowe API dla autorów frameworków:

```javascript
import { Signal } from "signal-polyfill";

const count = new Signal.State(0);
const doubled = new Signal.Computed(() => count.get() * 2);

count.get();     // 0
count.set(5);
doubled.get();   // 10 — liczane leniwko, potem bufforowane
```

`Signal.State` to value, `Signal.Computed` pochodzi z niego, a `Signal.subtle.Watcher` to to, na czym budują efekty. Logika: jak Promises — ustandaryzować prymityw, żeby biblioteki były kompatybilne, a runtime mogły go nativnie zoptymalizować.

Ale realismu — propozycja siedzi w Stage 1 od ponad dwóch lat, bez ruchu do Stage 2. Zamierzone. Autorzy najpierw chcą dowieść sensowności polyfillami, dopiero potem szukać promocji. Nie czekaj. Traktuj to jako potwierdzenie kierunku branży, nie datę wydania.

## Dlaczego React powiedział „nie"

To, co ginie w emocjonalnych dyskusjach: React to **świadoma decyzja filozoficzna**, która ma merytoryczne uzasadnienie.

Dana Abramov streszcza to mniej więcej tak: *„React był rozwiązaniem problemu, którym są sygnały."* Przed Reactem fine-grained reactivity (Knockout, wczesny Angular) oznaczała ręczne tworzenie zależności — a co za tym idzie, miejsca na błędy. Pomysł Reacta: przestań myśleć o zależnościach. Opisz interfejs jako funkcję stanu, framework wylicza resztę. W React **wszystko jest reaktywne domyślnie** — computed to opcjonalna optymalizacja, nie wymóg.

Sygnały to się odwraca. Musisz z góry zdefiniować stan, computed i efekty — i strukturować kod wokół tego grafu. Kontrargument Reacta: to przywraca dokładnie tę kognitywną złożoność, którą React usuwał. A na brzegach semantyka się rozpada (co znaczy sygnał deklarowany wewnątrz `if`, między renderami?).

Dlatego React postawił na **kompilator**. Jeśli problem to zbędne przerenderowania, React Compiler memoizuje komponenty i hooki automatycznie — dostajesz wydajność bliską sygnałom bez zmiany myślenia. Cel ten sam (minimalna praca), droga inna. Jeśli chcesz zagłębić się w mechanikę, mam [Deep Dive w React Compiler](/pl/blog/react-compiler-automatyczna-memoizacja) i [artykuł o React Fiber](/pl/blog/react-fiber-silnik-nowoczesnego-reacta).

Czy ten zakład wyjdzie, czekamy i zobaczymy. Nawet w samym Reaccie nie wszystkie drzwi są zamknięte — niektórzy rozważają sygnały jako *implementacyjny szczegół kompilatora*, ale nie jako publiczne API.

## Czy możesz używać sygnałów w React już dziś? (Tak, na swój sposób)

Nie musisz opuszczać Reacta. Kilka dróg:

**Preact Signals dla Reacta.** `@preact/signals-react` wstrzykuje prawdziwe sygnały do Reacta. Czytając `.value`, rejestrujesz komponent jako słuchacza — i tylko komponenty, które go czytają, renderują się na nowo. Faktycznie granularnie, nie jak surowy `useState`.

```jsx
import { signal } from "@preact/signals-react";

const count = signal(0); // żyje poza komponentem — nie resetuje się przy remouncie

function Counter() {
  // .value łączy TEN komponent z sygnałem; siostrzane komponenty się nie renderują
  return <button onClick={() => count.value++}>{count.value}</button>;
}
```

**Store'y w stylu sygnałów.** Jotai, Valtio, Zustand opierają się na tej samej idei — subskrybuj fragmenty stanu, żeby niepowiązane komponenty się nie rysowały. Nie są to prymitywy TC39, ale rozwiązują ten sam problem wewnątrz Reacta i łączą się z `useSyncExternalStore`.

**Albo po prostu `useSyncExternalStore`.** Jeśli piszesz bibliotekę i chcesz reaktywności na sygnałach bez bałaganu, to hook jest dla Ciebie. Inne biblioteki stanu na nim się opierają.

Szczera rada: nie doklejaj Preact Signals do starych bazodanowych aplikacji Reacta dla trendu. Sensowne tylko wtedy, gdy konkretne gorące miejsce — pozycja kursora, live chart, canvas narzędzia — się zawieszą. Wtedy sięgnij po store. Do reszty masz React Compiler.

## Gdzie to naprawdę się liczy: opóźnienie interakcji

Fine-grained reactivity to nie teoretyczna piękność. Pokazuje się w **Interaction to Next Paint (INP)**, metryce Core Web Vitals, która zastąpiła First Input Delay. INP to czas od akcji użytkownika do wizualnej odpowiedzi — a cykl „przerenderuj, porównaj, zatwierdź" to dokładnie ten rodzaj pracy na głównym wątku, która ją spowalnia.

Na dashboardzie z tysiącami wierszy sygnałowy framework aktualizuje trzy komórki i koniec. Virtual DOM uzgadnia drzewo, opiera się na memoizacji, żeby nie robić niepotrzebnej pracy. Oba mogą mieć dobry INP — ale sygnały osiągają go strukturalnie (domyślnie), React musi pracować (Ty lub kompilator). Jeśli INP przy skali Cię obchodzi, to nie jest czysto estetyczna dyskusja — to łączy się z [budżetami wydajności](/pl/blog/budzety-wydajnosci-frontend).

## Pułapki, o których nikt nie opowiada

Gdy przechodzisz z mentalnością Reacta na sygnały:

- **Czytanie za wcześnie psuje reaktywność.** W Solidzie `count()` zamiast `count` to snapshot. W Vue destrukturyzacja traci śledzenie. Zależności działają tylko jeśli czytasz *w środku* scope'a reaktywnego.
- **Effect nie to co `useEffect`.** Sygnałowy `effect` nie ma tablicy zależności — śledzi automatycznie. Mniej kodu, ale pułapka: odczytaj sygnał, którego nie chciałeś, a effect będzie się palić zbyt często.
- **Stare domknięcia mają bliźnię.** Sygnały zabijają bug nieaktualnych domknięć w React (zawsze masz nową wartość), ale wprowadzają przeciwnie — zapomnij sygnałować coś, a to nigdy się nie zmieni.
- **Komponent uruchamia się raz, nie wielokrotnie.** Console.log, lokalne zmienne, setup — wszystko jeden raz. Devowie Reacta spodziewają się przebiegu ciała; się nie dzieje.
- **LLM'y tutaj zawdzięczają.** ChatGPT i inne były uczone na latach Reacta i Svelte 3/4 — wciąż hallucynują starą składnię. Store'y zamiast runes, `$:` zamiast `$effect`. Sprawdzaj w dokumentacji.

## Testowanie i debugowanie

Testy sygnałów to odśwież: stan żyje poza renderowaniem, tworzysz sygnał, mutojesz, sprawdzasz wartość computed — bez render harnesu, bez `act()`, bez fake timerów. Solid daje `createRoot` na scope reaktywny; Svelte `.svelte.js` testuje się jak normalne funkcje.

Debugging zmienia się z „czemu to się renderowało?" na „czemu to się (nie) zaktualizowało?". Solid devtools rysują graf; Svelte 5 ma `$inspect` do śledzenia zmian rune'a. Pytania są inne, ale mniej ich — gonisz jedną krawędź grafu, nie kaskadę renderów.

## Więc co teraz?

Praktycznie:

- **Nowy projekt, wydajność ma znaczenie?** Solid albo Svelte 5 to świetne wybory — model reaktywności faktycznie pomaga. Wybieraj po ekosystemie i dostępnych devach, nie benchmarkach — oba są wystarczające.
- **Głęboki React?** Nie przepisuj. Włącz [React Compiler](/pl/blog/react-compiler-automatyczna-memoizacja), sięgnij po store na sygnały (Jotai/Valtio/Zustand) tylko gdzie trzeba — gorące miejsca. Tu trzymam się pragmatyzmu — tak jak w [unikaniu przedwczesnej abstrakcji](/pl/blog/wet-aha-przedwczesna-abstrakcja-frontend).
- **Angular lub Vue?** Już jesteś na ścieżce sygnałów — `signal()`/`ref` i (w Angularze) brak Zone.js.
- **Język, nie framework?** Obserwuj TC39, ale pisz na dziś. Stage 1 to konsensus, nie data.

„Sygnały wygrały" — w połowie. Wygrały *debatę o modelu reaktywności* wszędzie oprócz Reacta, gdzie stawia się na kompilator, czyniąc całą dyskusję szczegółem implementacyjnym. W 2026 nie musisz wybierać strony, żeby być dobrym frontend devem. Ale musisz rozumieć oba, bo framework, po który sięgniesz, to definiuje.
