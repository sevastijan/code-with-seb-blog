Last year I watched a senior backend engineer spend four days getting a PostgreSQL database provisioned for a new microservice. Four days. Not because the infrastructure was complex — it was a standard RDS instance with a predefined security group. It took four days because the request sat in a Jira queue, got assigned to a DevOps engineer who was mid-sprint on something else, required a Slack thread to clarify naming conventions, needed a Terraform plan review, and then failed the first apply because someone had hardcoded a subnet ID three months ago.

The engineer could have been building features. Instead, they were refreshing a Jira ticket and DMing the ops team asking "any update?"

That story isn't unusual. Humanitec's 2025 State of Platform Engineering report found that developers at mid-size companies spend an average of 8.2 hours per week on infrastructure-adjacent tasks — provisioning, configuring, debugging environments, waiting on tickets. That's an entire day per developer per week, gone.

This is the problem Internal Developer Platforms solve. And in 2026, they've gone from "thing Spotify built" to "thing your company probably needs."

## The Shift: From Individual Tools to Platforms

For the past decade, we've been accumulating tools. GitHub for code. Jenkins or GitHub Actions for CI. Terraform for infrastructure. ArgoCD for deployments. Datadog for monitoring. PagerDuty for incidents. Each tool excellent at its job, each with its own dashboard, its own auth, its own mental model.

The problem isn't any individual tool. The problem is that nobody stitched them together from the developer's perspective. A new engineer joining your team doesn't need to learn Terraform — they need to provision a database. They don't need to understand ArgoCD — they need to deploy their service. The gap between "what the developer wants to do" and "which tools they need to operate" kept widening.

Platform engineering emerged to close that gap. Instead of teaching every developer how to use 12 different tools, you build a platform layer that exposes the right abstractions: "create a service," "add a database," "set up monitoring." The tools still exist underneath. But developers interact with a single, opinionated interface.

If you're already [optimizing your CI/CD pipelines](/blog/optimizing-cicd-pipelines-dx-deploy-times-guide) for speed, an IDP is the natural next step — it wraps that fast pipeline into a self-service experience where developers don't even think about the pipeline configuration. They push code, and the platform handles the rest.

## What an IDP Actually Is

An Internal Developer Platform (IDP) is a self-service portal backed by pre-approved, pre-configured infrastructure and tooling. It's the layer between your developers and your infrastructure that lets them do things without filing tickets.

At its core, an IDP provides:

- **A service catalog.** Every service, API, library, and data pipeline your company runs, listed in one place with ownership, dependencies, documentation, and health status.
- **Self-service provisioning.** "I need a Redis cache" becomes a form submission that triggers approved Terraform, validates naming conventions, tags resources correctly, and provisions in minutes — not days.
- **Golden paths.** Opinionated templates for creating new services, APIs, or frontend apps. Instead of "clone this old repo and delete the parts you don't need," developers run a scaffolder that produces a production-ready starting point with CI, monitoring, and security baked in.
- **A plugin ecosystem.** Integrations with your existing tools — GitHub, PagerDuty, Datadog, Kubernetes, cloud providers — displayed in one unified interface.

The key word is "pre-approved." The platform team defines what's allowed — which instance sizes, which regions, which security configurations. Developers get freedom within guardrails. They don't need to ask permission because the guardrails already enforce policy.

## The Problem It Solves: "I Need a Database" Shouldn't Require a Jira Ticket

Let me paint the before-and-after.

**Before IDP (most companies today):**

1. Developer needs a PostgreSQL database for a new feature.
2. Developer creates a Jira ticket in the DevOps queue.
3. Ticket sits for 1-3 days until a DevOps engineer picks it up.
4. DevOps engineer asks clarifying questions via Slack (what size? what region? what naming convention?).
5. DevOps engineer writes Terraform, creates PR.
6. Another DevOps engineer reviews the PR (1-day turnaround).
7. Terraform applies. Database is provisioned.
8. Connection strings are manually added to the service's environment.
9. Developer can finally start building.

**Total time: 3-7 business days.**

**After IDP:**

1. Developer opens the platform portal.
2. Developer selects "Add PostgreSQL database" from the service catalog.
3. Developer fills in a form: database name, size (small/medium/large), environment.
4. Platform validates the request against policy, triggers Terraform, provisions the database, injects connection strings into the service's config.
5. Developer receives a Slack notification: "Your database is ready."

**Total time: 15 minutes.**

Same infrastructure. Same Terraform modules. Same security policies. The only difference is who initiates the workflow and how much manual coordination is required.

## Key IDP Tools in 2026

The IDP space has matured significantly. Here are the tools that matter.

### Backstage (Spotify, Open Source)

Backstage is the 800-pound gorilla. Open-sourced by Spotify in 2020, it's now a CNCF Incubating project with 28,000+ GitHub stars and contributions from hundreds of companies. It's a framework for building your own developer portal — not a turnkey product.

**Best for:** Teams that want full control and have the engineering capacity to build and maintain a custom portal.

### Port

Port is the "buy" option. It provides a no-code/low-code platform for building a service catalog and self-service actions without writing a custom frontend. You define your data model, connect your integrations, and build workflows through their UI.

**Best for:** Teams that want IDP capabilities without dedicating 2-3 engineers to building and maintaining Backstage.

### Cortex

Cortex focuses on service quality and standards. It's less about provisioning and more about ensuring every service meets your engineering standards — has runbooks, has on-call rotation, has SLOs defined, passes security scans.

**Best for:** Teams where the primary pain point is "we don't know what we own" rather than "provisioning is slow."

### Humanitec

Humanitec takes a different approach entirely. Instead of building a portal, it provides a Platform Orchestrator that sits between developers and infrastructure. Developers describe what they need (a workload with a database and a DNS entry), and Humanitec figures out how to provision it based on platform team-defined rules.

**Best for:** Teams that want to abstract away Kubernetes complexity without building a full portal.

## Backstage Deep Dive

Backstage deserves a deeper look because it's what most teams evaluate first.

### The Service Catalog

The catalog is Backstage's foundation. Every entity in your organization — services, APIs, libraries, teams, infrastructure components — is registered as a YAML descriptor:

```yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: payments-api
  description: Handles all payment processing
  annotations:
    github.com/project-slug: myorg/payments-api
    pagerduty.com/service-id: PXXXXXX
spec:
  type: service
  lifecycle: production
  owner: team-payments
  dependsOn:
    - resource:payments-db
    - component:auth-service
  providesApis:
    - payments-api
```

This file lives in the service's repo. Backstage reads it and builds a searchable, browsable catalog. Click on `payments-api` and you see: who owns it, what it depends on, its CI status, its PagerDuty incidents, its API docs, its Kubernetes deployments — all in one page.

For companies managing a [monorepo with Turborepo or Nx](/blog/monorepo-turborepo-nx-react-nextjs-guide), the catalog becomes essential for tracking which packages and apps live inside the monorepo and how they relate to each other.

### The Software Scaffolder

The scaffolder is where "golden paths" come to life. You define templates that generate entire project structures:

```yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: nextjs-service
  title: Next.js Microservice
spec:
  parameters:
    - title: Service Details
      properties:
        name:
          type: string
          description: Name of the service
        owner:
          type: string
          ui:field: OwnerPicker
  steps:
    - id: fetch-template
      action: fetch:template
      input:
        url: ./skeleton
    - id: create-repo
      action: publish:github
    - id: register-catalog
      action: catalog:register
    - id: provision-infra
      action: custom:terraform-apply
```

A developer clicks "Create New Service," fills in a form, and Backstage creates the repo, sets up CI, provisions infrastructure, registers the service in the catalog, and configures monitoring — all in one click. No Jira tickets, no Slack threads, no copy-pasting from another repo.

### TechDocs

TechDocs is Backstage's built-in documentation system. It reads Markdown files from your repos and renders them inside the Backstage portal, right next to the service they document. This solves the "where's the documentation?" problem because the answer is always "in Backstage."

### The Plugin Ecosystem

Backstage has 200+ community plugins covering everything from Kubernetes to cost management to security scanning. The plugin architecture means you're building a portal tailored to your company's specific toolchain rather than adapting to someone else's opinions about what tools you should use.

## Platform Engineering vs DevOps: What's the Actual Difference?

This distinction confuses a lot of people, so let me be direct.

**DevOps** is a philosophy: break down silos between development and operations. In practice, it manifested as DevOps engineers who manage infrastructure, write Terraform, maintain CI/CD, and respond to production incidents.

**Platform engineering** is a practice: build an internal product (the platform) that serves developers as customers. Platform engineers don't respond to individual requests — they build self-service capabilities that eliminate the need for requests.

Here's the key insight: **DevOps scales linearly. Platform engineering scales logarithmically.**

With DevOps, every new service means more work for the DevOps team. More Terraform to write, more pipelines to configure, more tickets to process. The ratio of DevOps engineers to developers stays roughly constant (typically 1:6 to 1:10).

With platform engineering, every new capability you build into the platform serves all current and future teams. Once "provision a database" is self-service, it costs zero additional DevOps time regardless of whether 5 or 500 developers use it.

That said, platform engineering doesn't replace DevOps. It's built on top of DevOps practices. You still need Terraform, CI/CD, monitoring, and incident response. The platform just wraps these capabilities in a developer-friendly interface.

## When You Need an IDP (And When You Don't)

Not every team needs an Internal Developer Platform. Here's my honest assessment.

### You probably need an IDP if:

- **Your engineering team has 15-20+ developers.** Below this threshold, informal processes work. A quick Slack message to your DevOps person gets a database provisioned in a few hours. Above this threshold, the DevOps team becomes a bottleneck and informal processes break down.
- **Provisioning infrastructure takes days, not minutes.** If "I need a staging environment" triggers a multi-day Jira workflow, you have a problem an IDP solves.
- **Onboarding new engineers takes more than a week.** If a new hire spends their first week learning which repo is which, who owns what, and how to deploy, a service catalog would cut that dramatically.
- **You can't answer "what services do we run?" in under 5 minutes.** If the answer requires checking three different dashboards and asking two people, you need a catalog.
- **Your DevOps team spends more time on tickets than on building.** If your infrastructure engineers are stuck in a ticket queue instead of improving the platform, an IDP breaks the cycle.

### You probably don't need an IDP if:

- **Your team is under 10 engineers.** The overhead of building and maintaining a platform isn't justified. A well-organized Notion doc and quick Slack communication will do.
- **You have a single product with a single deploy target.** If everyone works on one Next.js app deployed to Vercel, you don't need a service catalog.
- **Your infrastructure is fully serverless.** If everything runs on Vercel, Supabase, and managed services, there's not much to abstract away.

If you're a startup working with a [fractional CTO](/blog/fractional-cto-playbook-startup-guide), the IDP conversation usually starts around the Series A stage when the engineering team begins scaling past that 15-20 person threshold. Before that, the investment doesn't make sense.

## Building vs Buying: Backstage vs Port/Cortex

This is the biggest decision in the IDP space.

### Build: Backstage

**Pros:**
- Fully customizable — your portal, your way
- Open source — no vendor lock-in, large community
- Deep plugin ecosystem — 200+ integrations
- Free (aside from engineering time)

**Cons:**
- Requires 2-3 dedicated engineers to build and maintain
- Steep initial setup (plan 2-4 months for a basic portal)
- React frontend — your platform team needs frontend skills
- Upgrades can be painful — breaking changes between versions

**Total cost of ownership (year one):** $300-500K in engineering time for a mid-size deployment.

### Buy: Port / Cortex

**Pros:**
- Up and running in weeks, not months
- No engineering team required for maintenance
- Built-in best practices from hundreds of deployments
- Regular feature updates without migration headaches

**Cons:**
- Monthly SaaS cost ($20-50K/year for mid-size teams)
- Less customizable — you work within the product's constraints
- Vendor dependency — switching costs are real
- May not support your specific, niche integration

**Total cost of ownership (year one):** $50-100K in licensing plus 0.5-1 engineer for configuration.

**My recommendation:** If you have fewer than 100 engineers, start with a buy option. The time-to-value is dramatically faster, and you'll learn what you actually need before investing in a custom build. If you're above 200 engineers and have unique requirements, Backstage is likely the right choice.

## The Developer Experience Angle: Golden Paths, Not Golden Cages

The best IDPs follow a principle I call "guardrails, not gates." The difference matters.

**Gates** block developers until someone approves their request. "You can't create a database until DevOps reviews your ticket." Gates create bottlenecks and frustration.

**Guardrails** let developers move fast within safe boundaries. "You can create any database you want, but it will be encrypted, tagged, and sized within these approved tiers." Guardrails enable autonomy while maintaining standards.

Golden paths are the ultimate expression of guardrails. A golden path is an opinionated, pre-built way to accomplish a common task. Need a new microservice? Here's the golden path: use this template, it creates the repo with CI, monitoring, security scanning, and an SLO dashboard. You can deviate from the golden path if you need to — but most developers won't, because the golden path is genuinely the best way to do it.

The result is consistency without bureaucracy. Every service looks similar, follows the same conventions, and integrates with the same tooling — not because someone enforces it through code review, but because the easy path is also the correct path.

## Real Example: Onboarding in 2 Days Instead of 2 Weeks

A 50-person engineering team I worked with last year had a brutal onboarding problem. New engineers needed two full weeks before they could ship their first meaningful change. The first week was spent understanding the codebase, getting local environments running, and figuring out who owned what. The second week was navigating deployment workflows, understanding the monitoring setup, and learning the team's conventions.

We implemented Port as their IDP layer over eight weeks. Here's what changed:

**Day 1 for a new engineer:**
- Open the service catalog. Browse every service the company runs, who owns it, what it does, its health status, and its documentation.
- Click on the service they'll be working on. See its dependencies, deployment history, recent incidents, and tech stack.
- Use the "Set Up Dev Environment" self-service action to get a local environment running in 20 minutes instead of following a 47-step wiki page.

**Day 2 for a new engineer:**
- Use the scaffolder to create a small feature branch with all the boilerplate pre-configured.
- Push code, watch the CI pipeline (which they can monitor in the portal, not by digging through GitHub Actions logs).
- Deploy to staging via the portal's deployment workflow.
- Ship their first PR.

Onboarding went from 2 weeks to 2 days. Not because the codebase got simpler or the documentation got better — but because everything a new engineer needs to know is in one place, accessible through one interface. Teams that have already invested in [structured onboarding with Claude Code](/blog/claude-code-team-leads-onboarding-guide) can layer that on top of an IDP for an even smoother experience — the IDP handles infrastructure context while the AI assistant handles code context.

## AI + IDP: The Convergence That's Happening Now

This is where things get genuinely interesting in 2026. AI coding tools and IDPs are converging in ways that amplify each other.

### AI-Powered Scaffolding

Backstage's scaffolder is template-based — you define the template, developers fill in parameters. But Claude Code and similar tools can take this further. Instead of choosing from a fixed set of templates, a developer describes what they need in natural language:

> "I need a new Node.js service that consumes events from our Kafka cluster, processes payment webhooks, and stores results in PostgreSQL."

An AI agent, integrated with Backstage's catalog (so it knows what infrastructure exists) and the scaffolder (so it can trigger approved workflows), can:

1. Identify the right template (Node.js event consumer).
2. Pre-fill parameters based on the description (Kafka cluster name, database type).
3. Add the appropriate dependencies from the catalog (payment-events topic, payments-db).
4. Generate service-specific boilerplate code that Claude Code or Copilot can then extend.

This isn't hypothetical — Backstage's plugin architecture makes it straightforward to build an AI layer that wraps existing scaffolder templates with natural language input. Several companies have shipped internal versions of this in the past six months.

### Intelligent Service Discovery

When an AI coding assistant like Claude Code has access to the service catalog via MCP (Model Context Protocol), it can answer questions that previously required tribal knowledge:

- "Which service handles user authentication?" — answered by querying the catalog.
- "What's the schema for the orders API?" — answered by pulling API specs from the catalog.
- "Who should I talk to about the payments service?" — answered by looking up ownership in the catalog.

This turns the service catalog from a thing humans browse into a knowledge base that AI agents query, making every developer effectively an expert on the entire system from day one.

## The ROI Argument: How to Pitch an IDP to Leadership

If you're convinced your team needs an IDP but need to convince leadership, here's the framework I use.

### Quantify the waste

Track these numbers for two weeks:

- **Hours spent on infrastructure tickets** — count ticket creation time, wait time, and resolution time.
- **Onboarding time for new engineers** — how many days until first meaningful commit.
- **Incidents caused by configuration drift** — production issues from manual, inconsistent infrastructure setup.
- **DevOps team utilization** — percentage of time spent on repeatable tasks vs. strategic improvements.

### Build the cost model

For a 50-person engineering team with an average fully-loaded cost of $180K/year per engineer:

- **8 hours/week per developer on infra tasks** = 20% productivity loss = **$1.8M/year in wasted capacity**.
- **2-week onboarding** for 15 new hires/year at $3,500/week = **$105K/year in slow ramp**.
- **2 configuration-related incidents/month** at $5K average cost = **$120K/year in incident costs**.

**Total annual cost of not having an IDP: ~$2M.**

A Port or Cortex deployment costs $50-100K/year in licensing plus $150-200K in engineering setup. You break even within months.

### Frame it as a product investment, not an infrastructure cost

The language matters. Don't pitch "we need a new DevOps tool." Pitch "we're building an internal product that makes every engineer 20% more productive." Leadership understands product investment. They're skeptical of infrastructure spending.

Show the curve: the platform team is 3-4 engineers whose work makes 50+ engineers more effective. That's 12-15x leverage. No feature team delivers that kind of ROI.

## Getting Started: The Pragmatic Path

You don't need to build a full IDP on day one. Start with the highest-pain problem:

1. **Week 1-2:** Build a service catalog. Even a spreadsheet is better than nothing. Document every service, who owns it, and where the code lives.
2. **Month 1:** Evaluate Port, Cortex, or Backstage. Run a proof of concept with one team.
3. **Month 2-3:** Automate your top 3 most-requested infrastructure tasks as self-service actions.
4. **Month 4-6:** Roll out the platform to all engineering teams. Measure time saved.
5. **Month 6+:** Add golden path templates, AI integrations, and advanced workflows.

The goal isn't perfection — it's removing the friction that keeps your engineers waiting instead of building. Start with the Jira ticket that takes 4 days and turn it into a button that takes 4 minutes.

The companies that figured this out early are shipping faster, onboarding faster, and retaining engineers better. The gap between "platform-enabled" and "ticket-driven" engineering organizations is widening every quarter. In 2026, an IDP isn't a luxury — it's table stakes for any team serious about developer experience.
