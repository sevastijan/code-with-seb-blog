I have been building with the Claude API for over a year. Custom prompts, tool-use chains, retry logic, streaming handlers — the full stack of plumbing you write when you want an LLM to actually do things. It works, but every project starts with the same 400 lines of boilerplate before you get to the interesting part.

Then Anthropic released the Claude Agent SDK and I deleted most of that code.

The Agent SDK is not another wrapper around the Messages API. It is the same runtime that powers Claude Code — the same tool implementations, the same permission model, the same agent loop — exposed as a TypeScript library you `npm install` into your own projects. If you have used Claude Code and wished you could script it, this is exactly that. The shared substrate beneath all of it is Bun, which [Anthropic acquired in December 2025](/blog/anthropic-bun-acquisition-claude-code-strategy) precisely because Claude Code's distribution and the Agent SDK's runtime story both depend on it.

## What the Agent SDK Actually Is

Claude Code is an interactive CLI agent. You type a prompt, it reasons, calls tools, edits files, runs commands. The Agent SDK extracts the engine behind that experience and gives you programmatic control.

You get:

- **The same built-in tools** — Read, Write, Edit, Bash, Glob, Grep, WebSearch, WebFetch, Monitor, Agent
- **The same agentic loop** — the model decides which tools to call, processes results, and iterates until the task is done
- **The same permission model** — sandbox modes, allow/deny lists, approval callbacks
- **TypeScript-first** — full type safety, async/await, streaming support

The mental model: Claude Code is the GUI. The Agent SDK is the API. Same engine, different interface.

## Installation and Quickstart

```bash
npm install @anthropic-ai/claude-agent-sdk
```

You need an Anthropic API key. Set it as an environment variable or pass it directly:

```typescript
import { Agent } from '@anthropic-ai/claude-agent-sdk';

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  apiKey: process.env.ANTHROPIC_API_KEY,
});

const result = await agent.run('Read the README.md and summarize it');
console.log(result.text);
```

That is a complete working agent. It can read files, reason about them, and return a structured answer. The `run` method starts the agentic loop — the model calls tools as needed, processes results, and returns when it has a final answer.

For streaming (which you will want in production for progress feedback):

```typescript
const stream = agent.stream('Analyze the src/ directory and find security issues');

for await (const event of stream) {
  if (event.type === 'tool_use') {
    console.log(`Using tool: ${event.tool}`);
  }
  if (event.type === 'text') {
    process.stdout.write(event.content);
  }
}
```

## Built-in Tools: What You Get Out of the Box

The SDK ships with the same tool suite that makes Claude Code effective. Here is what each one does and when the model reaches for it:

| Tool | Purpose | When it fires |
|------|---------|---------------|
| **Read** | Read file contents | Analyzing code, configs, docs |
| **Write** | Create new files | Generating code, configs |
| **Edit** | Modify existing files | Refactoring, fixing bugs |
| **Bash** | Execute shell commands | Running tests, builds, git |
| **Glob** | Find files by pattern | Discovering project structure |
| **Grep** | Search file contents | Finding usages, patterns |
| **WebSearch** | Search the web | Fetching documentation, research |
| **WebFetch** | Fetch a URL | Reading web pages, APIs |
| **Monitor** | Watch processes | Long-running builds, servers |
| **Agent** | Spawn sub-agents | Parallel work, specialized tasks |

You do not need to register these. They are available by default. You can disable any of them:

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  disabledTools: ['Bash', 'Write', 'Edit'], // read-only agent
});
```

This is how you build a safe analysis agent that can explore a codebase but never modify it. I use this pattern for code review bots — they need Read, Glob, and Grep, but should never touch a file.

## Building Custom Tools

Here is where the SDK gets interesting. Custom tools are registered as in-process [MCP](/blog/mcp-vs-a2a-ai-agent-protocols-guide-2026) servers. You do not need to run a separate process or set up HTTP endpoints — the SDK hosts the MCP server inside your Node.js process.

```typescript
import { Agent, Tool } from '@anthropic-ai/claude-agent-sdk';

const jiraLookup: Tool = {
  name: 'jira_lookup',
  description: 'Look up a Jira ticket by ID and return its details',
  inputSchema: {
    type: 'object',
    properties: {
      ticketId: { type: 'string', description: 'Jira ticket ID, e.g. PROJ-123' },
    },
    required: ['ticketId'],
  },
  async execute({ ticketId }) {
    const response = await fetch(
      `https://your-org.atlassian.net/rest/api/3/issue/${ticketId}`,
      {
        headers: {
          Authorization: `Basic ${Buffer.from(`${JIRA_EMAIL}:${JIRA_TOKEN}`).toString('base64')}`,
        },
      }
    );
    const issue = await response.json();
    return {
      summary: issue.fields.summary,
      status: issue.fields.status.name,
      assignee: issue.fields.assignee?.displayName ?? 'Unassigned',
      description: issue.fields.description,
    };
  },
};

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  tools: [jiraLookup],
});

await agent.run('Look up PROJ-456 and tell me if it is ready for code review');
```

The model sees your custom tool alongside the built-in ones and decides when to use it based on the description. Good descriptions matter — they are the model's only documentation for your tool.

You can register as many custom tools as you need. I have agents with 15+ custom tools that connect to Slack, GitHub, Jira, internal APIs, and databases. The model handles tool selection surprisingly well as long as descriptions are clear and non-overlapping.

## Sub-agent Spawning for Parallel Work

One of the most powerful patterns in the SDK is spawning sub-agents for parallel work. If you have read my guide on [Claude Code sub-agents and multi-agent orchestration](/blog/claude-code-sub-agents-multi-agent-systems-guide), you know the orchestrator-worker pattern can deliver 90%+ performance gains on complex tasks. The SDK makes this pattern programmable.

```typescript
const orchestrator = new Agent({
  model: 'claude-opus-4-20250514',
  systemPrompt: `You are a code review orchestrator.
    Break the review into parallel tasks and delegate to sub-agents.
    Synthesize their findings into a single report.`,
  maxSubAgents: 5,
  subAgentConfig: {
    model: 'claude-sonnet-4-20250514',
    disabledTools: ['Write', 'Edit', 'Bash'], // read-only workers
  },
});
```

The orchestrator spawns sub-agents via the built-in Agent tool. Each sub-agent gets its own context window, runs independently, and returns results to the orchestrator. You control the model, tools, and limits for sub-agents separately from the orchestrator.

The cost implication is significant: your orchestrator uses Opus for strategic reasoning, but the workers run on Sonnet for the mechanical analysis work. This is the same `opusplan` principle I covered in my [token optimization guide](/blog/claude-code-token-optimization-bill-cut-guide) — Opus-grade planning, Sonnet-grade execution.

## Hooks for Lifecycle Events

The SDK exposes lifecycle hooks that let you intercept and modify agent behavior at every stage. If you are familiar with [Claude Code hooks](/blog/claude-code-hooks-custom-agents-ai-pipelines-guide), the SDK hooks follow the same event model but with a TypeScript callback interface instead of shell scripts.

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  hooks: {
    onToolUse: async (event) => {
      // Log every tool call to your observability stack
      await telemetry.track('tool_use', {
        tool: event.toolName,
        input: event.input,
        sessionId: event.sessionId,
      });

      // Block dangerous operations
      if (event.toolName === 'Bash' && event.input.command.includes('rm -rf')) {
        return { decision: 'deny', reason: 'Destructive command blocked by policy' };
      }

      return { decision: 'allow' };
    },

    onToolResult: async (event) => {
      // Add context after tool execution
      if (event.toolName === 'Read' && event.result.includes('TODO')) {
        return {
          additionalContext: 'Note: this file contains TODO comments that may need attention.',
        };
      }
    },

    onAgentStop: async (event) => {
      // Post-processing when the agent completes
      await saveReport(event.finalOutput);
    },
  },
});
```

Hooks are how you add guardrails, observability, and custom logic without modifying the agent's core behavior. I use them heavily for:

- **Audit logging** — every tool call recorded with inputs and outputs
- **Cost tracking** — token counts per session and per tool
- **Security gates** — blocking file writes to certain directories
- **Auto-formatting** — running Prettier after every Write or Edit

## MCP Server Integration

Beyond in-process tools, you can connect to external MCP servers. This is how you integrate with the broader [MCP ecosystem](/blog/mcp-vs-a2a-ai-agent-protocols-guide-2026) — databases, SaaS platforms, internal services that already expose MCP endpoints.

```typescript
const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  mcpServers: [
    {
      name: 'postgres',
      command: 'npx',
      args: ['-y', '@modelcontextprotocol/server-postgres'],
      env: {
        POSTGRES_CONNECTION_STRING: process.env.DATABASE_URL,
      },
    },
    {
      name: 'github',
      command: 'npx',
      args: ['-y', '@modelcontextprotocol/server-github'],
      env: {
        GITHUB_TOKEN: process.env.GITHUB_TOKEN,
      },
    },
  ],
});

await agent.run('Check the latest PRs on our repo and summarize any that touch the auth module');
```

The SDK manages the MCP server lifecycle — starting them, maintaining connections, shutting them down when the agent finishes. You declare servers, the SDK handles the rest.

This is particularly powerful when combined with custom tools. Your in-process tools handle business-specific logic (Jira lookups, Slack notifications) while MCP servers handle standardized integrations (databases, file systems, APIs).

## Production Example: A Code Review Agent

Let me show you a real agent I run in CI. It reviews every PR, checks for security issues, verifies test coverage, and posts a summary comment on GitHub.

```typescript
import { Agent, Tool } from '@anthropic-ai/claude-agent-sdk';

const postGithubComment: Tool = {
  name: 'post_review_comment',
  description: 'Post a code review comment on a GitHub pull request',
  inputSchema: {
    type: 'object',
    properties: {
      prNumber: { type: 'number' },
      body: { type: 'string' },
      repo: { type: 'string' },
    },
    required: ['prNumber', 'body', 'repo'],
  },
  async execute({ prNumber, body, repo }) {
    const response = await fetch(
      `https://api.github.com/repos/${repo}/issues/${prNumber}/comments`,
      {
        method: 'POST',
        headers: {
          Authorization: `Bearer ${process.env.GITHUB_TOKEN}`,
          'Content-Type': 'application/json',
        },
        body: JSON.stringify({ body }),
      }
    );
    return { success: response.ok, status: response.status };
  },
};

const reviewer = new Agent({
  model: 'claude-sonnet-4-20250514',
  systemPrompt: `You are a senior code reviewer. For each PR:
    1. Read the changed files using Read and Grep
    2. Check for security vulnerabilities (SQL injection, XSS, auth bypasses)
    3. Verify test coverage exists for new functionality
    4. Check for performance anti-patterns
    5. Post a structured review comment using post_review_comment

    Format your review as:
    ## Summary
    ## Security
    ## Testing
    ## Suggestions

    Be concise. Flag issues, not style preferences.`,
  tools: [postGithubComment],
  disabledTools: ['Write', 'Edit'], // never modify code
  hooks: {
    onToolUse: async (event) => {
      if (event.toolName === 'Bash') {
        const allowed = ['git diff', 'git log', 'git show', 'npm test'];
        const isAllowed = allowed.some((cmd) => event.input.command.startsWith(cmd));
        if (!isAllowed) {
          return { decision: 'deny', reason: 'Only git and npm test commands are allowed' };
        }
      }
      return { decision: 'allow' };
    },
  },
});

// Called from GitHub Actions
const prNumber = parseInt(process.env.PR_NUMBER!);
const repo = process.env.GITHUB_REPOSITORY!;

await reviewer.run(
  `Review PR #${prNumber} in ${repo}. Run git diff origin/main...HEAD to see changes. Then post your review.`
);
```

This agent runs in under 60 seconds for typical PRs. Total cost per review: roughly $0.03–$0.08 depending on diff size. At 20 PRs per day, that is about $1.20/day for automated code review that catches real issues.

The key design decisions:

1. **Read-only** — Write and Edit are disabled. The agent can never modify code.
2. **Bash whitelist** — only git and test commands are allowed via the hook.
3. **Custom tool for output** — instead of writing to a file, it posts directly to GitHub.
4. **Sonnet, not Opus** — code review does not need the most expensive model. Sonnet catches the same categories of issues at a fraction of the cost.

## Cost Considerations

The SDK itself is free and open source. You pay for Anthropic API tokens, same as direct API usage. But the SDK's built-in tools do add token overhead — tool descriptions consume input tokens, and tool results add to the context.

Practical cost ranges I have seen in production:

| Use Case | Model | Typical Cost |
|----------|-------|-------------|
| File analysis (5-10 files) | Sonnet | $0.01–$0.05 |
| Code review (PR diff) | Sonnet | $0.03–$0.08 |
| Multi-file refactoring | Sonnet | $0.10–$0.30 |
| Research + report | Opus | $0.50–$2.00 |
| Multi-agent orchestration | Opus + Sonnet | $0.20–$1.00 |

The biggest cost lever is model selection. Use Opus for orchestration and complex reasoning. Use Sonnet for everything else. If you want the full cost optimization playbook, my [token optimization guide](/blog/claude-code-token-optimization-bill-cut-guide) covers ten specific changes that cut my Claude Code bill by 70% — most of those techniques apply directly to SDK usage too.

Other cost tips specific to the SDK:

- **Disable unused tools** — each tool description adds ~200-500 input tokens. Disable tools the agent will not need.
- **Limit sub-agents** — set `maxSubAgents` to prevent runaway spawning.
- **Use streaming** — stream results to detect early when the agent is going off-track, then cancel.
- **Cache system prompts** — the SDK supports prompt caching for system prompts, which reduces input costs by up to 90% on repeated runs.

## When to Use Agent SDK vs Claude Code CLI vs API Directly

This is the decision framework I use:

**Use the Agent SDK when:**

- You are building an automated pipeline (CI/CD, scheduled jobs, webhooks)
- You need custom tools that connect to your infrastructure
- You want programmatic control over agent behavior (hooks, permissions, sub-agent config)
- You are embedding AI agents into a larger application
- You need to run agents in environments where the CLI is not available

**Use Claude Code CLI when:**

- You are working interactively at your terminal
- You want the full IDE-like experience (permissions UI, conversation history, slash commands)
- You are prototyping agent workflows before codifying them in the SDK
- You need the [skills system](/blog/claude-code-skills-reusable-ai-workflows-guide) for reusable prompt templates

**Use the API directly when:**

- You need fine-grained control over every message in the conversation
- You are building a chat interface where you manage the message history
- You do not need tool use at all (pure text generation)
- You are working in a language other than TypeScript/JavaScript
- You need features the SDK does not expose yet

The SDK sits between the CLI and the raw API. It gives you the power of Claude Code's tool system without the interactive terminal interface, and it saves you from reimplementing the agentic loop that the raw API requires.

## Getting Started

If you are already using Claude Code, the mental shift to the SDK is small. Start with a simple read-only agent, add one custom tool, then build up from there.

```typescript
import { Agent } from '@anthropic-ai/claude-agent-sdk';

const agent = new Agent({
  model: 'claude-sonnet-4-20250514',
  disabledTools: ['Write', 'Edit', 'Bash'],
});

const result = await agent.run(
  'Read the package.json and tell me which dependencies are outdated'
);

console.log(result.text);
```

Run it. See what happens. Then add a hook. Then add a custom tool. Then spawn a sub-agent. The SDK rewards incremental adoption.
