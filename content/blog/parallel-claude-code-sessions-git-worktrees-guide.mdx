An engineer I know at a mid-size fintech runs six Claude Code sessions simultaneously. One writes tests for a new payment endpoint. One refactors the notification service. One drafts a migration. One reviews yesterday's PR. Two are in plan mode for tomorrow's work. He ships at roughly 3x my throughput and he is not particularly faster at typing.

This is the git worktrees pattern, and Anthropic's internal team calls it their #1 productivity tip. Every tutorial describes the mechanics and none describes the system. A production parallel setup is not "open three terminals." It is a workflow architecture with specific tradeoffs that bite in specific ways.

Here is what a year of running parallel Claude sessions has taught me — including the merge strategy that prevents conflicts, the review cadence that prevents drift, and why six sessions is probably too many for you.

## What Git Worktrees Actually Give You

A git worktree is a linked working directory for a repository. One `.git` directory, multiple checked-out branches on disk, each in its own folder.

```bash
# From your main repo
git worktree add ../my-app-feature-auth feature/auth
git worktree add ../my-app-bugfix-ssr bugfix/ssr-hydration

# Now you have three directories:
# ~/code/my-app              (main branch)
# ~/code/my-app-feature-auth (feature/auth branch)
# ~/code/my-app-bugfix-ssr   (bugfix/ssr-hydration branch)
```

Unlike branch switching, nothing unmounts. You can open all three in separate editor windows, run tests in all three simultaneously, and run a Claude Code session in each one — all pointing at the same `.git` metadata.

Claude Code added first-class support via the `-w` flag in late 2025:

```bash
claude -w feature/refactor-billing "Refactor the billing module"
```

This creates an isolated worktree at `.claude/worktrees/feature-refactor-billing/`, checks out a new branch, and starts a session scoped entirely to that directory. Cleanup is one command away.

## The Single-Session Problem

Before you understand the parallel setup, understand what it solves.

A single long-running Claude session has a context ceiling. Once you are 40 minutes in, the context is 60K+ tokens of accumulated back-and-forth. The model starts making small mistakes it would not have made at the start — confusing related function names, forgetting that it already fixed this bug yesterday, re-reading files it has already read.

The usual fixes — `/compact`, `/clear` — either lose detail or force you to re-explain. Parallel sessions are the third option: don't let any single context grow in the first place.

Each session is short. Each session is scoped. Each session is killed and replaced, not extended. The cognitive overhead moves from "what have I told Claude in this session" to "what tasks do I have in flight."

## The Two-Session Setup (Your First Step)

Start here. Do not start with six.

```bash
# In terminal 1, the "work" session
cd ~/code/my-app
claude

# In terminal 2, the "review" session
cd ~/code/my-app
git worktree add ../my-app-review
cd ../my-app-review
claude
```

Rule of engagement: the work session edits, the review session reviews the commits the work session produces. When work commits, review pulls, reviews the diff, leaves notes. Work incorporates the notes. Loop.

This is the minimal viable parallel setup and it doubles throughput immediately, because:

1. You are never waiting for Claude to think. While work is running, you are reading review output.
2. Review catches drift that work's own "final check" would not catch, because review has a fresh context with no investment in the work.
3. The context of each session stays tight — work stays focused on implementation, review stays focused on evaluation.

If you do nothing else from this article, do this. Then try to live without it for a day.

## The Four-Session Pattern

Once the two-session rhythm is natural, the next stable point is four. Not three. Three is unstable because it breeds a "third thing" that does not fit the work/review cycle.

```
┌───────────────────────┬───────────────────────┐
│  Session A: Feature   │  Session B: Feature   │
│  (current work)       │  (next work)          │
│                       │                       │
│  Branch: feat/a       │  Branch: feat/b       │
│  Mode: implementing   │  Mode: planning       │
├───────────────────────┼───────────────────────┤
│  Session C: Review    │  Session D: Background│
│  (PR review, notes)   │  (tests, audits)      │
│                       │                       │
│  Branch: main         │  Branch: main         │
│  Mode: reviewing      │  Mode: batch tasks    │
└───────────────────────┴───────────────────────┘
```

Each quadrant has a role. Feature A is your primary focus. Feature B is ready-in-planning so when A ships you do not idle. C reviews finished work. D runs long tasks (test generation, dependency audits, doc generation) that would block A if they shared context.

What makes this work is **role discipline**. Each session has one job. The moment Session A starts drifting into a refactor of Session B's code, you stop. Commit whatever is safe, reset Session A, restart on-task. Parallel sessions fail when they blur into each other.

## tmux Layout That Works

A terminal-per-session model collapses under its own weight past two windows. Use a multiplexer. Here is the tmux layout I landed on after trying six variations:

```bash
# ~/.tmux/claude-parallel.sh
#!/usr/bin/env bash
SESSION="claude-parallel"

tmux new-session -d -s $SESSION -n work

# Pane 0 (top-left): Feature A implementation
tmux send-keys -t $SESSION:work "cd ~/code/app-a && claude" C-m

# Pane 1 (top-right): Feature B planning
tmux split-window -h -t $SESSION:work
tmux send-keys -t $SESSION:work "cd ~/code/app-b && claude --permission-mode plan" C-m

# Pane 2 (bottom-left): Review
tmux select-pane -t 0
tmux split-window -v -t $SESSION:work
tmux send-keys -t $SESSION:work "cd ~/code/app-review && claude" C-m

# Pane 3 (bottom-right): Background
tmux select-pane -t 2
tmux split-window -h -t $SESSION:work
tmux send-keys -t $SESSION:work "cd ~/code/app-bg && claude" C-m

tmux attach-session -t $SESSION
```

Run once, always get the same layout. Muscle memory becomes workflow. I have this bound to a keyboard shortcut. Claude Code itself handles the per-pane session isolation.

## The Merge Strategy That Prevents Conflicts

The #1 failure mode of parallel worktrees is not productivity — it is merge hell when two sessions edit overlapping code. Prevent it at the boundary, not at merge time.

**Rule 1: Scope worktrees by module, not by task.**

```
feat/billing-*    → always in billing-worktree
feat/auth-*       → always in auth-worktree
feat/ui-*         → always in ui-worktree
```

Tasks in the same module share a worktree (sequentially). Tasks across modules go to parallel worktrees. Two Claude sessions will never be editing the same file because the filesystem boundary prevents it.

(If wiring this up manually feels like too much, [Google Antigravity ships the same parallel-workspace model natively](/blog/google-antigravity-gemini-3-agentic-ide-developer-review) — it branches, isolates, and orchestrates approval flows for you, at the cost of running on Gemini 3.1 Pro instead of Opus.)

**Rule 2: Rebase, do not merge, between worktrees.**

```bash
# In the feat/a worktree, when main has advanced
git fetch origin
git rebase origin/main
```

Merges create merge commits that pollute history and confuse Claude when it reads `git log`. Rebases keep each worktree's history linear from main. Claude's reasoning about history is much better with linear chains.

**Rule 3: Commit at session boundaries.**

Every time a session ends — not every time Claude stops — you commit. Uncommitted changes are the only state that does not survive a session reset. Make that state tiny.

## Review Cadence: The 20-Minute Tick

The review session needs a heartbeat. Without one, "review" becomes "whenever I remember," which is never.

My cadence: every 20 minutes, the review session does one thing:

1. `git fetch --all` — pull in any new commits from sibling worktrees
2. Check `git log` for commits since last review
3. Run `git diff <last-reviewed-commit>..<latest>` for each commit
4. Ask Claude: "Review this diff. What did it actually change? Are there obvious issues?"
5. If issues, annotate and send back to the author session

Twenty minutes is the rhythm that matches how fast a well-prompted Claude session produces commit-worthy chunks. Faster and review is ahead of work. Slower and work drifts without feedback.

This maps almost exactly to the review rhythm from my [Plan Mode production guide](/blog/claude-code-plan-mode-ultraplan-production-guide) — annotate and send back. The mechanics are the same; only the channel is different.

## The 18% Throughput Number

A claim you will see in this space: teams using this pattern report 18% throughput improvement. Mindstudio's post on parallel sessions cites this number. It is real but it is also misleading.

The 18% is measured against sequential single-session work on tasks that can be cleanly parallelized. When tasks cannot be cleanly parallelized — if they touch the same module, if one blocks the other architecturally, if they share a critical dependency — parallel sessions either break even or go slower because of merge overhead.

The tasks that parallelize cleanly:

- Independent features in different modules
- Test generation for existing code
- Documentation updates
- Dependency audits
- Refactors contained to a single module

The tasks that do not:

- Architectural changes spanning the codebase
- Bug fixes where the root cause is uncertain
- Anything requiring a shared design discussion

Split your backlog by parallelizability before you split your sessions. The multiplier is real but not universal.

## Where the Wheels Fall Off

I have tried six sessions. I have tried eight. I peaked at nine during a mad week and shipped less than I did on a well-run three-session day. The failure modes, in the order they hit:

**4 sessions:** review queue starts to backlog. Work sessions produce faster than review can absorb.

**5 sessions:** you start losing track of which branch is for what. You need a written list — spreadsheet or text file — of "what is session N doing right now."

**6 sessions:** merge conflicts become weekly, not rare. Two sessions that "should not" overlap do overlap because the human orchestrator missed a module boundary.

**7+ sessions:** you are now a project manager, not an engineer. Most of your time is spent switching between sessions, not producing or reviewing. Productivity per session collapses.

The practical ceiling for most developers is **2–4 sessions**. Past that, you need either a second human (review partner) or specialized tooling like [Claude Code sub-agents](/blog/claude-code-sub-agents-multi-agent-systems-guide) for delegation within a single orchestrating session.

## Cost Implications

Parallel sessions multiply token spend. Four sessions is not four times the cost — compounded, closer to five times, because each session has its own context overhead that does not amortize across sessions.

If you are budget-conscious, see the [token optimization playbook](/blog/claude-code-token-optimization-bill-cut-guide). Specifically, `opusplan` becomes non-negotiable at this scale — all-Opus across four sessions gets expensive fast.

One practical trick: use Haiku for the review session. Review is mostly reading and commentary, not reasoning. Haiku handles diff commentary at a fraction of Sonnet's cost.

```bash
# The review pane, using Haiku
claude --model claude-haiku-4-5-20251001
```

The work sessions stay on Sonnet. The background session runs Haiku for batch tasks. The planning session uses Opus briefly. Model selection by role, not by default.

## Session Hygiene Across Worktrees

A few practices that prevent slow decay:

**Kill sessions that are stuck.** A session that has been spinning on the same problem for 10 minutes is dead. Start a fresh one with a different framing. Do not resurrect.

**Write session notes to a shared file.** `~/code/.claude-notes.md` with lines like "Session A: implementing billing partial refunds, waiting on payment provider API clarification." When you context-switch between panes, you do not lose the thread.

**Clean up worktrees weekly.**

```bash
git worktree list
git worktree remove ../my-app-feature-old
git worktree prune
```

Stale worktrees accumulate uncommitted state, confuse Claude's view of the repo, and eat disk space.

**Do not share `.claude/` directories across worktrees.** Each worktree should have its own `.claude/settings.local.json` if you have per-task configuration. The base `.claude/` at the root is fine.

## When to Not Parallelize

Not every task benefits. Signs you should run sequentially:

- You are learning a new codebase. Multiple sessions split your attention across a context you have not yet built.
- The task is small. A 15-minute bugfix does not need orchestration overhead.
- You are debugging. Debugging needs deep single-threaded attention; parallel sessions interrupt the flow.
- The tasks are architecturally coupled. Implementing X and Y where X depends on Y is sequential no matter how many sessions you spin up.

Parallelization is a throughput tool. For latency-sensitive work, it is noise.

## Onboarding a Teammate

If you want to introduce this to a colleague without overwhelming them:

1. Week 1: just two sessions. Work + review. No fancy tmux.
2. Week 2: add a planning session for tomorrow's work. Now three.
3. Week 3: add a background session for batch tasks. Now four.
4. Week 4: review what works, throw out what does not.

Skip steps and it feels like too much overhead for the payoff. Done gradually, each step demonstrably saves time before the next is added.

## What This Unlocks

The deeper shift is the one I did not see coming. Parallel sessions change how I think about tasks.

Before, every task was a thing to grind through. Now, every task is a thing to assign to a session with a defined scope, a review plan, and a merge boundary. I think about my work in terms of queues and handoffs, the way production systems think about work.

It is not universally better. For creative, exploratory work, single-session immersion is still the right shape. But for the 70% of engineering that is "turn this spec into working code," parallel worktrees change the economics.

Two sessions, twenty-minute review cadence, rebase-not-merge, scope by module. That is the whole playbook. Start there. Scale when it feels like you need more, not because someone online does six.
