I shipped a bug last year that TypeScript should have caught. A third-party API changed its response format — a field that used to return a number started returning a string. TypeScript didn't catch it because the data came in at runtime, long after the compiler had signed off. The type said `number`, the value was `"42"`, and my arithmetic silently produced `NaN` that cascaded through three downstream services before anyone noticed.

That was the week I stopped treating runtime validation as optional.

TypeScript is incredible for catching mistakes during development. But it evaporates at compile time — every `interface` and `type` annotation gets stripped from the JavaScript that actually runs. If your data comes from an API, a form, a database, a file, or an LLM, TypeScript has zero opinions about whether it matches your types. This is especially critical when dealing with [untrusted input in AI-powered apps](/blog/prompt-injection-llm-security-guide). **Zod fills that gap.**

## Why Zod Won

There are dozens of validation libraries. Joi, Yup, Ajv, class-validator, io-ts, Valibot. Zod won the TypeScript ecosystem for three reasons:

1. **Schema-first type inference** — define your schema once, infer the TypeScript type from it. No duplication.
2. **TypeScript-native** — built from the ground up for TypeScript, not retrofitted.
3. **Ecosystem integration** — first-class support in React Hook Form, tRPC, Next.js, Astro, Hono, and every major framework.

```typescript
import { z } from 'zod';

// Define the schema once
const UserSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1).max(100),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'viewer']),
  createdAt: z.coerce.date(),
});

// Infer the type — no duplication
type User = z.infer<typeof UserSchema>;
// { id: string; name: string; email: string; role: "admin" | "user" | "viewer"; createdAt: Date }
```

No more maintaining a `User` interface and a separate validation function that might drift apart. The schema IS the type.

## The Fundamentals: Primitives and Objects

### Primitives

```typescript
// Strings with constraints
const Name = z.string().min(1, 'Name is required').max(100);
const Email = z.string().email('Invalid email format');
const Slug = z.string().regex(/^[a-z0-9-]+$/, 'Must be lowercase with hyphens');
const URL = z.string().url();

// Numbers with constraints
const Age = z.number().int().min(0).max(150);
const Price = z.number().positive().multipleOf(0.01);
const Port = z.number().int().min(1).max(65535);

// Booleans, dates, enums
const IsActive = z.boolean();
const CreatedAt = z.coerce.date(); // Coerces strings/numbers to Date
const Status = z.enum(['active', 'inactive', 'pending']);
```

### Objects and Nesting

```typescript
const AddressSchema = z.object({
  street: z.string().min(1),
  city: z.string().min(1),
  state: z.string().length(2),
  zip: z.string().regex(/^\d{5}(-\d{4})?$/),
  country: z.string().default('US'),
});

const CompanySchema = z.object({
  name: z.string().min(1),
  website: z.string().url().optional(),
  address: AddressSchema,
  employees: z.number().int().nonnegative(),
  tags: z.array(z.string()).min(1).max(10),
  metadata: z.record(z.string(), z.unknown()).optional(),
});

type Company = z.infer<typeof CompanySchema>;
```

### Arrays, Tuples, Records

```typescript
// Arrays with constraints
const Tags = z.array(z.string()).min(1).max(10);
const Scores = z.array(z.number()).nonempty(); // At least one element

// Tuples — fixed-length arrays with per-element types
const Coordinate = z.tuple([z.number(), z.number()]);
const NameAge = z.tuple([z.string(), z.number().int().positive()]);

// Records — string-keyed objects with uniform value types
const Config = z.record(z.string(), z.string());
const Scores2 = z.record(z.string(), z.number());
```

## parse() vs safeParse() — Know the Difference

This is the most important API decision you'll make with Zod.

### parse() — throws on failure

```typescript
try {
  const user = UserSchema.parse(unknownData);
  // user is fully typed here
} catch (error) {
  if (error instanceof z.ZodError) {
    console.error(error.issues);
  }
}
```

Use `parse()` when invalid data is exceptional — configuration loading, environment variables, data you genuinely don't expect to be wrong.

### safeParse() — returns a result object

```typescript
const result = UserSchema.safeParse(unknownData);

if (result.success) {
  const user = result.data; // fully typed
} else {
  const errors = result.error.issues; // structured error info
  console.error(errors);
}
```

Use `safeParse()` everywhere else — form validation, API input, user-facing flows. It's faster than try/catch when failures are common, and it gives you structured error data without exception overhead.

**My rule of thumb:** `safeParse()` is the default. `parse()` is the exception.

## Transforms and Pipes — Where Zod Gets Powerful

Validation is just the beginning. Zod can also **transform** data as it validates:

```typescript
// String to number
const StringToNumber = z.string().transform((val) => parseInt(val, 10));
StringToNumber.parse("42"); // => 42

// Trim and lowercase
const NormalizedEmail = z.string().email().transform((val) => val.trim().toLowerCase());

// Parse date strings
const DateString = z.string().transform((val) => new Date(val));

// Complex transforms
const ApiResponseSchema = z.object({
  user_name: z.string(),
  created_at: z.string(),
  is_active: z.union([z.boolean(), z.string()]),
}).transform((data) => ({
  userName: data.user_name,
  createdAt: new Date(data.created_at),
  isActive: data.is_active === true || data.is_active === 'true',
}));

type ApiResponse = z.infer<typeof ApiResponseSchema>;
// { userName: string; createdAt: Date; isActive: boolean }
```

### Pipes — Chain Validations

`.pipe()` lets you chain schemas sequentially — validate, transform, then validate the transformed result:

```typescript
// Parse a string as JSON, then validate the structure
const JsonConfig = z.string()
  .transform((val) => JSON.parse(val))
  .pipe(z.object({
    host: z.string(),
    port: z.number().int().min(1).max(65535),
    debug: z.boolean().default(false),
  }));

const config = JsonConfig.parse('{"host":"localhost","port":3000}');
// { host: "localhost", port: 3000, debug: false }
```

## Discriminated Unions — Type-Safe Polymorphism

This is where Zod really shines. Discriminated unions let you model data that can take different shapes based on a discriminator field:

```typescript
const NotificationSchema = z.discriminatedUnion('type', [
  z.object({
    type: z.literal('email'),
    to: z.string().email(),
    subject: z.string(),
    body: z.string(),
  }),
  z.object({
    type: z.literal('sms'),
    phoneNumber: z.string().regex(/^\+\d{10,15}$/),
    message: z.string().max(160),
  }),
  z.object({
    type: z.literal('push'),
    deviceToken: z.string(),
    title: z.string(),
    body: z.string(),
    badge: z.number().int().nonnegative().optional(),
  }),
]);

type Notification = z.infer<typeof NotificationSchema>;

// TypeScript knows the exact shape based on `type`
function sendNotification(notification: Notification) {
  switch (notification.type) {
    case 'email':
      // TypeScript knows: to, subject, body
      sendEmail(notification.to, notification.subject, notification.body);
      break;
    case 'sms':
      // TypeScript knows: phoneNumber, message
      sendSMS(notification.phoneNumber, notification.message);
      break;
    case 'push':
      // TypeScript knows: deviceToken, title, body, badge
      sendPush(notification.deviceToken, notification.title, notification.body);
      break;
  }
}
```

Unlike regular `z.union()`, discriminated unions use the discriminator key to pick the right schema immediately — no need to try each option in order. This is both faster and gives better error messages.

## Custom Validation with refine() and superRefine()

When built-in validators aren't enough:

```typescript
// Simple refinement
const Password = z.string()
  .min(8, 'Must be at least 8 characters')
  .refine((val) => /[A-Z]/.test(val), 'Must contain an uppercase letter')
  .refine((val) => /[0-9]/.test(val), 'Must contain a number')
  .refine((val) => /[^A-Za-z0-9]/.test(val), 'Must contain a special character');

// Cross-field validation with superRefine
const RegistrationSchema = z.object({
  password: z.string().min(8),
  confirmPassword: z.string(),
  email: z.string().email(),
}).superRefine((data, ctx) => {
  if (data.password !== data.confirmPassword) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Passwords must match',
      path: ['confirmPassword'],
    });
  }
});

// Async refinement — e.g., check if email exists in database
const UniqueEmailSchema = z.string().email().refine(
  async (email) => {
    const exists = await checkEmailInDB(email);
    return !exists;
  },
  { message: 'Email already registered' }
);
```

## Real-World Pattern: API Boundary Validation

This is where Zod earns its keep. Validate at the boundary, trust everywhere else:

```typescript
// schemas/api.ts — define once, use everywhere
export const CreateOrderSchema = z.object({
  items: z.array(z.object({
    productId: z.string().uuid(),
    quantity: z.number().int().positive().max(100),
    price: z.number().positive(),
  })).min(1, 'Order must have at least one item'),
  shippingAddress: AddressSchema,
  paymentMethod: z.enum(['card', 'paypal', 'bank_transfer']),
  couponCode: z.string().optional(),
  notes: z.string().max(500).optional(),
});

export type CreateOrderInput = z.infer<typeof CreateOrderSchema>;

// api/orders/route.ts — validate at the API boundary
export async function POST(request: Request) {
  const body = await request.json();
  const result = CreateOrderSchema.safeParse(body);

  if (!result.success) {
    return Response.json(
      { errors: result.error.flatten().fieldErrors },
      { status: 400 }
    );
  }

  // result.data is fully typed and validated
  // No more runtime surprises
  const order = await createOrder(result.data);
  return Response.json(order, { status: 201 });
}
```

### Flattening Errors for APIs

Zod's `flatten()` method turns nested error objects into a clean, API-friendly format:

```typescript
const result = CreateOrderSchema.safeParse(badData);

if (!result.success) {
  const flat = result.error.flatten();
  // {
  //   formErrors: [],
  //   fieldErrors: {
  //     items: ["Order must have at least one item"],
  //     paymentMethod: ["Invalid enum value..."]
  //   }
  // }
}
```

## React Hook Form + Zod + Next.js Server Actions

The killer combo for full-stack form validation. Define the schema once, validate on both client and server:

```typescript
// schemas/contact.ts
import { z } from 'zod';

export const ContactSchema = z.object({
  name: z.string().min(1, 'Name is required').max(100),
  email: z.string().email('Please enter a valid email'),
  subject: z.enum(['general', 'support', 'partnership', 'other']),
  message: z.string().min(10, 'Message must be at least 10 characters').max(2000),
});

export type ContactFormData = z.infer<typeof ContactSchema>;
```

```typescript
// components/ContactForm.tsx
'use client';

import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { ContactSchema, type ContactFormData } from '@/schemas/contact';
import { submitContact } from '@/actions/contact';

export function ContactForm() {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<ContactFormData>({
    resolver: zodResolver(ContactSchema),
  });

  const onSubmit = async (data: ContactFormData) => {
    const result = await submitContact(data);
    if (result.error) {
      // Handle server-side validation errors
    }
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')} placeholder="Name" />
      {errors.name && <span>{errors.name.message}</span>}

      <input {...register('email')} placeholder="Email" />
      {errors.email && <span>{errors.email.message}</span>}

      <select {...register('subject')}>
        <option value="general">General</option>
        <option value="support">Support</option>
        <option value="partnership">Partnership</option>
        <option value="other">Other</option>
      </select>
      {errors.subject && <span>{errors.subject.message}</span>}

      <textarea {...register('message')} placeholder="Message" />
      {errors.message && <span>{errors.message.message}</span>}

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Sending...' : 'Send'}
      </button>
    </form>
  );
}
```

```typescript
// actions/contact.ts
'use server';

import { ContactSchema } from '@/schemas/contact';

export async function submitContact(data: unknown) {
  // Server-side validation — never trust the client
  const result = ContactSchema.safeParse(data);

  if (!result.success) {
    return { error: result.error.flatten().fieldErrors };
  }

  // data is validated and typed
  await sendEmail(result.data);
  return { success: true };
}
```

**The key insight:** the same `ContactSchema` validates on both client (instant feedback via React Hook Form) and server (security via Server Action). If someone bypasses the client, the server still catches it.

## Environment Variable Validation

One of my favorite Zod patterns — validate your entire environment at startup:

```typescript
// env.ts
import { z } from 'zod';

const EnvSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']),
  DATABASE_URL: z.string().url(),
  API_SECRET: z.string().min(32, 'API_SECRET must be at least 32 characters'),
  REDIS_URL: z.string().url().optional(),
  PORT: z.coerce.number().int().min(1).max(65535).default(3000),
  LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
  ALLOWED_ORIGINS: z.string().transform((val) => val.split(',')).pipe(
    z.array(z.string().url()).min(1)
  ),
});

// Validate on startup — fail fast
export const env = EnvSchema.parse(process.env);

// Now env is fully typed and validated
// env.PORT is a number, not a string
// env.ALLOWED_ORIGINS is string[], not string
```

If any environment variable is missing or malformed, your app crashes immediately at startup with a clear error message — not 3 hours later when that code path first executes.

## Validating API Responses

Don't trust external APIs. Validate what they return:

```typescript
const GitHubUserSchema = z.object({
  login: z.string(),
  id: z.number(),
  avatar_url: z.string().url(),
  name: z.string().nullable(),
  bio: z.string().nullable(),
  public_repos: z.number().int(),
  followers: z.number().int(),
  created_at: z.coerce.date(),
});

type GitHubUser = z.infer<typeof GitHubUserSchema>;

async function fetchGitHubUser(username: string): Promise<GitHubUser> {
  const response = await fetch(`https://api.github.com/users/${username}`);
  const data = await response.json();

  // Validate AND type the response
  return GitHubUserSchema.parse(data);
}
```

When GitHub changes their API response shape (and they will), you'll get an immediate, descriptive error instead of silent data corruption.

## Zod v4: What Changed

Zod v4 shipped with significant improvements:

- **10-20x faster parsing** for most schemas
- **Smaller bundle size** — better tree-shaking
- **Improved error messages** — more helpful by default
- **Better discriminated unions** — compose them, use union/pipe discriminators
- **`z.interface()`** — for recursive types without workarounds

```typescript
// Zod v4: Recursive types with z.interface()
const CategorySchema: z.ZodType<Category> = z.interface({
  name: z.string(),
  children: z.array(z.lazy(() => CategorySchema)).default([]),
});

type Category = z.infer<typeof CategorySchema>;
```

If you're still on v3, the migration is mostly painless — the API is backward-compatible for 95% of use cases.

## Zod vs Valibot vs Yup — When to Use What

| Feature | Zod | Valibot | Yup |
|---------|-----|---------|-----|
| **Bundle size** | ~17.7 kB | ~1.4 kB | ~12 kB |
| **TypeScript inference** | Excellent | Excellent | Good |
| **API style** | Chained methods | Functional/modular | Chained methods |
| **Ecosystem** | Massive | Growing | Large (React/Formik) |
| **Tree-shaking** | Good (v4) | Excellent | Limited |
| **Performance** | Fast (v4) | Very fast | Moderate |
| **Learning curve** | Low | Low | Low |

**Choose Zod** when you need the broadest ecosystem support, the best documentation, and you're working with tRPC, React Hook Form, or Next.js.

**Choose Valibot** when bundle size is critical — edge functions, client-heavy apps, or when you need under 2 kB.

**Choose Yup** if your team already uses it with Formik and migration cost isn't justified.

For new projects in 2026, I default to Zod. The ecosystem support alone is worth the extra kilobytes. If you're using TanStack in your stack, Zod pairs perfectly with it — I covered the full ecosystem in my [TanStack guide](/blog/tanstack-ecosystem-complete-guide-2026).

## Common Gotchas

### 1. Forgetting coerce for form data

HTML forms always submit strings. If you expect a number, use `z.coerce.number()`:

```typescript
// Wrong — form data is always strings
const schema = z.object({ age: z.number() });
schema.parse({ age: "25" }); // ZodError!

// Right — coerce strings to numbers
const schema = z.object({ age: z.coerce.number().int().positive() });
schema.parse({ age: "25" }); // { age: 25 } ✓
```

### 2. Optional vs nullable vs nullish

```typescript
z.string().optional()  // string | undefined
z.string().nullable()  // string | null
z.string().nullish()   // string | null | undefined

// They're different! Pick the right one for your data.
```

### 3. Object schemas strip unknown keys by default

```typescript
const schema = z.object({ name: z.string() });
schema.parse({ name: 'Seb', age: 30 }); // { name: 'Seb' } — age is stripped!

// To keep unknown keys:
schema.passthrough().parse({ name: 'Seb', age: 30 }); // { name: 'Seb', age: 30 }

// To reject unknown keys:
schema.strict().parse({ name: 'Seb', age: 30 }); // ZodError!
```

### 4. Error formatting confusion

```typescript
const result = schema.safeParse(badData);
if (!result.success) {
  // For APIs: flat object with field names as keys
  result.error.flatten();

  // For forms: nested format matching object structure
  result.error.format();

  // For logging: array of individual issues
  result.error.issues;
}
```

## Practical Takeaways

1. **Define schemas at the boundary** — API routes, form handlers, config loading. Validate once, trust internally.
2. **Infer types from schemas** — never maintain types and schemas separately.
3. **Use `safeParse()` by default** — only use `parse()` when invalid data is truly exceptional.
4. **Validate environment variables at startup** — fail fast, not at 3 AM.
5. **Validate external API responses** — third parties will change their APIs without telling you.
6. **Use discriminated unions** for polymorphic data — they're faster and give better errors than regular unions.
7. **Combine with React Hook Form** for full-stack validation with a single schema.
8. **Consider Valibot** if bundle size is your primary concern.

Zod isn't just a validation library — it's the bridge between TypeScript's compile-time promises and runtime reality. Pair it with [clean code principles](/blog/principles-of-clean-frontend-code-dry-kiss-solid) and [advanced React hooks patterns](/blog/advanced-react-hooks-best-practices-in-react-with-nextjs-and-remix), and you've got a rock-solid foundation for any TypeScript project. If you're not using it at your system boundaries, you're trusting data you shouldn't trust.
