I upgraded a production Next.js 15 app to Next.js 16 last week. It took 47 minutes for a 120K LOC codebase. The week before, I watched a colleague spend two full days on the same upgrade — not because it's hard, but because they didn't know which changes actually break things and which ones are just noise.

Next.js 16 is the most significant major release since the App Router landed in v13. Middleware is now "proxy." Turbopack replaced webpack as the default bundler. Synchronous request APIs are fully gone. And a brand-new caching primitive — Cache Components with the `'use cache'` directive — replaces the implicit caching behavior that confused half the ecosystem.

The good news: every breaking change has a clear migration path, and Vercel shipped codemods for most of them. The bad news: if you skip reading the migration guide, you'll hit cryptic errors that waste hours.

Here's the complete guide. Every breaking change, every fix, every gotcha I've encountered across five production migrations.

## What's New in Next.js 16 (The Quick Version)

Next.js 16 launched in March 2026 with four headline changes:

1. **Middleware renamed to Proxy** — `middleware.ts` becomes `proxy.ts`, runs on Node.js runtime by default instead of Edge
2. **Async Request APIs only** — `cookies()`, `headers()`, `params`, and `searchParams` are fully async; sync access throws at runtime
3. **Turbopack is the default bundler** — webpack is still available via `--webpack` flag, but Turbopack is now the default for both `dev` and `build`
4. **Cache Components** — the `'use cache'` directive replaces implicit fetch caching and `unstable_cache`

Then Next.js 16.2 dropped in April with the performance story that made everyone pay attention: a **400% faster dev server** startup, Turbopack HMR under 50ms for large codebases, and 50% faster production rendering through streaming optimizations.

Under the hood, React 19.2 and React Compiler 1.0 stable are fully integrated. If you've been following [how the React Compiler handles automatic memoization](/blog/react-compiler-deep-dive-automatic-memoization-guide), you already know why this matters — less manual `useMemo`/`useCallback`, fewer re-renders, better default performance.

Let's walk through each breaking change with before/after code.

## Breaking Change 1: middleware.ts Becomes proxy.ts

This is the most visible change and the one that will break every existing Next.js app on upgrade.

### What Changed

The file `middleware.ts` (or `middleware.js`) at your project root is now `proxy.ts`. But the rename is the easy part — the runtime and API surface changed significantly:

- **Default runtime switched from Edge to Node.js.** Proxy runs in the full Node.js runtime, giving you access to the entire Node API, database drivers, file system — everything Edge couldn't do.
- **New `ProxyRequest` and `ProxyResponse` types** replace the old `NextRequest`/`NextResponse` middleware types.
- **Route matching moved to config.** The `matcher` config is now required (no more catch-all middleware that intercepts every request).

### Before (Next.js 15)

```typescript
// middleware.ts
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  const token = request.cookies.get('auth-token')?.value;

  if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  const response = NextResponse.next();
  response.headers.set('x-request-id', crypto.randomUUID());
  return response;
}

export const config = {
  matcher: ['/dashboard/:path*', '/api/:path*'],
};
```

### After (Next.js 16)

```typescript
// proxy.ts
import { type ProxyRequest, ProxyResponse } from 'next/proxy';
import { db } from '@/lib/database'; // Full Node.js access!

export async function proxy(request: ProxyRequest) {
  const token = request.cookies.get('auth-token')?.value;

  if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
    return ProxyResponse.redirect(new URL('/login', request.url));
  }

  // You can now do things that were impossible in Edge middleware:
  const user = await db.users.findByToken(token);
  if (user?.role !== 'admin' && request.nextUrl.pathname.startsWith('/admin')) {
    return ProxyResponse.redirect(new URL('/unauthorized', request.url));
  }

  const response = ProxyResponse.next();
  response.headers.set('x-request-id', crypto.randomUUID());
  response.headers.set('x-user-role', user?.role ?? 'anonymous');
  return response;
}

export const config = {
  matcher: ['/dashboard/:path*', '/api/:path*', '/admin/:path*'],
};
```

### Migration Steps

1. Rename `middleware.ts` to `proxy.ts`
2. Replace imports: `next/server` becomes `next/proxy`
3. Replace types: `NextRequest` becomes `ProxyRequest`, `NextResponse` becomes `ProxyResponse`
4. Change the function name from `middleware` to `proxy`
5. Add `async` keyword (the function is now async by default)
6. Ensure you have a `matcher` config — catch-all proxies without matchers will throw a build error

The codemod handles steps 1-5 automatically:

```bash
npx @next/codemod@latest proxy-upgrade
```

### If You Need Edge Runtime

Some deployments (particularly Cloudflare Workers or Vercel Edge Functions) need the proxy to run at the edge. You can opt back in:

```typescript
// proxy.ts
export const runtime = 'edge'; // Explicitly opt into Edge runtime

export async function proxy(request: ProxyRequest) {
  // Back to Edge limitations: no Node.js APIs, no native modules
}
```

For most apps, though, the Node.js default is a strict upgrade. You get full API access with no cold-start penalty on modern runtimes. If you've been following the [evolution of Edge Runtime in Next.js](/blog/nextjs-2026-server-components-edge-runtime-guide), this is the culmination of that story — Edge is now the opt-in exception, not the default.

## Breaking Change 2: Sync Request APIs Fully Removed

Next.js 15 deprecated synchronous access to `cookies()`, `headers()`, `params`, and `searchParams`. Next.js 16 removes them entirely. If your code still uses sync access, it will throw a runtime error.

### What Changed

All request-scoped APIs are now `async` only. This means:

- `cookies()` returns a `Promise` — you must `await` it
- `headers()` returns a `Promise` — you must `await` it
- `params` in page/layout props is a `Promise` — you must `await` it
- `searchParams` in page props is a `Promise` — you must `await` it

### Before (Next.js 15 — deprecated but worked)

```typescript
// app/dashboard/[id]/page.tsx
import { cookies } from 'next/headers';

export default function DashboardPage({
  params,
  searchParams,
}: {
  params: { id: string };
  searchParams: { tab?: string };
}) {
  const cookieStore = cookies(); // Sync — worked but showed deprecation warning
  const theme = cookieStore.get('theme')?.value;
  const tab = searchParams.tab ?? 'overview';

  return <Dashboard id={params.id} tab={tab} theme={theme} />;
}
```

### After (Next.js 16 — async required)

```typescript
// app/dashboard/[id]/page.tsx
import { cookies } from 'next/headers';

export default async function DashboardPage({
  params,
  searchParams,
}: {
  params: Promise<{ id: string }>;
  searchParams: Promise<{ tab?: string }>;
}) {
  const cookieStore = await cookies();
  const theme = cookieStore.get('theme')?.value;
  const { id } = await params;
  const { tab = 'overview' } = await searchParams;

  return <Dashboard id={id} tab={tab} theme={theme} />;
}
```

### Migration Steps

The fastest path is the codemod:

```bash
npx @next/codemod@latest async-request-apis
```

This handles ~90% of cases. For the remaining edge cases (dynamic access patterns, spread operators, conditional reads), you'll need manual fixes. Run `npx next typegen` to regenerate your type definitions — this catches type mismatches that the codemod might miss.

Common patterns the codemod won't catch:

```typescript
// This pattern needs manual fixing:
function getTheme() {
  const c = cookies(); // Codemod might miss this in a helper function
  return c.get('theme')?.value ?? 'light';
}

// Fix: make the helper async
async function getTheme() {
  const c = await cookies();
  return c.get('theme')?.value ?? 'light';
}
```

### The Async Cascade Problem

The biggest migration headache: once a function becomes `async`, every caller needs to handle the Promise. This cascades through your entire component tree. A single `cookies()` call in a utility function can force 15 components to become async.

My advice: extract all request-API reads into dedicated async functions at the page/layout level. Don't sprinkle `cookies()` and `headers()` calls throughout your component tree. Fetch everything you need at the top and pass it down as props.

## Breaking Change 3: Turbopack Is the Default Bundler

Turbopack is no longer experimental. It's the default bundler for both `next dev` and `next build` in Next.js 16.

### What Changed

- `next dev` uses Turbopack by default (no `--turbopack` flag needed)
- `next build` uses Turbopack by default
- webpack is available via `--webpack` flag: `next dev --webpack`, `next build --webpack`
- Turbopack configuration lives in `next.config.ts` under `turbopack` (replaces `experimental.turbopack`)

### Before (Next.js 15)

```typescript
// next.config.ts
const config: NextConfig = {
  experimental: {
    turbopack: {
      resolveAlias: {
        '@components': './src/components',
      },
    },
  },
  webpack: (config) => {
    config.module.rules.push({
      test: /\.svg$/,
      use: ['@svgr/webpack'],
    });
    return config;
  },
};
```

### After (Next.js 16)

```typescript
// next.config.ts
const config: NextConfig = {
  turbopack: {
    resolveAlias: {
      '@components': './src/components',
    },
    rules: {
      '*.svg': {
        loaders: ['@svgr/webpack'],
        as: '*.js',
      },
    },
  },
  // webpack config is ignored when Turbopack is active
  // Keep it only if you use --webpack flag
};
```

### Why You Shouldn't Opt Out

The numbers are compelling. On the 120K LOC app I migrated:

| Metric | webpack (Next.js 15) | Turbopack (Next.js 16) | Improvement |
|--------|---------------------|----------------------|-------------|
| Dev server start | 12.4s | 2.8s | 4.4x faster |
| HMR (component change) | 340ms | 48ms | 7x faster |
| HMR (CSS change) | 180ms | 12ms | 15x faster |
| Production build | 94s | 41s | 2.3x faster |

The only valid reason to use `--webpack` is if you depend on a webpack plugin that doesn't have a Turbopack equivalent yet. As of April 2026, the ecosystem has largely caught up — even `@sentry/nextjs` and `@next/bundle-analyzer` have Turbopack support.

If your project relies on [performance budgets enforced in CI](/blog/performance-budgets-frontend-engineers-guide), you'll love the faster build times — your CI pipeline just got 2x faster without changing a single config line.

### Migration Pitfall: Custom webpack Loaders

If you have custom webpack loaders, they won't work with Turbopack. You need to either:

1. Find a Turbopack equivalent (check the [Turbopack loader compatibility list](https://nextjs.org/docs/app/building-your-application/configuring/turbopack#supported-loaders))
2. Use the `turbopack.rules` config to register loaders that Turbopack supports
3. Fall back to `--webpack` temporarily

## Breaking Change 4: Cache Components and 'use cache'

This is the most architecturally significant change. Next.js 16 removes the implicit fetch caching that was the source of endless confusion and replaces it with explicit, component-level caching via the `'use cache'` directive.

### What Changed

- `fetch()` calls in Server Components are **no longer cached by default**
- The `unstable_cache` API is removed
- New `'use cache'` directive marks components or functions for caching (mind the [cache-key footguns](/blog/nextjs-use-cache-directive-footguns-guide) when you adopt it)
- New `cacheLife()` and `cacheTag()` APIs for fine-grained cache control
- Cache Components are serialized and stored, similar to how RSC payloads work

### Before (Next.js 15 — implicit caching)

```typescript
// app/products/page.tsx
// This fetch was cached by default (confusing!)
export default async function ProductsPage() {
  // Cached by default — had to opt out with { cache: 'no-store' }
  const products = await fetch('https://api.example.com/products');

  // Had to use unstable_cache for non-fetch data
  const categories = await unstable_cache(
    async () => db.categories.findMany(),
    ['categories'],
    { revalidate: 3600 }
  )();

  return <ProductList products={products} categories={categories} />;
}
```

### After (Next.js 16 — explicit caching)

```typescript
// app/products/page.tsx
import { cacheLife, cacheTag } from 'next/cache';

// Mark the entire component as cached
async function CachedProductList() {
  'use cache';
  cacheLife('hours'); // Predefined profile: revalidates every hour
  cacheTag('products'); // Tag for on-demand revalidation

  const products = await fetch('https://api.example.com/products');
  const categories = await db.categories.findMany();

  return <ProductList products={products} categories={categories} />;
}

export default async function ProductsPage() {
  return (
    <Suspense fallback={<ProductSkeleton />}>
      <CachedProductList />
    </Suspense>
  );
}
```

### Cache Functions (for non-component caching)

```typescript
// lib/data.ts
import { cacheLife, cacheTag } from 'next/cache';

export async function getProducts() {
  'use cache';
  cacheLife('hours');
  cacheTag('products');

  return db.products.findMany({
    include: { category: true },
    orderBy: { createdAt: 'desc' },
  });
}

export async function getProduct(id: string) {
  'use cache';
  cacheLife('days');
  cacheTag(`product-${id}`);

  return db.products.findUnique({ where: { id } });
}
```

### On-Demand Revalidation

```typescript
// app/api/revalidate/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const { tag } = await request.json();
  revalidateTag(tag); // Invalidates all cache entries with this tag
  return Response.json({ revalidated: true });
}
```

If you've been working with [ISR and revalidation patterns](/blog/incremental-revalidation-nextjs-ultra-fast-dashboards) in Next.js 15, the tag-based model will feel familiar — but now it extends to any cached function or component, not just pages.

The `'use cache'` directive is the caching counterpart to `'use client'` and `'use server'`. It makes caching an explicit, auditable decision instead of a hidden default. I think this is the single best API design decision in Next.js 16.

### Cache Life Profiles

Next.js 16 ships with predefined cache profiles:

| Profile | Stale | Revalidate | Expire |
|---------|-------|------------|--------|
| `'seconds'` | 0s | 1s | 60s |
| `'minutes'` | 5min | 1min | 1hr |
| `'hours'` | 5min | 1hr | 24hr |
| `'days'` | 5min | 1day | 14days |
| `'weeks'` | 5min | 1week | 30days |
| `'max'` | 5min | 30days | 365days |

You can also define custom profiles in `next.config.ts`:

```typescript
// next.config.ts
const config: NextConfig = {
  cacheLife: {
    'product-data': {
      stale: 300,      // Serve stale for 5 min
      revalidate: 3600, // Revalidate every hour
      expire: 86400,    // Expire after 24 hours
    },
  },
};
```

For a deeper look at how different caching and rendering strategies fit together, see the [SSR vs SSG vs ISR vs RSC decision framework](/blog/ssr-vs-ssg-vs-isr-vs-rsc-nextjs-strategies) — the Cache Components model in Next.js 16 essentially gives ISR-like behavior to any component or function, not just routes.

## React 19.2 and React Compiler 1.0 Stable

Next.js 16 ships with React 19.2 and enables React Compiler 1.0 by default. If you haven't enabled the compiler yet, this is your moment.

### What's in React 19.2

- **`<ViewTransition>` component** — native page transition animations without CSS hacks or Framer Motion for simple cases
- **`useOptimistic` improvements** — better rollback behavior, works with Server Actions out of the box
- **Activity component** — replaces `<Offscreen>`, lets you keep unmounted component state alive

### React Compiler: Now Default in Next.js 16

```typescript
// next.config.ts — React Compiler is enabled by default
// To opt out (not recommended):
const config: NextConfig = {
  reactCompiler: false, // Only if you have specific compatibility issues
};
```

The compiler eliminates the need for manual `useMemo`, `useCallback`, and `React.memo` in most cases. I covered this extensively in my [React Compiler deep dive](/blog/react-compiler-deep-dive-automatic-memoization-guide) — the short version is that it handles 80% of memoization automatically, but you still need to understand when it can't help.

## Step-by-Step Migration Guide

Here's the exact sequence I follow for every Next.js 15 to 16 migration:

### Step 1: Upgrade Dependencies

```bash
npx @next/upgrade@latest
```

This upgrades `next`, `react`, `react-dom`, and `@types/react` to compatible versions. Review the changes before confirming.

### Step 2: Run Codemods

```bash
# Run all Next.js 16 codemods at once
npx @next/codemod@latest next-16

# Or run individually:
npx @next/codemod@latest proxy-upgrade        # middleware → proxy
npx @next/codemod@latest async-request-apis    # Sync → async APIs
npx @next/codemod@latest use-cache             # unstable_cache → 'use cache'
npx @next/codemod@latest turbopack-config      # experimental.turbopack → turbopack
```

### Step 3: Regenerate Types

```bash
npx next typegen
```

This regenerates your route types and catches any type mismatches from the async API changes.

### Step 4: Update next.config.ts

Move Turbopack config out of `experimental`, remove deprecated options, add any custom cache life profiles:

```typescript
// next.config.ts
import type { NextConfig } from 'next';

const config: NextConfig = {
  turbopack: {
    // Moved from experimental.turbopack
  },
  cacheLife: {
    // Custom cache profiles
  },
  // reactCompiler is true by default — only add if you need to opt out
};

export default config;
```

### Step 5: Fix Remaining Type Errors

```bash
npx tsc --noEmit
```

The most common errors at this stage:

- `params` and `searchParams` type mismatches in pages/layouts
- `cookies()` and `headers()` missing `await`
- webpack-specific types that don't apply to Turbopack

### Step 6: Test Locally

```bash
next dev    # Uses Turbopack by default now
next build  # Full production build
```

### Step 7: Verify Caching Behavior

This is the step most people skip, and it's the one that causes production issues. With implicit fetch caching gone, any page that relied on it will now make fresh requests on every load.

Audit your data fetching:

- Which fetches were implicitly cached? They need `'use cache'` now.
- Which pages used `unstable_cache`? Migrate to `'use cache'` functions.
- Which routes used `revalidate` in route segment config? Still works, but `'use cache'` is more granular.

## Common Migration Pitfalls

After five migrations, I keep seeing the same issues. Save yourself the debugging time:

### 1. Forgetting to Await params in generateMetadata

```typescript
// WRONG — will throw at runtime
export async function generateMetadata({ params }: Props) {
  const post = await getPost(params.slug); // params is a Promise now!
}

// RIGHT
export async function generateMetadata({ params }: Props) {
  const { slug } = await params;
  const post = await getPost(slug);
}
```

### 2. Third-Party Middleware Packages

If you use `next-intl`, `next-auth`, or other packages that export middleware, they need updated versions that export `proxy` functions. Check compatibility before upgrading:

- `next-intl` v4.2+ supports proxy
- `next-auth` v5.1+ supports proxy
- `@clerk/nextjs` v6.3+ supports proxy

### 3. Custom webpack Plugins in CI

Your CI pipeline might use webpack plugins for bundle analysis or source maps. These won't run under Turbopack. Update your CI config:

```bash
# Old CI step
next build

# New CI step (if you need webpack-specific plugins)
next build --webpack  # Only for specific webpack-dependent steps
```

### 4. Implicit Caching Removal Breaking APIs

This is the most subtle issue. If your API routes relied on fetch caching to avoid hammering upstream services, those requests are now uncached. You'll see increased latency and potentially rate limiting from upstream APIs.

Fix: wrap your data fetching functions with `'use cache'`.

### 5. The proxy.ts Export Name

The function exported from `proxy.ts` must be named `proxy`, not `default` and not `middleware`. The codemod handles this, but manual migrations sometimes miss it.

## Should You Migrate Now?

Here's my decision framework:

**Migrate now if:**
- You're starting a new project (obvious — start on 16)
- Your team is comfortable with the App Router and Server Components
- You're blocked by Edge Runtime limitations in middleware
- Build times are a pain point (Turbopack is a game-changer)
- You need explicit caching control (the `'use cache'` model is much clearer)

**Wait 4-8 weeks if:**
- You depend heavily on third-party middleware packages that haven't updated yet
- You have complex custom webpack configurations with no Turbopack equivalents
- You're mid-sprint on a critical feature — don't mix migration with feature work

**Wait longer if:**
- You're still on Pages Router (migrate to App Router first, then upgrade to 16)
- Your app has extensive `unstable_cache` usage across 50+ files (the codemod helps, but manual review is needed)
- Your team hasn't adopted TypeScript yet (the async API changes are much harder to catch without types)

For most production apps on Next.js 15 with the App Router, I'd say: **migrate now**. The codemods handle 90% of the work, the performance gains are immediate, and the new caching model eliminates an entire category of bugs.

The 47-minute migration I mentioned at the start? Most of that was waiting for the build. The actual code changes took about 15 minutes — 12 of which were the codemod running, and 3 were fixing two edge cases the codemod missed.

Next.js 16 is the release where the App Router vision fully comes together. Server Components, explicit caching, full Node.js proxy, Turbopack performance — it's all finally here, and it's all finally stable.

Go upgrade. Your dev server will thank you.
