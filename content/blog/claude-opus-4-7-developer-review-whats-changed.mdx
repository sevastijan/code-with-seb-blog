Anthropic dropped Claude Opus 4.7 on April 16 and the release notes looked straightforward — better vision, a new tokenizer, task budgets. Then I updated my API calls and one of them started returning 400 errors. No deprecation warning. No migration period. Just a hard break.

That experience is why I am writing this instead of a changelog summary. I spent the last six days running the same prompts through 4.6 and 4.7, measuring tokens, comparing outputs, and updating my tooling. Here is what actually matters for developers, what will bite you if you are not paying attention, and what is genuinely better.

## What Shipped

The April 16 release included five headline changes:

1. **Hi-res vision** — input images now processed at up to 2576px longest edge / 3.75 megapixels, up from 1568px / 1.15MP
2. **Task budgets** — a new `task_budget` parameter that gives Claude a running countdown of remaining resources
3. **New tokenizer** — a rebuilt BPE tokenizer that produces up to 35% more tokens for the same text
4. **Adaptive thinking** — replaces extended thinking with automatic depth-of-reasoning calibration
5. **Claude Code updates** — `xhigh` effort level, `/ultrareview`, Auto mode for Max subscribers

I will cover each one with the details that matter for working developers. But first, the thing that broke my code.

## The Breaking Change: budget_tokens Is Gone

If you are using the Messages API with `budget_tokens` inside the `thinking` block, your call now returns a 400 error on Opus 4.7. Not a warning. Not a fallback. A hard 400.

Here is what the old call looked like:

```json
{
  "model": "claude-opus-4-7-20260416",
  "max_tokens": 16000,
  "thinking": {
    "type": "enabled",
    "budget_tokens": 10000
  },
  "messages": [...]
}
```

And here is the error you get:

```json
{
  "type": "error",
  "error": {
    "type": "invalid_request_error",
    "message": "thinking.budget_tokens is not supported for this model. Use task_budget instead."
  }
}
```

The fix is to remove the `thinking` block entirely and use the new `task_budget` parameter (more on that below), or simply omit `budget_tokens` and let adaptive thinking handle depth automatically.

**Migration in practice:**

```python
# Before (4.6)
response = client.messages.create(
    model="claude-opus-4-6-20260320",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 10000},
    messages=messages
)

# After (4.7) — Option A: adaptive thinking (recommended)
response = client.messages.create(
    model="claude-opus-4-7-20260416",
    max_tokens=16000,
    messages=messages
)

# After (4.7) — Option B: task budget for controlled spend
response = client.messages.create(
    model="claude-opus-4-7-20260416",
    max_tokens=16000,
    task_budget={"tokens": 50000},
    messages=messages
)
```

If you are pinning to `claude-opus-4-6-20260320`, your existing code still works. But if you are using the `claude-opus-latest` alias, or you plan to upgrade, this is the first thing to fix.

I maintain a detailed guide on [token optimization and cost management for Claude Code](/blog/claude-code-token-optimization-bill-cut-guide) — the `budget_tokens` removal has direct implications for the capping strategies described there.

## The New Tokenizer: Same Text, More Tokens

This is the change that will quietly cost you money if you are not watching.

Opus 4.7 ships a rebuilt BPE tokenizer. Anthropic says it improves "representation of code, structured data, and multilingual text." What that means in practice: the same input text now produces more tokens.

I ran my standard test corpus — 50 prompts I use daily, mix of English prose, TypeScript code, JSON payloads, and markdown — through both tokenizers:

| Content type | 4.6 tokens | 4.7 tokens | Increase |
| --- | --- | --- | --- |
| English prose | 12,400 | 13,600 | +9.7% |
| TypeScript code | 18,200 | 24,100 | +32.4% |
| JSON payloads | 8,900 | 11,800 | +32.6% |
| Mixed markdown | 15,100 | 18,700 | +23.8% |

Code and structured data take the biggest hit. English prose is relatively mild. The 35% figure Anthropic cites is for code-heavy workloads, and my numbers confirm it.

**Pricing has not changed** — it is still $5 per million input tokens and $25 per million output tokens for Opus. But if the same prompt now costs 30% more tokens, your effective cost per prompt goes up 30%. There is no way around that math.

The silver lining: Anthropic claims the finer-grained tokenization improves output quality on code tasks. In my testing, this is true — Opus 4.7 generates slightly more accurate TypeScript (fewer type errors on first pass, better import resolution). Whether that quality bump justifies the cost increase depends on your rework rate. If you are already tracking rework like I describe in the [token optimization guide](/blog/claude-code-token-optimization-bill-cut-guide), compare your before/after numbers over a full week before drawing conclusions.

## Task Budgets: A Countdown Clock for Claude

Task budgets are the replacement for `budget_tokens`, and they work differently. Instead of capping thinking tokens specifically, you set a total token budget for the entire task — input, thinking, and output combined.

```python
response = client.messages.create(
    model="claude-opus-4-7-20260416",
    max_tokens=16000,
    task_budget={"tokens": 50000},
    messages=messages
)
```

What makes this interesting: Claude can see the budget. It is not just an external limit — the model receives a running countdown of remaining tokens and adjusts its behavior accordingly.

In practice, this means:

- **High budget (100K+):** Claude explores thoroughly, considers alternatives, writes longer reasoning chains
- **Medium budget (30K–100K):** Claude focuses on the main path, skips edge cases unless prompted
- **Low budget (10K–30K):** Claude prioritizes the most important output, wraps up gracefully instead of cutting off mid-sentence
- **Very low budget (<10K):** Claude gives concise answers, minimal reasoning

The "finishes gracefully" part is the key improvement over `max_tokens` truncation. With the old approach, hitting the limit meant your output got sliced at an arbitrary point. With task budgets, Claude sees the limit approaching and structures its response to reach a natural conclusion.

I tested this with code generation tasks at various budgets:

```
50K budget: Full implementation with tests and docs
30K budget: Full implementation, tests but no docs
15K budget: Implementation only, notes about what was skipped
8K budget:  Core function with inline comments about remaining work
```

Every response was complete and usable at its budget level. No truncation artifacts. That alone makes task budgets worth adopting.

## Hi-Res Vision: Finally Useful for Real Work

The old vision limit of 1568px meant that screenshots of code editors, architecture diagrams, and whiteboard photos lost critical detail after downscaling. I had stopped sending images to Claude for anything that required reading small text.

The new limit — 2576px longest edge, 3.75 megapixels — changes that calculation.

**What works now that did not before:**

- **Full-screen IDE screenshots** at 1440p or 4K — Claude can now read the code, the file tree, and the terminal output in a single image
- **Architecture diagrams** from tools like Excalidraw or Miro — text labels are legible, connection lines are traceable
- **Whiteboard photos** from planning sessions — handwriting recognition is noticeably better at the higher resolution
- **Error screenshots** from staging environments — stack traces in browser devtools are readable without zooming

I tested this on a React component library I maintain. Sent a 1440p screenshot of Storybook with six component variants visible. On 4.6, Claude could identify the components but misread prop values and missed a padding inconsistency. On 4.7, it read every prop label correctly and flagged the padding issue unprompted. That is the difference the resolution bump makes for real code work.

**What still does not work well:**

- Screenshots of entire Figma files with dozens of artboards — too much detail even at 3.75MP
- Photos of printed text from more than a meter away — camera quality matters more than model resolution
- Heavily compressed JPEGs — artifacts at high resolution look worse than artifacts at low resolution

The practical workflow I have settled on: screenshot at native resolution, resize to fit within 2576px if needed, send as PNG. For code reviews involving UI changes, I now include a screenshot alongside the diff. Claude catches layout issues that would have been invisible in the old resolution.

## Adaptive Thinking vs Extended Thinking

Extended thinking in Opus 4.6 was explicit: you turned it on, set a budget, and Claude wrote out its reasoning chain in a `thinking` block. Adaptive thinking in 4.7 is implicit: Claude decides how deeply to reason based on the complexity of the prompt.

**How it works:**

- Simple factual question → minimal internal reasoning, fast response
- Multi-step code problem → moderate reasoning chain
- Complex architectural decision → deep reasoning, comparable to old extended thinking
- Ambiguous or contradictory requirements → extensive reasoning to resolve conflicts

You do not control the depth directly. If you need to nudge it, you use prompt phrasing: "Think carefully about edge cases" triggers deeper reasoning, "Give me a quick answer" keeps it shallow.

**In my testing, adaptive thinking is better than manual extended thinking about 80% of the time.** The 20% where it falls short are cases where I know the problem is deceptively complex and Claude underestimates it. For those, I add explicit reasoning instructions to the prompt.

The net effect on my workflow: I stopped managing thinking budgets. One fewer parameter to tune. That maps well to the broader principle I wrote about in my [context engineering guide](/blog/context-engineering-claude-code-guide) — reduce the decisions you make per prompt, and you reduce the surface area for mistakes.

## Claude Code Changes

Three updates landed in Claude Code alongside the model release:

### xhigh Effort Level

Claude Code already had `low`, `medium`, and `high` effort levels. Now there is `xhigh`, which tells the model to be extremely thorough — check more files, write more tests, consider more edge cases.

In practice, `xhigh` roughly doubles the token spend of `high` and increases task completion time by 40–60%. I use it for:

- Initial project scaffolding where getting the architecture right matters
- Debugging issues I have already failed to solve at `high`
- Code reviews on PRs that touch more than 10 files

For daily development, `high` is still the right default. `xhigh` is the "I am stuck and willing to pay for thoroughness" option. Think of it as the escalation path: try `high` first, and if the result is incomplete or misses context from related files, re-run at `xhigh` before reaching for manual debugging.

### /ultrareview

A new slash command that runs an exhaustive code review. It reads every changed file, checks for security issues, performance problems, accessibility gaps, and stylistic inconsistencies.

Compared to a standard review at `high` effort, `/ultrareview`:

- Catches 2–3x more issues in my testing
- Takes 3–4x longer
- Costs roughly 4x more tokens

I run it before merging anything to main on production projects. For feature branches and drafts, the standard review is sufficient.

### Auto Mode for Max Subscribers

Max plan subscribers ($100/month for Sonnet, $200/month for Opus) now get Auto mode in Claude Code. Auto mode lets Claude execute multi-step tasks without confirming each action — file creation, terminal commands, git operations.

This is the biggest quality-of-life improvement if you are a Max subscriber. The confirmation prompts were the main friction point in Claude Code workflows, and Auto mode eliminates them for trusted operations.

If you maintain a [CLAUDE.md configuration](/blog/claude-md-memory-persistent-context-guide), update it to account for Auto mode — you will want explicit guardrails in your instructions now that Claude can act without asking.

## Real-World Testing: 4.6 vs 4.7

I ran 30 identical prompts through both models across three categories: code generation, code review, and architecture questions. Same system prompts, same context, same temperature.

### Code Generation (10 prompts)

| Metric | Opus 4.6 | Opus 4.7 |
| --- | --- | --- |
| First-pass correctness | 7/10 | 8/10 |
| Type errors | 4 | 1 |
| Missing imports | 3 | 1 |
| Average output tokens | 2,800 | 3,200 |
| Average input tokens | 4,100 | 5,400 |

Quality is up. Cost is also up. The one regression: 4.7 is slightly more verbose in comments and documentation, even when I ask for minimal output. Prompt engineering can fix that, but it is annoying.

### Code Review (10 prompts)

| Metric | Opus 4.6 | Opus 4.7 |
| --- | --- | --- |
| Issues found (real) | 23 | 31 |
| False positives | 5 | 3 |
| Security issues caught | 2 | 4 |
| Precision | 82% | 91% |

This is the category where 4.7 shines. Fewer false positives and more real catches, especially around security. If you are using Claude for code review, upgrade immediately.

### Architecture Questions (10 prompts)

| Metric | Opus 4.6 | Opus 4.7 |
| --- | --- | --- |
| Actionable recommendations | 6.2/prompt | 7.1/prompt |
| Cited trade-offs | 3.8/prompt | 5.4/prompt |
| Hallucinated libraries | 2 | 1 |

Better trade-off analysis is the standout here. 4.7 is more likely to say "this approach has these downsides" instead of presenting one solution as universally correct. That makes it more useful for actual architectural decisions.

## Migration Checklist

If you are upgrading from 4.6 to 4.7, here is what to do, in priority order:

1. **Search your codebase for `budget_tokens`** — remove it or replace with `task_budget`. This is the only breaking change.
2. **Re-benchmark your token costs** — run your standard workload for a day and compare. Expect 15–35% increase depending on content mix.
3. **Update cost alerts** — if you have spending thresholds set, bump them by 30% to avoid false alarms while you adjust.
4. **Test vision-dependent workflows** — if you were avoiding image inputs due to quality, re-test at the new resolution.
5. **Update CLAUDE.md** — if you reference extended thinking, thinking budgets, or model-specific behavior, revise those sections. My [CLAUDE.md guide](/blog/claude-md-memory-persistent-context-guide) covers the hierarchy and what belongs where.
6. **Review your [AI tooling stack](/blog/ai-coding-tools-2026-developer-guide)** — if you use Claude through third-party tools (Cursor, Continue, etc.), check whether they have updated their integration for the new tokenizer and task budgets. Cursor's in-house Composer 2.5 is now its own answer to Opus 4.7 — see how they actually stack up in [Cursor Composer 2.5 vs Claude Code Opus 4.7](/blog/cursor-composer-2-5-vs-claude-code-opus-4-7).
7. **Try adaptive thinking before fighting it** — give it a week before adding manual reasoning instructions. It is better than you expect.

## Pricing Reality

The sticker price has not changed:

| Tier | Input | Output |
| --- | --- | --- |
| Opus 4.7 | $5 / 1M tokens | $25 / 1M tokens |
| Opus 4.6 | $5 / 1M tokens | $25 / 1M tokens |

But the tokenizer change means you are buying the same text with more tokens. My daily Claude Code spend went from roughly $15 to $19 on the same workload — a 27% increase, right in line with the tokenizer overhead on mixed content.

You can offset some of this with task budgets (cap what you were not capping before) and with the quality improvements (fewer rework cycles). In my case, the net cost after accounting for reduced rework was about $17/day — a 13% real increase. Annoying but not catastrophic.

If you are cost-sensitive, pin to `claude-opus-4-6-20260320` until you have a plan. The old model is not going anywhere for now.

## Should You Upgrade?

Yes, with preparation.

The quality improvements in code review and code generation are real. Adaptive thinking removes a knob you were probably tuning wrong anyway. Hi-res vision opens workflows that were not viable before. Task budgets are a better abstraction than thinking budgets.

The tokenizer cost increase is the main downside, and it is permanent — there is no setting to opt out. Budget accordingly.

The breaking `budget_tokens` removal is the urgent item. Fix it before it hits production.

Everything else is upside. And if you are building [context-aware workflows around Claude Code](/blog/context-engineering-claude-code-guide), the adaptive thinking change simplifies your prompt design — you no longer need to predict reasoning depth upfront, which removes one of the trickiest variables from context engineering.
