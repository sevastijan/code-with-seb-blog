Every Next.js project I review has rendering strategy problems. Either everything is server-rendered (slow TTFB, high compute costs), everything is statically generated (stale data, huge build times), or — most commonly — the team picked one strategy and applied it everywhere without thinking about which pages need what.

The right approach is hybrid: different pages use different strategies based on their data requirements, update frequency, and performance needs. A marketing page should be statically generated. A user dashboard should be server-rendered. A blog post should use ISR. A product page should use streaming RSC.

Here's the decision framework with benchmarks.

## The Four Strategies at a Glance

| Strategy | When HTML is Generated | Data Freshness | TTFB | Compute Cost | Best For |
|----------|----------------------|----------------|------|-------------|----------|
| **SSG** | Build time | Stale (until rebuild) | ~50ms | Lowest | Marketing, docs, blog |
| **ISR** | Build + background revalidation | Fresh within revalidate window | ~80ms (cached) | Low | Blog, product pages, catalogs |
| **SSR** | Every request | Always fresh | ~200-800ms | Highest | Dashboards, personalized pages |
| **RSC + Streaming** | Every request (streamed) | Always fresh | ~100ms (shell) | Medium | Complex pages with mixed data |

## SSG: Static Site Generation

HTML is generated at build time. Every user gets the same pre-built page served from a CDN.

```typescript
// app/about/page.tsx — fully static, generated at build
export default function AboutPage() {
  return (
    <div>
      <h1>About Us</h1>
      <p>We build things.</p>
    </div>
  );
}
// No data fetching = automatically SSG in Next.js App Router
```

```typescript
// app/blog/[slug]/page.tsx — SSG with dynamic params
export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map(post => ({ slug: post.slug }));
}

export default async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug);
  return <Article post={post} />;
}
// Generated at build time for each slug from generateStaticParams
```

**Performance:** ~50ms TTFB from CDN edge. Zero server compute per request.

**The catch:** Data is frozen at build time. Any change requires a full rebuild and redeploy. With 10,000 blog posts, build times can exceed 30 minutes.

**Use SSG when:** Content never changes between deployments — marketing pages, documentation, legal pages, landing pages.

## ISR: Incremental Static Regeneration

The best of both worlds — static speed with fresh data. Pages are served from cache and revalidated in the background.

```typescript
// app/products/[id]/page.tsx — ISR with 60-second revalidation
export const revalidate = 60; // Revalidate every 60 seconds

export default async function ProductPage({ params }: { params: { id: string } }) {
  const product = await fetch(`${API_URL}/products/${params.id}`, {
    next: { tags: ['products', `product-${params.id}`] },
  }).then(res => res.json());

  return <ProductDetail product={product} />;
}
```

**The revalidation flow:**

```
Request 1 (cache empty):
  → Cache MISS → render on server → serve → cache result
  TTFB: ~300ms

Request 2 (within 60s):
  → Cache HIT → serve from cache
  TTFB: ~80ms

Request 3 (after 60s):
  → Cache STALE → serve stale content immediately → revalidate in background
  TTFB: ~80ms (user gets stale data, but it's fast)

Request 4 (after revalidation):
  → Cache HIT → serve fresh content
  TTFB: ~80ms
```

**Use ISR when:** Data changes periodically but doesn't need to be real-time — product catalogs, blog posts, pricing pages, news articles.

I covered ISR in depth in my [ultra-fast dashboards guide](/blog/incremental-revalidation-nextjs-ultra-fast-dashboards) — including on-demand revalidation with `revalidateTag`.

## SSR: Server-Side Rendering

Every request renders fresh HTML on the server. The most flexible but most expensive strategy.

```typescript
// app/dashboard/page.tsx — SSR (dynamic by default when using cookies/headers)
import { cookies } from 'next/headers';

export default async function DashboardPage() {
  const cookieStore = await cookies();
  const session = cookieStore.get('session');

  const data = await fetch(`${API_URL}/dashboard`, {
    headers: { Authorization: `Bearer ${session?.value}` },
    cache: 'no-store', // Force SSR — no caching
  }).then(res => res.json());

  return <Dashboard data={data} />;
}
```

**Performance:** TTFB depends on your server and API speed. Typically 200-800ms. No caching means every request hits your server and your APIs.

**Cost:** At 100K requests/day, SSR costs 10-50x more in compute than ISR serving cached pages.

**Use SSR when:** Data must be fresh on every request AND is personalized — user dashboards, account pages, admin panels, real-time trading platforms.

**Don't use SSR when:** The page isn't personalized — use ISR instead.

## RSC + Streaming: The Modern Approach

React Server Components with Suspense streaming give you the best TTFB while still rendering fresh data. The shell streams immediately, and data-dependent sections stream in as they resolve.

```typescript
// app/dashboard/page.tsx — Streaming RSC
import { Suspense } from 'react';

export default function DashboardPage() {
  return (
    <div>
      {/* Static shell — streams instantly */}
      <DashboardHeader />
      <DashboardNav />

      {/* Each section streams independently */}
      <div className="grid grid-cols-2 gap-6">
        <Suspense fallback={<MetricsSkeleton />}>
          <MetricsSection />  {/* Fetches and streams when ready */}
        </Suspense>

        <Suspense fallback={<ChartSkeleton />}>
          <RevenueChart />    {/* Fetches in parallel, streams when ready */}
        </Suspense>
      </div>

      <Suspense fallback={<TableSkeleton />}>
        <RecentOrders />      {/* Streams when API responds */}
      </Suspense>
    </div>
  );
}

// Each component is an async Server Component that fetches its own data
async function MetricsSection() {
  const metrics = await fetchMetrics(); // 100ms
  return <MetricsGrid data={metrics} />;
}

async function RevenueChart() {
  const revenue = await fetchRevenue(); // 500ms
  return <Chart data={revenue} />;
}

async function RecentOrders() {
  const orders = await fetchOrders(); // 300ms
  return <OrderTable orders={orders} />;
}
```

**Performance:**
```
Time:     0ms    100ms    300ms    500ms
          │       │        │        │
Shell:    ████    │        │        │   ← Instant (static HTML)
Metrics:  ─────── ████     │        │   ← Streams at 100ms
Orders:   ──────────────── ████     │   ← Streams at 300ms
Chart:    ─────────────────────── ████  ← Streams at 500ms
```

TTFB for the shell: ~50ms. User sees the layout immediately and content fills in progressively. **The slowest API doesn't block the page.**

**Use Streaming RSC when:** Pages have multiple data sources with different response times — dashboards, aggregation pages, complex product pages.

## The Decision Framework

```
Is the page personalized (different per user)?
├── No → Does the content change?
│        ├── No → SSG (build once, serve forever)
│        ├── Rarely (daily/weekly) → ISR (revalidate: 3600+)
│        └── Frequently (minutes) → ISR (revalidate: 30-60)
│
└── Yes → Does it have multiple data sources?
          ├── No → SSR (simple, always fresh)
          └── Yes → Streaming RSC + Suspense
                    (shell instant, data streams in)
```

### Per-Page Strategy for a Typical SaaS

| Page | Strategy | Why |
|------|----------|-----|
| **Homepage** | SSG | Static content, no data |
| **Pricing** | ISR (3600s) | Changes rarely, should load fast |
| **Blog post** | ISR (60s) | Fresh-ish content, instant TTFB |
| **Product catalog** | ISR (300s) | Updates periodically |
| **Product detail** | ISR (60s) + on-demand | Fresh on purchase, cached otherwise |
| **Search results** | SSR | Dynamic based on query |
| **User dashboard** | Streaming RSC | Personalized, multiple data sources |
| **Account settings** | SSR | Personalized, simple data |
| **Admin panel** | Streaming RSC | Multiple data sources, real-time |
| **Checkout** | SSR | Personalized, security-sensitive |

## Combining Strategies on One Page

The most powerful pattern: mix strategies within a single route.

```typescript
// app/product/[id]/page.tsx — ISR for product data, dynamic for user-specific
export const revalidate = 60; // ISR for the page

export default async function ProductPage({ params }: { params: { id: string } }) {
  // This fetch is cached (ISR)
  const product = await fetch(`${API_URL}/products/${params.id}`, {
    next: { revalidate: 60 },
  }).then(res => res.json());

  return (
    <div>
      {/* Static/ISR content */}
      <ProductHero product={product} />
      <ProductDescription description={product.description} />

      {/* Dynamic user-specific content (streaming) */}
      <Suspense fallback={<Skeleton />}>
        <PersonalizedRecommendations productId={product.id} />
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <UserReviews productId={product.id} />
      </Suspense>
    </div>
  );
}
```

The product data loads from ISR cache (instant). The recommendations and reviews stream in as separate Suspense boundaries. The page feels instant while still showing personalized content.

## Performance Benchmarks

Real-world measurements from a Next.js SaaS app (Vercel deployment):

| Page | Strategy | TTFB (p50) | TTFB (p99) | Monthly Cost |
|------|----------|-----------|-----------|-------------|
| Homepage | SSG | 42ms | 87ms | ~$0 |
| Blog post | ISR (60s) | 78ms | 142ms | ~$5 |
| Product page | ISR (60s) | 83ms | 156ms | ~$8 |
| Dashboard | SSR | 412ms | 1,247ms | ~$120 |
| Dashboard | Streaming RSC | 89ms (shell) | 203ms (shell) | ~$85 |

The streaming RSC dashboard has a **78% lower TTFB** than the SSR dashboard — because the shell streams immediately. Total time to full content is similar, but the perceived performance is dramatically better.

## Practical Takeaways

1. **Default to ISR** — most pages benefit from cached static content with background revalidation.
2. **Use SSG for truly static pages** — marketing, docs, legal. Zero cost, maximum speed.
3. **Use Streaming RSC for dashboards** — the shell renders instantly, data streams in progressively.
4. **Avoid SSR unless necessary** — it's the most expensive strategy. Use ISR with on-demand revalidation instead.
5. **Mix strategies per page** — ISR for public content, Suspense streaming for personalized sections.
6. **Wrap each data source in its own Suspense** — the slowest API shouldn't block the rest of the page.
7. **Monitor TTFB per route** — different strategies produce wildly different numbers.
8. **On-demand revalidation for mutations** — `revalidateTag` after user actions keeps ISR fresh.
9. **Don't over-cache** — staleTime: 0 on the server means every unique request fetches fresh data. Set revalidation windows based on how often data actually changes.
10. **The goal is perceived performance** — users care about when they can interact, not when the last byte arrives.

The framework gives you four strategies. The skill is knowing which to use where. Start with ISR everywhere, add streaming for dashboards, keep SSG for static pages, and reserve SSR for the rare cases where caching truly isn't an option.
