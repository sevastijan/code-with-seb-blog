The first time I logged into my own app with a passkey, there was no password field. I clicked "Sign in," my laptop asked for my fingerprint, and I was in. No password to forget, nothing to phish, nothing in my database that a breach could leak into a credential-stuffing list. That's the pitch for **passkeys in Next.js**, and it's real тАФ but the reason most developers bounce off WebAuthn is that they try to map it onto passwords, and it doesn't map. It's a different model.

So let's start with the model, because once it clicks, the two ceremonies and the SimpleWebAuthn code read like the obvious implementation of an obvious idea. Then I'll spend the back half on the production footguns тАФ the ones that don't show up in the tutorial's happy path and instead show up as "why is authentication silently failing for some users" three weeks after launch.

## The Mental Model: Challenge, Sign, Verify

Passwords are a shared secret: the user knows it, your server stores a hash of it, and authentication is "do these match?" Every part of that is a liability тАФ the secret can be phished, the hash can be leaked, users reuse it everywhere.

Passkeys are public-key cryptography instead. When a user registers, their device generates a keypair. **The private key never leaves the device** тАФ it's locked behind the biometric or PIN. Your server stores only the *public* key. Authentication is challenge-response:

1. Your server sends a random **challenge**.
2. The device unlocks the private key (fingerprint/PIN) and **signs** the challenge.
3. Your server **verifies** the signature against the stored public key.

That's the entire idea. There's no secret on your server to leak тАФ a public key is useless to an attacker. There's nothing to phish тАФ the signature is bound to your domain (the same domain-and-certificate trust that [HTTPS establishes](/blog/how-https-works-under-the-hood-of-secure-web-communication)), so a fake site can't elicit a usable one. The "password" is a signature that's different every time and only your server can validate. Hold this picture; everything below is plumbing around it.

## The Two Ceremonies and What Crosses the Wire

WebAuthn has exactly two flows, called **ceremonies**, and each is a round-trip pair (server offers options тЖТ device responds):

- **Registration** тАФ the server sends `PublicKeyCredentialCreationOptions` (who the user is, who you are, a challenge). The device creates a keypair and returns an **attestation** containing the new public key. You store it.
- **Authentication** тАФ the server sends options with a fresh challenge. The device signs it and returns an **assertion** (the signature). You verify it.

The library that does the cryptographic heavy lifting is **SimpleWebAuthn** тАФ roughly 4.9M weekly downloads as of mid-2026, effectively the standard. It has a server half (`@simplewebauthn/server`) and a browser half (`@simplewebauthn/browser`). It handles the crypto correctly; your job is the challenge lifecycle, the storage, and the verification config тАФ which is also exactly where the bugs live.

## Registration in Next.js

Two server actions (or route handlers) and a browser call. Generate the options:

```typescript
'use server'
import { generateRegistrationOptions } from '@simplewebauthn/server'

export async function getRegistrationOptions() {
  const user = await requireUser()

  const options = await generateRegistrationOptions({
    rpName: 'My App',
    rpID: process.env.RP_ID!,        // e.g. "myapp.com" тАФ see the footgun below
    userName: user.email,
    // Ask the authenticator to store a discoverable credential for username-less login
    authenticatorSelection: {
      residentKey: 'preferred',
      userVerification: 'preferred',
    },
  })

  // CRITICAL: persist the challenge server-side, tied to this user/session
  await saveChallenge(user.id, options.challenge)
  return options
}
```

The browser triggers the native prompt with those options:

```typescript
'use client'
import { startRegistration } from '@simplewebauthn/browser'

async function register() {
  const options = await getRegistrationOptions()
  const attestation = await startRegistration({ optionsJSON: options }) // fingerprint/PIN prompt
  await verifyRegistration(attestation) // send it back to the server
}
```

And verify + store on the server:

```typescript
'use server'
import { verifyRegistrationResponse } from '@simplewebauthn/server'

export async function verifyRegistration(attestation: unknown) {
  const user = await requireUser()
  const expectedChallenge = await getChallenge(user.id) // the one we saved

  const { verified, registrationInfo } = await verifyRegistrationResponse({
    response: attestation as any,
    expectedChallenge,
    expectedOrigin: process.env.ORIGIN!, // e.g. "https://myapp.com" тАФ strict!
    expectedRPID: process.env.RP_ID!,
  })

  if (!verified || !registrationInfo) return { error: 'Registration failed' }

  const { credential } = registrationInfo
  await db.passkey.create({
    data: {
      userId: user.id,
      credentialId: credential.id,
      publicKey: Buffer.from(credential.publicKey),
      counter: credential.counter,
      transports: credential.transports ?? [],
    },
  })
  return { success: true }
}
```

## The Database Schema You Must Store

This is the part every "it just works" tutorial underspecifies. A passkey is not one column тАФ you must persist, per credential:

- **`credentialId`** тАФ the handle the browser sends at login to say which passkey it's using.
- **`publicKey`** тАФ the key you verify signatures against (store the bytes, mind the encoding).
- **`counter`** тАФ the signature counter (more on this in Footgun 3).
- **`transports`** тАФ how the authenticator talks (`internal`, `hybrid`, `usb`тАж), used to hint the browser.

One user can have many passkeys (laptop, phone, hardware key), so this is a one-to-many table, not a column on `users`. Getting the encoding wrong тАФ storing the public key or credential ID as a mangled string instead of the right bytes/base64url тАФ is the classic "registration succeeds, authentication always fails" bug.

## Authentication and the Username-Less Flow

The magic UX тАФ no username field at all тАФ comes from **discoverable credentials** (resident keys). Because the passkey is stored on the device with your `rpID`, the browser's account chooser can offer it directly:

```typescript
'use server'
import { generateAuthenticationOptions } from '@simplewebauthn/server'

export async function getAuthOptions() {
  const options = await generateAuthenticationOptions({
    rpID: process.env.RP_ID!,
    userVerification: 'preferred',
    // No allowCredentials -> browser offers any passkey for this rpID (username-less)
  })
  await saveAnonymousChallenge(options.challenge) // no user yet тАФ key by session
  return options
}
```

```typescript
'use client'
import { startAuthentication } from '@simplewebauthn/browser'

async function signIn() {
  const options = await getAuthOptions()
  const assertion = await startAuthentication({ optionsJSON: options })
  await verifyAuth(assertion) // server looks up the credentialId, verifies, creates session
}
```

On the server you look up the stored passkey by the `credentialId` in the assertion, verify the signature with `verifyAuthenticationResponse`, update the counter, and mint your session тАФ the same session your app already uses. If you built auth with [NextAuth v5 or Clerk](/blog/nextauth-v5-vs-clerk-nextjs-auth-rbac), passkeys slot in as an additional sign-in method, not a replacement for your whole session layer.

## Footgun 1: rpID and the Domain Trap

The **Relying Party ID (`rpID`)** must match your domain exactly, and it is the single most dangerous thing to get wrong тАФ because **changing your rpID invalidates every passkey your users have ever created.** Passkeys are cryptographically bound to the rpID; move from `app.example.com` to `example.com`, or "clean up" the value, and every stored credential becomes unusable. Users get locked out en masse with no error you'll see in testing (you registered fresh).

Rules to tattoo on the deploy runbook:

- Pick the rpID **once**, deliberately, ideally the registrable domain (`example.com`) so subdomains can share it тАФ and never change it.
- The rpID **cannot be on the Public Suffix List** (you can't use `vercel.app` or `github.io` as your rpID). Custom domain required for production passkeys.
- `expectedOrigin` on verification must match the full origin (`https://example.com`) strictly. And yes тАФ passkeys require **HTTPS**; there's no passkey on plain HTTP (localhost excepted for dev).

## Footgun 2: Unique Challenges and Strict Origin

The challenge is what makes each authentication un-replayable, and it only works if you treat it as single-use:

- **Generate a fresh, random challenge for every request** and store it server-side tied to the session. Reusing a challenge, or accepting one the client sends back without checking it against what you issued, reopens replay attacks тАФ the exact thing the challenge exists to prevent.
- **Validate the origin strictly** on the server. Lax origin validation (e.g. accepting any subdomain, or user-controlled content origins) throws away one of WebAuthn's core security properties тАФ the binding of the signature to *your* site. Match against a known-good origin, not a pattern that a subdomain takeover could satisfy.

Server misconfiguration here fails silently and insecurely, which is the worst combination: it *works*, so you ship it, and it's quietly not protecting anyone. This is the same "the framework won't save you, validate on the server" discipline I harp on for [server actions security](/blog/nextjs-server-actions-security) тАФ WebAuthn is no exception.

## Footgun 3: The Sign Counter

Each authenticator keeps a **signature counter** that increments every time it signs. You store the last value; on each authentication, the new counter should be *higher* than the stored one. If it ever comes back lower or equal, that's a signal the credential may have been **cloned** тАФ two copies of a private key that shouldn't exist.

The footgun is forgetting to persist and check it: many tutorials store the counter at registration and never update it, which silently disables cloning detection. Update it on every successful authentication:

```typescript
await db.passkey.update({
  where: { credentialId },
  data: { counter: newCounter }, // from verifyAuthenticationResponse
})
```

(Caveat worth knowing: some platform authenticators, notably Apple's synced passkeys, always report a counter of 0. So "0 and stayed 0" is normal for those; "went backwards after being positive" is the real alarm. Handle both.)

## Fitting Passkeys Alongside Your Existing Auth

Don't rip out your current login on day one. The pragmatic rollout:

1. Ship passkeys as an *additional* method тАФ let users add a passkey from account settings while email/OAuth still works.
2. Offer passkey sign-in on the login page as the primary button, with the existing method as fallback (some users, some devices, some corporate environments still can't complete a passkey flow тАФ cross-device hybrid flows are more fragile than same-device).
3. Only consider going passwordless-only once adoption and device coverage are proven.

Passkeys are an *authentication* method; they hand off to the *same* session and [authorization](/blog/nextauth-v5-vs-clerk-nextjs-auth-rbac) you already have. They change how the user proves who they are, not what they're allowed to do.

## Checklist

1. Store per-credential: `credentialId`, `publicKey` (bytes), `counter`, `transports` тАФ one-to-many with users.
2. `rpID` chosen once, matches your domain, not on the Public Suffix List, never changed.
3. Fresh unique challenge per request, persisted server-side, single-use.
4. `expectedOrigin` and `expectedRPID` validated strictly on verification; HTTPS only.
5. Sign counter persisted and checked (and handle always-0 platform authenticators).
6. Discoverable credentials enabled for the username-less flow.
7. Passkeys added alongside existing auth first; passwordless-only only after coverage is proven.

Passkeys really are the endgame for authentication тАФ no shared secret, phishing-resistant by construction, and a genuinely better UX. SimpleWebAuthn makes the cryptography a non-event. The work that's actually yours is the boring, high-stakes plumbing: the challenge lifecycle, the storage encoding, the rpID you must never change, and the counter you must not forget. Get those right and you delete an entire category of security liability from your app.
