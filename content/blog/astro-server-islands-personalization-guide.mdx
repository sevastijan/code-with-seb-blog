For years, one dynamic thing on a page forced a bad trade. A logged-in user's avatar in the header, a live cart count, a "recommended for you" row — any single per-request element meant you couldn't fully cache the page. So the whole page went dynamic to serve one personalized corner, and your beautifully cacheable content paid the latency tax for a 40-pixel avatar.

**Astro Server Islands** kill that trade. The page stays static and cached at the edge; the handful of personalized components render fresh on the server *per request* and get injected into the page after it loads. You get the CDN-fast shell *and* the personalization, on the same page, without either compromising the other. Here's the mental model that makes it click, the `server:defer` patterns, and the caching and fallback gotchas that decide whether it feels magic or janky.

## What a Server Island Actually Is

A server island is a normal server-rendered Astro component with one instruction added: **delay your rendering until your content is ready.** You mark it with the `server:defer` directive. Astro then does something clever with the page around it:

- The page renders and ships **immediately**, with a **fallback placeholder** where the island will go.
- The island's real HTML is fetched from the server **after** the page loads, then slotted into place.

Crucially, the rest of the page — the static shell — doesn't wait for the island. It's already at the user's screen, cached from the edge, while the personalized piece resolves in the background. The dynamic content is server-rendered HTML, not a client-side framework component, so you're not shipping a React/Vue runtime just to show a cart count.

## How It Works: The Slot and the Injected Fetch

The mechanism is worth understanding because it explains every behavior below. When Astro builds the page, for each `server:defer` component it:

1. Leaves a **slot** (a placeholder hole) in the statically-rendered HTML shell.
2. Injects a **small script** into the page that, on load, fetches each deferred component's real HTML from the server.
3. Drops the fetched HTML into its slot.

So the shell is one cacheable artifact, and each island is an independent little server request that fills its hole. Two consequences fall out immediately: the shell can be cached globally on a CDN because it contains no per-user data, and **each island loads independently** — a slow recommendations island won't hold up the cart island or anything else. They race in parallel and fill in as they arrive.

## server:defer With a Good Fallback

The directive is trivial; the fallback is where the UX lives. Give every island a placeholder that occupies the *same space* the real content will, so nothing jumps when it arrives:

```astro
---
// UserAvatar.astro — a normal server component, rendered per request
const user = await getUser(Astro.request)
---
<img src={user.avatarUrl} alt={user.name} class="avatar" />
```

```astro
---
// In your page/layout — defer it and provide a fallback
import UserAvatar from '../components/UserAvatar.astro'
---
<header>
  <nav><!-- static nav --></nav>

  <UserAvatar server:defer>
    <!-- Fallback: shown instantly, replaced when the island arrives -->
    <div slot="fallback" class="avatar avatar--skeleton" />
  </UserAvatar>
</header>
```

The `slot="fallback"` content renders into the static shell immediately. When the deferred fetch resolves, Astro swaps in the real avatar. Size the skeleton to match the final element and you get a smooth fill with **no layout shift** — the difference between "feels instant" and "the header jumps a beat after load."

## Server Islands vs Client Islands vs Full SSR

Astro now gives you three tools for dynamic content, and picking the right one is the actual skill:

- **Client island** (`client:load`, `client:visible`, …) — ships a JS component that hydrates and runs *in the browser*. Use it for genuine **interactivity**: a search box, a carousel, a form with live validation.
- **Server island** (`server:defer`) — renders **per-request HTML on the server**, injected after load. Use it for **personalized or dynamic content that isn't interactive**: avatar, cart count, "recommended for you," A/B-tested blocks. No client framework shipped.
- **Full SSR** (dynamic page) — the *entire* page renders per request. Use it only when *most* of the page is dynamic and there's no meaningful static shell to cache.

The trap is reaching for full SSR because one component is dynamic. That's exactly the old bad trade. If the page is mostly static with a few personalized spots, server islands are the answer — and if a spot needs to *react to clicks*, that's a client island, not a server one. This is the same [static-shell-plus-dynamic-holes instinct as Next.js Partial Prerendering](/blog/partial-prerendering-nextjs-production-guide); Astro just makes the holes explicit components.

## The Caching Nuance: Cached Shell, Fresh Island

Here's the thing people miss and then ship a bug over: **the shell and the island have different cache lifetimes by design.** The static shell is cached — potentially for a long time, globally, on the CDN. Each server island is fetched **per request**, so it's fresh. That's the whole point.

But it means you must be careful about *what lives where*. Anything user-specific must be **inside** an island, never baked into the shell — because the shell is shared across all users and cached. Put an avatar URL directly in the cached shell and you'll serve one user's face to everyone who hits that edge cache. The rule: **static, shared, cacheable content in the shell; per-user, per-request content in a `server:defer` island.** Draw that line wrong and you either leak personalization (private data in the shell) or lose it (dynamic data cached stale).

## Auth and Cookies Inside an Island

Because each island is its own server request, it has access to the incoming request — cookies included — so reading the session inside an island works:

```astro
---
// Cart.astro (server:defer) — reads the session from the request
const session = await getSession(Astro.request)
const cart = session ? await getCart(session.userId) : { items: [] }
---
<div class="cart-badge">{cart.items.length}</div>
```

Two things to keep straight. First, the island runs as a *separate* fetch from the shell, so don't assume state computed in the page is available — re-derive what you need from the request inside the island. Second, treat that island endpoint like any other server endpoint: it's reachable, so the [auth and validation rules you'd apply anywhere](/blog/nextjs-server-actions-security) apply here too. An island that returns a user's cart must confirm *which* user is asking.

## When to Use Them (and When Not)

**Reach for server islands when:** a mostly-static page has a few personalized or frequently-changing spots (dashboards headers, storefront carts, logged-in nav, recommendations, live counts) and those spots don't need client-side interactivity.

**Don't when:** the element is interactive (use a client island), the whole page is dynamic (just SSR it), or the "dynamic" content is actually the same for everyone and could be cached with a short TTL (then it belongs in the shell with revalidation, not an island).

## Checklist

1. Only the genuinely per-request/personalized components get `server:defer` — not the whole page.
2. Every island has a `slot="fallback"` sized to match, so there's no layout shift.
3. All user-specific data lives *inside* islands; the cached shell contains only shared content.
4. Interactive elements use client islands, not server islands.
5. Islands re-derive what they need from `Astro.request` rather than assuming page state.
6. Island endpoints enforce their own auth — they're independently reachable.

Server islands are the feature that lets Astro stay true to its "mostly static, minimal JS" philosophy without conceding the personalized web to heavier frameworks. Cache the shell globally, defer the personal bits, give each a clean fallback — and you ship a page that's both instant and yours-specifically, which used to be a contradiction.
